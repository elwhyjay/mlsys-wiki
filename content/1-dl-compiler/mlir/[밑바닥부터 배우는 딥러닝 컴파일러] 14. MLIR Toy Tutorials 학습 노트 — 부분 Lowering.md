# [밑바닥부터 배우는 딥러닝 컴파일러] 14. MLIR Toy Tutorials 학습 노트 — 부분 Lowering

# 0x0. 머리말

이 노트는 Toy Tutorials 5장을 읽고 정리한 것이다. 이 장에서 주로 다루는 내용은 Toy Dialect의 일부 Operation을 Affine Dialect, MemRef Dialect, Standard Dialect로 Lowering하는 것이다. 단, `toy.print` 연산은 그대로 유지하기 때문에 이를 부분 Lowering(partial lowering)이라고 부른다. 이 Lowering을 통해 Toy Dialect의 Operation을 더 저수준의 구현 로직으로 표현할 수 있고, 더 많은 최적화 기회를 찾아 더 좋은 MLIR 표현식을 얻을 수 있다.

# 0x1. Dialect 변환

MLIR에는 수많은 Dialect가 있기 때문에, MLIR은 이러한 Dialect 간의 상호 변환을 지원하기 위해 통일된 `DialectConversion` 프레임워크를 제공한다. 이 프레임워크는 변환 대상이 되는 불법(illegal) 연산을 합법(legal) 연산으로 변환할 수 있게 해 준다. 이 프레임워크를 사용하기 위해서는 두 가지 조건(그리고 한 가지 선택적 조건)을 제공해야 한다.

  * **「변환 목표(Conversion Target)」** . 어떤 Dialect 연산이 합법적인 변환 대상인지를 명시하고, 합법화되지 않은 연산은 rewrite patterns를 통해 합법화해야 한다.
  * **「rewrite pattern 집합(Rewrite Pattern)」** . 불법 연산을 0개 또는 그 이상의 합법 연산으로 변환하는 데 사용되는 패턴 집합이다.
  * **「타입 변환기(Type Converter)(선택)」** . 제공되는 경우, block 인자의 타입을 변환하는 데 사용된다. 이 장에서는 이 변환이 필요하지 않다.



아래에서는 현재 MLIR 표현식을 어떻게 단계별로 부분 Lowering하여 새로운 MLIR 표현식으로 만들고, 더 많은 최적화 기회를 찾는지 살펴보자.

## 첫 번째 단계, 변환 목표(Conversion Target) 정의

새로운 최적화 기회를 찾기 위해, Toy Dialect 내의 계산 집약적인 연산들을 Affine, MemRef, Standard Dialect의 연산 조합으로 변환해야 한다. 코드 구현은 `mlir/examples/toy/Ch5/mlir/LowerToAffineLoops.cpp`에 있다.
    
    
    void ToyToAffineLoweringPass::runOnFunction() {  
      // The first thing to define is the conversion target. This will define the  
      // final target for this lowering.  
      mlir::ConversionTarget target(getContext());  
      
      // We define the specific operations, or dialects, that are legal targets for  
      // this lowering. In our case, we are lowering to a combination of the  
      // `Affine`, `Arithmetic`, `MemRef`, and `Standard` dialects.  
      target.addLegalDialect<AffineDialect, arith::ArithmeticDialect,  
                             memref::MemRefDialect, StandardOpsDialect>();  
      
      // We also define the Toy dialect as Illegal so that the conversion will fail  
      // if any of these operations are *not* converted. Given that we actually want  
      // a partial lowering, we explicitly mark the Toy operations that don't want  
      // to lower, `toy.print`, as *legal*. `toy.print` will still need its operands  
      // to be updated though (as we convert from TensorType to MemRefType), so we  
      // only treat it as `legal` if its operands are legal.  
      target.addIllegalDialect<ToyDialect>();  
      target.addDynamicallyLegalOp<toy::PrintOp>([](toy::PrintOp op) {  
        return llvm::none_of(op->getOperandTypes(),  
                             [](Type type) { return type.isa<TensorType>(); });  
      });  
      ...  
    }  
    

이 함수는 Toy Dialect 내의 일부 계산 집약적 연산(예: 행렬 곱셈 연산)을 Lowering하는 것을 구현한다. 우리의 목표는 원래의 MLIR 표현식을 새로운 표현식으로 Lowering하는 것이며, 여기서는 원래의 계산 집약적 연산을 더 저수준에 가까운 연산으로 변환하는 것이다. 먼저 `mlir::ConversionTarget target(getContext())`로 변환 목표를 정의한다. 그 다음 MLIR 표현식 Lowering 과정에서 합법적인 대상을 정의하는데, 여기에는 특정 Operation이나 Dialect가 포함된다. 이 예제에서는 MLIR 표현식을 Affine, MemRef, Standard Dialect 세 가지 Dialect의 Operation 조합으로 lowering한다. 따라서 코드에서는 `target.addLegalDialect` 함수를 사용해 이 세 Dialect를 합법 대상으로 추가한다. 동시에 불법 Dialect도 정의해야 하는데, 여기서는 이전의 Toy Dialect를 불법 대상으로 정의한다. 즉, `target.addIllegalDialect<ToyDialect>()`이다. 이는 변환이 끝난 후에도 MLIR 표현식 안에 Toy 연산이 남아 있다면 Lowering이 실패한다는 것을 의미한다.

특별한 점은, 이 과정에서 `toy.print`가 새로운 목표 Dialect에서는 지원되지 않으므로 보존하고 Lowering하지 않는다는 것이다. 즉 `target.addLegalOp<PrintOp>()`이다. 또한 MLIR에서는 단일 연산에 대한 정의가 항상 Dialect 정의보다 우선하기 때문에, 위 코드에서 합법, 불법 그리고 `PrintOp`를 선언하는 순서는 임의로 바꿀 수 있다.

## 두 번째 단계, 변환 패턴(Conversion Patterns) 명시

변환 목표를 정의한 후에는 불법 연산을 합법 연산으로 어떻게 변환할지를 정의할 수 있다. 3장에서 소개한 정규화 프레임워크와 유사하게, DialectConversion 프레임워크 역시 RewritePatterns를 사용하여 변환 로직을 수행한다. 이러한 패턴들은 이전에 본 RewritePatterns일 수도 있고, 변환 프레임워크 ConversionPattern에 특화된 새로운 형태의 패턴일 수도 있다. **「ConversionPatterns」** 는 전통적인 RewritePatterns와는 다르다. 이는 추가적인 operand 인자를 받는데, 그 안에는 이미 재매핑/대체된 operand들이 포함되어 있다. 공식 문서에서는 Toy Dialect 내의 transpose 연산을 Lowering하는 예제를 보여 준다. 코드는 다음과 같다.
    
    
    /// Lower the `toy.transpose` operation to an affine loop nest.  
    struct TransposeOpLowering : public mlir::ConversionPattern {  
      TransposeOpLowering(mlir::MLIRContext *ctx)  
          : mlir::ConversionPattern(TransposeOp::getOperationName(), 1, ctx) {}  
      
      /// Match and rewrite the given `toy.transpose` operation, with the given  
      /// operands that have been remapped from `tensor<...>` to `memref<...>`.  
      mlir::LogicalResult  
      matchAndRewrite(mlir::Operation *op, ArrayRef<mlir::Value> operands,  
                      mlir::ConversionPatternRewriter &rewriter) const final {  
        auto loc = op->getLoc();  
      
        // Call to a helper function that will lower the current operation to a set  
        // of affine loops. We provide a functor that operates on the remapped  
        // operands, as well as the loop induction variables for the inner most  
        // loop body.  
        lowerOpToLoops(  
            op, operands, rewriter,  
            [loc](mlir::PatternRewriter &rewriter,  
                  ArrayRef<mlir::Value> memRefOperands,  
                  ArrayRef<mlir::Value> loopIvs) {  
              // Generate an adaptor for the remapped operands of the TransposeOp.  
              // This allows for using the nice named accessors that are generated  
              // by the ODS. This adaptor is automatically provided by the ODS  
              // framework.  
              TransposeOpAdaptor transposeAdaptor(memRefOperands);  
              mlir::Value input = transposeAdaptor.input();  
      
              // Transpose the elements by generating a load from the reverse  
              // indices.  
              SmallVector<mlir::Value, 2> reverseIvs(llvm::reverse(loopIvs));  
              return rewriter.create<mlir::AffineLoadOp>(loc, input, reverseIvs);  
            });  
        return success();  
      }  
    };  
    

이 코드는 `tor.transpose` 연산을 Affine loop nest로 Lowering하는 것이다. 여기서 `lowerOpToLoops` 함수와 `TransposeOpAdaptor`의 정의는 이 코드를 이해하는 데 핵심이 된다. 관심이 있다면 소스 코드를 직접 살펴보기 바란다.

## 세 번째 단계, Lowering 패턴 집합에 위에서 정의한 변환 패턴 추가

이 코드는 `mlir/examples/toy/Ch5/mlir/LowerToAffineLoops.cpp`에 있다.
    
    
    void ToyToAffineLoweringPass::runOnFunction() {

      // Now that the conversion target has been defined, we just need to provide  
      // the set of patterns that will lower the Toy operations.  
      mlir::RewritePatternSet patterns(&getContext());  
      patterns.add<..., TransposeOpLowering>(&getContext());

여기서는 Toy Dialect의 연산을 Lowering하기 위한 패턴 집합 `patterns`를 제공하고, 그 집합에 일련의 Lowering 패턴(예: `TransposeOpLowering`)을 추가한다.

## 네 번째 단계, 실제 Lowering 과정 실행

Lowering 패턴을 명확히 한 후에는 실제 Lowering 과정을 실행할 수 있다. `DialectConversion` 프레임워크는 여러 가지 Lowering 모드를 제공하는데, 여기서는 부분 Lowering을 사용한다. `toy.print` 연산은 Lowering하지 않기 때문이다. 이는 계산 집약적인 Operation이 아니라 단지 출력 연산일 뿐이다. 아래 코드의 `mlir::applyPartialConversion(function, target, patterns)`은 현재 MLIR 표현식의 Operation에 Lowering을 적용한다는 의미이다.
    
    
    void ToyToAffineLoweringPass::runOnFunction() {  
      ...  
      
      // With the target and rewrite patterns defined, we can now attempt the  
      // conversion. The conversion will signal failure if any of our *illegal*  
      // operations were not converted successfully.  
      auto function = getFunction();  
      if (mlir::failed(mlir::applyPartialConversion(function, target, patterns)))  
        signalPassFailure();  
    }  
      
    

## 다섯 번째 단계, 부분 Lowering 시 주의 사항

Lowering 과정에서 우리는 값 타입 `TensorType`에서 이미 할당된(버퍼와 유사한) 타입 `MemRefType`으로 변환한다. 그러나 `toy.print` 연산은 Lowering하지 않으려 하는데, 여기서는 주로 계산 집약적인 연산자를 처리해 최적화 기회를 찾는 것이 목적이고, `toy.print`는 단지 출력 기능을 가진 연산자일 뿐이기 때문이다. 그래서 이 장에서는 이 연산자를 Lowering하지 않는다. 그런데 우리가 알다시피 `toy.print` 연산의 정의는 입력 타입이 `F64Tensor`인 입력 데이터만 출력하도록 지원한다. 따라서 이를 MemRef Dialect와 연계할 수 있도록, `F64MemRef` 타입을 추가해야 한다. 즉, `mlir/examples/toy/Ch5/mlir/Ops.td` 안의 `toy.print` 연산 정의를 다음과 같이 수정한다.
    
    
    def PrintOp : Toy_Op<"print"> {  
      ...  
      // The print operation takes an input tensor to print.  
      // We also allow a F64MemRef to enable interop during partial lowering.  
      let arguments = (ins AnyTypeOf<[F64Tensor, F64MemRef]>:$input);  
      ...  
    }  
    

## 여섯 번째 단계, 위에서 정의한 부분 Lowering 기능을 최적화 pipeline에 추가

이 부분 코드는 `mlir/examples/toy/Ch5/toyc.cpp`에 있다.
    
    
    /// Create a pass for lowering operations in the `Affine` and `Std` dialects,  
    /// for a subset of the Toy IR (e.g. matmul).  
    std::unique_ptr<Pass> mlir::toy::createLowerToAffinePass() {  
      return std::make_unique<ToyToAffineLoweringPass>();  
    }  
      
    if (isLoweringToAffine) {  
     mlir::OpPassManager &optPM = pm.nestmlir::FuncOp();  
      
     // Partially lower the toy dialect with a few cleanups afterwards.  
     optPM.addPass(mlir::toy::createLowerToAffinePass());  
     optPM.addPass(mlir::createCanonicalizerPass());  
     optPM.addPass(mlir::createCSEPass());  
     ...  
    }  
    

이 코드는 `ToyToAffineLoweringPass`를 최적화 pipeline에 추가하는 것을 구현한다. MLIR을 생성하는 명령에서 `-emit=mlir-affine` 옵션을 사용하면 `isLoweringToAffine`이 참이 되어 이 부분 Lowering 과정이 실행된다.

부분 Lowering을 도입한 후 출력되는 MLIR 표현식이 어떤 모습인지 살펴보자. 먼저 원래의 MLIR 표현식, 즉 `mlir/test/Examples/Toy/Ch5/affine-lowering.mlir`을 보자.
    
    
    func @main() {  
      %0 = toy.constant dense<[[1.000000e+00, 2.000000e+00, 3.000000e+00], [4.000000e+00, 5.000000e+00, 6.000000e+00]]> : tensor<2x3xf64>  
      %2 = toy.transpose(%0 : tensor<2x3xf64>) to tensor<3x2xf64>  
      %3 = toy.mul %2, %2 : tensor<3x2xf64>  
      toy.print %3 : tensor<3x2xf64>  
      toy.return  
    }  
    

그 다음 `./toyc-ch5 ../../mlir/test/Examples/Toy/Ch5/affine-lowering.mlir -emit=mlir-affine`을 실행하면 이 절의 부분 Lowering이 적용된 MLIR 표현식을 확인할 수 있다. 다음과 같다.
    
    
    func @main() {  
      %cst = arith.constant 1.000000e+00 : f64  
      %cst_0 = arith.constant 2.000000e+00 : f64  
      %cst_1 = arith.constant 3.000000e+00 : f64  
      %cst_2 = arith.constant 4.000000e+00 : f64  
      %cst_3 = arith.constant 5.000000e+00 : f64  
      %cst_4 = arith.constant 6.000000e+00 : f64  
      
      // Allocating buffers for the inputs and outputs.  
      %0 = memref.alloc() : memref<3x2xf64>  
      %1 = memref.alloc() : memref<3x2xf64>  
      %2 = memref.alloc() : memref<2x3xf64>  
      
      // Initialize the input buffer with the constant values.  
      affine.store %cst, %2[0, 0] : memref<2x3xf64>  
      affine.store %cst_0, %2[0, 1] : memref<2x3xf64>  
      affine.store %cst_1, %2[0, 2] : memref<2x3xf64>  
      affine.store %cst_2, %2[1, 0] : memref<2x3xf64>  
      affine.store %cst_3, %2[1, 1] : memref<2x3xf64>  
      affine.store %cst_4, %2[1, 2] : memref<2x3xf64>  
      
      // Load the transpose value from the input buffer and store it into the  
      // next input buffer.  
      affine.for %arg0 = 0 to 3 {  
        affine.for %arg1 = 0 to 2 {  
          %3 = affine.load %2[%arg1, %arg0] : memref<2x3xf64>  
          affine.store %3, %1[%arg0, %arg1] : memref<3x2xf64>  
        }  
      }  
      
      // Multiply and store into the output buffer.  
      affine.for %arg0 = 0 to 3 {  
        affine.for %arg1 = 0 to 2 {  
          %3 = affine.load %1[%arg0, %arg1] : memref<3x2xf64>  
          %4 = affine.load %1[%arg0, %arg1] : memref<3x2xf64>  
          %5 = arith.mulf %3, %4 : f64  
          affine.store %5, %0[%arg0, %arg1] : memref<3x2xf64>  
        }  
      }  
      
      // Print the value held by the buffer.  
      toy.print %0 : memref<3x2xf64>  
      memref.dealloc %2 : memref<2x3xf64>  
      memref.dealloc %1 : memref<3x2xf64>  
      memref.dealloc %0 : memref<3x2xf64>  
      return  
    }  
    

간단히 해석해 보자. 먼저 여기에는 여섯 개의 `f64` 데이터 타입의 상수가 있고, `%cst`로 시작하는 변수로 표현된다. 그 다음 입력과 출력에 대한 버퍼를 할당하는데, 예를 들어 `%0 = memref.alloc() : memref<3x2xf64>`는 `memref<3x2xf64>` 타입의 버퍼 영역을 하나 할당한 것이다. 그 다음 앞서 선언한 여섯 개의 데이터를 위에서 할당한 버퍼에 차례로 저장하며, `affine.store`는 데이터를 저장하는 연산을 의미한다. 그 다음 첫 번째 루프에서는 입력 데이터를 로드(데이터 로드 연산 `affine.load`)하여 다른 데이터 컨테이너에 저장하고, 최종적으로 transpose 연산을 수행한다. 이어서 두 번째 루프에서는 앞서 두 데이터 컨테이너에 정의된 데이터를 로드해 곱한 뒤 출력 데이터 컨테이너에 저장한다. 마지막으로 `toy.print`로 결과를 출력하고 버퍼를 해제한다.

## Affine Dialect에서 최적화 기회 찾기

Affine Dialect를 사용한 후에는 Operation의 더 저수준 로직을 노출시켜 코드의 중복을 더 쉽게 드러낼 수 있다. 여기서 최적화할 수 있는 부분은 두 중첩 루프의 루프 경계가 같다는 점이며, 루프 융합(loop fusion)을 수행할 수 있다. 같은 루프 안에서 처리하면 루프 횟수를 줄일 수 있고, 동시에 불필요한 데이터 컨테이너 할당을 줄이며, 데이터 로드 시간도 줄일 수 있어 자연스럽게 프로그램 실행 효율이 향상된다.

이는 TVM의 scheduler에서의 다양한 루프 관련 primitive에 비유할 수 있다.

이제 위에서 언급한 루프 융합과 불필요한 데이터 컨테이너 할당 및 로드를 줄이는 최적화를 pipeline에 추가하려고 하는데, 구체적으로는 `mlir/examples/toy/Ch5/toyc.cpp` 안의 다음 코드이다.
    
    
    if (isLoweringToAffine) {  
      mlir::OpPassManager &optPM = pm.nestmlir::FuncOp();  
      
      // Partially lower the toy dialect with a few cleanups afterwards.  
      optPM.addPass(mlir::toy::createLowerToAffinePass());  
      optPM.addPass(mlir::createCanonicalizerPass());  
      optPM.addPass(mlir::createCSEPass());  
       
      // Add optimizations if enabled.  
      if (enableOpt) {  
        optPM.addPass(mlir::createLoopFusionPass());  
        optPM.addPass(mlir::createMemRefDataFlowOptPass());  
      }  
    }  
    

마지막 `if` 조건에 주목하자. `createLoopFusionPass`와 `createMemRefDataFlowOptPass`가 추가되어 있는데, **「이 두 가지 MLIR 자체 제공 Pass는 각각 동일한 루프 경계 융합 최적화와 MemRef에 대한 데이터 흐름 최적화 기능을 수행한다」** . 우리는 이 두 최적화를 추가한 후 위의 MLIR 표현식이 어떻게 변하는지 궁금하다. 구체적으로는 위에서 부분 Lowering된 MLIR 표현식을 생성하는 명령에 `-opt` 옵션 하나만 추가하면 이 두 최적화 Pass가 적용된 새로운 MLIR 표현식을 생성할 수 있다. 명령은 다음과 같다. `./toyc-ch5 ../../mlir/test/Examples/Toy/Ch5/affine-lowering.mlir -emit=mlir-affine -opt`. 생성된 MLIR 표현식은 다음과 같다.
    
    
    func @main() {  
      %cst = arith.constant 1.000000e+00 : f64  
      %cst_0 = arith.constant 2.000000e+00 : f64  
      %cst_1 = arith.constant 3.000000e+00 : f64  
      %cst_2 = arith.constant 4.000000e+00 : f64  
      %cst_3 = arith.constant 5.000000e+00 : f64  
      %cst_4 = arith.constant 6.000000e+00 : f64  
      
      // Allocating buffers for the inputs and outputs.  
      %0 = memref.alloc() : memref<3x2xf64>  
      %1 = memref.alloc() : memref<2x3xf64>  
      
      // Initialize the input buffer with the constant values.  
      affine.store %cst, %1[0, 0] : memref<2x3xf64>  
      affine.store %cst_0, %1[0, 1] : memref<2x3xf64>  
      affine.store %cst_1, %1[0, 2] : memref<2x3xf64>  
      affine.store %cst_2, %1[1, 0] : memref<2x3xf64>  
      affine.store %cst_3, %1[1, 1] : memref<2x3xf64>  
      affine.store %cst_4, %1[1, 2] : memref<2x3xf64>  
      
      affine.for %arg0 = 0 to 3 {  
        affine.for %arg1 = 0 to 2 {  
          // Load the transpose value from the input buffer.  
          %2 = affine.load %1[%arg1, %arg0] : memref<2x3xf64>  
      
          // Multiply and store into the output buffer.  
          %3 = arith.mulf %2, %2 : f64  
          affine.store %3, %0[%arg0, %arg1] : memref<3x2xf64>  
        }  
      }  
      
      // Print the value held by the buffer.  
      toy.print %0 : memref<3x2xf64>  
      memref.dealloc %1 : memref<2x3xf64>  
      memref.dealloc %0 : memref<3x2xf64>  
      return  
    }  
    

여기서 보다시피, 불필요한 데이터 버퍼 할당이 제거되었고, 두 중첩 루프가 하나로 융합되었으며, 일부 불필요한 데이터 로드 연산들도 제거되었다.

따라서 우리는 부분 Lowering을 통해 확실히 더 많은 최적화 기회를 찾았고, MLIR 표현식의 실행 효율을 더 높일 수 있었다.

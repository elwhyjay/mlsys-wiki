# Chapter1: Toy 언어와 AST

MLIR은 Toy 언어라는 것을 제공하여 MLIR의 정의 및 실행 흐름을 설명합니다. Toy 언어는 tensor 기반 언어로, 함수 정의, 일부 수학 연산 수행, 결과 출력 등에 사용할 수 있습니다. 아래에서 소개할 예시에서는 Tensor의 차원을 2 이하로 제한하며, Toy 언어의 유일한 데이터 타입은 64비트 부동소수점 타입으로 C 언어의 "double"에 해당합니다. 또한 `Values`는 재작성할 수 없는데, 즉 모든 operation은 새로 할당된 value를 반환하며 해제는 자동으로 관리됩니다. 다음 예시를 바로 보겠습니다.


    def main() {
      # Define a variable `a` with shape <2, 3>, initialized with the literal value.
      # The shape is inferred from the supplied literal.
      var a = [[1, 2, 3], [4, 5, 6]];

      # b is identical to a, the literal tensor is implicitly reshaped: defining new
      # variables is the way to reshape tensors (element count must match).
      var b<2, 3> = [1, 2, 3, 4, 5, 6];

      # transpose() and print() are the only builtin, the following will transpose
      # a and b and perform an element-wise multiplication before printing the result.
      print(transpose(a) * transpose(b));
    }


타입 검사는 type 추론(type inference)을 통해 정적으로 수행됩니다. Toy 언어는 필요한 경우에만 Tensor shape의 type 선언을 지정하면 됩니다. 다음에서는 `multiply_transpose` 함수를 정의하는데, 이 함수의 인자 `a`와 `b`의 shape는 사전에 알 수 없고, 이 함수가 호출될 때에야 비로소 알 수 있습니다. 아래 예시의 shape 변화를 주의 깊게 살펴보세요.


    # User defined generic function that operates on unknown shaped arguments.
    def multiply_transpose(a, b) {
      return transpose(a) * transpose(b);
    }

    def main() {
      # Define a variable `a` with shape <2, 3>, initialized with the literal value.
      var a = [[1, 2, 3], [4, 5, 6]];
      var b<2, 3> = [1, 2, 3, 4, 5, 6];

      # This call will specialize `multiply_transpose` with <2, 3> for both
      # arguments and deduce a return type of <3, 2> in initialization of `c`.
      var c = multiply_transpose(a, b);

      # A second call to `multiply_transpose` with <2, 3> for both arguments will
      # reuse the previously specialized and inferred version and return <3, 2>.
      var d = multiply_transpose(b, a);

      # A new call with <3, 2> (instead of <2, 3>) for both dimensions will
      # trigger another specialization of `multiply_transpose`.
      var e = multiply_transpose(b, c);

      # Finally, calling into `multiply_transpose` with incompatible shape will
      # trigger a shape inference error.
      var f = multiply_transpose(transpose(a), c);
    }

# Chapter2. 초급 MLIR 생성하기

MLIR은 완전히 확장 가능한 기반 프레임워크로 설계되어 있어, attribute, operation, type의 집합이 닫혀 있지 않습니다. MLIR은 **Dialect**(https://mlir.llvm.org/docs/LangRef/#dialects)라는 개념을 통해 이러한 확장성을 지원합니다. **Dialect**는 특정한 `namespace` 아래에서 추상화에 대한 그룹화 메커니즘을 제공합니다.

MLIR에서 Operation은 추상화와 계산의 핵심 단위로, 여러 면에서 LLVM의 명령(instruction)과 유사합니다. 애플리케이션 특화된 의미를 가지며, LLVM의 모든 핵심 IR 구조(명령, globals(function과 유사), 모듈)를 표현하는 데 사용할 수 있습니다. 다음은 Toy 언어가 만들어낸 `transpose` Operation의 예시입니다.


    %t_tensor = "toy.transpose"(%tensor) {inplace = true} : (tensor<2x3xf64>) -> tensor<3x2xf64> loc("example/file/path":12:1)


구조 분석:

  * `%t_tensor`: 이 Operation이 정의하는 result의 이름이며, 앞의 `%`는 충돌을 피하기 위한 것입니다. https://mlir.llvm.org/docs/LangRef/#identifiers-and-keywords 참고. 하나의 Operation은 0개 또는 여러 개의 result를 정의할 수 있으며(Toy 언어에서는 단일 result Operation만 존재), 이들은 SSA value입니다. 이 이름은 파싱 동안에만 사용되며 영속적이지 않습니다(예: SSA value의 메모리 표현에서는 추적되지 않습니다).
  * `"toy.transpose"`: Operation의 이름입니다. 유일한 문자열이어야 하며, Dialect의 namespace가 "."로 접두어로 붙습니다. 즉 Toy Dialect의 transpose Operation으로 이해할 수 있습니다.
  * `(%tensor)`: 0개 또는 여러 개의 입력 operand(또는 인자) 목록이며, 다른 operation에 의해 정의되거나 block 인자를 참조하는 SSA value들입니다.
  * `{ inplace = true }`: 0개 또는 여러 개의 attribute로 이루어진 dictionary로, 항상 상수인 특수한 operand입니다. 여기서는 "inplace"라는 boolean attribute를 정의했고 그 상수 값은 true입니다.
  * `(tensor<2x3xf64>) -> tensor<3x2xf64>`: 함수 형태로 표현된 operation의 type이며, 앞은 입력, 뒤는 출력입니다. `<2x3xf64>` 안의 내용은 tensor의 크기 `2x3`과 tensor에 저장되는 데이터 type `f64`를 나타내며, 사이를 `x`로 연결합니다.
  * `loc("example/file/path":12:1)`: 이 operation의 소스 코드상 위치입니다.



MLIR 명령의 기본 구조를 알아보았으니 Chapter2가 무엇을 하는지로 시선을 돌려봅시다. 즉 **초급 MLIR 생성하기**입니다. 다음 명령을 실행하여 Chapter2의 테스트 예시인 `codegen.toy`로부터 MLIR을 생성합니다.

## MLIR 생성 흐름

![이미지](images/img_02.png)AST에서 MLIR로, Dialect와 관련된 부분

먼저 AST를 순회하는 MLIRGen 함수가 있습니다. 구체적으로는 `mlir/examples/toy/Ch2/mlir/MLIRGen.cpp` 파일에 구현되어 있고, 그 안에 `mlirGen` 함수가 다음과 같이 구현되어 있습니다.


    /// Dispatch codegen for the right expression subclass using RTTI.
      mlir::Value mlirGen(ExprAST &expr) {
        switch (expr.getKind()) {
        case toy::ExprAST::Expr_BinOp:
          return mlirGen(cast<BinaryExprAST>(expr));
        case toy::ExprAST::Expr_Var:
          return mlirGen(cast<VariableExprAST>(expr));
        case toy::ExprAST::Expr_Literal:
          return mlirGen(cast<LiteralExprAST>(expr));
        case toy::ExprAST::Expr_Call:
          return mlirGen(cast<CallExprAST>(expr));
        case toy::ExprAST::Expr_Num:
          return mlirGen(cast<NumberExprAST>(expr));
        default:
          emitError(loc(expr.loc()))
              << "MLIR codegen encountered an unhandled expr kind '"
              << Twine(expr.getKind()) << "'";
          return nullptr;
        }
      }


이 함수는 AST의 노드 타입에 따라 다른 `mlirGen` 하위 함수를 재귀적으로 호출하며, 각 하위 함수에서 실제로 MLIR 표현으로 변환하는 작업을 수행합니다. 위 `codege.toy`의 `transpose(a)` operation을 예로 들면, 대응하는 `mlirGen` 하위 함수는 다음과 같습니다.


    /// Emit a call expression. It emits specific operations for the `transpose`
      /// builtin. Other identifiers are assumed to be user-defined functions.
      mlir::Value mlirGen(CallExprAST &call) {
        llvm::StringRef callee = call.getCallee();
        auto location = loc(call.loc());

        // Codegen the operands first.
        SmallVector<mlir::Value, 4> operands;
        for (auto &expr : call.getArgs()) {
          auto arg = mlirGen(*expr);
          if (!arg)
            return nullptr;
          operands.push_back(arg);
        }

        // Builtin calls have their custom operation, meaning this is a
        // straightforward emission.
        if (callee == "transpose") {
          if (call.getArgs().size() != 1) {
            emitError(location, "MLIR codegen encountered an error: toy.transpose "
                                "does not accept multiple arguments");
            return nullptr;
          }
          return builder.create<TransposeOp>(location, operands[0]);
        }

        // Otherwise this is a call to a user-defined function. Calls to
        // user-defined functions are mapped to a custom call that takes the callee
        // name as an attribute.
        return builder.create<GenericCallOp>(location, callee, operands);
      }


여기서 `if (callee == "transpose")` 부분은 함수 시그니처를 판별하는 코드입니다. `transpose`라면 `TransposeOp` type의 MLIR 노드를 새로 생성해야 하며, 즉 `builder.create<TransposeOp>(location, operands[0])`입니다. 이 코드는 MLIR의 Dialect 및 TableGen과 관련되므로 자세히 설명하겠습니다.

[[밑바닥부터 배우는 딥러닝 컴파일러] 11. MLIR 첫걸음](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247499292&idx=1&sn=149110d0d6fcec34856125ec01e4d38b&scene=21#wechat_redirect>)에서 이미 언급했듯, MLIR은 Dialect를 통해 다양한 수준의 IR을 통일합니다. 즉 다양한 Operation의 정의와 파싱을 담당하면서 동시에 확장성을 가집니다. Toy 언어에서도 Dialect를 정의했으며, 이 Dialect를 정의할 때 TableGen 규격을 사용해 `mlir/examples/toy/Ch2/include/toy/Ops.td`에 작성했습니다.


    // Provide a definition of the 'toy' dialect in the ODS framework so that we
    // can define our operations.
    def Toy_Dialect : Dialect {
      let name = "toy";
      let cppNamespace = "::mlir::toy";
    }


MLIR에서 Dialect와 Operation(연산자라고도 할 수 있음)의 정의 프레임워크는 TableGen(선언적 프로그래밍 언어의 일종)의 규격을 기반으로 구성되어 있으며, 소스 코드에서는 `.td` 형식으로 존재합니다. 컴파일 시점에 자동으로 대응하는 C++ 파일이 생성되어 정의된 Dialect가 만들어집니다. TableGen을 사용하는 이점은 단지 선언적 언어라서 Dialect와 Operation을 새로 추가하기 쉽다는 것뿐만 아니라, 수정과 유지보수도 쉽다는 점입니다. 제 설명이 직관적이지 않을 수도 있는데, Chapter2 코드인 `mlir/examples/toy/Ch2/include/toy/Ops.td`와 함께 직접 보면 이해하기 쉽습니다. 이후 Toy 언어 예시에서 `.td` 파일의 구성과 TableGen이 어떻게 자동으로 `.td`를 파싱해 C++ 코드를 생성하는지 보게 될 것입니다.

먼저 `td`에서 Toy Dialect를 정의하고 Dialect와의 연결을 설정합니다. 이는 이후 Toy Dialect 공간에서 정의되는 모든 Operation을 묶어주는 역할을 합니다. 즉:


    // Provide a definition of the 'toy' dialect in the ODS framework so that we
    // can define our operations.
    def Toy_Dialect : Dialect {
      let name = "toy";
      let cppNamespace = "::mlir::toy";
    }


그 다음 `Toy_Op` 클래스를 만들어 Toy Dialect 아래의 모든 Operation의 base class로 삼습니다. 이후 새로 추가하는 Operation은 모두 이 클래스를 상속해야 합니다.


    // Base class for toy dialect operations. This operation inherits from the base
    // `Op` class in OpBase.td, and provides:
    //   * The parent dialect of the operation.
    //   * The mnemonic for the operation, or the name without the dialect prefix.
    //   * A list of traits for the operation.
    class Toy_Op<string mnemonic, list<OpTrait> traits = []> :
        Op<Toy_Dialect, mnemonic, traits>;


다음은 `transpose` Operation의 정의 예시입니다. 어떤 느낌인지 살펴보세요.


    def TransposeOp : Toy_Op<"transpose"> {
      let summary = "transpose operation";

      let arguments = (ins F64Tensor:$input);
      let results = (outs F64Tensor);

      let assemblyFormat = [{
        `(` $input `:` type($input) `)` attr-dict `to` type(results)
      }];

      // Allow building a TransposeOp with from the input operand.
      let builders = [
        OpBuilder<(ins "Value":$input)>
      ];

      // Invoke a static verify method to verify this transpose operation.
      let verifier = [{ return ::verify(*this); }];
    }


`Toy_Op`를 상속하는 것에 더하여, TableGen 문법을 사용해 description, arguments, results, builder, verifier 등의 요소도 정의했습니다.

`td` 파일을 작성한 후에는 `mlir-tblgen` 도구를 사용해 C++ 코드를 생성할 수 있습니다. 먼저 다음 명령으로 Dialect의 C++ 코드를 생성합니다: `./mlir-tblgen -gen-dialect-decls llvm-project/mlir/examples/toy/Ch2/include/toy/Ops.td -I ../../mlir/include/`

![이미지](images/img_03.png)자동 생성된 Toy Dialect C++ 코드

위 명령을 `./mlir-tblgen -gen-op-defs llvm-project/mlir/examples/toy/Ch2/include/toy/Ops.td -I ../../mlir/include/`로 바꾸면 Operation의 C++ 코드를 생성할 수 있습니다. 관심 있는 독자는 직접 살펴보시기 바랍니다.

toyc-ch2 toolchain과의 연결은 CMakeList.txt 파일에서 확인할 수 있습니다(기본 위치는 `llvm-project/mlir/examples/toy/Ch2/include/toy`).


    set(LLVM_TARGET_DEFINITIONS Ops.td)
    mlir_tablegen(Ops.h.inc -gen-op-decls)
    mlir_tablegen(Ops.cpp.inc -gen-op-defs)
    mlir_tablegen(Dialect.h.inc -gen-dialect-decls)
    mlir_tablegen(Dialect.cpp.inc -gen-dialect-defs)
    add_public_tablegen_target(ToyCh2OpsIncGen)


mlir-tblgen에 `-gen-op-decls`와 `-gen-op-defs` 옵션을 함께 사용해 Ops.h.inc 선언 코드와 Ops.cpp.inc 정의 코드를 생성하고, 이 둘을 toolchain toyc-ch2를 빌드하기 위한 코드 의존으로 사용합니다.

정리하면, Chapter2에서는 MLIR의 핵심 구성 요소인 MLIRGen, Dialect, Operation, TableGen에 대해 소개하고, 이들이 어떻게 상호작용하는지를 살펴보았습니다. 이들의 관계는 중국과학원의 Zhang Hongbin(장훙빈) 학생의 PPT를 빌려와 더 잘 설명할 수 있습니다.

# [밑바닥부터 배우는 딥러닝 컴파일러] 18. MLIR의 Interfaces

# 0x0. 머리말

이 글은 MLIR의 Interfaces(인터페이스)에 대해 알아보기 위한 글입니다. MLIR은 범용적이고 확장 가능한 프레임워크로, 서로 다른 계층의 특정 attribute, Operation, Type을 가진 Dialects로 구성됩니다. 바로 이 Dialects의 계층적 설계 덕분에 MLIR은 다양한 의미와 추상화 수준의 Operation을 표현할 수 있습니다. 그러나 이 계층적 설계에는 단점도 존재하는데, 서로 다른 Dialect 계층 간에 Operation 변환을 수행하거나 변환(Pass)을 적용할 때, 각 Dialect 아래의 각 Operation의 구체적인 의미를 명확히 알고 있어야 한다는 점입니다. 그렇지 않으면 변환에 실패할 수 있습니다. 사실 MLIR을 기반으로 개발해본 독자라면, 일부 MLIR Pass를 조합하여 MLIR 파일에 대해 lowering을 수행할 때 Op 변환에 실패하는 상황을 겪어봤을 것입니다. 이러한 상황을 완화하기 위해 MLIR은 Interfaces를 제안했습니다. 사실 [[밑바닥부터 배우는 딥러닝 컴파일러] 13. MLIR에서 Pass를 어떻게 작성하는가?](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247499540&idx=1&sn=c88146d7f00d83d8671fe6214d4416d4&scene=21#wechat_redirect>) 에서 우리는 이미 Interfaces를 활용하여 인라이닝과 형상 추론(shape inference) Pass를 구현했었습니다. 이번 절에서는 MLIR의 Interfaces에 대해 더 깊이 알아보고, 마지막에는 OneFlow IR의 UserOpCompatibleInterface 예시와 결합하여 이해를 한층 더 다지도록 하겠습니다.

> ❝
> 
> 본문에서 언급되는 Operation과 op는 같은 것이며, 모두 MLIR Dialect 아래의 op를 의미합니다.
> 
> ❞

# 0x1. 동기

Interfaces는 인터페이스로 번역할 수 있으며, MLIR의 Interfaces는 IR과 상호작용하는 일반적인 방법을 제공합니다. Interfaces의 설계 목표는 특정 Dialect의 특정 Operation이나 Dialect의 특정 지식에 침습하지 않고도 MLIR 표현식을 변환하고 분석할 수 있도록 하는 것입니다. 이렇게 하면 변환, 분석과 새로운 Dialect 및 그에 대응하는 Operation 추가를 디커플링하여 MLIR의 확장성을 크게 향상시킬 수 있습니다.

# 0x2. Dialect Interfaces 정의(자세히 보기)

Dialect Interfaces는 일반적으로 한 그룹의 attribute, Operation, type에 대해 일반적인 변환(Pass)이나 분석을 수행하고자 할 때 사용되며, 이러한 attribute, Operation, type은 서로 다른 Dialect에 의해 정의된 것일 수 있습니다. 이러한 Interfaces는 일반적으로 다양한 수준의 Dialects에 광범위하게 걸쳐 있으며, 소수의 분석과 변환에만 사용됩니다. 따라서 Interface는 Operation의 핵심이 아니라 일부 일반적인 변환의 핵심임을 명확히 해야 합니다. [[밑바닥부터 배우는 딥러닝 컴파일러] 13. MLIR에서 Pass를 어떻게 작성하는가?](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247499540&idx=1&sn=c88146d7f00d83d8671fe6214d4416d4&scene=21#wechat_redirect>) 에는 인라이닝 Interface를 사용하여 인라이닝 Pass를 구현하는 예시가 있습니다. 인라이닝은 일반적으로 Dialect의 Operation에 대한 고수준 정보(예: cost modeling과 합법성)를 조회하는데, 이러한 정보는 일반적으로 특정 Dialect의 특정 Operation에만 국한되어 존재하지 않습니다.

Dialect Interface는 CRTP 기반 클래스 `DialectInterfaceBase::Base<>`를 상속하여 정의할 수 있습니다. CRTP에 대한 소개는 다음을 참고하세요: `https://zh.wikipedia.org/wiki/奇异递归模板模式`. 제가 이해하기로는 정적 다형성(CRTP)을 사용하는 이유는 MLIR 안에는 이 `DialectInterfaceBase::Base<>` 기반 클래스에서 파생되는 Dialect Interface가 많이 존재하기 때문에, 성능을 고려할 때 CRTP를 사용하는 것이 적합하기 때문입니다. 이 기반 클래스는 Dialect Interface 등록에 필요한 일부 인터페이스를 제공하여, 향후 그것들을 참조하는 데 편리합니다. Interface가 정의된 후, Dialects는 Dialect 고유 정보를 사용하여 그것을 재정의(override)할 수 있습니다. 한 Dialect에 의해 정의된 Interfaces는 `addInterfaces<>`를 통해 등록되며, 이는 attribute, Operation, Type의 등록 메커니즘과 유사합니다. 다음 예시를 살펴봅시다.
    
    
    // 기본적인 인라이닝 Interface 클래스를 정의하여 Dialect가 인라이닝에 옵트인할 수 있도록 한다.
    class DialectInlinerInterface :  
        public DialectInterface::Base<DialectInlinerInterface> {  
    public:  
      /// 주어진 region 'src'가 해당 region에 인라이닝될 수 있으면 true를 반환한다.
      /// 'dest'는 현재 Dialect에 등록된 Operation에 부착된다.
      /// 'valueMapping'은 'src' region 내에서 재매핑된 모든 값을 포함한다.
      /// 예를 들어, 이는 어떤 값들이 "src" region의 진입(entry) 인자를 대체할지를 검사하는 데 사용될 수 있다.
      virtual bool isLegalToInline(Region *dest, Region *src,  
                                   BlockAndValueMapping &valueMapping) const {  
        return false;  
      }  
    };  
      
    /// 인라이닝 인터페이스를 재정의하여 AffineDialect에 대한 지원을 추가하고 Affine Dialect의 Operation 인라이닝을 활성화한다.
    struct AffineInlinerInterface : public DialectInlinerInterface {  
      /// Affine 구조는 특정한 인라이닝 제약을 가진다.
      bool isLegalToInline(Region *dest, Region *src,  
                           BlockAndValueMapping &valueMapping) const final {  
        ...  
      }  
    };  
      
    /// Dialect 아래에 인라이닝 Interfaces를 등록한다.
    AffineDialect::AffineDialect(MLIRContext *context) ... {  
      addInterfaces<AffineInlinerInterface>();  
    }  
    

이러한 Interfaces가 등록된 후에는, MLIR의 변환과 분석을 실행할 때 Dialect에서 조회할 수 있으며, 특정 Dialect 서브클래스(예: 특정 Operation으로 좁혀진 것)를 결정할 필요가 없습니다. 예를 들어:
    
    
    Dialect *dialect = ...;  
    if (DialectInlinerInterface *interface  
          = dialect->getRegisteredInterface<DialectInlinerInterface>()) {  
      // Dialect는 이 Interface의 구현을 제공한다
      ...  
    }  
    

예를 들어, `llvm/mlir/lib/IR/Dialect.cpp` 파일의 `registerDelayedInterfaces` 함수가 위와 같은 사용법을 보여주는데, 이 함수는 로드된 Dialect의 Interfaces를 등록하는 데 사용됩니다.
    
    
    void DialectRegistry::registerDelayedInterfaces(Dialect *dialect) const {  
      auto it = interfaces.find(dialect->getTypeID());  
      if (it == interfaces.end())  
        return;  
      
      // Add an interface if it is not already present.  
      for (const auto &kvp : it->getSecond().dialectInterfaces) {  
        if (dialect->getRegisteredInterface(kvp.first))  
          continue;  
        dialect->addInterface(kvp.second(dialect));  
      }  
      
      // Add attribute, operation and type interfaces.  
      for (const auto &info : it->getSecond().objectInterfaces)  
        std::get<2>(info)(dialect->getContext());  
    }  
    

# 0x3. DialectInterfaceCollection (선택적, 아직 사용해보지 않음)

`DialectInterfaceCollection`은 추가적인 유틸리티를 제공합니다. 이 클래스는 `MLIRContext` 인스턴스에 주어진 Interface가 등록된 모든 Dialect를 수집할 수 있게 해줍니다. 이는 등록된 Dialect Interface 조회를 숨기고 최적화하는 데 유용합니다.
    
    
    class InlinerInterface : public  
        DialectInterfaceCollection<DialectInlinerInterface> {  
      // 이 클래스의 hook은 DialectInlinerInterface의 hook의 미러이며, 기본 구현은 주어진 Dialect의 Interface 위 hook을 호출하는 것이다.
      virtual bool isLegalToInline(Region *dest, Region *src,  
                                   BlockAndValueMapping &valueMapping) const {  
        auto *handler = getInterfaceFor(dest->getContainingOp());  
        return handler ? handler->isLegalToInline(dest, src, valueMapping) : false;  
      }  
    };  
      
    MLIRContext *ctx = ...;  
    InlinerInterface interface(ctx);  
    if(!interface.isLegalToInline(...))  
       ...  
    

# 0x4. Attribute, Operation, Type Interfaces (선택적)

이름에서 알 수 있듯이, attribute/operation/type Interface는 특정 attribute/operation/type 수준에서 등록되는 것들입니다. 이러한 Interface는 반드시 구현해야 하는 가상(virtual) 인터페이스를 **「제공함으로써」** 파생 객체에 대한 접근을 제공합니다. 예를 들어, 많은 분석과 변환에서는 성능과 정확성을 향상시키기 위해 Operation의 부수효과(side effect)를 알고자 합니다. Operation의 부수효과는 일반적으로 특정 Operation의 의미와 관련이 있습니다. 예를 들어 affine.load Operation은 (이름에서 알 수 있듯이) 읽기 효과를 가집니다.

이러한 Interface는 특정 IR 엔티티의 CRTP 클래스를 재정의하여 정의됩니다. 각각 `AttrInterface`, `OpInterface`, `TypeInterface` 입니다. 이 클래스들은 `Concept`과 `Model` 클래스를 정의하는 `Traits` 클래스를 템플릿 인자로 받습니다. 이 클래스들은 concept 기반 다형성의 구현을 제공하는데, 여기서 `Concept`은 가상 메서드 집합을 정의하고, 이 메서드들은 구체적인 엔티티 타입에 대해 템플릿화된 `Model`에 의해 재정의됩니다. 주의할 점은, 이 클래스들은 순수해야 하며, 비정적 데이터 멤버나 다른 가변 데이터를 포함해서는 안 된다는 점입니다. Interface를 객체에 부착하기 위해, 기반 클래스는 해당 객체의 trait 목록에 부착할 수 있는 `Trait` 클래스를 제공합니다(아래 예시 코드를 건너뛰면 설명을 볼 수 있습니다).
    
    
    struct ExampleOpInterfaceTraits {  
      /// 구현해야 할 가상 인터페이스를 지정하는 기본 Concept 클래스를 정의한다.
      struct Concept {  
        virtual ~Concept();  
      
        /// 이는 Operation의 비정적 hook의 예시이다.
        virtual unsigned exampleInterfaceHook(Operation *op) const = 0;  
      
        /// 이는 Operation의 정적 hook의 예시이다. 정적 hook은 Operation의 구체 인스턴스를 필요로 하지 않는다.
        /// 구현은 비정적인 경우와 마찬가지로 가상 hook이며, hook 자체의 구현은 여전히 간접적으로 구현되어야 한다.
        virtual unsigned exampleStaticInterfaceHook() const = 0;  
      };  
      
      /// 주어진 Operation 타입의 concept에 대한 model 클래스를 정의한다.
      template <typename ConcreteOp>  
      struct Model : public Concept {  
        /// 구체 Operation에 디스패치할 메서드를 재정의한다.
        unsigned exampleInterfaceHook(Operation *op) const final {  
          return llvm::cast<ConcreteOp>(op).exampleInterfaceHook();  
        }  
      
        /// 구체 Operation에 디스패치할 메서드를 재정의한다.
        unsigned exampleStaticInterfaceHook() const final {  
          return ConcreteOp::exampleStaticInterfaceHook();  
        }  
      };  
    };  
      
    /// 분석과 변환이 상호작용할 메인 Interface 클래스를 정의한다.
    class ExampleOpInterface : public OpInterface<ExampleOpInterface,  
                                                  ExampleOpInterfaceTraits> {  
    public:  
      /// LLVM 스타일 캐스팅을 지원하기 위해 기반 클래스의 생성자를 상속한다.
      using OpInterface<ExampleOpInterface, ExampleOpInterfaceTraits>::OpInterface;  
      
      /// Interface는 `getImpl()`로 디스패치되며, 이는 기본 `OpInterface` 클래스가 제공하는 메서드로 concept의 인스턴스를 반환한다.
      unsigned exampleInterfaceHook() const {  
        return getImpl()->exampleInterfaceHook(getOperation());  
      }  
      unsigned exampleStaticInterfaceHook() const {  
        return getImpl()->exampleStaticInterfaceHook(getOperation()->getName());  
      }  
    };  
      
    

Interface가 정의되면, 앞서 언급한 것처럼 제공된 trait `ExampleOpInterface::Trait`를 추가하여 op에 등록할 수 있습니다. 이 인터페이스를 사용하는 것은 다른 파생 op 타입을 사용하는 것과 같으며, 즉 강제 캐스팅하면 됩니다.
    
    
      
    /// Operation을 정의할 때, Interface는 `OpInterface<>` 기반 클래스가 제공하는 중첩된 `Trait` 클래스를 통해 등록된다.
    class MyOp : public Op<MyOp, ExampleOpInterface::Trait> {  
    public:  
      /// The definition of the interface method on the derived operation.  
      unsigned exampleInterfaceHook() { return ...; }  
      static unsigned exampleStaticInterfaceHook() { return ...; }  
    };  
      
    /// 나중에, 우리는 특정 Operation(예: "MyOp")이 주어진 Interface를 재정의하는지 조회할 수 있다.
    Operation *op = ...;  
    if (ExampleOpInterface example = dyn_cast<ExampleOpInterface>(op))  
      llvm::errs() << "hook returned = " << example.exampleInterfaceHook() << "\n";  
    

여기까지 읽으셨고 이전의 [[밑바닥부터 배우는 딥러닝 컴파일러] 13. MLIR에서 Pass를 어떻게 작성하는가?](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247499540&idx=1&sn=c88146d7f00d83d8671fe6214d4416d4&scene=21#wechat_redirect>) 에서 인라이닝 Interface를 사용하는 예시를 보셨다면, Toy Dialect 아래에 인라이닝 Pass를 등록하는 몇 가지 단계를 더 잘 이해할 수 있을 것이라 믿습니다.

# 0x5. Attribute, Operation 및 Type Interfaces의 외부 Model (선택적)

이는 IR 객체에 대한 Interface 구현을 해당 객체의 정의를 수정하지 않고 제공해야 할 때 필요할 수 있습니다. 주목할 점은, 이를 통해 attribute, Operation, type의 Interface를 그것들이 정의된 Dialect 외부에서 구현할 수 있다는 점입니다. 예를 들어, built-in 타입에 대해 Interface를 제공할 수 있습니다. 이는 `Concept`에서 파생된 두 가지 concept 기반 다형성 Model 확장 클래스를 사용하여 구현됩니다. 다음과 같이(주석을 잘 보세요):
    
    
    struct ExampleTypeInterfaceTraits {  
      struct Concept {  
        virtual unsigned exampleInterfaceHook(Type type) const = 0;  
        virtual unsigned exampleStaticInterfaceHook() const = 0;  
      };  
      
      template <typename ConcreteType>  
      struct Model : public Concept { /*...*/ };  
      
      /// Model과 달리, FallbackModel은 type 객체를
      /// hook에 전달하여, 메서드가 클래스 자체에서 정의되지 않아 "this" 접근 권한이 없더라도
      /// 메서드 본문에서 접근할 수 있도록 한다. ODS는 모든 Interface에 대해 자동으로 이 클래스를 생성한다.
      template <typename ConcreteType>  
      struct FallbackModel : public Concept {  
        unsigned exampleInterfaceHook(Type type) const override {  
          getImpl()->exampleInterfaceHook(type);  
        }  
        unsigned exampleStaticInterfaceHook() const override {  
          ConcreteType::exampleStaticInterfaceHook();  
        }  
      };  
      
      /// `ExternalModel`은 Interface를 구현하는 model 클래스와 Interface를 구현하는 type 클래스를 명시적으로 분리하여,
      /// Interface 메서드의 기본 구현을 위한 위치를 제공한다. 그러면 `cast<ConcreteType>`을 사용하여 기본 구현을 정의할 수 있다.
      /// `ConcreteType`이 기본 구현에 필요한 API를 제공하지 않으면, 사용자 정의 구현은 `FallbackModel`을 직접 사용하여 기본 구현을 재정의할 수 있다.
      /// 클래스 템플릿 안에 위치하므로, 결코 인스턴스화되지 않으며 컴파일 오류를 일으키지 않는다.
      /// ODS는 이 클래스를 자동으로 생성하고 그 안에 기본 메서드 구현을 배치한다.
      template <typename ConcreteModel, typename ConcreteType>  
      struct ExternalModel : public FallbackModel<ConcreteModel> {  
        unsigned exampleInterfaceHook(Type type) const override {  
          // Default implementation can be provided here.  
          return type.cast<ConcreteType>().callSomeTypeSpecificMethod();  
        }  
      };  
    };  
      
    

`FallbackModel` 또는 `ExternalModel`을 파생하고 주어진 context에서 관련 클래스에 Model 클래스를 등록함으로써, attribute, Operation, type 인터페이스에 대한 외부 Model을 제공할 수 있습니다. 등록되지 않으면, 다른 context는 해당 Interface를 볼 수 없습니다. 예를 들어:
    
    
    /// 구체 클래스의 외부 Interface 구현. 이는 type 클래스 자체의 정의를 수정할 필요가 없다.
    struct ExternalModelExample  
        : public ExampleTypeInterface::ExternalModel<ExternalModelExample,  
                                                     IntegerType> {  
      static unsigned exampleStaticInterfaceHook() {  
        // 구현은 여기에 제공된다
        return IntegerType::someStaticMethod();  
      }  
      // "ExternalModel"에서 이미 기본 구현이 있는 "exampleInterfaceHook"는 정의할 필요가 없다. 하지만 필요하다면 재정의할 수 있다.
    }  
      
    int main() {  
      MLIRContext context;  
      /* ... */;  
      
      // 사용 전에, interface model을 주어진 context의 type에 부착한다. 이 시점에 해당 type을 포함하는 Dialect가 이미 로드되어 있을 것으로 예상된다.
      IntegerType::attachInterface<ExternalModelExample>(context);  
    }  
    

문서 마지막에는 한 가지 권고사항도 제시되었는데, 외부에서 적용하는 Interface를 우리가 "소유"할 때만 이 메커니즘을 사용하라는 것입니다. 이는 객체를 포함하는 Dialect의 소유자와 interface의 소유자 모두 Interface 구현을 알지 못하는 상황을 방지할 수 있으며, 이러한 상황은 중복되거나 분기된 구현을 초래할 수 있습니다. 아직 이 메커니즘을 사용해야 할 상황을 만나본 적이 없어, 여기서는 더 깊이 들어가지 않겠습니다.

# 0x6. OpInterface의 Dialect Fallback (선택적)

일부 Dialect는 개방된 생태계를 가지고 있어, 가능한 모든 Operation을 등록하지는 않습니다. 이러한 경우에도 이러한 op를 구현하는 `OpInterface`에 대한 지원을 제공할 수 있습니다. op가 등록되지 않았거나 Interface 구현을 제공하지 않을 때, 조회는 Dialect 자체로 fallback됩니다.

두 번째 Model이 이러한 경우에 사용되며, ODS가 `FallbackModel`(아래 참조)이라는 이름을 사용할 때 자동으로 생성됩니다. 이 Model은 특정 dialect를 위해 구현될 수 있습니다.
    
    
    // This is the implementation of a dialect fallback for `ExampleOpInterface`.  
    struct FallbackExampleOpInterface  
        : public ExampleOpInterface::FallbackModel<  
              FallbackExampleOpInterface> {  
      static bool classof(Operation *op) { return true; }  
      
      unsigned exampleInterfaceHook(Operation *op) const;  
      unsigned exampleStaticInterfaceHook() const;  
    };  
    

그런 다음, Dialect는 이 구현을 인스턴스화하고, `getRegisteredInterfaceForOp` 메서드를 재정의하여 특정 Operation에 대해 그것을 반환할 수 있습니다.
    
    
    void *TestDialect::getRegisteredInterfaceForOp(TypeID typeID,  
                                                   StringAttr opName) {  
      if (typeID == TypeID::get<ExampleOpInterface>()) {  
        if (isSupported(opName))  
          return fallbackExampleOpInterface;  
        return nullptr;  
      }  
      return nullptr;  
    }  
    

이 절도 잘 이해되지 않으니, 일단 기록만 해두겠습니다. 위에서 Interfaces에 대한 소개를 통해 우리는 기본적인 인상을 남길 수 있는 것으로 충분하다고 생각합니다. 다음으로 다룰 내용은 ODS를 기반으로 Interfaces를 정의하는 방법이며, 이것이 이 글의 핵심입니다.

# 0x7. ODS 프레임워크를 활용한 Interface 정의 (중요)

위에서 언급한 것처럼, Interface는 attribute, Operation, Type이 호출자가 특정 파생 타입을 알 필요 없이 메서드를 호출할 수 있도록 노출시킵니다. 이러한 인프라의 단점은 모든 부분을 연결하기 위해 일정량의 보일러플레이트가 필요하다는 점입니다. MLIR은 ODS에서 선언적으로 인터페이스를 정의하고 C++ 정의를 자동으로 생성하는 메커니즘을 제공합니다. 예를 들어:
    
    
    def ExampleOpInterface : OpInterface<"ExampleOpInterface"> {  
      let description = [{  
        This is an example interface definition.  
      }];  
      
      let methods = [  
        InterfaceMethod<  
          "This is an example of a non-static hook to an operation.",  
          "unsigned", "exampleInterfaceHook"  
        >,  
        StaticInterfaceMethod<  
          "This is an example of a static hook to an operation.",  
          "unsigned", "exampleStaticInterfaceHook"  
        >,  
      ];  
    }  
    

`AttrInterface`, `OpInterface`, `TypeInterface` 클래스의 정의를 제공하면, Interface의 C++ 클래스가 자동으로 생성됩니다. Interface는 다음과 같은 구성 요소로 구성됩니다.

  * C++ 클래스 이름 (템플릿 인자를 통해 제공).
  * 설명. (`description`).
  * C++ Namespace. (`cppNamespace`). 즉, Interface 클래스가 어떤 C++ 네임스페이스 아래에서 생성되어야 하는지.
  * Methods (`methods`). IR 객체에 의해 정의된 Interfaces hook 메서드 목록.
  * Extra Class Declarations. 선택적: `extraClassDeclaration`. Interface 클래스의 선언에서 생성될 추가 C++ 코드. 이는 사용자 대면 Interface 클래스에서 메서드 등을 정의할 수 있도록 하며, IR 엔티티에 hook할 필요가 없습니다. 이러한 선언은 인터페이스 메서드의 기본 구현 안에서는 암시적으로 보이지 않지만, 전체 이름 한정자(full name qualifier)를 사용하여 정적 선언에 접근할 수 있습니다.



`OpInterface` 클래스에는 추가로 Verifier(`verify`)가 포함될 수도 있습니다. 이는 Interface가 부착된 Operation에 적용되는 추가 검증을 포함하는 C++ 코드 블록입니다. 이 코드 블록의 구조는 `Trait::verifyTrait` 메서드의 구조와 1대1로 대응됩니다.

Interface와 함께 사용할 수 있는 메서드는 두 가지 유형이 있는데, `InterfaceMethod`와 `StaticInterfaceMethod`입니다. 둘 다 동일한 핵심 구성 요소로 이루어지며, 차이점은 `StaticInterfaceMethod`가 파생된 IR 객체의 정적 메서드를 모델링한다는 점입니다. Interface 메서드는 다음과 같은 구성 요소를 가집니다.

  * Description: 메서드의 설명 정보, 문자열.
  * ReturnType: 메서드의 C++ 반환 타입에 대응하는 문자열.
  * MethodName: 메서드의 C++ 이름에 대응하는 문자열.
  * Arguments (Optional): 각각 C++ 타입과 변수 이름에 대응하는 문자열.
  * MethodBody (Optional)과 DefaultImplementation. 아직 사용되지 않았으며, 나중에 필요할 때 다시 찾아볼 예정입니다.



Operation이 `DeclareOpInterfaceMethods`를 사용하여 Interface를 지정하는 경우, ODS는 Operation의 `InterfaceMethods`에 대한 선언도 생성할 수 있습니다(아래 예시 참조).
    
    
    ef MyInterface : OpInterface<"MyInterface"> {  
      let description = [{  
        This is the description of the interface. It provides concrete information  
        on the semantics of the interface, and how it may be used by the compiler.  
      }];  
      
      let methods = [  
        InterfaceMethod<[{  
          This method represents a simple non-static interface method with no  
          inputs, and a void return type. This method is required to be implemented  
          by all operations implementing this interface. This method roughly  
          correlates to the following on an operation implementing this interface:  
      
          ```c++  
          class ConcreteOp ... {  
          public:  
            void nonStaticMethod();  
          };  
          ```  
        }], "void", "nonStaticMethod"  
        >,  
      
        InterfaceMethod<[{  
          This method represents a non-static interface method with a non-void  
          return value, as well as an `unsigned` input named `i`. This method is  
          required to be implemented by all operations implementing this interface.  
          This method roughly correlates to the following on an operation  
          implementing this interface:  
      
          ```c++  
          class ConcreteOp ... {  
          public:  
            Value nonStaticMethod(unsigned i);  
          };  
          ```  
        }], "Value", "nonStaticMethodWithParams", (ins "unsigned":$i)  
        >,  
      
        StaticInterfaceMethod<[{  
          This method represents a static interface method with no inputs, and a  
          void return type. This method is required to be implemented by all  
          operations implementing this interface. This method roughly correlates  
          to the following on an operation implementing this interface:  
      
          ```c++  
          class ConcreteOp ... {  
          public:  
            static void staticMethod();  
          };  
          ```  
        }], "void", "staticMethod"  
        >,  
      
        StaticInterfaceMethod<[{  
          This method corresponds to a static interface method that has an explicit  
          implementation of the method body. Given that the method body has been  
          explicitly implemented, this method should not be defined by the operation  
          implementing this method. This method merely takes advantage of properties  
          already available on the operation, in this case its `build` methods. This  
          method roughly correlates to the following on the interface `Model` class:  
      
          ```c++  
          struct InterfaceTraits {  
            /// ... The `Concept` class is elided here ...  
      
            template <typename ConcreteOp>  
            struct Model : public Concept {  
              Operation *create(OpBuilder &builder, Location loc) const override {  
                return builder.create<ConcreteOp>(loc);  
              }  
            }  
          };  
          ```  
      
          Note above how no modification is required for operations implementing an  
          interface with this method.  
        }],  
          "Operation *", "create", (ins "OpBuilder &":$builder, "Location":$loc),  
          /*methodBody=*/[{  
            return builder.create<ConcreteOp>(loc);  
        }]>,  
      
        InterfaceMethod<[{  
          This method represents a non-static method that has an explicit  
          implementation of the method body. Given that the method body has been  
          explicitly implemented, this method should not be defined by the operation  
          implementing this method. This method merely takes advantage of properties  
          already available on the operation, in this case its `build` methods. This  
          method roughly correlates to the following on the interface `Model` class:  
      
          ```c++  
          struct InterfaceTraits {  
            /// ... The `Concept` class is elided here ...  
      
            template <typename ConcreteOp>  
            struct Model : public Concept {  
              Operation *create(Operation *opaqueOp, OpBuilder &builder,  
                                Location loc) const override {  
                ConcreteOp op = cast<ConcreteOp>(opaqueOp);  
                return op.getNumInputs() + op.getNumOutputs();  
              }  
            }  
          };  
          ```  
      
          Note above how no modification is required for operations implementing an  
          interface with this method.  
        }],  
          "unsigned", "getNumInputsAndOutputs", (ins), /*methodBody=*/[{  
            return $_op.getNumInputs() + $_op.getNumOutputs();  
        }]>,  
      
        InterfaceMethod<[{  
          This method represents a non-static method that has a default  
          implementation of the method body. This means that the implementation  
          defined here will be placed in the trait class that is attached to every  
          operation that implements this interface. This has no effect on the  
          generated `Concept` and `Model` class. This method roughly correlates to  
          the following on the interface `Trait` class:  
      
          ```c++  
          template <typename ConcreteOp>  
          class MyTrait : public OpTrait::TraitBase<ConcreteType, MyTrait> {  
          public:  
            bool isSafeToTransform() {  
              ConcreteOp op = cast<ConcreteOp>(this->getOperation());  
              return op.getNumInputs() + op.getNumOutputs();  
            }  
          };  
          ```  
      
          As detailed in [Traits](Traits.md), given that each operation implementing  
          this interface will also add the interface trait, the methods on this  
          interface are inherited by the derived operation. This allows for  
          injecting a default implementation of this method into each operation that  
          implements this interface, without changing the interface class itself. If  
          an operation wants to override this default implementation, it merely  
          needs to implement the method and the derived implementation will be  
          picked up transparently by the interface class.  
      
          ```c++  
          class ConcreteOp ... {  
          public:  
            bool isSafeToTransform() {  
              // Here we can override the default implementation of the hook  
              // provided by the trait.  
            }  
          };  
          ```  
        }],  
          "bool", "isSafeToTransform", (ins), /*methodBody=*/[{}],  
          /*defaultImplementation=*/[{  
        }]>,  
      ];  
    }  
      
    // Operation interfaces can optionally be wrapped inside  
    // DeclareOpInterfaceMethods. This would result in autogenerating declarations  
    // for members `foo`, `bar` and `fooStatic`. Methods with bodies are not  
    // declared inside the op declaration but instead handled by the op interface  
    // trait directly.  
    def OpWithInferTypeInterfaceOp : Op<...  
        [DeclareOpInterfaceMethods<MyInterface>]> { ... }  
      
    // Methods that have a default implementation do not have declarations  
    // generated. If an operation wishes to override the default behavior, it can  
    // explicitly specify the method that it wishes to override. This will force  
    // the generation of a declaration for those methods.  
    def OpWithOverrideInferTypeInterfaceOp : Op<...  
        [DeclareOpInterfaceMethods<MyInterface, ["getNumWithDefault"]>]> { ... }  
    

참고: ODS 프레임워크에서는 `OpInterfaceTrait` 클래스를 통해 C++로 정의된 기존 Operation 인터페이스에 접근할 수 있습니다.

# 0x9. OneFlow의 Interface

다음으로 OneFlow IR을 예로 들어, OneFlow Dialect에서 어떤 Interface가 정의되어 있는지 살펴보겠습니다. 코드는 `oneflow/ir/include/OneFlow/OneFlowInterfaces.td` 에 있습니다. OneFlow는 ODS를 기반으로 `UserOpCompatibleInterface`, `ControlEdgeCompatibleInterface`, `NoGrad` 등의 Interface 타입을 정의했습니다. `UserOpCompatibleInterface`를 예로 그 구현을 살펴봅시다.
    
    
    def UserOpCompatibleInterface : OpInterface<"UserOpCompatible"> {  
      let description = [{  
        Interface to getting the hard-coded bn  
      }];  
      
      let methods = [  
        StaticInterfaceMethod<"",  
            "const std::vector<std::string>*", "inputKeys", (ins), [{  
            static std::vector<std::string> val(mlir::oneflow::support::GetInputKeys(ConcreteOp::getOperationName().split('.').second.str()));  
            return &val;  
        }]>,  
        StaticInterfaceMethod<"",  
            "const std::vector<std::string>*", "outputKeys", (ins), [{  
            static std::vector<std::string> val(mlir::oneflow::support::GetOutputKeys(ConcreteOp::getOperationName().split('.').second.str()));  
            return &val;  
        }]>,  
        InterfaceMethod<"",  
            "std::pair<unsigned, unsigned>", "getODSOperandIndexAndLength", (ins "unsigned":$index), [{  
            return $_op.getODSOperandIndexAndLength(index);  
        }]>,  
        InterfaceMethod<"",  
            "std::pair<unsigned, unsigned>", "getODSResultIndexAndLength", (ins "unsigned":$index), [{  
            return $_op.getODSResultIndexAndLength(index);  
        }]>  
      ];  
    }  
    

`UserOpCompatibleInterface`는 7절의 Interface ODS 규약에 있는 StaticInterfaceMethod와 InterfaceMethod를 사용하여 이 Interface에 대해 Operation의 입력 operand 이름, 출력 operand 이름, operand 및 길이, result 및 길이 등을 가져오는 메서드들을 지정한 것을 볼 수 있습니다. 그런 다음 OneFlow의 `oneflow/ir/include/OneFlow/OneFlowUserOps.td`에서 `DeclareOpInterfaceMethods<UserOpCompatibleInterface>`를 사용하여 다양한 Operation에 Interface를 지정하면, 생성된 Operation 코드에 이 Interface 선언이 포함됩니다.

그렇다면 이렇게 하면 어떤 이점이 있을까요? 첫 번째는 OneFlow의 UserOp 모두가 UserOpCompatibleInterface를 가지고 있기 때문에, OneFlow의 UserOp에 대해 일반적인 `GetInputKeys` 함수만 구현하면, UserOp에서 파생된 모든 Operation이 이 함수의 기능을 가지게 된다는 것입니다. 그것들 모두가 UserOpCompatibleInterface 인터페이스를 가지고 있기 때문입니다.

더 일반적인 예시는 Interface를 기반으로 일부 일반적인 Pass를 개발하는 것입니다. 예를 들어 인라이닝과 형상 추론(shape inference) Pass가 있습니다. [[밑바닥부터 배우는 딥러닝 컴파일러] 13. MLIR에서 Pass를 어떻게 작성하는가?](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247499540&idx=1&sn=c88146d7f00d83d8671fe6214d4416d4&scene=21#wechat_redirect>) 를 참고하세요.

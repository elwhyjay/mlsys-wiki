# 11\. Constraint (이것이 매우 중요하다)

Constraint는 표 기반 Operation 정의에서 핵심 개념이다. Operation 검증과 그래프 Operation 매칭은 모두 Constraint에 기반하여 이루어진다. 따라서 Operation 정의와 재작성 규칙(rewrite rule)은 모두 Constraint를 작성하는 일과 직접적으로 관련된다. MLIR은 `OpBase.td`(`https://github.com/llvm/llvm-project/blob/main/mlir/include/mlir/IR/OpBase.td`)에서 `Constraint` base class를 정의한다. 한 Operation의 Constraint는 서로 다른 범위를 다룰 수 있다. 예를 들면 다음과 같다.

  * 단일 attribute에만 관심을 두는 경우 (예: 5보다 큰 32비트 정수)
  * 여러 operand와 result에 관심을 두는 경우 (예: 첫 번째 result의 shape이 첫 번째 operand(Tensor로 이해 가능)의 shape과 동일해야 함)
  * Operation 자체에 내재된 경우 (예: 부수 효과가 없음, Transpose Op 제거 사례 참고)



이들을 각각 단일 엔티티 Constraint, 다중 엔티티 Constraint, Trait이라고 부른다. 여기에서는 개념만 알아두면 되고, 새로운 Constraint를 작성하는 것이 가장 중요하다고 생각한다.

  * **단일 엔티티 Constraint**. 단일 엔티티 Constraint는 단일 operand, attribute 또는 result를 작용 범위로 한다. 해당 Constraint는 엔티티가 선언되는 위치에서 지정한다. 예를 들어 **Operation arguments** 와 **Operation results** 에서 지정한다([[밑바닥부터 배우는 딥러닝 컴파일러] 16. MLIR ODS 핵심 정리 (상)](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247499828&idx=1&sn=d7ed4550bcf76e2267ac5a1eb9d5154e&scene=21#wechat_redirect>) 에서 Operation arguments와 Operation results에 대해 알아두어야 할 지식을 정리하였다).

  * **다중 엔티티 Constraint**. 다중 엔티티 Constraint는 `https://github.com/llvm/llvm-project/blob/main/mlir/include/mlir/IR/OpBase.td`에서 `PredOpTrait` class(`OpTrait`의 한 sub class)로 모델링된다. 전체 목록은 `OpBase.td`를 참고하자.

  * **Trait**. Trait은 Operation의 본질적 속성이다. 예를 들어 부수 효과 유무, 교환 가능 여부, 종결자(terminator) 여부 등이다. 이러한 Constraint는 Op class의 template 매개변수로 지정해야 한다. [[밑바닥부터 배우는 딥러닝 컴파일러] 16. MLIR ODS 핵심 정리 (상)](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247499828&idx=1&sn=d7ed4550bcf76e2267ac5a1eb9d5154e&scene=21#wechat_redirect>) 의 3절 Op의 Trait과 Constraint(Operation traits and constraints)에서 다룬 바와 같다. Trait은 `https://github.com/llvm/llvm-project/blob/main/mlir/include/mlir/IR/OpBase.td`에서 `NativeOpTrait` class(`OpTrait`의 한 sub class)로 모델링된다. 이들은 지원되며 대응되는 C++ `mlir::OpTrait` class로 변환된다.

  * **새로운 Constraint를 어떻게 지정하는가**? 새로운 Constraint를 작성하려면 술어(predicate)를 제공하고 설명용 이름을 지정해야 한다. `Pred` class를 사용하여 모델링된 술어는 Constraint를 구성하는 핵심이다. Constraint의 술어는 보통 중첩된 방식으로 구성된다. 술어에는 두 가지 종류가 있다. 1. `CPred`: 가장 원초적인 leaf 노드 술어. 2. 합성 술어: 술어 결합자를 사용하여 sub 술어로부터 구성되는 술어 (conjunction: `And`, disjunction: `Or`, negation: `Neg`, substitution: `SubstLeaves`, concatenation: `Concat`). `CPred`는 더 복잡한 술어를 구성하는 토대이다. 그것은 TableGen 시각에서의 "원자" 술어이며, TableGen과 C++ 사이의 "인터페이스"이다. 그 안은 이미 C++ 코드이며, 특수한 placeholder를 가진 불투명한 문자열로 처리되고 치환된다. 우리는 boolean 값을 반환하는 어떠한 C++ 코드든 `CPred`에 넣을 수 있다. 표현식 계산, 함수 호출, class 메서드 호출 등 무엇이든 가능하다.




C++ 환경과의 상호작용을 돕기 위해, 해당 술어가 사용되는 context의 엔티티를 참조할 수 있는 몇 가지 특수 placeholder가 제공된다. 이들은 둘러싼 환경의 "후크(hook)" 역할을 한다. `$_builder`, `$_op`, `$_self`가 여기에 해당한다.

  * `$_builder`는 `mlir::Builder` 인스턴스로 치환되며, 이를 통해 일반적인 build 메서드에 접근할 수 있다.
  * `$_op`은 현재 Operation으로 치환되며, 이를 통해 현재 Operation의 정보에 접근할 수 있다.
  * `$_self`는 해당 술어가 부착된 엔티티로 치환된다. 예를 들어 `BoolAttr`는 `CPred<"$_self.isa<BoolAttr>()">`를 포함하는 attribute Constraint이다. 따라서 `BoolAttr:$attr`의 경우 `$_self`는 `$attr`로 치환된다. type Constraint의 경우는 조금 특별한데, 각 type 정의의 Constraint를 자연스럽게 읽을 수 있게 하고 싶고, 또 type Constraint를 operand/result에 직접 부착하고 싶기 때문에 `$_self`는 operand/result의 type으로 치환된다. 예를 들어 `F32:$operand`의 `F32`에서 그 `$_self`는 `operand(...).getType()`으로 확장된다.



예를 들어 attribute `attr`이 `IntegerAttr`인지 검사하려면 C++에서는 `attr.isa<IntegerAttr>()`를 호출하면 된다. 이 코드는 `$_self.isa<IntegerAttr>()` 형태로 `CPred`에 wrapping될 수도 있는데, 여기서 `$_self`는 특수 placeholder로서 확장 시 현재 attribute `attr`로 치환되어 동일한 기능을 수행한다(여기서는 TableGen에서의 의미).

더 복잡한 술어의 경우, 단일 `CPred`로 wrapping할 수도 있고, 술어 결합자를 사용하여 조합할 수도 있다. 예를 들어 attribute `attr`이 32비트 또는 64비트 정수라는 Constraint를 작성하려면 다음과 같이 쓸 수 있다.
    
    
    And<[  
      CPred<"$_self.isa<IntegerAttr>()">,  
      Or<[  
        CPred<"$_self.cast<IntegerAttr>().getType().isInteger(32)">,  
        CPred<"$_self.cast<IntegerAttr>().getType().isInteger(64)">  
      ]>  
    ]>  
    

(주의: 위는 익숙한 예를 통해 `CPred`와 술어 결합자를 사용하여 복잡한 술어를 작성하는 방법을 보여준 것일 뿐이다. 구체적으로 정수 attribute의 경우 `OpBase.td`에 이미 `I32Attr`과 `I64Attr`이 정의되어 있다. 따라서 실제로는 이를 재사용하여 `Or<[I32Attr.predicate, I64Attr.predicate]>`로 쓸 수 있다.)

여기에서 OneFlow의 한 예시로 다시 한번 설명해 본다. IsGPU라는 Constraint를 정의했다.
    
    
    def IsGPU: Constraint<CPred<"$0.getValue().equals(\"gpu\")">, "is GPU device">;  
    

그 다음 OneFlow는 Transformer 부분에 맞춤형 최적화를 추가하였는데, Scale과 Tril 두 개의 연속된 Kernel을 하나의 큰 Kernel로 fuse하여 메모리 읽기/쓰기 시간을 일부 절약하는 것이다. 다만 이 fuse된 kernel은 GPU 환경에서만 적용되므로, 현재 계산 그래프에서 검출된 Scale과 Tril 두 Operation의 device가 GPU인지 판단해야 하고, 이 때문에 위와 같은 Constraint가 필요하다. FusedScaleTrilPattern Pass의 구현은 다음과 같으며, 마지막에 IsGPU Constraint가 사용된 것을 확인할 수 있다.
    
    
    def FusedScaleTrilPattern : Pat<  
      (  
        OneFlow_TrilOp  
        (  
          OneFlow_ScalarMulOp  
            $x,  
            $scale_op_name,  
            $scale_trainable,  
            $scale_device_tag,  
            $scale_device_name,  
            $scale_scope_symbol_id,  
            $scale_hierarchy,  
            $has_int_operand,  
            $has_float_operand,  
            $int_operand,  
            $float_operand  
        ),  
        $tril_op_name,  
        $tril_trainable,  
        $tril_device_tag,  
        $tril_device_name,  
        $tril_scope_symbol_id,  
        $tril_hierarchy,  
        $diagonal,  
        $floating_fill_value,  
        $integer_fill_value,  
        $is_floating_fill_value  
      ),  
      (OneFlow_FusedScaleTrilOp $x,  
        $tril_op_name,  
        $tril_trainable,  
        $tril_device_tag,  
        $tril_device_name,  
        $tril_scope_symbol_id,  
        $tril_hierarchy,  
        $diagonal,  
        $floating_fill_value,  
        $integer_fill_value,  
        $is_floating_fill_value,  
        $float_operand,  
        $int_operand,  
        $has_float_operand  
      ),  
      [  
        (IsGPU $tril_device_tag),  
        (IsGPU $scale_device_tag)  
      ]  
    >;  
    

이 Pass의 기능은 연속된 Scale+Tril Operation을 검출하면 두 Operation을 하나의 FusedScaleTril Operation으로 fuse하는 것이다.

만약 술어를 `CPred`와 술어 결합자만으로 작성하는 것이 너무 복잡하다면, 일반적인 C++ 함수로 작성한 뒤 `CPred`를 함수를 "호출"하는 수단으로 사용할 수도 있다. 예를 들어 attribute `attr`이 어떤 속성을 가지는지 검증하기 위해 다음과 같은 C++ 함수를 작성할 수 있다.
    
    
    bool HasSomeProperty(Attribute attr) { ... }  
    

그리고 Op는 다음과 같이 정의한다.
    
    
    def HasSomeProperty : AttrConstraint<CPred<"HasSomeProperty($_self)">,  
                                         "has some property">;  
      
    def MyOp : Op<...> {  
      let arguments = (ins  
        ...  
        HasSomeProperty:$attr  
      );  
    }  
    

전체 표현식을 단일 `CPred`로 wrapping할지, 술어 결합자를 가진 여러 `CPred`를 사용할지, 또는 단일 `CPred`로 함수를 "호출"하여 술어를 정의할지에 대해 명확한 기준은 없다. `CPred`와 술어 결합자를 사용하여 정의하는 방식은 (C++ 함수 뒤에 모든 로직을 숨기는 대신) 더 많은 정보를 Operation 정의 명세에 노출하기 때문에 잠재적으로 더 많은 자동 생성 사례를 끌어낼 수 있어 권장된다. 다만 그러려면 중복을 피할 수 있는 좋은 범용 술어 라이브러리가 building block으로서 필요한데, 현재 연구 중이다.

# 12\. attribute 정의 (매우 중요 +1)

attribute는 컴파일 시점에 알 수 있는 Operation의 상수이다. ODS는 C++ attribute class에 대한 attribute wrapper를 제공한다. MLIR의 핵심 IR 라이브러리에는 일부 일반적인 C++ attribute class가 정의되어 있다(`https://github.com/llvm/llvm-project/blob/main/mlir/include/mlir/IR/Attributes.h`). ODS는 TableGen에서 이러한 attribute를 사용하여 Operation을 정의할 수 있도록 하며, 더 세분화된 Constraint도 가질 수 있다. 예를 들어 `StrAttr`는 `StringAttr`로 직접 매핑되며, `F32Attr/F64Attr`는 `FloatAttr`이 추가로 일정한 비트 폭을 가질 것을 요구한다. ODS attribute는 storage type(attribute를 저장하는 `mlir::Attribute` class에 대응), return type(생성되는 `getters` 도우미 함수의 C++ return type에 대응), 그리고 내부 storage type과 도우미 함수 사이를 변환하는 메서드를 가지도록 정의된다.

**attribute 데코레이터**. ODS attribute에 적용하여 선택성, 기본값 등 흔히 쓰이는 부가 속성을 지정할 수 있는 중요한 attribute adapter/decorator/modifier가 몇 가지 있다.

  * `DefaultValuedAttr`: attribute에 기본값을 지정한다.
  * `OptionalAttr`: attribute를 optional로 지정한다.
  * `Confined`: `Confined`는 값 type이 도입하는 attribute Constraint를 추가로 모델링하는 데 도움을 주기 위한 범용 메커니즘으로 제공된다. `Confined`를 통해 비교적 원초적인 Constraint를 조합하여 복잡한 Constraint를 만들 수 있다. 예를 들어 최솟값이 10인 32비트 정수형은 `Confined<I32Attr, [IntMinValue<10>]>`로 표현할 수 있다. 그 외에 `IntMinValue<N>`(N 이상인 정수형 attribute를 지정) 같은 다른 예도 있다.



**enum attribute**. 어떤 attribute는 미리 정의된 enum에서만 값을 가져올 수 있다. 예를 들어 비교 op의 비교 종류가 그렇다. 이러한 attribute를 정의하기 위해 ODS는 `StrEnumAttr`, `IntEnumAttr`, `BitEnumAttr` 등 여러 메커니즘을 제공한다.

  * `StrEnumAttr`: 각 enum case는 문자열이며, attribute는 op에서 `StringAttr`로 저장된다.
  * `IntEnumAttr`: 각 enum case는 정수이며, attribute는 op에서 `IntegerType`으로 저장된다.
  * `BitEnumAttr`: 각 enum case는 한 비트이며, attribute는 op에서 `IntegerAttr`로 저장된다.



이러한 모든 `*EnumAttr` attribute는 대응되는 `*EnumAttrCase`를 통해 허용되는 모든 case를 완전히 명시해야 한다. 이를 통해 ODS는 허용된 case만 받아들이도록 추가 검증을 생성할 수 있다. `*EnumAttrs`와 그 C++ 사용자 사이의 상호작용을 촉진하기 위해, EnumsGen(`https://github.com/llvm/llvm-project/blob/main/mlir/tools/mlir-tblgen/EnumsGen.cpp`) TableGen backend는 일반적인 유틸리티들을 생성할 수 있다. C++ enum class, enum class용 `llvm::DenseMapInfo`, 문자열로/문자열로부터의 변환 함수가 그 예이다. 이는 `mlir-tblgen`의 `-gen-enum-decls`와 `-gen-enum-defs` 명령행 옵션으로 제어된다.

예를 들어 다음과 같은 `EnumAttr`이 주어졌을 때:
    
    
    def Case15: I32EnumAttrCase<"Case15", 15>;  
    def Case20: I32EnumAttrCase<"Case20", 20>;  
      
    def MyIntEnum: I32EnumAttr<"MyIntEnum", "An example int enum",  
                               [Case15, Case20]> {  
      let cppNamespace = "Outer::Inner";  
      let stringToSymbolFnName = "ConvertToEnum";  
      let symbolToStringFnName = "ConvertToString";  
    }  
    

다음 코드가 `mlir-tblgen -gen-enum-decls`을 통해 생성된다.
    
    
    namespace Outer {  
    namespace Inner {  
    // An example int enum  
    enum class MyIntEnum : uint32_t {  
      Case15 = 15,  
      Case20 = 20,  
    };  
      
    llvm::Optional<MyIntEnum> symbolizeMyIntEnum(uint32_t);  
    llvm::StringRef ConvertToString(MyIntEnum);  
    llvm::Optional<MyIntEnum> ConvertToEnum(llvm::StringRef);  
    inline constexpr unsigned getMaxEnumValForMyIntEnum() {  
      return 20;  
    }  
      
    } // namespace Inner  
    } // namespace Outer  
      
    namespace llvm {  
    template<> struct DenseMapInfo<Outer::Inner::MyIntEnum> {  
      using StorageInfo = llvm::DenseMapInfo<uint32_t>;  
      
      static inline Outer::Inner::MyIntEnum getEmptyKey() {  
        return static_cast<Outer::Inner::MyIntEnum>(StorageInfo::getEmptyKey());  
      }  
      
      static inline Outer::Inner::MyIntEnum getTombstoneKey() {  
        return static_cast<Outer::Inner::MyIntEnum>(StorageInfo::getTombstoneKey());  
      }  
      
      static unsigned getHashValue(const Outer::Inner::MyIntEnum &val) {  
        return StorageInfo::getHashValue(static_cast<uint32_t>(val));  
      }  
      
      static bool isEqual(const Outer::Inner::MyIntEnum &lhs, const Outer::Inner::MyIntEnum &rhs) {  
        return lhs == rhs;  
      }  
    };  
    }  
    

다음 코드는 `mlir-tblgen -gen-enum-defs`을 통해 생성된다.
    
    
    namespace Outer {  
    namespace Inner {  
    llvm::StringRef ConvertToString(MyIntEnum val) {  
      switch (val) {  
        case MyIntEnum::Case15: return "Case15";  
        case MyIntEnum::Case20: return "Case20";  
      }  
      return "";  
    }  
      
    llvm::Optional<MyIntEnum> ConvertToEnum(llvm::StringRef str) {  
      return llvm::StringSwitch<llvm::Optional<MyIntEnum>>(str)  
          .Case("Case15", MyIntEnum::Case15)  
          .Case("Case20", MyIntEnum::Case20)  
          .Default(llvm::None);  
    }  
    llvm::Optional<MyIntEnum> symbolizeMyIntEnum(uint32_t value) {  
      switch (value) {  
      case 15: return MyIntEnum::Case15;  
      case 20: return MyIntEnum::Case20;  
      default: return llvm::None;  
      }  
    }

    // An example bit enum  
    enum class MyBitEnum : uint32_t {  
      None = 0,  
      Bit1 = 1,  
      Bit2 = 2,  
      Bit3 = 4,  
    };  
      
    llvm::Optional<MyBitEnum> symbolizeMyBitEnum(uint32_t);  
    std::string stringifyMyBitEnum(MyBitEnum);  
    llvm::Optional<MyBitEnum> symbolizeMyBitEnum(llvm::StringRef);  
    inline MyBitEnum operator|(MyBitEnum lhs, MyBitEnum rhs) {  
      return static_cast<MyBitEnum>(static_cast<uint32_t>(lhs) | static_cast<uint32_t>(rhs));  
    }  
    inline MyBitEnum operator&(MyBitEnum lhs, MyBitEnum rhs) {  
      return static_cast<MyBitEnum>(static_cast<uint32_t>(lhs) & static_cast<uint32_t>(rhs));  
    }  
    inline bool bitEnumContains(MyBitEnum bits, MyBitEnum bit) {  
      return (static_cast<uint32_t>(bits) & static_cast<uint32_t>(bit)) != 0;  
    }  
      
    namespace llvm {  
    template<> struct DenseMapInfo<::MyBitEnum> {  
      using StorageInfo = llvm::DenseMapInfo<uint32_t>;  
      
      static inline ::MyBitEnum getEmptyKey() {  
        return static_cast<::MyBitEnum>(StorageInfo::getEmptyKey());  
      }  
      
      static inline ::MyBitEnum getTombstoneKey() {  
        return static_cast<::MyBitEnum>(StorageInfo::getTombstoneKey());  
      }  
      
      static unsigned getHashValue(const ::MyBitEnum &val) {  
        return StorageInfo::getHashValue(static_cast<uint32_t>(val));  
      }  
      
      static bool isEqual(const ::MyBitEnum &lhs, const ::MyBitEnum &rhs) {  
        return lhs == rhs;  
      }  
    };  
    
    
    
    std::string stringifyMyBitEnum(MyBitEnum symbol) {  
      auto val = static_cast<uint32_t>(symbol);  
      // 모든 비트가 unset인 경우의 특수 처리.  
      if (val == 0) return "None";  
      
      llvm::SmallVector<llvm::StringRef, 2> strs;  
      if (1u & val) { strs.push_back("Bit1"); val &= ~1u; }  
      if (2u & val) { strs.push_back("Bit2"); val &= ~2u; }  
      if (4u & val) { strs.push_back("Bit3"); val &= ~4u; }  
      
      if (val) return "";  
      return llvm::join(strs, "|");  
    }  
      
    llvm::Optional<MyBitEnum> symbolizeMyBitEnum(llvm::StringRef str) {  
      // 모든 비트가 unset인 경우의 특수 처리.  
      if (str == "None") return MyBitEnum::None;  
      
      llvm::SmallVector<llvm::StringRef, 2> symbols;  
      str.split(symbols, "|");  
      
      uint32_t val = 0;  
      for (auto symbol : symbols) {  
        auto bit = llvm::StringSwitch<llvm::Optional<uint32_t>>(symbol)  
          .Case("Bit1", 1)  
          .Case("Bit2", 2)  
          .Case("Bit3", 4)  
          .Default(llvm::None);  
        if (bit) { val |= *bit; } else { return llvm::None; }  
      }  
      return static_cast<MyBitEnum>(val);  
    }  
      
    llvm::Optional<MyBitEnum> symbolizeMyBitEnum(uint32_t value) {  
      // 모든 비트가 unset인 경우의 특수 처리.  
      if (value == 0) return MyBitEnum::None;  
      
      if (value & ~(1u | 2u | 4u)) return llvm::None;  
      return static_cast<MyBitEnum>(value);  
    }

# 13\. type 정의 (간단히 살펴보았을 뿐)

MLIR은 `TypeDef` class 계층을 정의하여, 명세에 따라 데이터 type을 생성하는 기능을 지원한다. type은 `TypeDef` class를 특수화함으로써 정의되며, 필요한 모든 필드의 구체적 내용을 가진다. 예를 들어 정수 type은 다음과 같이 정의할 수 있다.
    
    
    // 모든 type은 이 class를 상속한다.  
    class Test_Type<string name> : TypeDef<Test_Dialect, name> { }  
      
    // 또 다른 형태의 정수 type.  
    def IntegerType : Test_Type<"TestInteger"> {  
      let mnemonic = "int";  
      
      let summary = "An integer type with special semantics";  
      
      let description = [{  
        An alternate integer type. This type differentiates itself from the  
        standard integer type by not having a SignednessSemantics parameter, just  
        a width.  
      }];  
      
      let parameters = (ins "unsigned":$width);  
      
      // printer는 inline으로 정의한다.  
      let printer = [{  
        $_printer << "int<" << getImpl()->width << ">";  
      }];  
      
      // parser도 여기에서 정의한다.  
      let parser = [{  
        if ($_parser.parseLess())  
          return Type();  
        int width;  
        if ($_parser.parseInteger(width))  
          return Type();  
        if ($_parser.parseGreater())  
          return Type();  
        return get($_ctxt, width);  
      }];  
    }  
    

  * **Type name**: 생성되는 C++ class의 이름은 기본적으로 `<classParamName>Type`이다(위 예시의 `TestIntegerType`). 이 이름은 `cppClassName` 필드를 통해 override할 수 있다. `mnemonic`은 parsing에 사용되는 asm 이름을 지정한다. 이는 optional이며, 지정하지 않으면 이 class에 parser나 print 메서드가 부착되지 않음을 의미한다.

  * **Type documentation**: `summary`와 `description` 필드가 존재하며, 사용 방식은 Operation에서와 동일하다. 즉 `summary`는 한 줄이어야 하고, `description`은 더 긴 설명이어야 한다.

  * **Type parameters**: `parameters` 필드는 type parameter의 목록이다. 어떤 parameter도 지정하지 않으면(default), 이 type은 singleton type으로 간주된다. parameter는 `"c++Type":$paramName` 형식을 따른다. C++ type을 parameter로 사용하면서 storage 생성자 안에서 할당이 필요한 경우 두 가지 선택지가 있다. 1. `hasCustomStorageConstructor`를 설정하여, 방금 선언된 생성자를 가진 TypeStorage class를 생성하되 정의는 두지 않는다. 그래서 우리가 직접 작성할 수 있다. 2. "c++Type" 문자열 대신 `TypeParameter` tablegen class를 사용한다. (뒷부분은 잘 이해하지 못했고 사용해 본 적도 없다.)

  * **TypeParameter tablegen class**: 이는 각 type parameter에 대한 attribute를 한층 더 자세히 지정하는 데 쓰인다. 문서(`summary`와 `syntax`), 사용할 C++ type, storage 생성자 메서드에서 사용할 custom 할당자, 그리고 parameter type의 두 인스턴스가 같은지 판단하기 위한 custom 비교자 등을 포함한다.



    
    
    // 이렇게 하면 안 된다!  
    let parameters = (ins "ArrayRef<int>":$dims);  
    

기본 storage 생성자는 필드를 값으로 무작정 복사할 뿐이고, type에 대해서는 아무것도 모른다. 이 경우 ArrayRef는 `dims = allocator.copyInto(dims)`를 사용하여 할당해야 한다.
    
    
    class ArrayRefIntParam :  
        TypeParameter<"::llvm::ArrayRef<int>", "Array of ints"> {  
      let allocator = "$_dst = $_allocator.copyInto($_self);";  
    }  
      
    ...  
      
    let parameters = (ins ArrayRefIntParam:$dims);  
    

`allocator` 코드 블록은 `$_allocator`(객체를 할당할 TypeStorageAllocator)와 `$_dst`(할당된 데이터를 놓을 변수)로 구성된다. `comparator` 코드 블록은 `$_lhs`와 `$_rhs` parameter type 인스턴스로 구성된다.

custom Type에 관해서는 더 많은 내용이 있지만, 현재 그쪽 수요가 없어 더 이상 살펴보지 않았다. 여기서는 간단히 알아본 정도이다. 관심 있는 독자는 다음 문서를 참고하여 직접 더 깊이 연구해 보기 바란다: https://mlir.llvm.org/docs/OpDefinitions/.

# 14\. DEBUG 방법

`mlir-tblgen`을 사용하여 생성된 텍스트를 살펴본다. TableGen 문법은 때때로 매우 난해할 수 있다. 생성된 텍스트를 읽어 보는 것은 문제를 이해하고 디버깅하는 데 매우 유용하다. `mlir-tblgen`을 build하려면 build 디렉터리에서 `cmake --build . --target mlir-tblgen`을 실행하면 되며, `bin/` 하위 디렉터리에서 `mlir-tblgen` 바이너리를 찾을 수 있다. 지원되는 모든 generator는 `mlir-tblgen --help`를 통해 확인할 수 있다.

생성된 코드를 보려면 `-I`로 include 경로를 제공하고, `mlir-tblgen`을 호출하여 특정 generator를 사용한다. 예를 들면 다음과 같다.
    
    
    # op C++ class 선언을 보려면  
    mlir-tblgen --gen-op-decls -I /path/to/mlir/include /path/to/input/td/file  
    # op C++ class 정의를 보려면  
    mlir-tblgen --gen-op-defs -I /path/to/mlir/include /path/to/input/td/file  
    # op 문서를 보려면  
    mlir-tblgen --gen-dialect-doc -I /path/to/mlir/include /path/to/input/td/file  
      
    # op interface C++ class 선언을 보려면  
    mlir-tblgen --gen-op-interface-decls -I /path/to/mlir/include /path/to/input/td/file  
    # op interface C++ class 정의를 보려면  
    mlir-tblgen --gen-op-interface-defs -I /path/to/mlir/include /path/to/input/td/file  
    # op interface 문서를 보려면  
    mlir-tblgen --gen-op-interface-doc -I /path/to/mlir/include /path/to/input/td/file  
    

# 15\. 정리

이번 절에서는 [[밑바닥부터 배우는 딥러닝 컴파일러] 16. MLIR ODS 핵심 정리 (상)](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247499828&idx=1&sn=d7ed4550bcf76e2267ac5a1eb9d5154e&scene=21#wechat_redirect>) 을 바탕으로 ODS의 핵심 사항을 보충하여 마무리하였다. Constraint와 attribute의 정의는 모두 MLIR에서 상당히 중요한 요소이며, type 정의에 대해서는 개인적으로 알아두는 정도면 충분하고, 직접 type을 커스터마이즈해야 할 때 자세히 연구해도 늦지 않다고 생각한다. 마지막으로 MLIR의 문법은 다소 난해하므로 초보자는 `mlir-tblgen`을 활용하여 debug에 도움을 받을 수 있다.

이 두 편의 글에서 MLIR의 ODS 규범을 따라 한 번 완전히 훑어보면서 14개의 핵심 사항을 정리하였다. 각 핵심 사항마다 OneFlow MLIR의 Op 정의를 대조하여 살펴보았고, 일부 예시 코드와 위치도 함께 제시하였다. 독자가 MLIR에 입문하는 데 도움이 되기를 바란다.

# 0x1. 본문 개요 (전체 글의 너무 길어서 안 읽음 버전)

본 글에서 소개하는 Pattern Rewrite 메커니즘은 사실 [[밑바닥부터 배우는 딥러닝 컴파일러] 19. MLIR의 Pass 메커니즘 실습](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247501841&idx=1&sn=d48ed145df3d18ca383abad82e0f035e&scene=21#wechat_redirect>) 글의 이론 부분에 대한 보충 자료이다. MLIR에서 Pattern Rewrite 메커니즘은 매우 중요하다. IR에 대한 일반적인 변환 최적화뿐만 아니라, Op의 규범화(canonicalization)와 Dialect 간 및 Dialect 내부의 Op 변환(conversion)도 담당하기 때문이다. 따라서 MLIR의 핵심 컴포넌트를 이해하려면 Pattern Rewrite 메커니즘을 알아 두는 것이 필수적이다.

본 글의 0x2절에서는 Pattern Rewrite 메커니즘을 소개하기 전에 일반적인 Dag Rewriter 메커니즘을 먼저 다룬다. Pattern Rewriter 아키텍처도 Dag Rewriter 메커니즘 위에 구축되어 있다. 이 절에서는 먼저 Dag 변환이 컴파일러 분야에서 매우 흔한 변환이라는 점을 설명하고, MLIR이 동일한 기반 아키텍처 위의 다단계(multi-level) IR이기 때문에 Dag Rewrite 메커니즘을 도입하면 다양한 분야의 여러 문제를 해결할 수 있음을 설명한다. 여기서 상수 폴딩(constant folding) 예시를 언급하는데, MLIR에서는 `fold` 메커니즘으로 손쉽게 처리할 수 있다. 그 다음 Dag Rewriter 메커니즘과 관련된 작업들을 소개하면서 MLIR의 Dag Rewriter 메커니즘이 LLVM의 DAG-to-DAG Instruction Selection Infrastructure와 비교적 유사하다는 점을 언급하고, 그 LLVM DAG-to-DAG 명령어 선택 인프라의 장단점도 함께 설명한다. 이 절에서는 주로 MLIR의 Dag Rewriter 메커니즘 도입 배경과 역사적 맥락, 관련 작업 등을 다루며, 너무 깊이 신경 쓸 필요는 없다.

본 글의 0x3절에서는 MLIR의 Pattern Rewriter 인프라(범용 DAG to DAG 변환 프레임워크)의 설계와 API를 자세히 다룬다. 이 프레임워크는 MLIR 전반에 걸쳐 규범화, conversion, 일반 변환(transformation)에 두루 사용된다. 그 다음 Pattern의 정의와 적용을 설명하는데, Pattern의 정의 부분에서는 Pattern이 `RewritePattern` 클래스를 상속하여 정의되며, 생성자의 핵심 멤버로 Benefit, Root Operation Name 등이 있다고 언급한다. 여기서 Benefit은 주어진 Pattern을 매칭하는 데 드는 비용(이득)이고, Root Operation Name은 현재 Pattern이 매칭하는 Op의 이름이다. 이 이름은 지정하지 않을 수도 있지만, 이 경우에는 임의의 Op 타입을 매칭하기 위해 Pattern이 `MatchAnyOpTypeTag` 타입임을 명시적으로 선언해야 한다. 그리고 Pattern은 `match` & `rewrite` 함수나 `matchAndRewrite` 함수를 재정의(override)하여 특정 Pattern 매칭과 rewrite 기능을 구현할 수 있다. 또한 Pattern rewrite 과정에서의 제약 사항, Pattern의 재귀 적용 지원 방법, Pattern의 매칭과 rewrite 과정을 디버깅하는 방법도 함께 소개한다. 이어서 우리가 자주 사용하는 `replaceOp/replaceOpWithNewOp/eraseOp` 등을 포함한 `PatternRewriter` 클래스의 API 목록과, Pattern이 드라이버(driver)에 의해 어떻게 적용되는지 설명한다. 다음으로 MLIR 시스템에 흔히 등장하는 Pattern 드라이버들을 소개하는데, Op 변환을 위한 Dialect Conversion Driver, 규범화를 위한 Greedy Pattern Rewrite Driver 등이 있다. 이 절의 마지막에서는 Pattern Rewriter 적용 과정에서 디버깅하는 팁도 제공한다. `match`와 `rewrite` 과정을 시각화할 수 있어 꽤 편리한 기능이다.

본 글의 0x4절에서는 MLIR의 Op 규범화를 다룬다. 먼저 MLIR이라는 다단계 IR에서는 어떤 단계든 규범화 Pass를 사용할 수 있음을 언급하고, 항등(identity) Op 제거, 피연산자(operand) 수 줄이기, 상수 폴딩 등 몇 가지 규범화 예시를 소개한다. 이어서 문서에서는 규범화를 정의하는 두 가지 방법인 일반적인 Pattern Rewriter 방식과 fold 방식을 소개한다. 구체적으로는 Op의 ODS 정의에서 `hasCanonicalizeMethod`를 설정하고 소스 파일에서 `getCanonicalizationPatterns` 메서드의 구현을 제공하는 방법, 또는 Op에서 `hasFolder` 필드를 설정한 뒤 소스 파일에서 `fold` 메서드 구현을 제공하는 방법으로 처리할 수 있다. 자세한 내용은 0x4절과 공식 문서를 참고하기 바란다.

# 0x2. Generic DAG Rewriter Infrastructure Rationale

제목은 "범용 Dag rewrite 아키텍처의 기본 원리"로 번역할 수 있다. `https://mlir.llvm.org/docs/Rationale/RationaleGenericDAGRewriter/` 문서에 해당하며, MLIR의 범용 Dag-to-Dag rewrite 아키텍처 뒤에 있는 기본 원리를 주로 다룬다.

## 0x2.1 소개와 동기

컴파일러 IR의 목표는 다양한 추상화 단계에서 코드를 표현하는 것이며, 이는 표현력과 변환의 용이성 측면에서 서로 다른 트레이드오프를 만들어낸다. 그러나 코드를 표현하는 능력 자체만으로는 큰 의미가 없다. 이러한 변환을 실제로 구현할 수 있어야 한다.

컴파일러의 변환은 매우 다양하지만, 이 글에서는 MLIR의 목표에 매우 중요하고 반복적으로 등장하는 한 가지 변환을 주로 다룬다. 바로 일련의 Op로 구성된 Dag를 매칭하여 다른 Dag로 교체하는 변환이다. 이는 많은 학습용 컴파일러에서도 빼놓을 수 없는 부분이며, "identity(직결) 노드 제거"나 "x+0"을 "x"로 교체하는 것과 같은 최적화, 범용 규범화 프레임워크(예: LLVM의 instruction combiner), 그리고 컴파일러가 여러 중간 IR 위에서 최적화 알고리즘을 구현할 때 유용한 추상을 제공한다.

MLIR의 특별한 강점(그리고 LLVM, GCC, XLA, TensorFlow 같은 다른 컴파일러 인프라와의 주요 차이점) 중 하나는, 단일 컴파일러 IR을 사용해 여러 추상화 단계의 코드를 표현한다는 점이다. MLIR Operation은 "TensorFlow operation", "XLA HLO", affine 루프 중첩, LLVM IR 명령(전이적으로 X86, Lanai, PTX 등 target 특화 명령어 포함), 또는 MLIR operator 시스템이 합리적으로 표현 가능한 그 어떤 것이든 될 수 있다. MLIR이 이렇게 폭넓은 문제 영역을 다루고 있다는 점을 고려하면, graph-to-graph rewrite를 수행하는 단일 인프라는 다양한 분야의 도전 과제를 해결하는 데 도움이 될 수 있다.

MLIR처럼 SSA(Static Single Assignment) 기반의 IR은 Op의 operand와 "users"에 쉽게 접근할 수 있다. 따라서 이러한 graph-to-graph rewrite의 자연스러운 추상은 DAG Pattern 매칭이다. 즉, 클라이언트가 DAG tile pattern(여기서 tile은 DAG 부분 그래프를 정의하는 일련의 Op)을 정의하고, 각 Pattern은 결과 DAG와 그것을 생성하는 비용(반대로 표현하면 교체로 얻는 이득(benefit))을 포함한다. 범용 인프라는 이를 효율적으로 찾아 적용해 rewrite를 실행할 수 있다.

위에서 언급한 개념은 단순하지만 세부 사항은 미묘하다. 이 문서에서는 광범위한 문제를 해결할 수 있는 일련의 추상을 정의하고 탐구하며, MLIR이 시간이 지남에 따라 마주하게 될 다양한 종류의 문제에 적용될 수 있을 것으로 예상한다.

## 상수 폴딩(Constant Folding)

DAG-to-DAG Pattern 매칭의 단순화된, 그러나 흔한 사례가 바로 상수 폴딩이다. operand가 상수인 Op는 보통 결과 상수 값으로 폴딩될 수 있다.

MLIR의 Op는 `fold`를 오버라이드하여 이를 구현할 수 있는데, 이는 일반적인 DAG-to-DAG Pattern 매처에 비해 더 단순한 API를 제공하며, 범용 매처가 적용되지 않는 경우에도 사용할 수 있다. 예를 들어, DAG rewrite는 현재 함수의 임의 노드를 삭제할 수 있는데 이는 iterator를 무효화할 가능성이 있다. 반면 API로서의 상수 폴딩은 어떠한 노드도 삭제하지 않고, 단지 상수 값(목록)을 제공하며 클라이언트가 필요에 따라 자신의 자료구조를 갱신하도록 한다.

상수 폴딩에 대한 내용은 뒤의 0x4절 예시 설명을 참고하기 바란다. 이는 `https://mlir.llvm.org/docs/Canonicalization` 문서를 번역한 것이다.

## 관련 작업

거의 모든 기존 컴파일러가 이 문제를 여러 차례 해결해야 했음을 감안하면, 살펴볼 만한 관련 작업은 매우 많다. 한 가지 일관된 점은 이러한 시스템들이 모두 특정한, 종종 좁은 문제를 해결하는 것을 목표로 한다는 점이다. 반면 MLIR은 단일 인프라 안에서 이러한 문제 다수를 해결하고자 한다. 다음은 일부 관련 Pattern Rewriter 시스템과 그 작업의 장단점이다(MLIR에 존재하는 인프라와 가장 유사한 설계는 LLVM의 DAG-to-DAG 명령어 선택 알고리즘이다).

  * AST 단계 Pattern 매처: 텍스트에는 등가 변환을 통해 성능을 향상시키는 source-to-source 변환기들이 다수 존재한다(예: x*0을 0으로 변환). 큰 예로 GCC의 fold 함수가 있는데, AST에 대해 많은 최적화를 수행한다. Clang에도 표현식에 적용되는 단순한 상수 폴딩의 유사한 예시가 있지만(C++의 요구사항), AST에 대해 일반적인 최적화를 수행하지는 않는다. AST 옵티마이저의 주된 단점은 한 Op가 여러 용도로 쓰이는 경우(multiple uses)를 볼 수 없다는 점이다. 잘 알려져 있듯이 DAG Pattern 매칭은 트리 Pattern 매칭보다 강력하지만, 다른 한편으로 DAG Pattern 매칭은 중복 계산을 유발할 수 있다.
  * 두 번째는 다루지 않으니, 관심 있다면 공식 문서를 참고하기 바란다.
  * LLVM's DAG-to-DAG Instruction Selection Infrastructure: LLVM의 명령어 선택 서브시스템은 여러 해에 걸친 반복과 연구의 결과물이다. LLVM이 다수의 target codegen을 지원해야 하고, 현대적 명령어 집합(예: X86)을 위한 코드 생성기가 복잡하며, target 간 코드 재사용을 열렬히 추구해 왔기 때문이다. Eli Bendersky가 동작 방식에 대한 짧은 개요를 작성했고, LLVM 문서에서는 그 장단점을 포함하여 더 깊이 있게 설명한다. 다음과 같은 Pattern을 작성할 수 있다.


    
    
    def : Pat<(or GR64:$src, (not (add GR64:$src, 1))),  
              (BLCI64rr GR64:$src)>;  
    

이 예시는 X86 target 기술에서 "blci" 명령어에 대한 매처를 정의한다. 이 파일에는 다른 명령어들도 다수 들어 있다(`Pat<>` Pattern을 찾아 보자. 어셈블러/디스어셈블러 생성 로직 같은 컴파일러 세부 사항과 얽혀 있지 않기 때문이다).

## 작은 정리

MLIR은 다양한 Pattern 매칭과 graph rewrite 문제를 마주하며, 다단계에서 범용 코드 표현을 사용하는 주요 장점 중 하나는 단일 인프라에 투자하고 그것을 적극적으로 활용해 이런 작업을 처리할 수 있다는 점이다.

이후로는 Dag rewrite 메커니즘의 목표, 즉 어떤 문제들을 해결하는지, 어떤 매칭 전략을 사용하는지, 좋은 에러 메시지 등과 같은 내용도 일부 소개된다.

# 0x3. Pattern Rewriting : Generic DAG-to-DAG Rewriting

이 문서는 MLIR에 존재하는 Pattern Rewriter 인프라(범용 DAG to DAG 변환 프레임워크)의 설계와 API를 자세히 설명한다. 이 프레임워크는 MLIR 전반에 걸쳐 규범화, conversion, 일반 변환(transformation)에 폭넓게 사용된다.

## 소개

Pattern Rewriter 프레임워크는 크게 두 부분으로 나눌 수 있다. Pattern 정의와 Pattern 적용이다.

## Pattern 정의

Pattern은 `RewritePattern` 클래스를 상속하여 정의된다. 이 클래스는 MLIR 내 모든 Rewrite Pattern의 베이스 클래스를 나타내며, 다음 컴포넌트로 구성된다.

### Benefit

이는 주어진 Pattern을 적용함으로써 기대되는 이득이다. 이 이득은 Pattern을 구성할 때 정적이지만, Pattern 초기화 시 동적으로 계산할 수도 있다. 예를 들어 target 아키텍처와 같은 특정 영역의 정보로부터 이득을 얻어내는 것이 가능하다. 이러한 제약 덕분에 Pattern fusion을 수행하고 Pattern을 효율적인 상태 머신으로 컴파일할 수 있다. 또한 Thier, Ertl, Krall은 거의 모든 경우에 매칭 술어(predicate)에 대해 동적인 계산 비용이 필요하지 않음을 증명했다. 단순히 가능한 각 이득별로 동일 Pattern을 한 번씩 인스턴스화하고, 술어를 사용해 매칭을 보호하면 된다.

### Root Operation Name (선택)

이 Pattern이 매칭하는 root operation의 이름이다. 지정되어 있으면 해당 이름을 가진 Op들만 `match`와 `rewrite` 구현을 제공하면 된다. 지정되어 있지 않으면 임의의 operation 타입을 받을 수 있다. 가능한 한 root operation 이름을 제공하는 것이 좋다. 비용 모델 적용 시 Pattern 분석을 단순화할 수 있기 때문이다. 임의의 Op 타입을 매칭하려면 의도를 명확히 하기 위한 특별한 태그인 `MatchAnyOpTypeTag`를 제공해야 한다.

#### match와 rewrite 구현

이는 주어진 root operation에 대해 매칭하고 IR rewrite를 수행하는 코드 블록이다. `RewritePattern`은 별도의 `match`와 `rewrite` 메서드, 또는 결합된 `matchAndRewrite` 메서드로 이 구현을 지정할 수 있다. 결합된 `matchAndRewrite` 메서드를 사용할 때는 매칭이 성공하기 전까지 IR 변경(mutation)이 발생해서는 안 된다. 매칭과 rewrite 단계에서 사소하지 않게(non-trivially) 재계산 가능한 정보가 필요할 때는 결합된 `matchAndRewrite`가 유용하다. 아래 예시를 참고하기 바란다.
    
    
    class MyPattern : public RewritePattern {  
    public:  
      /// This overload constructs a pattern that only matches operations with the  
      /// root name of `MyOp`.  
      MyPattern(PatternBenefit benefit, MLIRContext *context)  
          : RewritePattern(MyOp::getOperationName(), benefit, context) {}  
      /// This overload constructs a pattern that matches any operation type.  
      MyPattern(PatternBenefit benefit)  
          : RewritePattern(benefit, MatchAnyOpTypeTag()) {}  
      
      /// In this section, the `match` and `rewrite` implementation is specified  
      /// using the separate hooks.  
      LogicalResult match(Operation *op) const override {  
        // The `match` method returns `success()` if the pattern is a match, failure  
        // otherwise.  
        // ...  
      }  
      void rewrite(Operation *op, PatternRewriter &rewriter) {  
        // The `rewrite` method performs mutations on the IR rooted at `op` using  
        // the provided rewriter. All mutations must go through the provided  
        // rewriter.  
      }  
      
      /// In this section, the `match` and `rewrite` implementation is specified  
      /// using a single hook.  
      LogicalResult matchAndRewrite(Operation *op, PatternRewriter &rewriter) {  
        // The `matchAndRewrite` method performs both the matching and the mutation.  
        // Note that the match must reach a successful point before IR mutation may  
        // take place.  
      }  
    };  
    

### 제약 사항

Pattern의 `match` 부분에는 다음 제약이 적용된다.

  * IR 변경(mutation)은 허용되지 않는다.



Pattern의 `rewrite` 부분에는 다음 제약이 적용된다.

  * 생성을 포함한 모든 IR 변경은 반드시 주어진 `PatternRewriter`를 통해 수행해야 한다. 이 클래스는 Pattern 내에서 발생할 수 있는 모든 변경을 처리하기 위한 hook을 제공한다. 예를 들어 operation을 삭제할 때 그 자체의 `erase` 메서드를 호출해서는 안 된다. operation을 삭제하려면 적절한 `PatternRewriter` hook(이 경우에는 `eraseOp`)을 사용해야 한다.
  * root operation은 다음 중 하나여야 한다. inplace 업데이트, 교체, 또는 삭제.



### 재귀 적용

재귀는 Pattern rewrite 컨텍스트에서 핵심 주제 중 하나이다. 한 Pattern은 보통 자기 자신의 결과에도 적용 가능한 경우가 많기 때문이다. 그러나 재귀는 Pattern 매칭 과정을 무한 루프에 빠뜨릴 수도 있다. Pattern rewrite 인프라는 보수적으로 어떤 Pattern도 재귀하지 않는다고 가정하고, 재귀가 감지되면 실패 신호를 보낸다. 만약 어떤 Pattern이 재귀를 지원한다면, Pattern 초기화 시 명시적으로 `setHasBoundedRewriteRecursion`을 호출하여 해당 Pattern의 재귀 적용이 발생할 수 있으며 안전하게 처리할 수 있음을 알려야 한다.

### Debug Names and Labels

디버깅을 돕기 위해 Pattern은 다음을 지정할 수 있다. 디버그 이름(`setDebugName`을 통해)은 특정 Pattern을 고유하게 식별하는 식별자에 대응해야 하며, 디버그 라벨 집합(`addDebugLabels`를 통해)은 Pattern 그룹을 고유하게 식별하는 식별자들에 대응한다. 다양한 유틸리티가 이 정보를 사용해 Pattern rewrite 디버깅을 돕는다. 예를 들어 디버그 로그에서 Pattern 필터링을 제공하는 식이다. 간단한 코드 예시는 다음과 같다.
    
    
    class MyPattern : public RewritePattern {  
    public:  
      /// Inherit constructors from RewritePattern.  
      using RewritePattern::RewritePattern;  
      
      void initialize() {  
        setDebugName("MyPattern");  
        addDebugLabels("MyRewritePass");  
      }  
      
      // ...  
    };  
      
    void populateMyPatterns(RewritePatternSet &patterns, MLIRContext *ctx) {  
      // Debug labels may also be attached to patterns during insertion. This allows  
      // for easily attaching common labels to groups of patterns.  
      patterns.addWithLabel<MyPattern, ...>("MyRewritePatterns", ctx);  
    }  
    

### 초기화

일부 Pattern 상태는 명시적인 초기화가 필요하다. 예를 들어 Pattern이 재귀 적용을 안전하게 처리할 수 있다면 `setHasBoundedRewriteRecursion`을 설정해 주어야 한다. 이러한 Pattern 상태는 Pattern 생성자에서 초기화할 수도 있고, `initialize` hook 유틸리티를 통해 초기화할 수도 있다. `initialize` hook을 사용하면 추가 Pattern 상태 초기화를 주입하기 위해 Pattern 생성자를 다시 정의할 필요가 없다. 예시는 다음과 같다.
    
    
    class MyPattern : public RewritePattern {  
    public:  
      /// Inherit the constructors from RewritePattern.  
      using RewritePattern::RewritePattern;  
      
      /// Initialize the pattern.  
      void initialize() {  
        /// Signal that this pattern safely handles recursive application.  
        setHasBoundedRewriteRecursion();  
      }  
      
      // ...  
    };  
    

### 생성

RewritePattern을 생성할 때는 정적 유틸리티 메서드 `RewritePattern::create<T>`를 사용해야 한다. 이 메서드는 Pattern이 올바르게 초기화되어 `RewritePatternSet`에 삽입할 준비가 되도록 보장한다.

## Pattern Rewriter

`PatternRewriter`는 Pattern이 Pattern 적용 드라이버와 통신할 수 있도록 해 주는 특수한 클래스이다. 위에서 설명한 대로, 생성을 포함한 모든 IR 변경은 반드시 `PatternRewriter` 클래스를 거쳐 수행되어야 한다. 이는 하부 Pattern 드라이버가 변경 발생 시 무효화될 수 있는 상태를 가질 수 있기 때문에 필수적이다. 자주 쓰이는 `PatternRewriter` API의 예시는 아래에 나와 있다. 사용 가능한 API의 최신 목록은 클래스 문서(`https://github.com/llvm/llvm-project/blob/main/mlir/include/mlir/IR/PatternMatch.h#L235`)를 참고하기 바란다.

  * Op 삭제: `eraseOp`



이 메서드는 결과(result)가 없거나 결과가 모두 사용되지 않는다고 알려진 Op를 지운다(IR에서 매달려 있는(dangling) Op).

  * `match` 실패 통지: `notifyMatchFailure`



이 메서드는 `matchAndRewrite` 안에서 Pattern이 매칭에 실패한 이유에 대한 진단 메시지를 제공할 수 있게 해 준다. 메시지가 사용자에게 어떻게 표시되는지는 특정 Pattern 드라이버에 따라 다르다.

  * Op 교체: `replaceOp/replaceOpWithNewOp`



이 메서드는 제공된 값들의 집합으로 Op의 결과를 대체하고 해당 Op를 삭제한다.

  * Op를 inplace로 갱신: `(start|cancel|finalize)RootUpdate`



이는 Pattern 안에서 operation의 attribute, 위치, operand, 후속자(successor)를 inplace로 갱신할 수 있는 트랜잭션 유사 API를 제공하는 메서드들의 집합이다. inplace 업데이트 트랜잭션은 `startRootUpdate`로 시작하며, `cancelRootUpdate`와 `finalizeRootUpdate`로 각각 취소하거나 종료할 수 있다. 또한 `start`와 `finalize`를 하나의 콜백으로 감싸 주는 wrapper인 `updateRootInPlace`도 제공된다.

  * OpBuilder API



`PatternRewriter`는 `OpBuilder` 클래스를 상속하므로, `OpBuilder`에 있는 모든 동일 기능을 제공한다. 여기에는 Op 생성과 다양한 유용한 attribute 및 type 생성 메서드들이 포함된다.

## Pattern Application

Pattern 집합을 정의한 후에는 이를 모아 특정 드라이버에 적용용으로 제공해야 한다. 드라이버는 다음과 같은 몇 가지 상위 수준의 부분으로 구성된다.

  * 입력 `RewritePatternSet`. 드라이버 입력 Pattern은 `RewritePatternSet` 형태로 제공된다. 이 클래스는 Pattern 목록을 만드는 단순화된 API를 제공한다.
  * 드라이버 특화 `PatternRewriter`. 드라이버 상태가 Pattern Rewriter 내 IR 변경에 의해 무효화되지 않도록, 드라이버는 필요한 hook을 오버라이드한 `PatternRewriter` 인스턴스를 제공해야 한다. 드라이버가 특정 변경에 hook을 걸 필요가 없다면, 그 변경을 직접 수행하는 기본 구현이 제공된다.
  * Pattern Application과 Cost Model. 각 드라이버는 자신만의 Op 방문 순서와 Pattern 비용 모델을 정의할 책임이 있지만, 최종 적용은 `PatternApplicator` 클래스를 통해 이루어진다. 이 클래스는 `RewritePatternSet`을 입력으로 받아 제공된 비용 모델에 따라 Pattern을 변환한다. 비용 모델은 드라이버 특화 정보들을 사용해 주어진 Pattern의 최종 비용을 계산한다. 비용 모델 계산이 끝나면, 드라이버는 `PatternApplicator::matchAndRewrite`를 사용해 Pattern을 Op와 매칭하기 시작할 수 있다.


    
    
    class MyPattern : public RewritePattern {  
    public:  
      MyPattern(PatternBenefit benefit, MLIRContext *context)  
          : RewritePattern(MyOp::getOperationName(), benefit, context) {}  
    };  
      
    /// Populate the pattern list.  
    void collectMyPatterns(RewritePatternSet &patterns, MLIRContext *ctx) {  
      patterns.add<MyPattern>(/*benefit=*/1, ctx);  
    }  
      
    /// Define a custom PatternRewriter for use by the driver.  
    class MyPatternRewriter : public PatternRewriter {  
    public:  
      MyPatternRewriter(MLIRContext *ctx) : PatternRewriter(ctx) {}  
      
      /// Override the necessary PatternRewriter hooks here.  
    };  
      
    /// Apply the custom driver to `op`.  
    void applyMyPatternDriver(Operation *op,  
                              const RewritePatternSet &patterns) {  
      // Initialize the custom PatternRewriter.  
      MyPatternRewriter rewriter(op->getContext());  
      
      // Create the applicator and apply our cost model.  
      PatternApplicator applicator(patterns);  
      applicator.applyCostModel([](const Pattern &pattern) {  
        // Apply a default cost model.  
        // Note: This is just for demonstration, if the default cost model is truly  
        //       desired `applicator.applyDefaultCostModel()` should be used  
        //       instead.  
        return pattern.getBenefit();  
      });  
      
      // Try to match and apply a pattern.  
      LogicalResult result = applicator.matchAndRewrite(op, rewriter);  
      if (failed(result)) {  
        // ... No patterns were applied.  
      }  
      // ... A pattern was successfully applied.  
    }  
    

## 흔한 Pattern 드라이버들

MLIR은 다양한 사용 사례를 지원하는 몇 가지 흔한 Pattern 드라이버를 제공한다.

### Dialect Conversion Driver

이 드라이버는 "legality" 개념을 사용하여 Dialect 간 및 Dialect 내부의 Op 변환을 수행하는 프레임워크를 제공한다. 이 프레임워크는 Pattern 기반의 Op rewrite Pattern 집합을 통해 비합법(illegal) Op를 제공된 conversion target이 지원하는 Op로 변환할 수 있게 한다. 또한 type conversion도 지원한다. 이 드라이버에 대한 자세한 내용은 다음에서 확인할 수 있다. `https://mlir.llvm.org/docs/DialectConversion/`.

### Greedy Pattern Rewrite Driver

이 드라이버는 제공된 Op들을 순회하며 지역적으로 가장 유리한 pattern을 탐욕적으로(greedy) 적용한다. pattern의 이득은 전적으로 pattern에 지정된 `benefit`과 pattern 목록 내에서의 상대적인 순서(두 pattern의 지역 이득이 같을 때)에 의해 결정된다. pattern은 fixed point에 도달할 때까지 Op에 반복적으로 적용되며, 그 시점에 드라이버가 종료된다. 이 드라이버는 다음의 방식으로 사용할 수 있다: `applyPatternsAndFoldGreedily`와 `applyOpPatternsAndFold`. 후자는 제공된 operation에만 Pattern을 적용하며 IR을 순회하지 않는다.

이 드라이버는 설정이 가능하며 두 가지 모드를 지원한다. 1) "top-down" 순회를 선택할 수 있다. Op를 순회하며 작업 목록(worklist)을 만들고, region tree에 대해 pre-order로 정렬한다. 일반적으로 컴파일 시간 측면에서 더 효율적이다. 2) 기본은 "bottom-up" 순회로, region tree의 post-order 순회를 사용해 초기 worklist를 만든다. 이는 모호한 pattern 집합 안에서 더 큰 Pattern과 매칭될 수 있다.

이런 종류의 드라이버는 MLIR의 규범화 Pass에서만 사용된다. 이는 본 글의 0x4절에서 다룰 내용이다.

### Debugging

Greedy Pattern Rewrite 드라이버의 실행을 디버깅하려면 `-debug-only=greedy-rewriter`를 사용할 수 있다. 이 명령행 플래그는 Greedy Pattern Rewriter에 대해서만 LLVM의 디버그 로깅 인프라를 활성화한다. 출력은 트리 구조로 포맷되어 Pattern 적용 과정의 구조를 반영한다. 이 출력에는 Rewriter가 처리하는 모든 Op, Op가 어떻게 처리되었는지, 어떤 Pattern이 적용되었는지, 그리고 실패한 이유가 포함된다.
    
    
    //===-------------------------------------------===//  
    Processing operation : 'cf.cond_br'(0x60f000001120) {  
      "cf.cond_br"(%arg0)[^bb2, ^bb2] {operand_segment_sizes = dense<[1, 0, 0]> : vector<3xi32>} : (i1) -> ()  
      
      * Pattern SimplifyConstCondBranchPred : 'cf.cond_br -> ()' {  
      } -> failure : pattern failed to match  
      
      * Pattern SimplifyCondBranchIdenticalSuccessors : 'cf.cond_br -> ()' {  
        ** Insert  : 'cf.br'(0x60b000003690)  
        ** Replace : 'cf.cond_br'(0x60f000001120)  
      } -> success : pattern applied successfully  
    } -> success : pattern matched  
    //===-------------------------------------------===//  
    

이 출력은 `cf.cond_br` Op의 처리를 설명한다. 먼저 `SimplifyConstCondBranchPred`를 적용해 보지만 실패한다. 그 후 다른 Pattern (`SimplifyCondBranchIdenticalSuccessors`)이 적용되어 `cf.cond_br`을 매칭하고 `cf.br`로 교체한다.

## 디버깅

### Pattern Filtering

테스트 케이스 정의와 축소를 단순화하기 위해, `FrozenRewritePatternSet` 클래스는 Pattern 적용 드라이버에 어떤 Pattern을 제공할지 필터링하는 기능을 내장 지원한다. `FrozenRewritePatternSet` 구성 시 `disabledPatterns`와 `enabledPatterns` 목록을 제공해 필터링 동작을 지정한다. `disabledPatterns` 목록에는 Pattern 적용 동안 비활성화할 Pattern들의 디버그 이름이나 라벨, 즉 필터링해 제외할 Pattern들을 담아야 한다. `enabledPatterns` 목록에는 Pattern 적용 동안 활성화할 Pattern들의 디버그 이름이나 라벨을 담아야 하며, 이 조건을 만족하지 않는 Pattern은 필터링되어 제외된다. 참고로, `disabledPatterns` 목록으로 지정된 pattern은 `enabledPatterns` 목록의 조건과 일치하더라도 필터링되어 제외된다. 예시는 다음과 같다.
    
    
    void MyPass::initialize(MLIRContext *context) {  
      // No patterns are explicitly disabled.  
      SmallVector<std::string> disabledPatterns;  
      // Enable only patterns with a debug name or label of `MyRewritePatterns`.  
      SmallVector<std::string> enabledPatterns(1, "MyRewritePatterns");  
      
      RewritePatternSet rewritePatterns(context);  
      // ...  
      frozenPatterns = FrozenRewritePatternSet(rewritePatterns, disabledPatterns,  
                                               enabledPatterns);  
    }  
    

### Common Pass Utilities

rewrite pattern을 활용하는 pass는 서로 다른 passes/projects 등 사이를 오갈 때 디버깅 경험을 단순화하기 위해 공통의 옵션 및 토글 집합을 제공하는 것을 목표로 해야 한다. 이를 돕기 위해, MLIR은 사용자 정의 pass에 손쉽게 포함시킬 수 있는 공통 유틸리티 집합을 제공한다. 이는 `mlir/RewritePassUtil.td`에 정의되어 있다. 사용 예시는 아래와 같다.
    
    
    def MyRewritePass : Pass<"..."> {  
      let summary = "...";  
      let constructor = "createMyRewritePass()";  
      
      // Inherit the common pattern rewrite options from `RewritePassUtils`.  
      let options = RewritePassUtils.options;  
    }  
    

#### Rewrite Pass Options

이 절에서는 rewrite pattern 적용 동작을 제어하는 데 사용할 수 있는 공통 pass 옵션을 기록한다.

##### Pattern Filtering

두 가지 흔한 Pattern 필터링 옵션이 노출되는데, `disable-patterns`와 `enable-patterns`이며, 위 pattern filtering 절에서 설명한 `disabledPatterns`와 `enabledPatterns` 목록의 동작과 일치한다. 이 옵션들의 `tablegen` 정의 조각은 아래와 같다.
    
    
    ListOption<"disabledPatterns", "disable-patterns", "std::string",  
               "Labels of patterns that should be filtered out during application",  
               "llvm::cl::MiscFlags::CommaSeparated">,  
    ListOption<"enabledPatterns", "enable-patterns", "std::string",  
               "Labels of patterns that should be used during application, all "  
               "other patterns are filtered out",  
               "llvm::cl::MiscFlags::CommaSeparated">,  
    

이 옵션들은 pass 안에서 어떤 `FrozenRewritePatternSet`을 구성할 때 필터링 동작을 제공하는 데 사용할 수 있다.
    
    
    void MyRewritePass::initialize(MLIRContext *context) {  
      RewritePatternSet rewritePatterns(context);  
      // ...  
      
      // When constructing the `FrozenRewritePatternSet`, we provide the filter  
      // list options.  
      frozenPatterns = FrozenRewritePatternSet(rewritePatterns, disabledPatterns,  
                                               enabledPatterns);  
    }  
    

# 0x4. Operation Canonicalization (operation 규범화)

규범화는 컴파일러 IR 설계에서 중요한 부분이다. 신뢰할 수 있는 컴파일러 변환을 구현하고 코드의 좋고 나쁨을 추론하기 쉽게 만들어 주며, 특정 IR 단계의 목표에 대한 흥미로운 논의를 불러일으킨다. Dan Gohman은 이러한 문제들을 탐구한 글을 썼다. 이러한 개념에 익숙하지 않다면 한 번 읽어볼 가치가 있다. 글 주소는 다음과 같다: `https://sunfishcode.github.io/blog/2018/10/22/Canonicalization.html`.

대부분의 컴파일러에는 규범화 pass가 있고, 때로는 다양한 종류의 pass가 여러 개 있기도 하다(예: LLVM의 instcombine, dag combine 등). MLIR은 다단계 IR이기 때문에, 단일한 규범화 인프라를 제공하고 그것이 표현하는 다양한 IR에서 재사용할 수 있다. 이 절에서는 일반적인 전역 규범화 방식을 설명하고, 참고용으로 IR 특화 규칙을 캡처하는 일부 내용을 함께 제공한다.

## 일반 설계

MLIR에는 단일 규범화 pass가 있으며, 이 pass는 IR이 수렴할 때까지 탐욕적인 방식으로 규범화 변환을 반복 적용한다. 이러한 변환은 Op 자신에 의해 정의되며, 각 dialect가 자신의 Op와 규범화 집합을 함께 정의할 수 있게 한다. 규범화 Pattern을 고려할 때 유의할 점은 다음과 같다.

  * Pattern을 반복 적용하면 수렴해야 한다. 불안정하거나 순환하는 rewrite는 규범화 프로그램에서 무한 루프를 유발한다.
  * operand가 중복되는 경우, 일반적으로 값의 사용이 더 적은 Op로 규범화하는 것이 좋다. 일부 Pattern은 값에 user가 단 하나 있을 때만 매칭되기 때문이다. 예를 들어 "x + x"를 "x * 2"로 규범화하는 것은 보통 좋다. x의 사용 횟수가 하나 줄어들기 때문이다.
  * 가능하다면 Op를 완전히 제거하는 것이 항상 좋다. 예를 들어 알려진 항등(identity)을 폴딩하는 것이다(예: "x + 0 = x").



## 전역 적용 규칙

다음 변환들은 모든 단계의 IR에 적용된다.

  * 부수 효과(side effect)가 없고 쓸모 없는 Op 제거.
  * 상수 폴딩. 예를 들어 "(addi 1, 2)"를 "3"으로. 상수 폴딩 hook은 Op가 지정한다.
  * 상수 operand를 교환 가능한 operator의 오른쪽으로 이동. 예를 들어 "(addi 4, x)"를 "(addi x, 4)"로.
  * `constant-like` Op는 유일하게 만들어지며, 첫 번째 부모 barrier region의 진입 블록(entry block)으로 끌어올려진다. 이는 함수의 진입 블록처럼 위쪽과 격리된 region이거나, `DialectFoldInterface`의 `shouldMaterializeInto` 메서드를 통해 barrier로 표시된 진입 블록을 의미한다.



## Canonicalizations 정의

규범화를 정의하는 데에는 두 가지 메커니즘이 있다. 일반적인 RewritePatterns와 fold 메서드이다.

### RewritePattern으로 규범화하기

이 메커니즘은 규범화를 RewritePatterns 집합으로 제공하거나, C++로 강제 정의하거나, Declarative Rewrite Rule(DRR)로 선언적으로 작성할 수 있게 한다. Pattern Rewriter 인프라는 다양한 종류의 규범화를 표현할 수 있다. 이러한 변환은 곱셈을 시프트로 교체하는 것처럼 단순할 수도 있고, 조건 분기를 무조건 분기로 교체하는 것까지도 가능하다.

ODS에서는 Op에 `hasCanonicalizer` 비트나 `hasCanonicalizeMethod` 비트를 설정해 `getCanonicalizationPatterns` 메서드를 생성할 수 있다.
    
    
    def MyOp : ... {  
      // I want to define a fully general set of patterns for this op.  
      let hasCanonicalizer = 1;  
    }  
      
    def OtherOp : ... {  
      // A single "matchAndRewrite" style RewritePattern implemented as a method  
      // is good enough for me.  
      let hasCanonicalizeMethod = 1;  
    }  
    

그런 다음 소스 파일에서 규범화 Pattern을 제공할 수 있다.
    
    
    void MyOp::getCanonicalizationPatterns(RewritePatternSet &patterns,  
                                           MLIRContext *context) {  
      patterns.add<...>(...);  
    }  
      
    LogicalResult OtherOp::canonicalize(OtherOp op, PatternRewriter &rewriter) {  
      // patterns and rewrites go here.  
      return failure();  
    }  
      
    

### `fold` 메서드로 규범화하기

`fold` 메커니즘은 의도적으로 제한되어 있지만 강력한 메커니즘이다. 컴파일러 전반의 여러 곳에서 규범화를 적용할 수 있게 해 준다. 예를 들어 규범화 pass 외에도, `fold`는 Dialect Conversion 인프라에서 합법화(legalization) 메커니즘으로 사용되며, 어디서든 `OpBuilder`가 있으면 `OpBuilder::createOrFold`를 통해 직접 호출할 수 있다.

`fold`의 제약은 새로운 Op를 만들 수 없으며, root Op만 교체할 수 있다는 것이다(삭제는 불가). 또한 Op의 inplace 갱신이 가능하며, Op를 대체할 기존 값(또는 attribute) 집합을 반환할 수도 있다. 이는 `fold` 메서드가 진정한 "in-place" 변환임을 보장하며, Pattern Rewriter 없이도 호출할 수 있게 한다.

ODS에서는 Op에 `hasFolder` 비트를 설정해 `fold` 메서드의 선언을 생성할 수 있다. 이 메서드는 Op의 구조에 따라 형태가 달라진다.
    
    
    def MyOp : ... {  
      let hasFolder = 1;  
    }  
    

Op의 결과(result)가 단 하나뿐이라면, 다음이 생성된다.
    
    
    /// Implementations of this hook can only perform the following changes to the  
    /// operation:  
    ///  
    ///  1. They can leave the operation alone and without changing the IR, and  
    ///     return nullptr.  
    ///  2. They can mutate the operation in place, without changing anything else  
    ///     in the IR. In this case, return the operation itself.  
    ///  3. They can return an existing value or attribute that can be used instead  
    ///     of the operation. The caller will remove the operation and use that  
    ///     result instead.  
    ///  
    OpFoldResult MyOp::fold(ArrayRef<Attribute> operands) {  
      ...  
    }  
    

그렇지 않으면 다음이 생성된다.
    
    
    /// Implementations of this hook can only perform the following changes to the  
    /// operation:  
    ///  
    ///  1. They can leave the operation alone and without changing the IR, and  
    ///     return failure.  
    ///  2. They can mutate the operation in place, without changing anything else  
    ///     in the IR. In this case, return success.  
    ///  3. They can return a list of existing values or attribute that can be used  
    ///     instead of the operation. In this case, fill in the results list and  
    ///     return success. The results list must correspond 1-1 with the results of  
    ///     the operation, partial folding is not supported. The caller will remove  
    ///     the operation and use those results instead.  
    ///  
    /// Note that this mechanism cannot be used to remove 0-result operations.  
    LogicalResult MyOp::fold(ArrayRef<Attribute> operands,  
                             SmallVectorImpl<OpFoldResult> &results) {  
      ...  
    }  
    

위에서 각 메서드에는 `ArrayRef<Attribute>`가 제공되는데, 이는 각 operand의 상수 attribute 값에 대응한다. 이러한 operand는 `ConstantLike` trait을 구현한 operand들이다. 어떤 operand가 상수가 아니라면 null `Attribute` 값이 제공된다. 예를 들어, MyOp가 세 operand [a, b, c]를 가지지만 b만 상수라면, operand 형식은 [Attribute(), b-value, Attribute()]가 된다.

위에서는 `OpFoldResult`의 사용도 보여준다. 이 클래스는 op를 `fold`한 가능한 결과를 표현한다: SSA `Value` 또는 (상수 결과의 경우) `Attribute`. SSA `Value`가 제공된다면, 그 값은 기존에 존재하는 값에 대응해야 한다. `fold` 메서드는 새로운 `Value`를 생성하는 것을 허용하지 않는다. 반환된 `Attribute` 값의 형식에는 특별한 제약이 없지만, 특정 `Type`의 `Attribute` 표현이 일관성을 유지하는 것이 중요하다.

Op의 `fold` hook이 성공하지 못한 경우, Dialect는 `DialectFoldInterface`를 구현하고 `fold` hook을 오버라이드함으로써 fallback을 제공할 수 있다.

### attribute로부터 상수 생성

`fold` 메서드가 결과로 `Attribute`를 반환할 때, 이는 결과가 "상수"임을 의미한다. `Attribute`는 값의 상수 표현이다. `fold` 메서드의 사용자, 예를 들어 canonicalizer pass는 이러한 `Attribute`를 받아 IR 안에서 그것을 표현하는 상수 Op를 구체화(materialize)한다. 이러한 구체화를 활성화하려면 Op의 Dialect가 `materializeConstant` hook을 구현해야 한다. 이 hook은 보통 `fold`가 반환하는 `Attribute` 값을 받아, 해당 값을 표현하는 "constant-like"한 Op를 생성한다.

ODS에서는 Dialect가 `hasConstantMaterializer` 비트를 설정해 `materializeConstant` 메서드의 선언을 생성할 수 있다.
    
    
    def MyDialect_Dialect : ... {  
      let hasConstantMaterializer = 1;  
    }  
    

그런 다음 소스 파일에서 상수를 구체화할 수 있다.
    
    
    /// Hook to materialize a single constant operation from a given attribute value  
    /// with the desired resultant type. This method should use the provided builder  
    /// to create the operation without changing the insertion position. The  
    /// generated operation is expected to be constant-like. On success, this hook  
    /// should return the value generated to represent the constant value.  
    /// Otherwise, it should return nullptr on failure.  
    Operation *MyDialect::materializeConstant(OpBuilder &builder, Attribute value,  
                                              Type type, Location loc) {  
      ...  
    }

# Chapter3: MLIR에서의 표현식 변형(Pass 작성 방법)

Chapter2에서 우리는 이미 초급 수준의 합법적인 MLIR 표현식을 생성하였지만, MLIR 표현식은 일반적으로 추가로 처리하고 단순화할 수 있으며, 이는 TVM의 Pass가 Relay IR을 최적화하는 것에 비유할 수 있습니다. 여기서는 초급 MLIR 표현식을 변형하는 방법을 살펴보겠습니다. MLIR에서는 표현식 매칭과 재작성을 기반으로 MLIR 표현식 변형을 수행합니다. 이 튜토리얼에서는 C++ 템플릿 매칭과 재작성 사용 방법, 그리고 DRR 프레임워크(`https://mlir.llvm.org/docs/DeclarativeRewrites/`)를 기반으로 표현식 재작성 규칙을 정의한 뒤 ODS 프레임워크를 사용하여 코드를 자동 생성하는 방법을 각각 소개합니다.

## C++ 패턴 매칭과 재작성 방법으로 transpose(Transpose) 연산 최적화하기

여기서의 목표는 서로 상쇄되는 효과를 가진 두 개의 transpose 시퀀스를 제거하는 것입니다: `transpose(transpose(X)) -> X`, 즉 동일한 입력에 연속으로 Transpose 연산을 수행하면 분명히 중복 연산이 존재합니다. 해당 연산에 대응하는 소스 코드는 다음과 같습니다 (`mlir/test/Examples/Toy/Ch3/transpose_transpose.toy` 안에 있음):
    
    
    def transpose_transpose(x) {  
      return transpose(transpose(x));  
    }  
    

만약 어떤 최적화 Pass도 사용하지 않는다면, 이 Toy 소스 프로그램이 생성하는 MLIR 표현식이 어떤 모습인지 살펴봅시다. 다음 명령으로 MLIR을 생성합니다: `./toyc-ch3 ../../mlir/test/Examples/Toy/Ch3/transpose_transpose.toy -emit=mlir`.
    
    
    func @transpose_transpose(%arg0: tensor<*xf64>) -> tensor<*xf64> {  
        %0 = toy.transpose(%arg0 : tensor<*xf64>) to tensor<*xf64>  
        %1 = toy.transpose(%0 : tensor<*xf64>) to tensor<*xf64>  
        toy.return %1 : tensor<*xf64>  
      }  
    

생성된 MLIR 표현식에서 `x`에 대해 두 번 실제 transpose 연산을 수행하고, 두 번의 transpose 후의 Tensor를 반환하는 것을 볼 수 있습니다. 그러나 실제로 이 두 번의 transpose는 불필요한데, 왜냐하면 출력 결과가 사실 입력된 `x` 그 자체이기 때문입니다. 따라서 이러한 상황을 최적화하기 위해, 먼저 C++ 방식으로 표현식 매칭 및 재작성 코드를 작성합니다 (`mlir/examples/toy/Ch3/mlir/ToyCombine.cpp` 안에 있음):
    
    
    /// This is an example of a c++ rewrite pattern for the TransposeOp. It  
    /// optimizes the following scenario: transpose(transpose(x)) -> x  
    struct SimplifyRedundantTranspose : public mlir::OpRewritePattern<TransposeOp> {  
      /// We register this pattern to match every toy.transpose in the IR.  
      /// The "benefit" is used by the framework to order the patterns and process  
      /// them in order of profitability.  
      SimplifyRedundantTranspose(mlir::MLIRContext *context)  
          : OpRewritePattern<TransposeOp>(context, /*benefit=*/1) {}  
      
      /// This method attempts to match a pattern and rewrite it. The rewriter  
      /// argument is the orchestrator of the sequence of rewrites. The pattern is  
      /// expected to interact with it to perform any changes to the IR from here.  
      mlir::LogicalResult  
      matchAndRewrite(TransposeOp op,  
                      mlir::PatternRewriter &rewriter) const override {  
        // Look through the input of the current transpose.  
        mlir::Value transposeInput = op.getOperand();  
        TransposeOp transposeInputOp = transposeInput.getDefiningOp<TransposeOp>();  
      
        // Input defined by another transpose? If not, no match.  
        if (!transposeInputOp)  
          return failure();  
      
        // Otherwise, we have a redundant transpose. Use the rewriter.  
        rewriter.replaceOp(op, {transposeInputOp.getOperand()});  
        return success();  
      }  
    };  
    

`matchAndRewrite` 함수에서 먼저 현재 operation의 operand를 가져온 뒤, 현재 위치의 operand에 대응하는 operation이 transpose인지를 판단하고, 그렇다면 표현식을 안쪽 transpose 연산의 operand로 재작성하며, 그렇지 않으면 최적화하지 않고 현재 상태를 유지하는 것을 볼 수 있습니다.

다음으로, 방금 만든 매칭 및 재작성 패턴을 정규화 프레임워크(Canonicalization Framework)에 등록하여 프레임워크가 호출할 수 있도록 해야 합니다. canonicalize에 대한 더 많은 소개는 `https://mlir.llvm.org/docs/Canonicalization/` 를 참조하세요. 등록 코드는 다음과 같습니다 (코드는 여전히 `mlir/examples/toy/Ch3/mlir/ToyCombine.cpp` 에 있습니다):
    
    
    /// Register our patterns as "canonicalization" patterns on the TransposeOp so  
    /// that they can be picked up by the Canonicalization framework.  
    void TransposeOp::getCanonicalizationPatterns(RewritePatternSet &results,  
                                                  MLIRContext *context) {  
      results.add<SimplifyRedundantTranspose>(context);  
    }  
      
    

표현식 재작성 규칙을 정규화 프레임워크에 추가한 후에는, operator를 정의하는 `td` 파일을 수정하여 정규화 프레임워크를 활성화하고, 동시에 operator 정의에 "부작용 없음"(`NoSideEffect`)이라는 새로운 trait를 추가해야 합니다. 현재 Transpose 연산의 정의는 다음과 같습니다:
    
    
    def TransposeOp : Toy_Op<"transpose", [NoSideEffect]> {  
      let summary = "transpose operation";  
      
      let arguments = (ins F64Tensor:$input);  
      let results = (outs F64Tensor);  
      
      let assemblyFormat = [{  
        `(` $input `:` type($input) `)` attr-dict `to` type(results)  
      }];  
      
      // Enable registering canonicalization patterns with this operation.  
      let hasCanonicalizer = 1;  
      
      // Allow building a TransposeOp with from the input operand.  
      let builders = [  
        OpBuilder<(ins "Value":$input)>  
      ];  
      
      // Invoke a static verify method to verify this transpose operation.  
      let verifier = [{ return ::verify(*this); }];  
    }  
    

마지막으로, 메인 프로그램에서 정규화 프레임워크 기반의 최적화를 실행 흐름에 추가해야 합니다. 이 부분 코드는 `mlir/examples/toy/Ch3/toyc.cpp` 의 `dumpMLIR` 함수 안에 있습니다. 아래 그림의 빨간 박스 부분과 같습니다:

![이미지](images/img_01.png)MLIR을 lowering할 때 최적화 Pass 활성화

여기까지 우리는 C++ 기반의 MLIR 표현식 매칭 및 재작성을 완료했습니다. 다음 명령을 통해 위 transpose 표현식 재작성 후 생성된 MLIR 표현식에서 transpose가 제거되었는지 확인할 수 있습니다. 명령은 다음과 같습니다: `./toyc-ch3 ../../mlir/test/Examples/Toy/Ch3/transpose_transpose.toy -emit=mlir -opt`. 결과는 다음과 같습니다:
    
    
    func @transpose_transpose(%arg0: tensor<*xf64>) -> tensor<*xf64> {  
        toy.return %arg0 : tensor<*xf64>  
      }  
    

최적화 후의 MLIR 표현식에서 transpose 연산이 제거되어 최적화 효과가 달성된 것을 볼 수 있습니다.

## DRR로 tensor 변형(Reshape) 연산 최적화하기

MLIR은 또 다른 표현식 재작성 방법을 제공합니다. DRR 규칙 기반으로 표현식 매칭 및 재작성 함수를 자동 생성하는 방식이며, 코드 생성 부분은 여전히 ODS 프레임워크를 기반으로 구현됩니다. DRR(Declarative, Rule-based Pattern-match and Rewrite)는 선언적이고 규칙 기반의 패턴 매칭 및 재작성 방법입니다. DAG 기반의 선언적 재작성기로, 테이블 기반의 패턴 매칭 및 재작성 규칙 문법을 제공합니다.

여기서는 MLIR 표현식에서 중복된 tensor reshape 연산을 제거하는 것을 예로 들겠습니다. 대응하는 Toy 소스 파일은 다음과 같습니다 (`mlir/test/Examples/Toy/Ch3/trivial_reshape.toy` 안에 있음):
    
    
    def main() {  
      var a<2,1> = [1, 2];  
      var b<2,1> = a;  
      var c<2,1> = b;  
      print(c);  
    }  
    

다음 명령으로 먼저 대응하는 MLIR 표현식을 생성해 봅시다: `./toyc-ch3 ../../mlir/test/Examples/Toy/Ch3/trivial_reshape.toy -emit=mlir`
    
    
    module  {  
      func @main() {  
        %0 = toy.constant dense<[1.000000e+00, 2.000000e+00]> : tensor<2xf64>  
        %1 = toy.reshape(%0 : tensor<2xf64>) to tensor<2x1xf64>  
        %2 = toy.reshape(%1 : tensor<2x1xf64>) to tensor<2x1xf64>  
        %3 = toy.reshape(%2 : tensor<2x1xf64>) to tensor<2x1xf64>  
        toy.print %3 : tensor<2x1xf64>  
        toy.return  
      }  
    }  
    

분명히 `a`, `b`, `c`의 shape과 값은 모두 동일하므로 이러한 reshape 연산은 중복입니다. 다음으로 DRR 프레임워크를 기반으로 표현식 매칭 및 재작성 규칙을 정의합니다. 여기서는 몇 가지 경우를 고려해야 합니다 (이 부분의 코드 구현은 모두 `mlir/examples/toy/Ch3/mlir/ToyCombine.td` 에 있습니다).

  * `Reshape(Reshape(x)) = Reshape(x)`로 인해 발생하는 중복 코드 해결하기.


    
    
    // Reshape(Reshape(x)) = Reshape(x)  
    def ReshapeReshapeOptPattern : Pat<(ReshapeOp(ReshapeOp $arg)),  
                                       (ReshapeOp $arg)>;  
    

즉, `ReshapeOp(ReshapeOp $arg)` 를 `ReshapeOp $arg` 로 대체합니다. 동일한 tensor 변형 연산이 여러 번 수행되더라도 한 번만 실행하면 됩니다.

  * reshape의 인자와 결과의 type이 동일할 때, 이는 해당 reshape 연산이 무용함을 의미하므로 입력 인자를 직접 반환하면 됩니다. 즉 `Reshape(x) = x` 입니다.


    
    
    // Reshape(x) = x, where input and output shapes are identical  
    def TypesAreIdentical : Constraint<CPred<"$0.getType() == $1.getType()">>;  
    def RedundantReshapeOptPattern : Pat<  
      (ReshapeOp:$res $arg), (replaceWithValue $arg),  
      [(TypesAreIdentical $res, $arg)]>;  
      
    

즉, `0.getType()` 과 `1.getType()` 이 같을 때가 곧 중복이며, operand `$arg` 로 대체합니다.

다음으로 ODS 프레임워크와 정의된 `ToyCombine.td` 파일을 사용하여 코드 파일 `ToyCombine.inc` 를 자동 생성할 수 있습니다. 다음 명령을 사용합니다:
    
    
    $   cd llvm-project/build  
    $   ./bin/mlir-tblgen --gen-rewriters ${mlir_src_root}/examples/toy/Ch3/mlir/ToyCombine.td -I ${mlir_src_root}/include/  
    

물론 프로젝트를 빌드할 때 이 생성 과정을 cmakelists.txt 에 구성할 수도 있습니다: `mlir/examples/toy/Ch3/CMakeLists.txt`. 다음과 같이:
    
    
    set(LLVM_TARGET_DEFINITIONS mlir/ToyCombine.td)  
    mlir_tablegen(ToyCombine.inc -gen-rewriters)  
    add_public_tablegen_target(ToyCh3CombineIncGen)  
    

마지막으로 `./toyc-ch3 ../../mlir/test/Examples/Toy/Ch3/trivial_reshape.toy -emit=mlir -opt` 를 실행하면 이 Pass들로 최적화된 MLIR 표현식을 생성할 수 있습니다:
    
    
    module  {  
      func @main() {  
        %0 = toy.constant dense<[[1.000000e+00], [2.000000e+00]]> : tensor<2x1xf64>  
        toy.print %0 : tensor<2x1xf64>  
        toy.return  
      }  
    }  
    

# Chapter4: 일반화된 표현식 변환 구현

Chapter3에서 우리는 MLIR에서 표현식을 재작성하는 방법을 배웠지만, 위 방법에는 매우 명백한 문제가 있습니다. 우리가 Toy 언어를 위해 구현한 Pass는 다른 Dialect 추상화에서는 재사용할 수 없는데, 이는 단지 Toy 언어의 일부 Operation에 특화된 연산이기 때문입니다. 만약 각 Dialect마다 각 변환을 구현해야 한다면 대량의 중복 코드가 발생할 것입니다. 그래서 이번 절에서는 두 가지 예시를 들어 MLIR에서 일반화된 표현식을 어떻게 구현하는지 설명합니다.

이 글에서는 다음 예시를 사용하여 소개합니다 (`mlir/test/Examples/Toy/Ch5/codegen.toy` 안에 있음):
    
    
    def multiply_transpose(a, b) {  
      return transpose(a) * transpose(b);  
    }  
      
    def main() {  
      var a<2, 3> = [[1, 2, 3], [4, 5, 6]];  
      var b<2, 3> = [1, 2, 3, 4, 5, 6];  
      var c = multiply_transpose(a, b);  
      var d = multiply_transpose(b, a);  
      print(d);  
    }  
    

먼저 이에 대응하는 MLIR 표현식을 살펴봅시다. `./toyc-ch4 ../../mlir/test/Examples/Toy/Ch4/codegen.toy -emit=mlir`:
    
    
    module  {  
      func private @multiply_transpose(%arg0: tensor<*xf64>, %arg1: tensor<*xf64>) -> tensor<*xf64> {  
        %0 = toy.transpose(%arg0 : tensor<*xf64>) to tensor<*xf64>  
        %1 = toy.transpose(%arg1 : tensor<*xf64>) to tensor<*xf64>  
        %2 = toy.mul %0, %1 : tensor<*xf64>  
        toy.return %2 : tensor<*xf64>  
      }  
      func @main() {  
        %0 = toy.constant dense<[[1.000000e+00, 2.000000e+00, 3.000000e+00], [4.000000e+00, 5.000000e+00, 6.000000e+00]]> : tensor<2x3xf64>  
        %1 = toy.reshape(%0 : tensor<2x3xf64>) to tensor<2x3xf64>  
        %2 = toy.constant dense<[1.000000e+00, 2.000000e+00, 3.000000e+00, 4.000000e+00, 5.000000e+00, 6.000000e+00]> : tensor<6xf64>  
        %3 = toy.reshape(%2 : tensor<6xf64>) to tensor<2x3xf64>  
        %4 = toy.generic_call @multiply_transpose(%1, %3) : (tensor<2x3xf64>, tensor<2x3xf64>) -> tensor<*xf64>  
        %5 = toy.generic_call @multiply_transpose(%3, %1) : (tensor<2x3xf64>, tensor<2x3xf64>) -> tensor<*xf64>  
        toy.print %5 : tensor<*xf64>  
        toy.return  
      }  
    }  
    

이는 최적화 전의 MLIR 표현식이며, Tensor를 인스턴스화하기 전에는 Tensor의 shape를 알 수 없음을 볼 수 있습니다. 즉, 표현식의 `tensor<*xf64>` 입니다. 이는 후속 Pass, 예를 들어 Chapter3에서 정의한 shape 관련 Pass에 영향을 미쳐 최적화가 충분히 이루어지지 못하게 합니다. 그래서 우리는 Reshape 관련 Pass를 실행하기 전에 각 Tensor의 shape를 알 수 있기를 원합니다. 그래서 여기서는 Shape 추론 Pass의 구현을 소개합니다. 또한 함수 호출 오버헤드를 줄이기 위한 인라인 Pass도 소개합니다.

## 인라인 Pass

위 코드를 살펴보면 `multiply_transpose` 같은 작은 함수가 빈번히 호출되는 것을 볼 수 있는데, 이때 함수 호출 자체의 오버헤드를 무시할 수 없습니다. 따라서 여기서 인라인 Pass를 정의하여 `multiply_transpose` 라는 함수를 인라인 함수로 만들어 실행 효율을 높이고자 합니다.

### 1단계

MLIR은 인라인을 처리하기 위한 범용 인터페이스 `DialectInlinerInterface` 를 제공합니다. 이는 Dialect가 재작성할 수 있는 가상 훅(virtual hook) 집합을 포함합니다. 우리는 이 클래스를 기반으로 Toy Operation을 위한 인라인 인터페이스와 표현식 재작성 규칙을 정의해야 합니다. 코드 구현은 `mlir/examples/toy/Ch5/mlir/Dialect.cpp` 에 있습니다:
    
    
    /// This class defines the interface for handling inlining with Toy operations.  
    /// We simplify inherit from the base interface class and override  
    /// the necessary methods.  
    struct ToyInlinerInterface : public DialectInlinerInterface {  
      using DialectInlinerInterface::DialectInlinerInterface;  
      
      /// This hook checks to see if the given callable operation is legal to inline  
      /// into the given call. For Toy this hook can simply return true, as the Toy  
      /// Call operation is always inlinable.  
      bool isLegalToInline(Operation *call, Operation *callable,  
                           bool wouldBeCloned) const final {  
        return true;  
      }  
      
      /// This hook checks to see if the given operation is legal to inline into the  
      /// given region. For Toy this hook can simply return true, as all Toy  
      /// operations are inlinable.  
      bool isLegalToInline(Operation *, Region *, bool,  
                           BlockAndValueMapping &) const final {  
        return true;  
      }  
      
      /// This hook is called when a terminator operation has been inlined. The only  
      /// terminator that we have in the Toy dialect is the return  
      /// operation(toy.return). We handle the return by replacing the values  
      /// previously returned by the call operation with the operands of the  
      /// return.  
      void handleTerminator(Operation *op,  
                            ArrayRef<Value> valuesToRepl) const final {  
        // Only "toy.return" needs to be handled here.  
        auto returnOp = cast<ReturnOp>(op);  
      
        // Replace the values directly with the return operands.  
        assert(returnOp.getNumOperands() == valuesToRepl.size());  
        for (const auto &it : llvm::enumerate(returnOp.getOperands()))  
          valuesToRepl[it.index()].replaceAllUsesWith(it.value());  
      }  
    };  
    

이 코드 부분은 Toy Operation을 위한 인라인 인터페이스와 표현식 변형 규칙을 정의합니다. 두 개의 `isLegalToInline` 오버로드 함수는 두 개의 훅입니다. 첫 번째 훅은 주어진 호출 가능 operation `callable` 을 주어진 호출 `call` 에 인라인하는 것이 합법적인지, 인라인 가능한지를 확인하는 데 사용됩니다. 두 번째 훅은 주어진 operation을 주어진 region에 인라인하는 것이 합법적인지를 확인하는 데 사용됩니다. `handleTerminator` 함수는 단지 `toy.return` 을 처리하며, 반환 연산의 operand `it.index()` 를 직접 반환값 `it.value()` 로 대체합니다 (이 부분은 잘 이해하지 못했습니다 QAQ).

### 2단계

이어서, 위의 표현식 변형 규칙을 Toy Dialect 정의에 추가해야 합니다. 위치는 `mlir/examples/toy/Ch5/mlir/Dialect.cpp` 입니다.
    
    
    /// Dialect initialization, the instance will be owned by the context. This is  
    /// the point of registration of types and operations for the dialect.  
    void ToyDialect::initialize() {  
      addOperations<  
    #define GET_OP_LIST  
    #include "toy/Ops.cpp.inc"  
          >();  
      addInterfaces<ToyInlinerInterface>();  
    }  
    

여기의 `addInterfaces<ToyInlinerInterface>()` 가 바로 인라인 Pass를 등록하는 과정이며, `ToyInlinerInterface` 가 우리가 정의한 표현식 변형 규칙입니다.

### 3단계

다음으로, inliner가 IR의 `toy.generic_call` 이 함수 호출을 나타낸다는 것을 알게 해야 합니다. MLIR은 어떤 Operation을 호출로 표시할 수 있는 Operation 인터페이스 `CallOpInterface` 를 제공합니다. 위 동작을 추가하려면 Toy Dialect 정의 (`mlir/examples/toy/Ch5/include/toy/Ops.td`) 파일에 `include "mlir/Interfaces/CallInterfaces.td"` 라인을 추가해야 합니다.

그런 다음 Dialect 정의 부분에 새 Operation을 추가합니다. 코드는 다음과 같습니다:
    
    
    def GenericCallOp : Toy_Op<"generic_call",  
        [DeclareOpInterfaceMethods<CallOpInterface>]> {  
      let summary = "generic call operation";  
      let description = [{  
        Generic calls represent calls to a user defined function that needs to  
        be specialized for the shape of its arguments. The callee name is attached  
        as a symbol reference via an attribute. The arguments list must match the  
        arguments expected by the callee. For example:  
      
        ```mlir  
         %4 = toy.generic_call @my_func(%1, %3)  
               : (tensor<2x3xf64>, tensor<2x3xf64>) -> tensor<*xf64>  
        ```  
      
        This is only valid if a function named "my_func" exists and takes two  
        arguments.  
      }];  
      
      // The generic call operation takes a symbol reference attribute as the  
      // callee, and inputs for the call.  
      let arguments = (ins FlatSymbolRefAttr:$callee, Variadic<F64Tensor>:$inputs);  
      
      // The generic call operation returns a single value of TensorType.  
      let results = (outs F64Tensor);  
      
      // Specialize assembly printing and parsing using a declarative format.  
      let assemblyFormat = [{  
        $callee `(` $inputs `)` attr-dict `:` functional-type($inputs, results)  
      }];  
      
      // Add custom build methods for the generic call operation.  
      let builders = [  
        OpBuilder<(ins "StringRef":$callee, "ArrayRef<Value>":$arguments)>  
      ];  
    }  
    

설명: 우리는 `DeclareOpInterfaceMethods` 를 사용하여 `CallOpInterface` 의 선언에서 사용할 인터페이스 메서드를 선언했습니다. `DeclareOpInterfaceMethods` 라는 trait는 프로그램이 `generic_call` operation(원본 MLIR 표현식의 `toy.generic_call` 에 해당)을 인식하고, 해당 위치에서 인터페이스 함수를 호출하도록 한다는 의미입니다.

그런 다음 `mlir/examples/toy/Ch5/mlir/Dialect.cpp` 에서 `GenericCallOp` 의 기능을 구현했습니다. 코드는 다음과 같습니다:
    
    
    /// Return the callee of the generic call operation, this is required by the  
    /// call interface.  
    CallInterfaceCallable GenericCallOp::getCallableForCallee() {  
      return getAttrOfType<SymbolRefAttr>("callee");  
    }  
      
    /// Get the argument operands to the called function, this is required by the  
    /// call interface.  
    Operation::operand_range GenericCallOp::getArgOperands() { return inputs(); }  
    

위의 `GenericCallOp::getCallableForCallee() {...}` 는 일반화 호출 Operation의 호출 대상을 반환합니다. 그리고 `GenericCallOp::getArgOperands(){...}` 는 호출되는 함수의 인자 operand를 가져오는 데 사용됩니다.

### 4단계

다음으로 Dialect 정의에 cast 연산을 추가하고 호출 인터페이스를 설정해야 합니다. 왜 cast 연산을 추가해야 할까요? 이는 함수 호출 시 입력 tensor의 type은 확정되어 있지만, 함수 정의 시에는 입력 tensor의 type이 확정되지 않기 때문입니다(일반화 type, 이 점은 위의 원본 MLIR 표현식에서 알 수 있습니다). 따라서 호출 시 숨겨진 데이터 type 변환이 필요하며, 그렇지 않으면 인라인 연산을 수행할 수 없으므로 여기서 cast를 도입합니다. cast 연산은 확정된 데이터 type을 함수가 기대하는 데이터 type으로 변환할 수 있습니다. 다음으로 `mlir/examples/toy/Ch5/include/toy/Ops.td` 에 cast 연산을 추가합니다:
    
    
    def CastOp : Toy_Op<"cast", [  
         DeclareOpInterfaceMethods<CastOpInterface>,  
         DeclareOpInterfaceMethods<ShapeInferenceOpInterface>,  
         NoSideEffect,  
         SameOperandsAndResultShape  
      ]> {  
      let summary = "shape cast operation";  
      let description = [{  
        The "cast" operation converts a tensor from one type to an equivalent type  
        without changing any data elements. The source and destination types must  
        both be tensor types with the same element type. If both are ranked, then  
        shape is required to match. The operation is invalid if converting to a  
        mismatching constant dimension.  
      }];  
      
      let arguments = (ins F64Tensor:$input);  
      let results = (outs F64Tensor:$output);  
      
      let assemblyFormat = "$input attr-dict `:` type($input) `to` type($output)";  
    }  
    

우리는 `DeclareOpInterfaceMethods` 를 사용하여 `CallOpInterface` 의 선언에서 사용할 인터페이스 메서드를 선언했습니다. `DeclareOpInterfaceMethods` 라는 trait는 프로그램이 `cast` 연산을 인식한다는 의미입니다.

다음으로 cast op의 `areCastCompatible` 메서드도 재작성해야 합니다 (`mlir/examples/toy/Ch5/mlir/Dialect.cpp` 에 있음):
    
    
    /// Returns true if the given set of input and result types are compatible with  
    /// this cast operation. This is required by the `CastOpInterface` to verify  
    /// this operation and provide other additional utilities.  
    bool CastOp::areCastCompatible(TypeRange inputs, TypeRange outputs) {  
      if (inputs.size() != 1 || outputs.size() != 1)  
        return false;  
      // The inputs must be Tensors with the same element type.  
      TensorType input = inputs.front().dyn_cast<TensorType>();  
      TensorType output = outputs.front().dyn_cast<TensorType>();  
      if (!input || !output || input.getElementType() != output.getElementType())  
        return false;  
      // The shape is required to match if both types are ranked.  
      return !input.hasRank() || !output.hasRank() || input == output;  
    }  
    

이 메서드는 type 변환을 수행해야 하는지 판단하는 데 사용됩니다. inputs와 outputs의 type이 호환된다면 true를 반환하고, 그렇지 않으면 type 변환(cast)이 필요하므로 false를 반환합니다.

또한 ToyInlinerInterface 의 훅, 즉 `materializeCallConversion` 함수도 재작성해야 합니다:
    
    
    struct ToyInlinerInterface : public DialectInlinerInterface {  
      ....  
      /// Attempts to materialize a conversion for a type mismatch between a call  
      /// from this dialect, and a callable region. This method should generate an  
      /// operation that takes 'input' as the only operand, and produces a single  
      /// result of 'resultType'. If a conversion can not be generated, nullptr  
      /// should be returned.  
      Operation *materializeCallConversion(OpBuilder &builder, Value input,  
                                           Type resultType,  
                                           Location conversionLoc) const final {  
        return builder.create<CastOp>(conversionLoc, resultType, input);  
      }  
    };  
    

이 함수가 인라인 Pass의 진입점입니다.

### 5단계

인라인 Pass를 최적화 pipeline에 추가합니다. `mlir/examples/toy/Ch5/toyc.cpp` 에서:
    
    
    if (enableOpt) {  
        mlir::PassManager pm(&context);  
        // Apply any generic pass manager command line options and run the pipeline.  
        applyPassManagerCLOptions(pm);  
      
        // Inline all functions into main and then delete them.

    ...

`pm.addPass(mlir::createInlinerPass());` 라인을 거치면, 최적화 pipeline에 인라인 Pass가 추가됩니다.

인라인 최적화 Pass 후에 원본 MLIR 표현식이 어떻게 바뀌는지 살펴봅시다:
    
    
    func @main() {  
      %0 = "toy.constant"() {value = dense<[[1.000000e+00, 2.000000e+00, 3.000000e+00], [4.000000e+00, 5.000000e+00, 6.000000e+00]]> : tensor<2x3xf64>} : () -> tensor<2x3xf64>  
      %1 = "toy.constant"() {value = dense<[[1.000000e+00, 2.000000e+00, 3.000000e+00], [4.000000e+00, 5.000000e+00, 6.000000e+00]]> : tensor<2x3xf64>} : () -> tensor<2x3xf64>  
      %2 = "toy.cast"(%1) : (tensor<2x3xf64>) -> tensor<*xf64>  
      %3 = "toy.cast"(%0) : (tensor<2x3xf64>) -> tensor<*xf64>  
      %4 = "toy.transpose"(%2) : (tensor<*xf64>) -> tensor<*xf64>  
      %5 = "toy.transpose"(%3) : (tensor<*xf64>) -> tensor<*xf64>  
      %6 = "toy.mul"(%4, %5) : (tensor<*xf64>, tensor<*xf64>) -> tensor<*xf64>  
      toy.print %6 : tensor<*xf64>  
      toy.return  
    }  
    

이제 MLIR 표현식에는 main 함수만 있으며, 이전의 transpose 함수는 인라인되었고, `toy.cast` 가 구현하는 기능도 볼 수 있습니다.

## Shape 추론 Pass

위의 인라인 Pass는 확정된 type의 Tensor를 일반화 type의 Tensor로 변환하여 인라인 연산이 완료되도록 했습니다. 이어서, shape가 확정된 Tensor를 바탕으로 일반화 Tensor의 shape를 추론해야 합니다. 여기서는 ODS 프레임워크를 활용하여 일반화 Tensor의 shape를 추론하는 커스텀 Operation 인터페이스를 생성합니다. 전체 Shape 추론 과정도 inline과 마찬가지로 Pass로 추상화되어 MLIR 표현식에 작용하게 됩니다.

### 1단계: ODS 프레임워크로 Shape 추론 Operation 인터페이스 정의

코드 구현은 `mlir/examples/toy/Ch5/include/toy/ShapeInferenceInterface.td` 에 있습니다.
    
    
    def ShapeInferenceOpInterface : OpInterface<"ShapeInference"> {  
      let description = [{  
        Interface to access a registered method to infer the return types for an  
        operation that can be used during type inference.  
      }];  
      
      let methods = [  
        InterfaceMethod<"Infer and set the output shape for the current operation.",  
                        "void", "inferShapes">  
      ];  
    }  
    

`ShapeInferenceOpInterface` 인터페이스는 `OpInterface` 를 상속하며, 이 상속은 생성될 C++ 인터페이스 클래스에 부여할 이름 "ShapeInference" 를 템플릿 인자로 받습니다. `description` 필드는 Operation에 대한 간략한 설명을 제공하고, `methods` 필드는 Operation이 제공해야 할 인터페이스 메서드를 정의합니다.

### 2단계: 필요한 Toy Operation 정의에 trait 추가

Toy 언어의 Mul Operation을 예로 들면, 구현은 `mlir/examples/toy/Ch5/include/toy/Ops.td` 에 있습니다:
    
    
    def MulOp : Toy_Op<"mul",  
        [NoSideEffect, DeclareOpInterfaceMethods<ShapeInferenceOpInterface>]> {  
      let summary = "element-wise multiplication operation";  
      let description = [{  
        The "mul" operation performs element-wise multiplication between two  
        tensors. The shapes of the tensor operands are expected to match.  
      }];  
      
      let arguments = (ins F64Tensor:$lhs, F64Tensor:$rhs);  
      let results = (outs F64Tensor);  
      
      // Specify a parser and printer method.  
      let parser = [{ return ::parseBinaryOp(parser, result); }];  
      let printer = [{ return ::printBinaryOp(p, *this); }];  
      
      // Allow building a MulOp with from the two input operands.  
      let builders = [  
        OpBuilder<(ins "Value":$lhs, "Value":$rhs)>  
      ];  
    }  
      
    

위 코드에서 `DeclareOpInterfaceMethods<ShapeInferenceOpInterface>` 는 Mul Operation에 shape 추론 trait를 추가합니다. 인라인 Pass에서 `CallOpInterface` trait를 cast Operation에 추가했던 것과 유사합니다.

### 3단계: 해당 Operation의 shape 추론 함수 정의

shape 추론이 필요한 각 Operation은 그에 해당하는 `inferShapes()` 함수를 정의해야 합니다. 예를 들어 Mul Operation의 경우 결과의 shape는 입력의 shape와 동일합니다(elementwise 연산이기 때문). 코드 구현은 `mlir/examples/toy/Ch5/mlir/Dialect.cpp` 에 있습니다:
    
    
    /// Infer the output shape of the MulOp, this is required by the shape inference  
    /// interface.  
    void MulOp::inferShapes() { getResult().setType(getOperand(0).getType()); }  
    

### 4단계: shape 추론 Pass 구현

이 단계는 shape 추론 Pass의 구체적인 구현을 소개합니다. 앞의 몇 단계는 이 단계의 전제 조건입니다. 이 단계에서는 Shape 추론 알고리즘을 구현하기 위한 shape 추론 Pass 클래스를 정의하고, 이 Pass 클래스를 기반으로 Shape 추론 Pass를 생성합니다. 코드 구현은 `mlir/examples/toy/Ch5/mlir/ShapeInferencePass.cpp` 에 있습니다.
    
    
    class ShapeInferencePass  
        : public mlir::PassWrapper<ShapeInferencePass, FunctionPass> {  
    public:  
      void runOnFunction() override {  
        auto f = getFunction();  
      
        // Populate the worklist with the operations that need shape inference:  
        // these are operations that return a dynamic shape.  
        llvm::SmallPtrSet<mlir::Operation *, 16> opWorklist;  
        f.walk([&](mlir::Operation *op) {  
          if (returnsDynamicShape(op))  
            opWorklist.insert(op);  
        });  
      
        // Iterate on the operations in the worklist until all operations have been  
        // inferred or no change happened (fix point).  
        while (!opWorklist.empty()) {  
          // Find the next operation ready for inference, that is an operation  
          // with all operands already resolved (non-generic).  
          auto nextop = llvm::find_if(opWorklist, allOperandsInferred);  
          if (nextop == opWorklist.end())  
            break;  
      
          Operation *op = *nextop;  
          opWorklist.erase(op);  
      
          // Ask the operation to infer its output shapes.  
          LLVM_DEBUG(llvm::dbgs() << "Inferring shape for: " << *op << "\n");  
          if (auto shapeOp = dyn_cast<ShapeInference>(op)) {  
            shapeOp.inferShapes();  
          } else {  
            op->emitError("unable to infer shape of operation without shape "  
                          "inference interface");  
            return signalPassFailure();  
          }  
        }  
      
        // If the operation worklist isn't empty, this indicates a failure.  
        if (!opWorklist.empty()) {  
          f.emitError("Shape inference failed, ")  
              << opWorklist.size() << " operations couldn't be inferred\n";  
          signalPassFailure();  
        }  
      }  
      
      /// A utility method that returns if the given operation has all of its  
      /// operands inferred.  
      static bool allOperandsInferred(Operation *op) {  
        return llvm::all_of(op->getOperandTypes(), [](Type operandType) {  
          return operandType.isa<RankedTensorType>();  
        });  
      }  
      
      /// A utility method that returns if the given operation has a dynamically  
      /// shaped result.  
      static bool returnsDynamicShape(Operation *op) {  
        return llvm::any_of(op->getResultTypes(), [](Type resultType) {  
          return !resultType.isa<RankedTensorType>();  
        });  
      }  
    };  
    } // end anonymous namespace  
      
    /// Create a Shape Inference pass.  
    std::unique_ptr<mlir::Pass> mlir::toy::createShapeInferencePass() {  
      return std::make_unique<ShapeInferencePass>();  
    }  
      
    

`ShapeInferencePass` 는 `FunctionPass` 를 상속하며, `runOnFunction()` 인터페이스를 재작성하여 Shape 추론 알고리즘을 구현합니다. 먼저 출력 반환값이 일반화 Tensor인 Operation 리스트를 생성한 뒤, 리스트를 순회하면서 입력 operand의 type이 확정된 Tensor인 Operation을 찾습니다. 만약 찾지 못하면 루프를 종료하고, 그렇지 않으면 해당 Operation을 루프에서 제거하고 해당하는 `inferShape()` 함수를 호출하여 그 Operation의 출력 반환 Tensor의 shape를 추론합니다. Operation 리스트가 비어 있으면 알고리즘이 종료됩니다.

### 5단계: shape 추론 Pass를 최적화 pipeline에 추가

인라인 Pass와 유사하게, shape 추론 Pass를 최적화 pipeline에 추가해야 합니다. 위의 인라인 Pass에서 이미 보여드렸으므로 코드를 다시 붙여 넣지는 않겠습니다.

여기까지, 우리는 인라인 Pass와 shape 추론 Pass의 구현을 완료했습니다. 이 두 Pass로 최적화한 후의 MLIR 표현식이 어떤 모습인지 살펴봅시다. `./toyc-ch4 ../../mlir/test/Examples/Toy/Ch4/codegen.toy -emit=mlir -opt` 를 실행하여 최적화된 MLIR 표현식을 얻었습니다:
    
    
    module  {  
      func @main() {  
        %0 = toy.constant dense<[[1.000000e+00, 2.000000e+00, 3.000000e+00], [4.000000e+00, 5.000000e+00, 6.000000e+00]]> : tensor<2x3xf64>  
        %1 = toy.transpose(%0 : tensor<2x3xf64>) to tensor<3x2xf64>  
        %2 = toy.mul %1, %1 : tensor<3x2xf64>  
        toy.print %2 : tensor<3x2xf64>  
        toy.return  
      }  
    }

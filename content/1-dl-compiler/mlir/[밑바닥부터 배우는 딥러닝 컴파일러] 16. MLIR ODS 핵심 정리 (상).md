분량 문제와 최근 시간이 빠듯한 관계로, 이 글에서는 주로 10가지 핵심 사항을 정리합니다. 나머지 핵심 사항들은 하편에서 다룰 예정입니다. 의견이나 지적은 언제든 환영합니다.

# 1\. 왜 ODS로 Operation을 정의해야 하는가

MLIR에서 Operation을 정의하는 방법은 C++로 직접 정의하는 것과 ODS 프레임워크 기반으로 정의하는 두 가지가 있습니다. C++로 직접 정의하려면 기반 클래스 Op의 일부 생성자 메서드를 상속하고 재정의해야 하며, 모든 Op마다 C++ 코드를 따로 작성해야 합니다. 짐작할 수 있듯이 이렇게 하면 시스템 전체의 Op 정의 부분이 매우 장황해지고, 반복되는 코드가 대량 발생하며 가독성도 떨어집니다. 반면 ODS를 기반으로 Operation을 정의하면, ODS의 규범에 따라 Op 정의를 하나의 `td` 파일에 모아 작성한 뒤, MLIR이 제공하는 코드 생성 도구로 Operation의 C++ 정의를 자동 생성하기만 하면 됩니다. 이러한 완전한 auto codegen 방식은 매우 우아하게 Operation 정의를 구현하고, 사용자가 신경 써야 할 부분(즉 ODS의 문법 규범)도 더 직관적입니다.

ODS는 MLIR에서 Operation을 정의하는 최선의 선택이므로, 우리는 ODS의 문법 규범을 학습할 필요가 있습니다.

# 2\. TableGen 문법

하나의 TableGen 파일(`.td` 확장자)은 다음과 같은 문법 요소를 포함합니다:

  * TableGen `class`는 C++의 class와 비슷하며, 템플릿 또는 기반 클래스로 사용해 하위 클래스를 파생시킬 수 있습니다.
  * TableGen `def`는 C++의 객체와 비슷합니다. TableGen `class`의 특수화로 선언할 수 있으며, 예: `def MyDef: MyClass<...>;`, 또는 단독으로 `def MyDef;`처럼 사용할 수도 있습니다. 이는 템플릿으로 사용할 수 없고 기반 클래스로서 하위 클래스를 파생시킬 수도 없습니다.
  * TableGen `dag`은 방향성 비순환 그래프(DAG) 요소 전용 type입니다. `dag` type은 하나의 연산자와 0개 이상의 인자를 가집니다. 문법은 (`operator arg0, arg1, argN`.) 형태이며, 여기서 `operator`는 임의의 TableGen `def`이 될 수 있습니다. 인자에는 `dag` 자체를 포함해 무엇이든 들어갈 수 있습니다. 연산자와 인자에 이름을 붙일 수도 있는데, 예: (`MyOp:$op_name MyArg:$arg_name`)입니다.



TableGen이 지원하는 더 많은 type과 표현식에 대해서는 다음 링크를 참조하세요: https://llvm.org/docs/TableGen/ProgRef.html.

# 3\. Operation 정의

MLIR은 Operation 정의를 돕기 위한 몇 가지 공통 구조를 정의하고, `TableGen backend : OpDefinitionsGen`을 통해 그 의미를 제공합니다. 이러한 공통 구조는 `OpBase.td` 파일에 정의되어 있습니다. 주요 항목은 다음과 같습니다:

  * `Op` class: Operation을 정의할 때 사용하는 주요 구조입니다. 이 class를 특수화할 때 아래 구조의 도움을 받아 Operation에 관한 모든 정보를 지정합니다.
  * `Dialect` class: 같은 논리 그룹에 속하는 Operation들은 하나의 Dialect 아래에 배치됩니다. Dialect는 dialect 등급 정보를 포함합니다.
  * `OpTrait` class와 그 하위 class: Operation의 특수한 속성과 제약(Constraint)을 지정하는 데 사용됩니다. 예를 들어 Operation이 부수효과(side effect)가 있는지, Op의 출력 형상이 입력 형상과 동일한지 등입니다.
  * `ins/outs` 마커: 이는 `OpDefinitionsGen` 백엔드에 내장된 두 개의 특수 마커로, 각각 operand/attribute와 result의 정의를 안내합니다.
  * `TypeConstraint` class와 그 하위 class: operand 또는 result에 대한 제약(Constraint)을 지정하는 데 사용됩니다. 주목할 만한 하위 class로 `Type`이 있는데, 이는 일반적인 C++ type에 대한 제약을 나타냅니다.
  * `AttrConstraint` class와 그 하위 class: attribute에 대한 제약(Constraint)을 지정하는 데 사용됩니다. 주목할 만한 하위 class로 `Attr`이 있는데, 이는 값이 일반 type인 attribute에 대한 제약을 나타냅니다.



하나의 Operation은 `Op` class를 특수화함으로써 정의되며, 특수화된 `Op` class는 필요한 모든 필드의 구체적인 내용을 포함합니다. 예를 들어 `tf.AvgPool`은 다음과 같이 정의됩니다:
    
    
    def TF_AvgPoolOp : TF_Op<"AvgPool", [NoSideEffect]> {  
      let summary = "Performs average pooling on the input.";  
      
      let description = [{  
    Each entry in `output` is the mean of the corresponding size `ksize`  
    window in `value`.  
      }];  
      
      let arguments = (ins  
        TF_FpTensor:$value,  
      
        Confined<I64ArrayAttr, [ArrayMinCount<4>]>:$ksize,  
        Confined<I64ArrayAttr, [ArrayMinCount<4>]>:$strides,  
        TF_AnyStrAttrOf<["SAME", "VALID"]>:$padding,  
        DefaultValuedAttr<TF_ConvertDataFormatAttr, "NHWC">:$data_format  
      );  
      
      let results = (outs  
        TF_FpTensor:$output  
      );  
      
      TF_DerivedOperandTypeAttr T = TF_DerivedOperandTypeAttr<0>;  
    }  
    

아래에서는 Operation 하나를 정의하는 데 필요한 모든 필드를 설명합니다. 지원되는 필드의 전체 목록은 `Op` class의 정의(즉 `OpBase.td`)를 참고하세요.

  * **「Operation name」** : Operation의 이름입니다. 예를 들어 TensorFlow Dialect의 `tf.Add`처럼요.
  * **「Operation documentation」** : Operation의 문서 설명으로, `summary`와 `description` 두 가지를 포함합니다. 보면 바로 이해되니 자세한 설명은 생략하겠습니다.
  * **「Operation arguments」** : Operation의 인자입니다. 하나의 Operation은 두 종류의 인자를 가집니다. 하나는 `operands` 즉 operand이고, 다른 하나는 `attributes` 즉 attribute 인자입니다. 그중 attribute 인자는 다시 `Natural attributes`와 `Derived attributes` 두 종류로 나뉩니다. 전자는 자연 attribute로 반드시 지정해야 하는데, 예를 들어 합성곱(Convolution)의 출력 채널 수가 그렇고, 후자는 파생 attribute로 예를 들면 출력 Tensor의 형상(shape) 같은 것입니다.



operand와 attribute는 모두 `dag` type의 `arguments`에서 `ins`로 안내되어 지정됩니다:
    
    
    let arguments = (ins  
      <type-constraint>:$<operand-name>,  
      ...  
      <attr-constraint>:$<attr-name>,  
      ...  
    );  
    

여기서 `<type-constraint>`는 `TypeConstraint` class 계층에서 온 TableGen `def`입니다. 이와 비슷하게 `<attr-constraint>`는 `AttrConstraint` class 계층에서 온 TableGen `def`입니다. Constraints 장에 더 자세한 내용이 있습니다.

  * **「가변 operand(Variadic operands)」** . 가변 operand를 정의하려면 `Variadic<...>`로 `TypeConstraint`를 감싸야 합니다. 보통 Operation은 가변 operand가 없거나 하나만 갖습니다. 후자의 경우, 정적 가변 operand 정의로부터 동적 가변 operand를 쉽게 추론할 수 있습니다. 그러나 만약 한 Operation이 여러 개의 가변 길이 operand(선택적이거나 가변 길이인)를 갖는다면, 해당 Operation으로부터의 추가 정보 없이는 동적 operand를 그에 대응되는 정적 가변 길이 operand 정의로 귀속시키는 것이 불가능합니다. 따라서 모든 가변 길이 operand가 그에 대응되는 동적 값을 가진다는 것을 표시하기 위해 `SameVariadicOperandSize` 또는 `AttrSizedOperandSegments` 특성(Trait)을 사용해야 합니다.
  * **「선택적 operand(Optional operands)」** . 선택적 operand를 정의하려면 `Optional<...>`로 `TypeConstraint`를 감싸야 합니다. 설명은 가변 operand와 동일합니다.
  * **「선택적 attribute(Optional attributes)」** . 선택적 attribute를 정의하려면 `OptionalAttr<...>`로 `AttrConstraint`를 감싸야 합니다.
  * **「기본값을 갖는 선택적 attribute」** . `DefaultValuedAttr<..., "...">`로 `AttrConstraint`를 감쌉니다. `DefaultValuedAttr`의 두 번째 인자는 C++ 기본값을 담은 문자열이어야 합니다. 예를 들어, 단정밀도 부동소수점의 기본값은 `"0.5f"`로 지정해야 하고, 정수 배열의 기본값은 `"{1, 2, 3}"`으로 지정해야 합니다.
  * **「제약 attribute(Confining attributes)」** . `Confined`는 값 type이 가져오는 attribute 제약(Constraint)을 한층 더 모델링하는 데 도움을 주는 일반적인 메커니즘으로 제공됩니다. `Confined`를 통해 비교적 원시적인 제약들을 결합하여 복잡한 제약을 만들 수 있습니다. 예를 들어, `32bit` 정수의 최소값이 10인 경우는 `Confined<I32Attr, [IntMinValue<10>]>`로 표현할 수 있습니다. 다른 예시로는 `IntMinValue<N>`이 있는데, 이는 `N` 이상인 정수 attribute를 지정합니다.
  * **「Operation results」** : operand와 비슷하게, result는 `dag` type의 `results`로 선언하며 `outs`로 안내됩니다.


    
    
    let results = (outs  
      <type-constraint>:$<result-name>,  
      ...  
    );  
    

  * **「Operation regions」** 와 **「Operation successors」** 도 있는데, 아직 사용해 보지 않아 적용 시나리오는 정확히 알지 못합니다.

  * **「Op의 특성과 제약(Operation traits and constraints)」** : Trait은 문법이나 의미에 영향을 미치는 Operation의 속성입니다. MLIR C++의 다양한 Trait은 `mlir::OpTrait` namespace에 있습니다. Operation의 Trait, 인터페이스 또는 제약(Constraint)이 여러 operand/attribute/result를 다룰 때는 `Op` class의 두 번째 템플릿 인자로 전달해야 합니다. 이들은 모두 `OpTrait` class를 상속해야 합니다. 자세한 내용은 Constraints 장을 참고하세요.




# 4\. Operation에 자동 생성되는 기본 builder 메서드

Operation을 정의했다면, 어떻게 생성할 수 있을까요? 모든 Operation에 대해, Operation의 인자와 반환값을 바탕으로 몇몇 `builder`가 자동으로 생성됩니다. 예를 들어 다음과 같은 Operation 정의가 있다고 합시다:
    
    
    def MyOp : ... {  
      let arguments = (ins  
        I32:$i32_operand,  
        F32:$f32_operand,  
        ...,  
      
        I32Attr:$i32_attr,  
        F32Attr:$f32_attr,  
        ...  
      );  
      
      let results = (outs  
        I32:$i32_result,  
        F32:$f32_result,  
        ...  
      );  
    }  
    

다음과 같은 `builder`들이 생성됩니다:
    
    
    // All result-types/operands/attributes have one aggregate parameter.  
    // 모든 result type/operand/attribute가 하나의 집계 파라미터로 묶입니다.  
    static void build(OpBuilder &odsBuilder, OperationState &odsState,  
                      ArrayRef<Type> resultTypes,  
                      ValueRange operands,  
                      ArrayRef<NamedAttribute> attributes);  
      
    // Each result-type/operand/attribute has a separate parameter. The parameters  
    // for attributes are of mlir::Attribute types.  
    // 각 result type/operand/attribute가 개별 파라미터입니다. attribute 파라미터는 mlir::Attribute type입니다.  
    static void build(OpBuilder &odsBuilder, OperationState &odsState,  
                      Type i32_result, Type f32_result, ...,  
                      Value i32_operand, Value f32_operand, ...,  
                      IntegerAttr i32_attr, FloatAttr f32_attr, ...);  
      
    // Each result-type/operand/attribute has a separate parameter. The parameters  
    // for attributes are raw values unwrapped with mlir::Attribute instances.  
    // (Note that this builder will not always be generated. See the following  
    // explanation for more details.)  
    // 각 result type/operand/attribute가 개별 파라미터입니다.  
    // attribute 파라미터는 mlir::Attribute 인스턴스로 감싸지지 않은 원시 값입니다.  
    // (주의: 이 builder는 항상 생성되는 것은 아닙니다. 자세한 내용은 아래 설명을 참고하세요.)  
    static void build(OpBuilder &odsBuilder, OperationState &odsState,  
                      Type i32_result, Type f32_result, ...,  
                      Value i32_operand, Value f32_operand, ...,  
                      APInt i32_attr, StringRef f32_attr, ...);  
      
    // Each operand/attribute has a separate parameter but result type is aggregate.  
    // 각 operand/attribute는 개별 파라미터이지만 result는 모두 하나의 집계 type으로 묶입니다.  
    static void build(OpBuilder &odsBuilder, OperationState &odsState,  
                      ArrayRef<Type> resultTypes,  
                      Value i32_operand, Value f32_operand, ...,  
                      IntegerAttr i32_attr, FloatAttr f32_attr, ...);  
      
    // All operands/attributes have aggregate parameters.  
    // Generated if return type can be inferred.  
    // 이 builder는 반환값 type을 추론할 수 있을 때만 생성됩니다.  
    static void build(OpBuilder &odsBuilder, OperationState &odsState,  
                      ValueRange operands, ArrayRef<NamedAttribute> attributes);  
      
    // (And manually specified builders depending on the specific op.)  
    

위 코드의 주석 번역에서 이 builder들의 차이점을 설명해 두었습니다. 이외에도 다른 builder들이 존재할 수 있으니, https://mlir.llvm.org/docs/OpDefinitions/#run-mlir-tblgen-to-see-the-generated-content 의 문서를 참조하세요.

# 5\. 사용자 정의 builder 메서드

만약 위에서 생성된 C++ 코드의 생성자 메서드가 우리가 원하는 형태가 아니라면, 사용자 정의 builder 메서드를 작성해야 합니다. 예를 들어:
    
    
    def MyOp : Op<"my_op", []> {  
      let arguments = (ins F32Attr:$attr);  
      
      let builders = [  
        OpBuilder<(ins "float":$val)>  
      ];  
    }  
    

`builders` 필드는 Op class에 추가되는 사용자 정의 builder 목록입니다. 이 예에서는 attribute 대신 float 값을 받는 편리한 builder를 제공했습니다. TableGen `dag`을 사용하는 ODS에서, 많은 함수 선언이 `ins` 접두사를 사용합니다. 그 뒤에는 쉼표로 구분된 목록이 따라오며, 목록의 각 항목은 type과 `$` 접두사를 가진 이름의 조합입니다. 위 정의는 다음과 같은 builder로 변환됩니다:
    
    
    class MyOp : /*...*/ {  
      /*...*/  
      static void build(::mlir::OpBuilder &builder, ::mlir::OperationState &state,  
                        float val);  
    };  
    

이 builder에는 두 개의 추가 선행 파라미터가 있다는 점에 유의하세요. 이 파라미터들은 Operation 생성에 매우 유용합니다. 특히, 이 메서드를 통해 Operation을 생성할 수 있도록 하려면, `state`에 해당 Operation의 attribute, operand, region, 반환값 type을 채워야 합니다. `builder`는 type이나 중첩된 Operation 등 Op에 속하는 임의의 IR 객체를 생성하는 데 사용할 수 있습니다. type과 이름이 C++ 코드로 변환될 때는 유효한 C++ 구조여야 합니다. 즉 (Op의 namespace 안에서) 유효한 type과 식별자여야 합니다(예를 들어 `class`는 유효한 식별자가 아닙니다). builder의 구현은 다음과 같이 TableGen 코드 블록을 사용해 ODS에서 직접 제공할 수도 있습니다:
    
    
    def MyOp : Op<"my_op", []> {  
      let arguments = (ins F32Attr:$attr);  
      
      let builders = [  
        OpBuilder<(ins "float":$val), [{  
          $_state.addAttribute("attr", $_builder.getF32FloatAttr(val));  
        }]>  
      ];  
    }  
    

두 특수 파라미터 `$_builder`와 `$_state`는 각각 `builder`와 `state`에 대응됩니다. `ins` 부분의 파라미터는 직접 사용할 수 있는데, 예를 들어 `val`처럼요. builder의 C++ 코드 구현은 ODS의 특수 변수를 치환함으로써 완성되며, builder ODS 구현의 나머지 부분이 유효한 C++ 구조임을 보장해야 합니다. 코드 크기에는 제한이 없지만, ODS에는 짧은 정의의 builder만 인라인으로 두고 더 긴 정의의 builder는 C++ 파일에 두는 것을 권장합니다. 마지막으로, 일부 파라미터에 기본값이 필요하다면 `CArg`를 사용해 type과 그 값을 함께 감싸 정의할 수 있으며, 다음과 같이 작성합니다:
    
    
    def MyOp : Op<"my_op", []> {  
      let arguments = (ins F32Attr:$attr);  
      
      let builders = [  
        OpBuilder<(ins CArg<"float", "0.5f">:$val), [{  
          $_state.addAttribute("attr", $_builder.getF32FloatAttr(val));  
        }]>  
      ];  
    }  
    

변환된 C++ 코드에서 기본 인자는 선언에만 나타나고 정의에는 나타나지 않으며, 이는 C++의 요구사항에 부합합니다.
    
    
    /// Header file.  
    class MyOp : /*...*/ {  
      /*...*/  
      static void build(::mlir::OpBuilder &builder, ::mlir::OperationState &state,  
                        float val = 0.5f);  
    };  
      
    /// Source file.  
    MyOp::build(::mlir::OpBuilder &builder, ::mlir::OperationState &state,  
                float val) {  
      state.addAttribute("attr", builder.getF32FloatAttr(val));  
    }  
    

# 6\. 선언적 명령 형식(Declarative Assembly Format)

Operation의 선언적 명령 형식은 Operation의 operand, attribute 등과 매칭되는 선언적 문자열로 지정할 수 있습니다. 또한 Operation을 빌드하기 위해 파싱이 필요한 추가 정보를 표현할 수도 있습니다:
    
    
    def CallOp : Std_Op<"call", ...> {  
      let arguments = (ins FlatSymbolRefAttr:$callee, Variadic<AnyType>:$args);  
      let results = (outs Variadic<AnyType>);  
      
      let assemblyFormat = [{  
        $callee `(` $args `)` attr-dict `:` functional-type($args, results)  
      }];  
    }  
    

크게 세 부분으로 구성됩니다:

  * **「Directives: 지시자」** . 지시자는 선택적 인자를 갖는 내장 함수의 일종입니다. 사용 가능한 지시자에는 `attr-dict`, `attr-dict-with-keyword`, `operands`, `ref` 등이 있습니다.
  * **「리터럴(Literals)」** . 리터럴은 ``로 감싼 키워드 또는 구두점입니다. 다음은 유효한 구두점의 집합입니다: `:, ,, =, <, >, (, ), {, }, [, ], ->, ?, +, *`. `\n` 구두점은 줄 바꿈 효과를 갖습니다. 다음과 같이요:


    
    
    let assemblyFormat = [{  
      `{` `\n` ` ` ` ` `this_is_on_a_newline` `\n` `}` attr-dict  
    }];  
    
    
    
    %results = my.operation {  
      this_is_on_a_newline  
    }  
    

내용이 빈 리터럴은 특정 리터럴 요소가 암묵적으로 삽입된 후의 공백을 제거하는 데 사용할 수 있습니다. 예를 들어 `)`나 `]` 등이요. 예를 들어, `]`이 output의 끝에 나타날 수 있는데, 이는 형식의 마지막 요소가 아닐 수 있습니다. 이 경우 `"]``"`를 사용해 그 뒤의 공백을 제거할 수 있습니다.

  * **「Variables(변수)」** . 변수는 Operation에 등록된 엔티티로, 예를 들어 Operation의 인자(attribute 또는 operand), region, result, successor 등이 있습니다. `CallOp`에서 변수는 `$callee`와 `$args`를 가리킵니다. attribute 변수는 자신의 값 type을 표시합니다. 단, 그 값의 type이 구성 가능한(constructable) 경우에는 attribute 변수의 값 type을 생략할 수 있습니다.



# 7\. 사용자 정의 지시자(Custom Directives) & 선택적 그룹(Optional Groups)

선언적 명령 형식 규범은 Operation을 형식화할 때 대부분의 일반적인 시나리오를 처리할 수 있습니다. 형식 안에서 Operation의 특정 부분을 지정하고 싶지만 선언식 문법으로 지원되지 않는 Op라면, 사용자 정의 지시자(Custom Directive)를 시도해 볼 수 있습니다.

어떤 경우에는 Operation이 "선택적" 정보를 가질 수 있습니다. 예를 들어 attribute 또는 비어 있는 가변 인자 operand 그룹이 그렇습니다. 이런 경우, 그 정보의 존재 여부에 따라 어셈블리 형식의 일부를 선택적으로 표시할 수 있습니다.

이 두 부분은 비교적 복잡하고, 저도 아직 사용해 본 적이 없어서 여기서는 자세히 다루지 않습니다. 관심이 있다면 공식 문서를 참고하시기 바랍니다.

# 8\. 타입 추론

형식의 한 가지 요구 사항은 operand와 result의 type이 항상 존재해야 한다는 것입니다. 어떤 경우에는 type 제약(Constraint)이나 다른 사용 가능한 정보를 통해 변수의 type을 추론할 수 있습니다. 이 경우 형식에서 해당 변수의 type을 생략할 수 있습니다.

  * **「Buildable Types. 구성 가능한 type」** . 일부 type 제약(Constraint)은 표현 방식이 단 하나뿐이라 직접 구성할 수 있습니다. 예를 들어 `I32`나 `Index` type이 그렇습니다. ODS의 type은 `builderCall` 필드를 설정하거나 `BuildableType` class를 상속함으로써 자신을 구성 가능한 것으로 표시할 수 있습니다.
  * **「Trait Equality Constraints. Trait 등가 제약」** . Operation에 대해 알려진 type 등가 Trait으로 등록된 제약을 가진 Operation들이 많이 있습니다. 예를 들어, `select` Operation의 true, false, result 값은 보통 동일한 type을 갖습니다. 어셈블리 형식은 이러한 등가 제약을 검사해 누락된 변수의 type을 식별할 수 있습니다. 현재 지원되는 Trait은 `AllTypesMatch`, `TypesMatchWith`, `SameTypeOperands`, `SameOperandsAndResultType`입니다.
  * **「InferTypeOpInterface」** . `InferTypeOpInterface`를 구현한 Operation은 result type이 operand로부터 추론될 수 있으므로 어셈블리 형식에서 result type을 생략할 수 있습니다.
  * **「hasCanonicalizer」** . 이 boolean 필드는 해당 Operation에 정규화(canonicalization) 패턴이 정의되어 있는지를 나타냅니다. 값이 `1`이면 `::getCanonicalizationPatterns()`를 정의해야 합니다.
  * **「hasCanonicalizeMethod」** . 이 boolean 필드가 `true`이면, 해당 Operation이 단순한 "matchAndRewrite" 스타일 정규화 패턴을 위한 `canonicalize` 메서드를 구현하고 있음을 나타냅니다. `hasCanonicalizer`가 0이면 이 함수를 호출하기 위해 `::getCanonicalizationPatterns()`의 구현이 제공됩니다.
  * **「hasFolder」** . 이 boolean 필드는 해당 Operation에 일반 folding 규칙이 정의되어 있는지를 나타냅니다. 값이 `1`이면 `::fold()`를 정의해야 합니다.



# 9\. 추가 선언

테이블 기반 Operation 정의의 목표 중 하나는 각 Operation에 대해 가능한 한 많은 로직과 메서드를 자동 생성하는 것입니다. 그렇긴 하지만 결국 다룰 수 없는 long-tail 케이스는 늘 존재하기 마련입니다. 이런 경우에는 `extraClassDeclaration`을 사용할 수 있습니다. `extraClassDeclaration` 안의 코드는 생성된 C++ op class에 그대로 복사됩니다.

`extraClassDeclaration`은 고급 사용자의 long-tail 케이스를 위한 메커니즘이라는 점에 유의하세요. 아직 구현되지 않았으나 광범위하게 적용 가능한 경우라면, 기반 인프라를 개선하는 편이 바람직합니다.

# 10\. C++ 코드 생성

`OpDefinitionsGen` (https://github.com/llvm/llvm-project/blob/main/mlir/tools/mlir-tblgen/OpDefinitionsGen.cpp)은 Operation 정의 명세 파일(`.td` 파일)을 처리하여 대응되는 C++ 코드를 담은 두 개의 파일을 생성합니다: 하나는 선언용, 다른 하나는 정의용입니다. 전자는 `-gen-op-decls` 명령줄 옵션으로 생성되고, 후자는 `-gen-op-defs` 옵션으로 생성됩니다.

정의 파일은 모든 op 메서드 정의를 담고 있으며, `GET_OP_CLASSES`를 정의함으로써 포함하고 활성화할 수 있습니다. OpDefinitionsGen은 각 Operation에 대해 하나의 Operation class와 하나의 operand adaptor class(`operand adaptor class`)를 생성합니다. 또한 정의된 모든 Operation의 콤마로 구분된 목록도 포함되어 있어, `GET_OP_LIST`를 정의함으로써 이 Operation들을 포함하고 활성화할 수 있습니다.

  * **「Class name and namespaces」** .



각 Operation에 대해, 생성되는 C++ class 이름은 TableGen `def`을 접두사로 한 이름에서 Dialect 접두사를 제거한 형태입니다. 첫 번째 `_`이 구분자로 사용됩니다. 예를 들어 `def TF_AddOp`의 경우 C++ class 이름은 `AddOp`이 됩니다. `TF` 접두사를 제거한 것은, 이는 여러 Operation의 작용 영역(scope)을 나타내기 때문입니다. 다른 Dialect도 자체 AddOps를 정의할 수 있습니다.

생성된 C++ class의 namespace는 Dialect의 `cppNamespace` 필드에서 옵니다. 예를 들어, 어떤 Dialect의 `Namespace`가 `A::B`라면, 그 Dialect의 Op는 `namespace A { namespace B { ... } }` 안에 배치됩니다. Dialect가 `cppNamespace`를 지정하지 않으면, dialect의 이름을 namespace로 사용합니다.

이는 생성된 C++ class의 이름이 반드시 Operation 이름의 op 이름과 정확히 일치하지는 않을 수 있음을 의미합니다. 이는 코딩 스타일 요구 사항을 충족시키기 위한 유연한 명명을 허용하기 위함입니다.

  * **「Operand adaptors」**



각 Operation에 대해, MLIR은 자동으로 operand adaptor를 생성합니다. 이 class는 list 값으로 제공되는 operand에 접근할 때 "매직 상수"를 사용하지 않아도 되도록 해줍니다. operand adaptor는 `Value` 배열을 참조하며, Operation class와 동일한 이름의 메서드를 제공해 그 값들에 접근할 수 있게 합니다. 예를 들어, 이항 산술 연산의 경우 `.lhs()`로 첫 번째 operand에, `.rhs()`로 두 번째 operand에 접근하도록 할 수 있습니다. operand adaptor class는 Operation class와 동일한 namespace에 있으며, class 이름은 Operation class 이름 뒤에 `Adaptor`를 붙인 형태입니다.

operand adaptor는 Operation을 처리하는 함수 템플릿에도 사용할 수 있습니다:
    
    
    template <typename BinaryOpTy>  
    std::pair<Value, Value> zip(BinaryOpTy &&op) {  
      return std::make_pair(op.lhs(), op.rhs());;  
    }  
      
    void process(AddOp op, ArrayRef<Value> newOperands) {  
      zip(op);  
      zip(Adaptor<AddOp>(newOperands));  
      /*...*/  
    }  
      
    

OneFlow에서 생성된 `UserOpAdaptor` 코드를 볼 수 있습니다. 그 안에는 Operation의 operand와 관련 attribute에 접근할 수 있는 일련의 인터페이스가 제공됩니다.
    
    
    //===----------------------------------------------------------------------===//  
    // ::mlir::oneflow::UserOp declarations  
    //===----------------------------------------------------------------------===//  
      
    class UserOpAdaptor {  
    public:  
      UserOpAdaptor(::mlir::ValueRange values, ::mlir::DictionaryAttr attrs, ::mlir::RegionRange regions = {});  
      UserOpAdaptor(UserOp &op);  
      ::mlir::ValueRange getOperands();  
      std::pair<unsigned, unsigned> getODSOperandIndexAndLength(unsigned index);  
      ::mlir::ValueRange getODSOperands(unsigned index);  
      ::mlir::ValueRange data_input();  
      ::mlir::ValueRange ctrl_inputs();  
      ::mlir::DictionaryAttr getAttributes();  
      ::mlir::StringAttr op_name();  
      ::mlir::BoolAttr trainable();  
      ::mlir::StringAttr device_tag();  
      ::mlir::ArrayAttr device_name();  
      ::mlir::IntegerAttr scope_symbol_id();  
      ::mlir::ArrayAttr hierarchy();  
      ::mlir::DenseIntElementsAttr operand_segment_sizes();  
      ::mlir::DenseIntElementsAttr result_segment_sizes();  
      ::mlir::StringAttr op_type_name();  
      ::mlir::ArrayAttr input_lbn_segment_keys();  
      ::mlir::ArrayAttr input_lbn_segment_sizes();  
      ::mlir::ArrayAttr output_lbn_segment_keys();  
      ::mlir::ArrayAttr output_lbn_segment_sizes();  
      ::mlir::ArrayAttr output_lbns();  
      ::mlir::LogicalResult verify(::mlir::Location loc);  
      
    private:  
      ::mlir::ValueRange odsOperands;  
      ::mlir::DictionaryAttr odsAttrs;  
      ::mlir::RegionRange odsRegions;  
    };  
    

# 11\. 제약(Constraint)

제약(Constraint)은 테이블 기반 Operation 정의의 핵심 개념입니다: Operation 검증과 그래프 Operation 매칭은 모두 제약(Constraint)의 만족을 기반으로 합니다. 따라서 Operation 정의와 재작성(rewrite) 규칙 모두 제약을 작성하는 것과 직접적으로 관련됩니다. MLIR은 `OpBase.td`(`https://github.com/llvm/llvm-project/blob/main/mlir/include/mlir/IR/OpBase.td`)에서 `Constraint` 기반 class를 정의합니다. Operation의 Constraint는 다음과 같이 다양한 범위를 다룰 수 있습니다:

  * 단일 attribute에만 관여(예: 5보다 큰 32비트 정수)
  * 여러 operand와 result에 관여(예: 첫 번째 result의 형상이 첫 번째 operand(Tensor로 이해할 수 있는)의 형상과 같아야 함)
  * Operation 자체에 본질적으로 내재(예: 부수효과(side effect) 없음. Transpose Op 제거 사례 참고)



우리는 이들을 각각 단일 엔티티 제약(single-entity constraint), 다중 엔티티 제약(multi-entity constraint), 그리고 Trait이라고 부릅니다.

# 0x1. 배경

현재 Transformer 아키텍처는 AI 알고리즘 개발자와 엔지니어들이 반드시 언급할 수밖에 없는 기반 아키텍처가 되었다. Transformer 기반 아키텍처에서 파생된 일련의 초대형 모델, 예컨대 Bert와 GPT-2는 업계에 매우 큰 영향을 끼쳤으며, 대형 모델의 흐름을 이끌었다. 그러나 대형 모델의 막대한 학습 비용은 많은 사람과 기업들을 망설이게 만들었고, 보통은 사전 학습된 대형 모델 위에서 일부 다운스트림 task만 수행하는 데에 그치게 된다. 따라서 대형 모델 학습을 어떻게 가속할 것인지가 매우 중요하다. 2019년 NVIDIA는 가장 큰 언어 모델인 GPT-2 8B를 성공적으로 구축하고 학습시켰는데, 이 모델은 83억 개의 파라미터를 가지고 있어 BERT-Large 모델의 24배, GPT-2의 5.6배에 달한다. NVIDIA는 이 모델을 「Megatron」(메가트론)이라 명명했으며, 이 모델을 학습시키는 데 사용한 pytorch 코드도 오픈소스로 공개했다: https://github.com/NVIDIA/Megatron-LM.

이 논문에서는 대형 모델 학습을 가속하기 위한 많은 방법들을 언급하고 있는데, 특히 모델 병렬 학습 기술 같은 것이 그러하다. 다만 본인은 분산 학습에 대한 이해가 적어 여기서는 다루지 않는다. 여기서 유일하게 주목하는 부분은 Megatron 논문(`https://arxiv.org/pdf/2104.04473.pdf`)의 4.2절에서 언급하는 컴파일 최적화를 통한 모델 학습 가속이다:

![이미지](images/img_01.png)Megatron 4.2절![이미지](images/img_02.png)Megatron 4.2절

이 절의 컴파일 최적화는 주로 PyTorch JIT 기술을 통해 일부 op fusion을 수행할 수 있다는 내용이다. 예를 들어 bias_add와 gelu를 하나의 op로 fusion하거나, bias_add+dropout을 하나의 op로 fusion하는 식이다. 이러한 op fusion을 적용하면 GPU의 반복적인 데이터 read/write를 피해 메모리(VRAM) 점유를 줄일 수 있을 뿐 아니라, cuda kernel launch 횟수를 줄여 전체 연산 과정을 가속할 수 있다.

논문에서 언급한 컴파일 최적화를 구현하려면 두 가지 전제 조건이 필요하다. 첫째, 프레임워크가 fusion된 op의 구현을 제공해야 하고, 둘째, 컴파일러를 기반으로 한 최적화 Pass를 구현하여 모델에서 fusion 가능한 Pattern을 자동으로 찾고 이를 등가의 fusion op로 rewrite함으로써 계산 그래프 실행을 가속하는 목적을 달성해야 한다.

# 0x2. BiasAdd, Dropout 및 fusion op 소개

OneFlow에서는 Megatron의 bias_add 및 dropout fuse에 대응하기 위해 `fused_bias_add_mask_scale`이라는 op를 구현했으며, 하는 일은 BiasAdd와 Dropout을 하나의 op로 fusion하여 가속하는 것이다. 이 op의 구현 과정은 여기서 자세히 다루지 않으며, 핵심은 모델에서 MLIR을 기반으로 이러한 Pattern을 자동으로 발견하고 자동으로 `fused_bias_add_mask_scale` op로 대체하는 방법이다.

다음 절에서 fusion Pass의 작동 방식을 더 잘 이해할 수 있도록, 여기서는 bias_add, dropout 및 fused_bias_add_mask_scale 이 세 가지 Op의 파라미터 목록에 대해 간략하게 소개한다.

  * bias_add op:


    
    
    >>> import oneflow as flow  
    >>> x = flow.randn(2, 3)  
    >>> y = flow.randn(3)  
    >>> z = flow._C.bias_add(x, y, axis=1)  
    

이 op는 3개의 파라미터를 가지는 것을 볼 수 있다. 하나는 입력 Tensor, 하나는 bias Tensor, 그리고 또 하나는 bias Tensor를 입력 Tensor의 어느 차원에 더할지를 나타내는 axis attribute이다. Transformer 구조에서 bias가 있는 linear layer(`nn.Linear`)는 행렬 곱(matmul) op와 bias_add op로 구현된다.

  * nn.Dropout op: Dropout op는 모두에게 매우 익숙할 것이라 따로 설명할 필요는 없을 것이며, 아래의 OneFlow op 문서를 참고하면 된다.



![이미지](images/img_03.png)예시:
    
    
    >>> import numpy as np  
    >>> import oneflow as flow  
      
    >>> m = flow.nn.Dropout(p=0)  
    >>> arr = np.array(  
    ...    [  
    ...        [-0.7797, 0.2264, 0.2458, 0.4163],  
    ...        [0.4299, 0.3626, -0.4892, 0.4141],  
    ...        [-1.4115, 1.2183, -0.5503, 0.6520],  
    ...    ]  
    ... )  
    >>> x = flow.Tensor(arr)  
    >>> y = m(x)  
    >>> y   
    tensor([[-0.7797,  0.2264,  0.2458,  0.4163],  
            [ 0.4299,  0.3626, -0.4892,  0.4141],  
            [-1.4115,  1.2183, -0.5503,  0.6520]], dtype=oneflow.float32)  
    

  * fused_bias_add_mask_scale: fused_bias_add_mask_scale op는 bias_add op의 input `a`와 `b`(bias)가 필요하고, 그 다음 input `a`로부터 `random_mask_like` Op를 호출해 생성된 mask Tensor `mask`가 세 번째 input으로 필요하며, 마지막으로 bias_add op의 `axis` attribute와 Dropout의 `p` attribute가 필요하다.



여기서 왜 mask가 필요한지 설명할 필요가 있다. 사실 Dropout op는 구현 시 두 개의 output을 만들어 내는데, 하나는 출력 Tensor이고, 하나는 mask이다. 이는 Dropout이 `p`와 우리가 입력한 random seed에 따라 어떤 위치의 뉴런을 보존하고 어떤 위치의 뉴런을 0으로 만들지 결정하는 mask를 생성하기 때문이며, 올바른 backward propagation을 위해 입력 Tensor에 대응하는 gradient를 구할 때 이 mask를 반드시 보존해야 한다. 따라서 fused_bias_add_mask_scale Op에서는 mask를 명시적으로 이 Op에 넘겨주어야 하는데, 이 Op의 output은 단 하나뿐이라 추가 mask를 별도로 출력하지 않기 때문이다. 그리고 이 mask의 생성은 oneflow 내부의 `random_mask_like` Op를 이용하여 생성하는데, 이 Op는 입력 Tensor와 `p` 그리고 random seed를 받아 일정한 확률 분포를 갖는 mask Tensor mask를 생성한다.

# 0x3. Pattern 매칭 및 rewrite

이러한 Op들의 operand, attribute 및 output을 이해한 후에는, MLIR을 기반으로 BiasAdd와 Dropout에 대한 Pattern 자동 매칭 및 rewrite를 수행할 수 있다. 이 기능의 구현 위치는: `https://github.com/Oneflow-Inc/oneflow/pull/7709`.

먼저, `oneflow/ir/include/OneFlow/OneFlowPatterns.td` 파일에서 MLIR의 DRR 프레임워크를 기반으로 자동 매칭 및 rewrite의 템플릿을 작성해야 하며, 구현은 다음과 같다:
    
    
    def GetDefaultSeed :  
      NativeCodeCall<"mlir::oneflow::GetDefaultSeed($_builder)">;  
      
    def FusedBiasAddMaskScale :  
      NativeCodeCall<"mlir::oneflow::CreateFusedBiasAddMaskScale($_builder, $0, $1, $2)">;  
      
    def IsAddToOutputNone: Constraint<CPred<"mlir::oneflow::IsAddToOutputNone($0)">, "">;  
      
    def FusedBiasAddDropoutPattern : Pattern<  
      (  
        OneFlow_DropoutOp: $dropout_res  
        (  
          OneFlow_BiasAddOp: $bias_add_res  
            $a,  
            $b,  
            $bias_add_op_name,  
            $bias_add_device_tag,  
            $bias_add_device_name,  
            $bias_add_scope_symbol_id,  
            $bias_add_hierarchy,  
            $bias_add_op_axis  
        ),  
        $_add_to_output,  
        $dropout_op_name,  
        $dropout_device_tag,  
        $dropout_device_name,  
        $dropout_scope_symbol_id,  
        $dropout_hierarchy,  
        $dropout_op_rate  
      ),  
      [  
        (  
          FusedBiasAddMaskScale  
          $dropout_res__0,  
          $bias_add_res,  
          (  
            OneFlow_RandomMaskLikeOp : $mask  
              $a,  
              $bias_add_op_name,  
              $dropout_device_tag,  
              $dropout_device_name,  
              $dropout_scope_symbol_id,  
              $dropout_hierarchy,  
              $dropout_op_rate,  
              (GetDefaultSeed)  
          )  
        ),  
        (replaceWithValue $mask)  
      ],  
      [(IsAddToOutputNone $_add_to_output)]  
    >;  
    

`NativeCodeCall`은 placeholder 코드로, `NativeCodeCall`을 통해 Dialect에서 직접 작성한 C++ 함수를 호출할 수 있다. 예를 들어:
    
    
    def GetDefaultSeed :  
      NativeCodeCall<"mlir::oneflow::GetDefaultSeed($_builder)">;  
    

여기서는 OneFlow Dialect에서 직접 작성한 `GetDefaultSeed` 함수를 호출하며, 이 함수는 OneFlow의 DefaultAutoGenerator 클래스가 생성한 random seed를 반환한다. 이 random seed는 Pattern 안에서 RandomMaskLikeOp의 attribute로 사용된다:
    
    
    mlir::IntegerAttr GetDefaultSeed(::mlir::PatternRewriter& rewriter) {  
      const auto gen = CHECK_JUST(::oneflow::one::DefaultAutoGenerator());  
      return getSI64IntegerAttr(rewriter, (int64_t)gen->current_seed());  
    }  
    

마찬가지로 `CreateFusedBiasAddMaskScale` 함수는 매칭된 Pattern(BiasAddOp+DropoutOp)을 FusedBiasAddMaskScaleOp로 rewrite하는 역할을 한다. 코드 구현은 다음과 같다:
    
    
    ::llvm::SmallVector<::mlir::Value, 4> CreateFusedBiasAddMaskScale(::mlir::PatternRewriter& rewriter,  
                                                                      OpResult dropout_result,  
                                                                      OpResult bias_add_result,  
                                                                      Operation* mask) {  
      if (auto dropout_op = llvm::dyn_cast<oneflow::DropoutOp>(dropout_result.getDefiningOp())) {  
        if (auto bias_add_op = llvm::dyn_cast<oneflow::BiasAddOp>(bias_add_result.getDefiningOp())) {  
          SmallVector<Value, 4> operands;  
          operands.push_back(bias_add_op.a());  
          operands.push_back(bias_add_op.b());  
          operands.push_back(mask->getResults()[0]);  
          NamedAttrList fused_bias_add_dropout_attributes = dropout_op->getAttrs();  
          fused_bias_add_dropout_attributes.append(llvm::StringRef("axis"), bias_add_op.axisAttr());  
          fused_bias_add_dropout_attributes.append(llvm::StringRef("scale"), dropout_op.rateAttr());  
          fused_bias_add_dropout_attributes.erase(dropout_op.rateAttrName());  
          auto res = rewriter  
                         .create<oneflow::FusedBiasAddMaskScaleOp>(  
                             dropout_op->getLoc(), dropout_op->getResultTypes().front(), operands,  
                             fused_bias_add_dropout_attributes)  
                         ->getResults();  
          // bias_add and dropout op is expected to be erased if it is not used  
          return res;  
        }  
      }  
      return {};  
    }  
    

이 함수는 PatternRewriter 객체와 DropoutOp 및 BiasAddOp의 output value를 받고, 이 두 value로부터 그것들을 정의하는 Op를 얻을 수 있으며, Op로부터 다시 대응하는 operand와 attribute 등을 얻을 수 있다. 그런 다음 PatternRewriter 객체를 기반으로 새로운 Op를 생성하는 과정을 완료하고, 현재 DropoutOp의 위치에서 대체를 완료하면 특정 Pattern의 rewrite 작업이 끝난다. 무효화된 BiasAddOp와 DropoutOp는 NoSideEffect이기 때문에, 생성된 IR에서 자동으로 제거된다.

이어서 IsAddToOutputNone이라는 Constraint를 살펴보자. `def IsAddToOutputNone: Constraint<CPred<"mlir::oneflow::IsAddToOutputNone($0)">, "">;` 여기서는 `CPred`를 사용하여 사용자 정의 Constraint를 정의했는데, 이 `CPred` 안에는 bool 타입을 반환하는 어떤 C++ 함수도 넣을 수 있다. 여기서의 구현은 다음과 같다:
    
    
    bool IsAddToOutputNone(ValueRange value) { return (int)value.size() > 0 ? false : true; }  
    

즉, Dropout Op의 `_add_to_output`이라는 optional input이 존재하는지 판단하며, 존재하지 않을 때만 우리가 구현한 이 Pass를 사용할 수 있다.

위의 일반적인 부분 외에, 여기서는 두 가지 특별한 점에 주목해야 하므로 따로 정리한다.

## NativeCodeCall의 제약으로 인해 발생한 문제

MLIR 문서에서 NativeCodeCall은 단 하나의 결과만 반환할 수 있다는 점을 확인할 수 있다. 따라서 위의 템플릿 매칭 및 rewrite 시 rewrite 부분에 두 개의 output을 설정했으며, 하나는 FusedBiasAddMaskScaleOp의 output(목표 output)이고, 다른 하나는 `(replaceWithValue $mask)`로 정의한 placeholder output이다. 그 이유는 Dropout Op가 2개의 output을 가지기 때문에, 만약 여기서 새로운 placeholder output을 정의하지 않으면 템플릿 매칭 rewrite 시 output 개수가 다르다는 오류가 발생한다. 여기서 `replaceWithValue`를 사용한 이유는, 단순하게 하나의 value(`mlir::Value`)를 대체하는 기능을 직접 수행할 수 있어 여기서의 placeholder 역할에 적합하기 때문이다.

## RandomMaskLikeOp는 왜 builder를 직접 정의해야 하는가

위의 구현에는 또 하나의 의존이 있는데, RandomMaskLikeOp의 builder를 직접 정의해야 한다는 점이다. 새로운 RandomMaskLikeOp의 정의는 다음과 같다:
    
    
    def OneFlow_RandomMaskLikeOp : OneFlow_BaseOp<"random_mask_like", [NoSideEffect, NoGrad, DeclareOpInterfaceMethods<UserOpCompatibleInterface>]> {  
      let input = (ins  
        OneFlow_Tensor:$like  
      );  
      let output = (outs  
        OneFlow_Tensor:$out  
      );  
      let attrs = (ins  
        DefaultValuedAttr<F32Attr, "0.">:$rate,  
        DefaultValuedAttr<SI64Attr, "0">:$seed  
      );  
      let builders = [  
        OpBuilder<(ins  
          "Value":$like,  
          "StringRef":$op_name,  
          "StringRef":$device_tag,  
          "ArrayAttr":$device_name,  
          "IntegerAttr":$scope_symbol_id,  
          "ArrayAttr":$hierarchy,  
          "FloatAttr":$rate,  
          "IntegerAttr":$seed  
        )>  
      ];  
      let has_check_fn = 1;  
      let has_logical_tensor_desc_infer_fn = 1;  
      let has_physical_tensor_desc_infer_fn = 1;  
      let has_get_sbp_fn = 1;  
      let has_data_type_infer_fn = 1;  
    }  
    

builder를 직접 정의한 후에는 이 builder를 C++로 구현해야 한다:
    
    
    void RandomMaskLikeOp::build(mlir::OpBuilder& odsBuilder, mlir::OperationState& odsState,  
                                 mlir::Value like, StringRef op_name, StringRef device_tag,  
                                 ArrayAttr device_name, IntegerAttr scope_symbol_id,  
                                 ArrayAttr hierarchy, mlir::FloatAttr rate, mlir::IntegerAttr seed) {  
      odsState.addOperands(like);  
      odsState.addAttribute(op_nameAttrName(odsState.name), odsBuilder.getStringAttr(op_name));  
      odsState.addAttribute(device_tagAttrName(odsState.name), odsBuilder.getStringAttr(device_tag));  
      odsState.addAttribute(device_nameAttrName(odsState.name), device_name);  
      if (scope_symbol_id) {  
        odsState.addAttribute(scope_symbol_idAttrName(odsState.name), scope_symbol_id);  
      }  
      if (hierarchy) { odsState.addAttribute(hierarchyAttrName(odsState.name), hierarchy); }  
      odsState.addAttribute(rateAttrName(odsState.name), rate);  
      odsState.addAttribute(seedAttrName(odsState.name), seed);  
      odsState.addTypes(like.getType());  
    }  
    

이렇게 하는 이유는, 여기서 RandomMaskLikeOp가 생성하는 mask가 대체하려는 Dag의 최외곽 Op가 아니기 때문에, MLIR이 RandomMaskLikeOp의 output value 타입을 추론할 수 없기 때문이다(만약 단일 Op라면, 이 Op의 output 타입은 그것이 replace하려는 Op의 output 타입이 된다). 따라서 출력 타입이 필요 없는 특별한 builder를 제공해야 한다. 이 builder는 type inference를 수행하는데, 이 예시에서는 like로부터 직접 타입을 가져온다. 즉, `odsState.addTypes(like.getType());` 이 줄의 코드이다.

만약 이를 수정하지 않으면 타입이 일치하지 않는다는 오류가 발생하며, 대체로 다음과 같은 형태이다:

# 0x4. 테스트

위에서 모든 구현 세부 사항을 다 다뤘으므로, OneFlow 프로그램을 하나 구성하여 이 IR fusion이 정상적으로 동작하는지 검증해 볼 수 있다. 테스트 코드는 다음과 같다:
    
    
    import unittest  
    import numpy as np  
    import os  
    os.environ["ONEFLOW_MLIR_ENABLE_ROUND_TRIP"] = "1"  
    import oneflow as flow  
    import oneflow.unittest  
      
      
    def do_bias_add_dropout_graph(test_case, with_cuda, prob):  
        x = flow.randn(2, 3, 4, 5)  
        bias = flow.randn(5)  
        dropout = flow.nn.Dropout(p=prob)  
        if with_cuda:  
            x = x.cuda()  
            bias = bias.to("cuda")  
            dropout.to("cuda")  
      
        eager_res = dropout(flow._C.bias_add(x, bias, axis=3))  
      
        class GraphToRun(flow.nn.Graph):  
            def __init__(self):  
                super().__init__()  
                self.dropout = dropout  
      
            def build(self, x, bias):  
                return self.dropout(flow._C.bias_add(x, bias, axis=3))  
      
        graph_to_run = GraphToRun()  
        lazy_res = graph_to_run(x, bias)  
        test_case.assertTrue(np.array_equal(eager_res.numpy(), lazy_res.numpy()))  
      
    @flow.unittest.skip_unless_1n1d()  
    class TestBiasAddDropout(oneflow.unittest.TestCase):  
        def test_bias_add_dropout_graph(test_case):  
            do_bias_add_dropout_graph(test_case, True, 1.0)  
      
      
    if __name__ == "__main__":  
        unittest.main()  
    

여기서는 nn.Graph로 연산 과정을 wrapping했는데, 즉 static graph 모드로 전체 프로그램을 실행하는 것이다. nn.Graph가 구축된 후에는 Job(OneFlow의 원본 계산 그래프 표현)이 생성되고, 그 후 이 Job이 MLIR 표현(OneFlow Dialect)으로 변환되어 위에서 설명한 Fuse Pass를 거친 후 다시 Job(최적화된 OneFlow 계산 그래프 표현)으로 되돌아온 다음 학습이나 추론을 수행한다.

# 0x5. 정리

이 글에서는 MLIR의 Pass 메커니즘 실습을 소개했으며, OneFlow Dialect에서는 이미 많은 자주 사용되는 Fuse Op을 구현했고 MLIR을 사용해 Pattern Match 및 Rewrite를 수행한다. 이를 통해 사용자가 어떠한 코드 수정도 하지 않은 상태에서도 자연스럽게 계산 그래프를 가속하고 메모리(VRAM)를 절약할 수 있다. 이 부분에 관심이 있다면, OneFlow 저장소에서 직접 살펴볼 수 있다.

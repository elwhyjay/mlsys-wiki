# 알고리즘에 최적화 여지가 있는지 어떻게 판단할까?

> ❝
>
> Armv7a 아키텍처의 theoretical GFLOPS와 직접 작성한 어떤 알고리즘의 GFLOPS를 계산하는 방법을 다룬다. 추가로 native 버전 matmul의 각 사이즈별 GFLOPS를 시각화해 주는 스크립트도 함께 제공한다.
>
> ❞

# 1. 머리말

지금까지는 알고리즘을 어떻게 최적화할지에 대해, 알고리즘 로직 차원이나 더 낮은 레벨까지 다룬 글을 주로 써 왔다. 그러나 측정 작업은 비교적 대충 처리해 왔다. 즉, 단순히 실행 시간을 거칠게 비교하는 수준이었다. 최근에 matmul 최적화를 학습하려고 준비하다 보니, 이런 비교 방식으로는 사실 많은 정보를 얻기 어렵다는 생각이 들었다. 예를 들어 현재 버전의 알고리즘이 특정 하드웨어에서 아직 최적화 여지가 남아 있는지 알 수 없다. 그래서 이번 글에서는 알고리즘 가속을 평가하는 또 다른 기준, 즉 알고리즘의 floating-point peak performance(GFLOPS)를 소개하려 한다.

> ❝
>
> GFLOPS는 계산량을 소요 시간으로 나눈 값을 의미한다.
>
> ❞

이전에 Gao(고숙숙) 선배가 하드웨어의 floating-point peak performance를 어떻게 계산하는지 알려 주는 글을 올렸는데(`https://zhuanlan.zhihu.com/p/28226956`), 그분의 오픈소스 코드는 x86 아키텍처를 대상으로 한 것이었다. 그래서 나는 모바일(ArmV7-a 아키텍처)을 대상으로 이를 흉내 내어, 하드웨어의 floating-point peak performance를 측정한 뒤, 직접 작성한 native 버전 matmul의 GFLOPS를 계산하여, 현재 버전의 알고리즘이 하드웨어의 theoretical peak performance까지 얼마만큼 최적화 여지가 남아 있는지 판단해 보았다.

# 2. Cortex-A17 하드웨어 floating-point peak performance 계산

자세한 원리는 다음을 참고하자: [floating-point peak performance에 관하여](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247490528&idx=1&sn=6e2be12bd5e64b5f8d1c016c6d7998d7&scene=21#wechat_redirect>) . 여기서는 floating-point peak performance를 계산하는 방법 부분만 다시 가져와 본다:

```text
Haswell 아키텍처에는 AVX2와 FMA 명령어가 추가되었다. 여기서 FMA(Fused Multiply-Add)는 256비트 융합 곱셈-덧셈 명령어이다. 마이크로아키텍처 다이어그램을 보면 port 0과 port 1에 각각 하나의 FMA 유닛이 존재하므로, Haswell의 이론적 피크 성능은 다음과 같이 계산할 수 있다.

2port × 8 × 2(mul + add) × 클럭 주파수 × 코어 수

예를 들어 최상위 i7 모델인 4790K를 생각해 보자. 이 프로세서는 4코어를 가지며, 터보 부스트를 비활성화하고 클럭을 4GHz로 고정했을 때의 이론적 피크 성능은 다음과 같다.

2 × 8 × 2 × 4 × 4 = 512 GFLOPS

즉, 코어당 피크 성능은 128 GFLOPS이다.

Haswell의 이론적 피크 성능을 측정하는 방법은 SNB(Sandy Bridge)와 기본적으로 동일하다. 다만 사용하는 명령어가 다를 뿐 아니라 몇 가지 차이점이 존재한다. 예를 들어 FMA 명령어는 세 개의 피연산자를 사용한다. 두 개의 소스 레지스터에서 데이터를 읽고, 결과를 별도의 목적지 레지스터에 기록한다. 따라서 이전의 두 피연산자 방식에서 존재했던 인접 반복(iteration) 간의 RAW(Read After Write) 의존성이 제거된다.

하지만 각 반복이 여전히 한 사이클마다 수행되며 두 개의 FMA 명령어를 발행(issue)한다고 가정하면, 반복 간 데이터 의존성이 완전히 사라지는 것은 아니다. 현재 반복의 두 FMA가 완료되어야 다음 반복의 두 FMA를 실행할 수 있기 때문이다. 이로 인해 파이프라인에 stall이 발생하고 실행 사이클이 낭비된다.

FMA 명령어의 지연 시간(latency)을 k 사이클이라고 하면, 이를 해결하는 방법은 한 번의 반복에서 서로 의존성이 없는 2k개의 FMA 명령어를 배치하는 것이다. 이렇게 하면 데이터 의존성으로 인한 대기를 피할 수 있다.

실험 결과, Haswell에서는 10개의 독립적인 FMA 명령어를 배치하면 파이프라인을 완전히 채울 수 있으며, 부동소수점 연산 성능이 이론적 최대치에 도달하는 것으로 확인되었다.
```

이 방법을 참고하면 모바일 단에서 floating-point peak performance를 측정할 수 있다. 먼저 테스트의 핵심 코드를 다음과 같이 작성한다. GFLOPS 계산 방법은 계산량을 프로그램 소요 시간으로 나누는 것임에 유의하자:


    #include <time.h>
    #include <stdio.h>

    #define LOOP (1e9)
    #define OP_FLOATS (80)

    void TEST(int);

    static double get_time(struct timespec *start,
                           struct timespec *end) {
        return end->tv_sec - start->tv_sec + (end->tv_nsec - start->tv_nsec) * 1e-9;
    }



    int main() {
        struct timespec start, end;
        double time_used = 0.0;

        clock_gettime(CLOCK_MONOTONIC_RAW, &start);

        TEST(LOOP);

        clock_gettime(CLOCK_MONOTONIC_RAW, &end);

        time_used = get_time(&start, &end);
        printf("perf: %.6lf \r\n", LOOP * OP_FLOATS * 1.0 * 1e-9 / time_used);
    }


여기서 TEST는 순수 어셈블리로 구현했다. 즉 test.S 파일이며, 코드는 다음과 같다. 한 번의 루프에서 왜 10개의 `vmla.f32` 명령을 발사하는지는 위에서 가져온 계산 방법 부분에 명확히 설명되어 있다. 이 부분은 직접 여러 값을 시도해 보면서 하드웨어 FLOPs를 더 정밀하게 얻을 수도 있다:


    .text
    .align 5

    .global TEST

    TEST:

    .loop2:
        vmla.f32 q0, q0, q0
        vmla.f32 q1, q1, q1
        vmla.f32 q2, q2, q2
        vmla.f32 q3, q3, q3
        vmla.f32 q4, q4, q4
        vmla.f32 q5, q5, q5
        vmla.f32 q6, q6, q6
        vmla.f32 q7, q7, q7
        vmla.f32 q8, q8, q8
        vmla.f32 q9, q9, q9

        subs r0,r0,    #1
        bne .loop2


Cortex-A17 위에서 single core의 floating-point peak performance를 측정했다. 결과는 다음과 같다:

# 3. Native matmul 구현 및 floating-point peak performance 기록

다음으로 `https://github.com/flame/how-to-optimize-gemm` 를 참고하여 native 버전 matmul을 구현한다. 즉 A 행렬의 한 행과 B 행렬의 한 열을 곱해 C 행렬의 한 원소를 얻는 방식이다(계산량은 `2 * M * N * K`). 그리고 그 실행 시간을 측정하여 GFLOPS를 계산한다. 또한 matmul의 GFLOPS와 행렬 크기 사이의 관계를 파악하기 위해, 각 사이즈에 대한 matmul의 GFLOPS를 `txt` 파일에 기록한 뒤, Python의 matplotlib 라이브러리를 사용해 이 데이터를 하나의 그래프에 그려 보여 주려 한다. 먼저 사이즈별 matmul을 구현한다:


    #define A( i, j ) a[ (i)*lda + (j) ]
    #define B( i, j ) b[ (i)*ldb + (j) ]
    #define C( i, j ) c[ (i)*ldb + (j) ]
    // gemm C = A * B + C
    void MatrixMultiply(int m, int n, int k, float *a, int lda, float *b, int ldb, float *c, int ldc)
    {
        for(int i = 0; i < m; i++){
            for (int j=0; j<n; j++ ){
                for (int p=0; p<k; p++ ){
                    C(i, j) = C(i, j) + A(i, p) * B(p, j);
                }
            }
        }
    }


측정 및 FLOPs 집계 부분의 코드는 다소 길어서 핵심 부분만 붙여 둔다. 전체 코드는 github 에서 받을 수 있다(`https://github.com/BBuf/ArmNeonOptimization/tree/master/optimize_gemm`):


    // i는 현재 행렬의 길이/너비를 나타내며, 길이와 너비가 모두 i와 같다
    for(int i = 40; i <= 800; i += 40){
           m = i;
           n = i;
           k = i;
           gflops = 2.0 * m * n * k * 1.0e-09;
           lda = m;
           ldb = n;
           ldc = k;
           a = (float *)malloc(lda * k * sizeof(float));
           b = (float *)malloc(ldb * n * sizeof(float));
           c = (float *)malloc(ldc * n * sizeof(float));
           prec = (float *)malloc(ldc * n * sizeof(float));
           nowc = (float *)malloc(ldc * n * sizeof(float));
           // 행렬을 무작위 값으로 채움
           random_matrix(m, k, a, lda);
           random_matrix(k, n, b, ldb);
           random_matrix(m, n, prec, ldc);

           memset(prec, 0, ldc * n * sizeof(float));

           copy_matrix(m, n, prec, ldc, nowc, ldc);

           // nowc를 기준으로, 행렬 연산 결과가 올바른지 판단
           MatrixMultiply(m, n, k, a, lda, b, ldb, nowc, ldc);

           // 20회 반복하여, 가장 빠른 실행 시간을 결과로 사용
           for(int j=0; j < 20; j++){

               copy_matrix(m, n, prec, ldc, c, ldc);

               clock_gettime(CLOCK_MONOTONIC_RAW, &start);

               MatrixMultiply(m, n, k, a, lda, b, ldb, c, ldc);

               clock_gettime(CLOCK_MONOTONIC_RAW, &end);

               time_tmp = get_time(&start, &end);

               if(j == 0)
                   time_best = time_tmp;
               else
                   time_best = min(time_best, time_tmp);
           }

           diff = compare_matrices(m, n, c, ldc, nowc, ldc);

           if(diff > 0.5f || diff < -0.5f){
               exit(0);
           }

           printf("%d %le %le \n", i, gflops / time_best, diff);
           fflush(stdout);

           free(a);
           free(b);
           free(c);
           free(prec);
           free(nowc);
    }
    printf("\n");
    fflush(stdout);


**「컴파일 후 실행할 때 리다이렉트 명령을 하나 추가하기만 하면, 행렬 크기와 GFLOPS가 기록된 txt 파일을 얻을 수 있다. 예: `./unit_test >> now.txt`. 단, now.txt는 사전에 직접 생성해 두어야 하며, 쓰기 권한이 있는지 확인해야 한다.」**

이어서 다음 스크립트를 사용하여 `now.txt`를 이미지로 시각화하고, 이미지를 로컬에 저장한다:


    import matplotlib.pyplot as plt
    import numpy as np

    def solve(filename):
        f = open(filename)
        sizes = [40]
        times = [0.0]
        title = 'origin'
        while True:
            line = f.readline()
            if line:
                slices = line.split(" ")
                if len(slices) <= 2:
                    break;
                size = int(slices[0])
                time = float(slices[1])
                sizes.append(size)
                times.append(time)
        return title, sizes, times

    if __name__ == '__main__':
        plt.xlabel('size')
        plt.ylabel('gflops')
        t, x, y = solve('now.txt')
        plt.plot(x, y, label=t)
        plt.legend()
        plt.savefig('origin.png')
        plt.show()


결과를 살펴보자:

![이미지](images/img_03.png) Native 버전 matmul의 GFLOPS

이 그래프를 보면, 행렬의 길이/너비가 100일 때 최고 GFLOPS인 약 0.25 GFLOPS에 도달함을 알 수 있다. 하드웨어의 theoretical floating-point peak performance에 비하면 2~3% 수준에 불과하므로, 이 알고리즘의 최적화 여지는 여전히 매우 크다. 다음으로 곱셈 횟수 줄이기, memory alignment, blocking 등의 전략을 적용해 이 알고리즘을 개선하고 더 나은 GFLOPS를 얻어 갈 수 있다. 이렇게 하면 알고리즘을 최적화하는 과정에서 알고리즘의 성능을 더 직관적으로 볼 수 있게 된다.

# 4. 정리

이 글은 matmul 최적화 시리즈의 시작 부분일 뿐이다. Gao 선배의 글에서 영감을 받아, 모바일 단 혹은 PC 단에 관심 있는 분들에게 GFLOPS 계산 예시 하나를 제공하고, GFLOPS를 더 직관적으로 보여 주는 스크립트 도구를 함께 제공하는 데 목적이 있다. 도움이 되었으면 한다.

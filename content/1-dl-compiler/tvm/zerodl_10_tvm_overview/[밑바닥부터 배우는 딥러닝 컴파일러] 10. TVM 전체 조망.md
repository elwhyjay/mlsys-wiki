# 0x1. TVM의 전체 아키텍처

먼저 TVM의 전체 아키텍처 그림을 본 다음, 이 그림을 따라가며 설명하겠다.

![이미지](images/img_02.png)TVM의 전체 아키텍처 그림

가장 위쪽의 Model from Frameworks와 Import(frontend)는 앞서 다루었던 내용으로, 각종 딥러닝 프레임워크의 계산 그래프를 Relay IR로 변환하는 부분이다. 즉 [【从零开始学TVM】三，基于ONNX模型结构了解TVM的前端 ](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247494189&idx=1&sn=a0f646dac459d3a47019f6f9bd0db545&scene=21#wechat_redirect>)에서 다루었던 내용이다. 그리고 TVM의 IR은 두 계층으로 나뉘는데, 위쪽은 frontend를 향한 Relay IR이고, 아래쪽은 LLVM을 향한 저수준 IR(Tir이라고도 부른다)이다.

TVM은 설계상 Relay IR과 Tir 두 계층을 추상화하고 있지만, 구현상으로는 둘 다 Object 메타클래스를 통해 통일된 AST Node 표현을 구현하고, IRModule을 통해 위아래 계층의 IR 표현을 관통한다. 이 점은 `include/tvm/ir/module.h`의 IRModuleNode 정의에서 확인할 수 있다.

![이미지](images/img_03.png)IRModuleNode는 Object 기반 클래스를 상속한다

이전에 [【从零开始学深度学习编译器】七，万字长文入门TVM Pass](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247494923&idx=1&sn=0cdde2ecdd1cee546b0847d03cc40b2c&scene=21#wechat_redirect>)에서 Relay AST 트리 구조를 설명할 때 언급했듯이, `include/tvm/relay/expr.h`에는 Relay 표현식 트리의 노드들이 정의되어 있다. `ConstantNode`, `VarNode`, `TupleNode`, `CallNode`, `LetNode`, `IfNode` 등이 그것이다. 이 노드들은 실제로 모두 통일된 노드 표현 클래스인 `ExprNode`를 상속하며, 이 클래스 또한 Object 클래스에서 파생되었다. 따라서 TVM이 두 계층의 IR을 추상화했다고 하더라도, 실제 구현에서는 공통점이 있고, 완전히 별개로 존재하는 두 종류의 IR이 아니다.

다음으로 아키텍처 그림에 있는 핵심 구성 요소들을 간단히 소개하겠다.

  * 이 그래프 구조에는 두 가지 **「핵심 노드」** 가 있다. 하나는 `Import`로, TVM의 frontend 컴포넌트가 각종 딥러닝 프레임워크의 모델을 TVM의 `IRModule`로 가져오는 역할을 한다. 다른 하나는 `Transform`으로, 하나의 `IRModule`을 기능이 동일한 다른 `IRModule`로 등가 변환하여 최적화 효과를 얻는다. 이 변환은 하드웨어와 무관한 것과 하드웨어에 관련된 것 두 종류로 나뉜다.

  * `IRModule`: function들의 집합이며, 그 중 가장 핵심적인 두 종류의 Function 집합인 `relay::Function`과 `tir::PrimFuc`을 포함한다.

  * 위쪽의 `relay::Function`은 `BaseFunction`을 상속하며, end2end 모델에 대응한다. 제어 흐름, 재귀, 복잡한 자료구조를 지원하는 계산 그래프로 이해하면 된다.

  * 아래쪽의 `tir::PrimFunc`도 `BaseFunction`을 상속하며, 저수준의 threading, vector/tensor 명령어들을 포함한다. 일반적으로 모델 내 한 개의 op 실행 단위에 해당한다.

  * `Target Translation`: 컴파일러가 IRModule을 타깃 하드웨어에서 실행 가능한 형식으로 변환하는 과정(즉 코드 생성)이며, 생성된 코드는 runtime으로 캡슐화된다.

  * `Passes`: pass는 계산 그래프에 대한 일련의 최적화와 변환이다. 예를 들어 상수 폴딩(constant folding), operator fusion, 데드 코드 제거 등이 있다.

  * 컴파일 단계에서 하나의 `relay::Function`은 여러 개의 `tir::PrimFunc`로 lowering될 수 있다.




다음 예시를 통해 `IRModule`이 어떻게 생겼는지 직관적으로 살펴보자.
    
    
    import tvm  
    from tvm import relay  
    import numpy as np  
    from tvm.contrib import graph_executor  
      
    # 모델 빌드  
    n = 2  
    x = relay.var("x", shape=(n,), dtype='float32')  
    y = relay.nn.softmax(x)  
    net = relay.Function([x], y)  
      
    # 빌드 및 lowering  
    module = tvm.IRModule.from_expr(net)  
    lib = relay.build(module, "llvm")  
      
      
    dev = tvm.cpu(0)  
    input = tvm.nd.array(np.random.uniform(size=[n]).astype('float32'), dev)  
    m = graph_executor.GraphModule(lib["default"](dev))  
    # 입력 설정  
    m.set_input("x", input)  
    # 실행  
    m.run()  
    # 출력 가져오기  
    tvm_output = m.get_output(0)  
      
    print(module)  
    # def @main(%x: Tensor[(2), float32]) {  
    #   nn.softmax(%x)  
    # }

## Relay Transform

Relay의 Transform은 앞서 소개한 TVM의 하드웨어 무관 Pass들이다. 예를 들어 일반적인 constant folding, dead-code elimination 그리고 tensor 연산과 관련된 특수한 Pass들, 예컨대 transformation, scaling factor folding 등이 있다.

Relay 최적화 pipeline 후반부에는 FuseOps Pass가 실행되어, MobileNet과 같은 end-to-end function을 conv2d-relu와 같은 sub-function 단위로 잘라낸다. 이는 sub-graph 분할이라고 부르는 것이 더 적절할 것 같다.

하드웨어 무관 Pass를 모두 실행한 뒤, TVM은 `relay::Function`을 여러 개의 `tir::PrimFunc`로 lowering한 다음, 각 `tir::PrimFunc`에 대해 컴파일과 최적화를 수행한다. 일부 특수한 backend의 경우 Tir을 거치지 않고 외부 코드 생성기로 타깃 backend 코드를 직접 생성할 수도 있다.

Relay/Backend 계층에서 실행 구조를 호출할 때는 몇 가지 경우로 나뉜다.

  * 모델이 정적 shape이고 제어 흐름이 없으면 graph runtime으로 lowering한다.
  * 모델이 동적 shape이고 제어 흐름이 있으면 **「virtual machine backend」** 를 사용할 수 있다.
  * 향후 sub-graph 수준의 프로그램을 executable and generated primitive functions 수준으로 직접 변환하는 것을 지원할 계획이다.



위의 모든 모드는 통일된 인터페이스인 `runtime.Module`로 캡슐화된다.

## Tir Transform

Tir Transform은 주로 Tir 수준의 다양한 Pass들을 포함한다. 이 부분의 주요 기능은 lowering이지만 일부 optimization도 있다. 예를 들어 다차원 데이터 접근을 1차원 포인터 접근으로 평탄화하거나, 특정 backend에 대해 intrinsic을 확장하거나, runtime 호출 규약에 따라 함수를 데코레이팅하는(이후 호출을 용이하게 함) 등이 있다. 주의할 점은 이 단계에서는 일부 저수준 최적화는 수행하지 않고 후단의 LLVM이나 CUDA C 컴파일러에 맡긴다는 것이다. 예를 들어 레지스터 할당 등이 그렇다.

## Relay의 Pass는 어떻게 구현되어 있는가

TVM의 Pass는 AST를 순회하며 node를 수정하는 방식으로 구현되며, `TVM_REGISTER_GLOBAL` 매크로를 통해 지원되는 Pass를 등록하고 노출한다. 자세한 소개는 [【从零开始学深度学习编译器】八，TVM的算符融合以及如何使用TVM Pass Infra自定义Pass](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247495225&idx=1&sn=5819431f3b1bca6687cb171bd4243bb6&scene=21#wechat_redirect>)를 참고하면 된다. 개발자 입장에서는 직접 Pass를 추가하는 방법에 더 관심이 있을 텐데, TVM 공식 문서에는 constant folding 예시가 제시되어 있다. TVM의 IR이 AST와 비슷하기 때문에, TVM에 새로운 Pass를 추가하려면 다음과 같은 단계가 필요하다.

  * 새로운 `AST Traversers`를 추가하여 어떤 node가 수정되어야 하는지 판단한다. constant folding의 경우 `ConstantChecker`가 구현되어 있고, map 구조의 `memo_`를 통해 어떤 node가 상수 node인지 기록한다. 이 Pass는 두 종류의 node에 대한 함수 오버로드만 설계되어 있는데, 바로 ConstantNode와 TupleNode이다.
  * 새로운 `Expression Mutators`를 추가하여 조건을 만족하는 node를 수정하고 교체한다. constant folding Pass에서는 `LetNode`, `TupleItemNode`, `CallNode` 세 종류의 node만 폴딩 대상이므로, 이 세 node에 대한 함수를 오버로드해야 한다.



TVM의 Pass 설계 사상과 아키텍처는 [【从零开始学深度学习编译器】七，万字长文入门TVM Pass ](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247494923&idx=1&sn=0cdde2ecdd1cee546b0847d03cc40b2c&scene=21#wechat_redirect>)에서 다루었으며, 그 글에서는 TVM Pass Infrastructure 문서도 함께 번역했다. TVM Pass는 전체적으로 LLVM Pass의 설계 사상을 많이 계승했으며, 다음과 같은 효과를 목표로 한다.

  * pass를 유연하게 배치할 수 있어 사용자가 자유롭게 pass pipeline을 커스터마이즈할 수 있다.
  * 친숙한 pass debug 경험을 제공한다.
  * 사용자가 pass 사이의 의존 관계를 수동으로 처리할 필요가 없도록 한다.
  * 개발자가 새로운 pass를 추가하는 절차를 단순화하고, Python 측에서 Pass를 커스터마이즈할 수 있도록 지원한다.



TVM의 Pass는 구현상 `Module Level Pass`, `Function Level Pass`, `Sequential Level Pass`로 나눌 수 있다. 그 중 `Module Level Pass`는 전역 정보를 기반으로 최적화하며 `Function`을 삭제할 수 있다(예: defuse_ops pass). `Function Level Pass`는 Module 안의 각 Function에 대해 최적화를 수행하며 지역 정보만을 가진다. 예를 들어 공통 부분식 제거가 있다. `Sequential Pass`는 일련의 Pass를 순차적으로 실행한다.

## AutoTVM

위에서 소개한 Pass들은 모두 결정론적인 변환이며, 이는 어떤 종류의 Pass든 IR lowering이든 마찬가지이다. TVM 설계 목표 중 하나는 다양한 플랫폼에 대해 고성능 텐서화 코드를 생성하는 것을 지원하는 것이며, 그래서 TVM은 scheduler가 형성한 탐색 공간 안에서 머신 러닝 알고리즘을 사용해 최적의 선택을 찾는 방안을 채택했다.

TVM에는 많은 전형적인 scheduler가 있는데, 관심 있다면 https://zhuanlan.zhihu.com/p/94846767 을 참고하면 된다. 이러한 작은 단위의 scheduler들을 기반으로, TVM은 프로그램 변환에 사용할 수 있는 연산들의 집합을 정의했고, 이를 schedule primitive(scheduling primitives)라고 부른다. 예를 들어 loop 변환(split, unroll...), inlining, vectorization 등이다. 이 단계에서는 동시에 target의 개념도 도입되어, target의 일부 특성을 고려하여 탐색을 수행한다. 예를 들어 target의 register가 동시에 처리할 수 있는 데이터 개수가 이 부분 코드의 vectorization 동작을 결정한다.

다음으로는 프로그램을 반복 실행하면서 성능을 기록하고 schedule 선택을 조정하며, 마지막으로 프로그램을 실행할 때 어떤 sub-graph 프로그램에 대해서는 log 파일에서 최적의 성능에 해당하는 schedule 방안을 선택해 실행하게 된다.

다만 TVM의 탐색 시간은 매우 길고, 일부 모델에서는 반드시 좋은 효과를 내는 것도 아니다. 이후 학자들은 Ansor를 제안하여 탐색 공간이 부족하고 탐색 시간이 너무 긴 문제를 완화시켰다. 관심 있는 분은 필자가 Ansor를 번역하고 조사한 글 [Ansor论文阅读笔记&&论文翻译](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247495841&idx=1&sn=419568fd8000487f11e3c94daf28a0d7&scene=21#wechat_redirect>)을 참고하면 된다.

## Target Translation

여기서는 Tir을 타깃 backend에 대응하는 실행 파일로 한 단계 더 변환한다. X86과 Arm CPU의 경우, TVM은 LLVM IR Builder를 사용해 메모리 안에서 LLVM IR을 구성한다. 또한 소스 코드 수준의 언어, 예를 들어 CUDA C나 OpenCL 소스 코드를 생성할 수도 있다. 그 외에 Relay Function에서 특정 backend로 직접 codegen하는 것도 지원한다(이 부분은 아직 사용해 본 적이 없으니, 관심 있다면 TVM 소스 코드의 Codegen 부분이 어떻게 구현되어 있는지 살펴보면 된다).

## Runtime Execution

Runtime 단계에서는 주로 `runtime.Module`, `runtime.PackedFunc`, `runtime.NDArray`라는 세 가지 핵심 요소가 있다. 그 중 `runtime.NDArray`는 실행 시 Tensor 구조를 캡슐화하고, `runtime.PackedFunc`는 backend가 생성한 함수로 딥러닝 프레임워크의 kernel 함수와 유사하며, `runtime.Module`은 컴파일된 DSO를 캡슐화하는 핵심 단위로 많은 `PackedFunc`를 포함하며 `name`을 통해 가져올 수 있다.

위에서 IRModule이 어떻게 생겼는지 보여주는 예시에서, `lib["default"]`로 이 `runtime.Module`의 기본 kernel 함수를 로드한 다음, TVM의 `graph runtime` 메커니즘을 기반으로 이 kernel 함수를 실행하여 추론 결과를 얻었다.

# 0x2. TVM의 Executor

위 마지막 부분에서 TVM의 `graph runtime`을 언급했는데, 마침 이 자리에서 TVM의 Executor를 소개하겠다. `tvm/python/tvm/relay/build_module.py` 파일에서 TVM 실행 시점의 몇 가지 모드를 확인할 수 있다.
    
    
    def create_executor(kind="debug", mod=None, device=None, target="llvm"):  
        """Factory function to create an executor.  
      
        Example  
        -------  
        .. code-block:: python  
      
            import tvm.relay  
            import numpy as np  
      
            x = tvm.relay.var("x", tvm.relay.TensorType([1], dtype="float32"))  
            expr = tvm.relay.add(x, tvm.relay.Constant(tvm.nd.array(np.array([1], dtype="float32"))))  
            tvm.relay.create_executor(  
                kind="vm", mod=tvm.IRModule.from_expr(tvm.relay.Function([x], expr))  
            ).evaluate()(np.array([2], dtype="float32"))  
            # returns `array([3.], dtype=float32)`  
      
        Parameters  
        ----------  
        kind : str  
            The type of executor. Avaliable options are `debug` for the  
            interpreter, `graph` for the graph executor, and `vm` for the virtual  
            machine.  
      
        mod : :py:class:`~tvm.IRModule`  
            The Relay module containing collection of functions  
      
        device : :py:class:`Device`  
            The device to execute the code.  
      
        target : :py:class:`tvm.Target`  
            The corresponding context  
      
        Returns  
        -------  
        executor : :py:class:`~tvm.relay.backend.interpreter.Executor`  
        """  
        if mod is None:  
            mod = IRModule()  
        if device is not None:  
            assert device.device_type == _nd.device(str(target), 0).device_type  
        else:  
            device = _nd.device(str(target), 0)  
      
        if isinstance(target, str):  
            target = Target(target)  
        if kind == "debug":  
            return _interpreter.Interpreter(mod, device, target)  
        if kind == "graph":  
            return GraphExecutor(mod, device, target)  
        if kind == "vm":  
            return VMExecutor(mod, device, target)  
        raise RuntimeError("unknown execution strategy: {0}".format(kind))  
    

위 코드를 통해 알 수 있듯이, Relay의 interpreter는 Relay의 표현식을 실행할 수 있지만 일반적으로 debug 용도로만 사용된다. 이는 interpreter가 AST를 순회하며 프로그램을 실행하는데, 이 순회 과정이 매우 비효율적이기 때문이다. 또한 interpreter 기반 실행 모드는 Dynamic Code를 친숙하게 지원하지 못한다. 예를 들어 동적 scheduler, 동적 Tensor shape, 그리고 제어 흐름 등이 그렇다.

실행 효율을 고려하여 TVM은 graph runtime 기술을 도입했고, 빠른 실행 메커니즘을 제공하지만 일부 Relay 프로그램만 지원한다. 내가 이해하기로는 정적 Tensor Shape이며 제어 흐름이 없는 프로그램을 가리킨다.

동적 Tensor shape과 제어 흐름 문제를 해결하기 위해 TVM은 Relay Virtual Machine Executor라는 한 계층을 더 추상화했다. 구체적인 설명은 https://zhuanlan.zhihu.com/p/354995641 을 참고하면 된다. 다만 TVM의 동적 Tensor shape 지원도 일부 전형적인 추론 시나리오만 지원할 뿐이며, 학습은 지원하지 않는다.

# 0x3. TVM의 컴포넌트

TVM에 있는 일부 기본 컴포넌트들의 상호작용 그림은 다음과 같다.

![이미지](images/img_04.png)TVM의 컴포넌트 상호작용 그림

아래에서 각 컴포넌트를 차례로 소개하겠다.

  * Support: TVM의 일부 범용 컴포넌트로 socketing, logging 등이 있다.
  * Frontends: TVM의 frontend로, 각종 딥러닝 프레임워크의 계산 그래프를 Relay IR로 변환하는 것을 담당한다.
  * Relay: high-level 계산 그래프 기술이며, 자체 IR 표현을 가지고 이 IR로 신경망 구조를 기술한다. 위에서 소개한 일부 하드웨어 무관 Pass들도 Relay IR을 기반으로 수행된다.
  * AutoTVM/Ansor: 선택적 자동 튜닝 모듈이다.
  * Topi: Tensor 계산 라이브러리로, 신경망에서 자주 쓰이는 많은 op들이 포함되어 있다. 예를 들어 행렬 곱셈, 점곱, 합성곱 등이다. 이 topi op들을 통해 계산을 수행할 수 있다.
  * Tir: Relay IR에 비해 이 계층의 IR은 저수준과 하드웨어 구현에 더 가깝다.
  * Te: Te는 Tensor Expression을 나타내며, 사용자는 te 안의 함수를 호출하여 Tir을 구성할 수 있다(이는 TVM이 사용자에게 Te를 통해 직접 신경망을 작성할 수 있도록 허용한다는 뜻이다).
  * Node: node는 IR을 기반으로 일부 새로운 특성을 추가한 것으로, 사용자가 일부 함수에 접근할 수 있게 해 준다. 이를 통해 더 복잡한 표현 함수를 구현할 수 있다.
  * Driver: 하드웨어 기반 driver이다.
  * Arith: Tir과 관련이 있으며, Tir 최적화 시 일부 분석을 수행할 수 있게 해 준다.
  * Runtime: runtime은 그래프 구조 변환, 최적화, 코드 생성, 그리고 프로그램의 하드웨어 상 실행을 캡슐화하여, 사용자에게 모든 컴파일 과정을 완료할 수 있는 API 인터페이스를 제공한다.



이 컴포넌트 상호작용 그림에서 화살표에 주의해야 한다. **「점선 화살표는 컴포넌트 사이에 구현 수준의 의존이 있음을 나타낸다. 예를 들어 Tir pass는 구현상 반드시 arith에 의존하여 분석을 수행해야 한다. 반면 실선 화살표는 두 컴포넌트가 frontend 수준에서 의존 관계를 가짐을 나타낸다」**. 예를 들어 Relay IR은 Tir에 의존하는데, 이는 앞서 말했듯이 Relay IR이 Tir로 lowering되기 때문이다.

frontend와 backend의 상호작용 측면에서, TVM은 모든 핵심 자료구조를 Python frontend에 노출시켜 충분한 유연성과 사용 편의성을 갖추도록 했다. 구체적으로 다음과 같은 점에서 드러난다.

  * 모든 핵심 객체는 Python API를 통해 직접 생성하고 조작할 수 있다. 예를 들어 IR Module이 그렇다.
  * frontend에서 Pass를 자유롭게 조합하여 커스터마이즈하는 것을 지원한다.
  * TVM의 API를 통해 IR을 직접 조작할 수 있어, Python 측에서 Pass를 작성하는 것을 지원한다.

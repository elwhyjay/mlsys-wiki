# [밑바닥부터 배우는 딥러닝 컴파일러] 9. TVM의 CodeGen 흐름

이 글에서는 TVM의 Codegen 흐름을 살펴보면서, Relay frontend부터 시작해 Graph 노드의 메모리 할당, Relay IR에서 TIR 노드로의 변환, TIR 그래프 노드의 Schedule 최적화, 그리고 Lower function이 어디서 일어나는지까지 정리해 보았다. 이 글은 호출 체인에 초점을 맞추었으며, Schedule 최적화, IR에서 TIR 노드로의 변환, Lower Function 같은 구체적인 동작은 자세히 설명하지 않았다. 이후에 더 많은 예시를 통해 이해해 볼 예정이다.

# 0x0. 소개

이 글에서는 주로 TVM의 CodeGen 흐름을 소개한다. TVM의 자동 코드 생성 인터페이스는 `tvm.build`와 `tvm.relay.build`인데, `tvm.build`는 op의 코드 생성에 쓰이고, `tvm.relay.build`는 Relay 계산 그래프의 자동 코드 생성에 쓰인다(여기서 코드 생성은 컴파일 흐름까지 포함한다). 이어서 이 두 함수에서 출발하여 TVM Codegen의 구체적인 구현까지 살펴보겠다. 이 글을 읽기 전에 TVM의 컴파일 흐름을 먼저 파악해 두기를 권장한다. # 0x1. 생성된 코드 확인 방법

Relay에서 생성된 코드를 확인하는 예시는 다음과 같다.
    
    
    from tvm import relay  
    from tvm.relay import testing  
    import tvm  
      
    # Resnet18 workload  
    resnet18_mod, resnet18_params = relay.testing.resnet.get_workload(num_layers=18)  
      
      
    with relay.build_config(opt_level=0):  
        graph, lib, params = relay.build_module.build(resnet18_mod, "llvm", params=resnet18_params)  
      
    # print relay ir  
    print(resnet18_mod.astext(show_meta_data=False))  
      
    # print source code  
    print(lib.get_source())  
      
    

TVM은 runtime Module에 대해 `get_source`를 제공해 생성된 코드를 확인할 수 있게 해주며, 동시에 IRModule의 `astext` 함수를 통해 IR의 중간 표현을 확인할 수 있다. 여기서 만들어지는 결과는 지정된 디바이스(CPU)에서 실행 가능한 머신 코드이므로 가독성이 떨어져 따로 첨부하지는 않는다.

op 단위 자동 코드 생성 예시를 통해 TVM이 생성하는 코드가 어떤 모습인지 직관적으로 느껴볼 수 있다. `tvm.build` 인터페이스에서 `target`을 `c`로 설정하면 C 언어 코드가 생성되기 때문이다. 예시는 다음과 같다.
    
    
    import tvm  
    from tvm import te  
      
    M = 1024  
    K = 1024  
    N = 1024  
      
    # Algorithm  
    k = te.reduce_axis((0, K), 'k')  
    A = te.placeholder((M, K), name='A')  
    B = te.placeholder((K, N), name='B')  
    C = te.compute(  
               (M, N),  
               lambda x, y: te.sum(A[x, k] * B[k, y], axis=k),  
               name='C')  
      
    # Default schedule  
    s = te.create_schedule(C.op)  
    ir_m = tvm.lower(s, [A, B, C], simple_mode=True,name='mmult')  
    rt_m = tvm.build(ir_m, [A, B, C], target='c', name='mmult')  
      
    # print tir  
    print("tir:\n", ir_m.astext(show_meta_data=False))  
    # print source code  
    print("source code:\n",rt_m.get_source())

TVM의 코드 생성 인터페이스(`tvm.build`와 `tvm.relay.build`)를 직관적으로 살펴봤으니, 이제 `https://zhuanlan.zhihu.com/p/139089239` 글에서 정리한 TVM 코드 생성 과정의 흐름도를 통해 더 잘 이해해 보자.

원문에서는 다음과 같이 소개하고 있다.

> TVM의 코드 생성 인터페이스는 IRModule에서 runtime module로의 변환이며, TIR 또는 Relay IR을 target 코드(예: C 또는 LLVM IR 등)로 컴파일한다. 아래 흐름도는 전체 코드의 컴파일 흐름을 나타내며, 진한 색은 C++ 코드를, 옅은 색은 Python 코드를 의미한다. op 컴파일 시에는 먼저 TIR 최적화를 수행하고 host와 device 부분을 분리한 뒤, 등록된 target.build.target 함수를 호출해 컴파일을 수행한다. Relay 그래프 컴파일은 op보다 다소 복잡하며, 핵심 코드는 C++로 개발되어 있다. relayBuildModule.Optimize를 통해 Relay 그래프를 최적화한 후, module 내 각 lower_funcs에 대해 컴파일을 진행하고 최종적인 runtime module로 합치며, 그 이후의 컴파일 흐름은 op 컴파일과 비슷하다.

![이미지](images/img_01.png)한 가지 주의할 점은, 오른쪽 그림의 relay.build_module.build가 이 글에서 말하는 tvm.relay.build에 해당한다는 것이다.

위 흐름도를 보면 TVM Codegen의 흐름을 직관적으로 이해할 수 있다. 여기서는 Relay를 예로 들어 TVM 코드 안에서 어떻게 대응되는지 간단히 살펴보겠다. 위 오른쪽 그림에는 Relay Graph의 전체 컴파일 흐름이 나타나 있다. GraphCodeGen 이전 부분은 [[밑바닥부터 배우는 딥러닝 컴파일러] 6. TVM의 컴파일 흐름 상세 설명](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247494801&idx=1&sn=b893c43133eea1343034bb0aca356e24&scene=21#wechat_redirect>) 에서 이미 다루었으므로, 여기서는 GraphCodegen 인스턴스를 생성하는 부분(`graph_codegen_ = std::unique_ptr<GraphCodegen>(new GraphCodegen());`)부터 GraphCodegen의 흐름을 정리하겠다. 해당 부분의 코드는 다음과 같다.
    
    
    void BuildRelay(IRModule relay_module,  
                      const std::unordered_map<std::string, tvm::runtime::NDArray>& params) {  
        Target target_host = GetTargetHost();  
        // If no target_host has been set, we choose a default one, which is  
        // llvm if "codegen.LLVMModuleCreate" is accessible.  
        const runtime::PackedFunc* pf = runtime::Registry::Get("codegen.LLVMModuleCreate");  
        if (!target_host.defined()) target_host = (pf != nullptr) ? Target("llvm") : Target("stackvm");  
      
        // Update all the targets in the targets_ TargetsMap  
        CheckAndUpdateHostConsistency(&targets_, &target_host);  
      
        // Relay IRModule -> IRModule optimizations.  
        relay_module = Optimize(relay_module, targets_, params);  
        // Get the updated function.  
        auto func = Downcast<Function>(relay_module->Lookup("main"));  
      
        // Generate code for the updated function.  
        graph_codegen_ = std::unique_ptr<GraphCodegen>(new GraphCodegen());  
        graph_codegen_->Init(nullptr, targets_);  
        graph_codegen_->Codegen(func);  
      
        ret_.graph_json = graph_codegen_->GetJSON();  
        ret_.params = graph_codegen_->GetParams();  
      
        auto lowered_funcs = graph_codegen_->GetIRModule();  
      
        // Generate a placeholder function that attaches linked params as its arguments.  
        if (target_host->GetAttr<Bool>("link-params").value_or(Bool(false))) {  
          CHECK(pf != nullptr) << "Unable to link-params with no target_host and no llvm codegen.";  
          auto param_ids = graph_codegen_->GetParamIds();  
          auto link_params = Map<String, tir::LinkedParam>();  
          for (auto param : ret_.params) {  
            link_params.Set(param.first, tir::LinkedParam(param_ids[param.first], param.second));  
          }  
      
          Map<String, ObjectRef> dict;  
          dict.Set(tvm::tir::attr::kLinkedParams, link_params);  
          dict.Set(tvm::attr::kGlobalSymbol, String(::tvm::runtime::symbol::tvm_lookup_linked_param));  
          DictAttrs attrs{dict};  
          auto prim = tir::PrimFunc(Array<tir::Var>(), tir::SeqStmt(Array<tir::Stmt>()), VoidType(),  
                                    Map<tir::Var, tir::Buffer>(), attrs);  
          if (lowered_funcs.find(target_host->str()) == lowered_funcs.end()) {  
            lowered_funcs.Set(target_host->str(), IRModule(Map<GlobalVar, BaseFunc>({})));  
          }  
          lowered_funcs[target_host->str()]->Add(  
              GlobalVar(::tvm::runtime::symbol::tvm_lookup_linked_param), prim);  
        }  
      
        // When there is no lowered_funcs due to reasons such as optimization.  
        if (lowered_funcs.size() == 0) {  
          if (target_host.defined() && target_host->kind->name == "llvm") {  
            // If we can decide the target is LLVM, we then create an empty LLVM module.  
            ret_.mod = (*pf)(target_host->str(), "empty_module");  
          } else {  
            // If we cannot decide the target is LLVM, we create an empty CSourceModule.  
            // The code content is initialized with ";" to prevent complaining  
            // from CSourceModuleNode::SaveToFile.  
            ret_.mod = tvm::codegen::CSourceModuleCreate(";", "", Array<String>{});  
          }  
        } else {  
          ret_.mod = tvm::build(lowered_funcs, target_host_);  
        }  
      
        auto ext_mods = graph_codegen_->GetExternalModules();  
        ret_.mod = tvm::codegen::CreateMetadataModule(ret_.params, ret_.mod, ext_mods, GetTargetHost());  
      }  
    

먼저 여기서는 `GraphCodegen` 객체를 하나 만든다. `GraphCodegen` 구조체의 정의는 `tvm/src/relay/backend/build_module.cc`의 `struct GraphCodegen` 부분에 있으며, `tvm/src/relay/backend/graph_executor_codegen.cc`의 `GraphExecutorCodegenModule`에서 Codegen과 관련된 몇 가지 함수(`init`, `codegen`, `get_graph_json`, `get_external_modules` 등)를 감싸고 있다. 예를 들어 초기화 함수 `init`의 구현은 다음과 같다.

# 0x2. Graph CodeGen 메모리 할당

이제 `GraphExecutorCodegen` 클래스의 핵심 함수인 `LoweredOutput Codegen(relay::Function func)`를 살펴보자. 이 함수는 메모리 할당, Relay IR 노드에서 TIR 노드로의 변환, TIR 그래프 노드의 schedule 최적화를 구현한다. 함수 코드와 대조하며 설명하겠다.
    
    
    LoweredOutput Codegen(relay::Function func) {  
        auto pf = GetPackedFunc("relay.backend.GraphPlanMemory");  
        storage_device_map_ = (*pf)(func);  
        // First we convert all the parameters into input nodes.  
        for (auto param : func->params) {  
          auto node_ptr = GraphInputNode::make_node_ptr(param->name_hint(), GraphAttrs());  
          var_map_[param.get()] = AddNode(node_ptr, param);  
        }  
        heads_ = VisitExpr(func->body);  
        std::ostringstream os;  
        dmlc::JSONWriter writer(&os);  
        GetJSON(&writer);  
        LoweredOutput ret;  
        ret.graph_json = os.str();  
        ret.params = std::unordered_map<std::string, std::pair<int, const tvm::runtime::NDArray>>();  
        for (auto param : params_) {  
          ret.params.emplace(std::make_pair(  
              param.first,  
              std::make_pair(static_cast<int>(param_storage_ids_[param.first]), param.second)));  
        }  
    

Graph Codegen의 첫 번째 단계는 메모리 할당, 즉 다음 두 줄의 코드가 하는 일이다.
    
    
    auto pf = GetPackedFunc("relay.backend.GraphPlanMemory");  
        storage_device_map_ = (*pf)(func);  
    

`GraphPlanMemory`의 정의를 따라가 보면, `tvm/src/relay/backend/graph_plan_memory.cc`에 다음과 같이 정의되어 있다.
    
    
    Map<Expr, Array<IntegerArray> > GraphPlanMemory(const Function& func) {  
      return StorageAllocator().Plan(func);  
    }  
    

이 부분은 주로 `StorageAllocator`와 `StorageAllocaInit` 두 클래스의 구현과 관련이 있다. `StorageAllocaInit`은 메모리 할당 정보를 담는 TokenMap을 만들고, 서로 다른 op들의 디바이스 정보를 수집하기 위한 것이다. `StorageAllocaInit`의 `GetInitTokenMap` 함수는 `func`의 노드들을 순회하며 각 노드의 디바이스 attribute를 얻는 데 쓰인다. `GetInitTokenMap` 함수의 구현은 다음과 같다.
    
    
    /*! \return The internal token map */  
      std::unordered_map<const ExprNode*, std::vector<StorageToken*> > GetInitTokenMap(  
          const Function& func) {  
        node_device_map_ = CollectDeviceInfo(func);  
        this->Run(func);  
        return std::move(token_map_);  
      }  
    

먼저 `CollectDeviceInfo` 함수를 호출해 `func`의 각 노드의 디바이스 attribute를 얻는다. 그렇다면 구체적으로 어떻게 동작할까? `tvm/src/relay/transforms/device_annotation.cc`에 있는 주석을 통해 그 아이디어를 알 수 있다.
    
    
    /*  
     * \brief Return device allocation map based on the post order traversed graph.  
     * For the following program:  
     * .. code-block:: python  
     *     x = relay.var("x")  
     *     y = relay.var("y")  
     *     add = relay.add(x, y)  
     *     sqrt = relay.sqrt(add)  
     *     log = relay.log(add)  
     *     subtract = relay.subtract(sqrt, log)  
     *     exp = relay.exp(subtract)  
     *  
     * Suppose we have annotated add, sqrt, and log with device 1, 2, and 3,  
     * respectively. The fallback/default device is 4. After Rewriting the  
     * program, we can have the following graph, where each copy op has both  
     * source and destination device type denoting which device the data should be  
     * copied from and to.  
     *  
     *         x     y  
     *          \   /  
     *          add/1  
     *          /   \  
     *       copy1  copy2  
     *         |     |  
     *      sqrt/2 log/3  
     *         |     |  
     *       copy3 copy4  
     *          \   /  
     *        subtract  
     *            |  
     *           exp  
     *  
     * To Get the device mapping of each expression, we need to propagate the  
     * device information from the copy ops. This can be done in two passes.  
     *  -Pass 1: Propagating the source device type to ops in a bottom-up way to the  
     *           ancestors until encountering another copy op. For example, this way  
     *           provides add, x, and y device types from the copy operator, `copy1`.  
     *  -Pass 2: Propagating the destination device type of "the last" copy op to the  
     *           remain nodes. For instance, this offers `subtract` and `exp` the  
     *           same device type as `copy3`.  
     */  
    

이 예시를 설명하면, add, sqrt, log 노드가 각각 1, 2, 3번 디바이스로 표시되어 있을 때 다른 노드의 디바이스 번호를 두 가지 방식으로 추론할 수 있다는 의미다.

  * 한 copy 노드에서 시작해 다음 copy를 만날 때까지 아래에서 위로 순회한다. 예를 들어 add, x, y 노드의 디바이스 번호가 copy1과 같다고 추론할 수 있다.
  * 마지막 copy 노드에서 아래로 순회하면 subtract, exp의 디바이스 번호가 copy3과 같다고 추론할 수 있다.



보다시피 `func`에서 각 노드의 device 정보를 얻는 일은 주로 `copy` op를 통해 추론한다. `copy` op는 서로 다른 디바이스 사이의 데이터 교환을 담당하는데, 이는 딥러닝 프레임워크의 `to`와 비슷하다. 그래서 `copy` 뒤에 연결된 op는 그 `copy` op와 같은 device 정보를 가지게 된다. 사실 여기서 노드의 device 정보를 얻는 것 역시 일종의 Pass이다. 알고리즘 구현 측면에서는 Pass Infra의 많은 부분을 상속받았으며, copy op에서 앞뒤로 순회하는 방식으로 copy가 아닌 노드의 디바이스 정보를 추론한다. 구체적인 코드 구현은 다음과 같다.
    
    
    static Map<Expr, Integer> GetDeviceMap(const Expr& expr) {  
        DeviceInfo device_info;  
        device_info.post_visitor_ = PostDfsOrderVisitor();  
        device_info.post_visitor_.Visit(expr);  
        if (device_info.post_visitor_.num_device_copy_ops_ > 0) {  
          device_info.PropagateDeviceId();  
          return device_info.device_map_;  
        } else {  
          return Map<Expr, Integer>();  
        }  
      }  
    

여기서는 먼저 `PostDfsOrderVisitor` 깊이 우선 순회를 실행해 `std::unordered_map<const ExprNode*, int> device_tag_;` map을 갱신한다. 이 map은 해당 노드에 연결된 copy 노드가 있는지를 기록하며, 이후 copy를 통해 노드의 디바이스 정보를 추론할 때 사용된다. 깊이 우선 순회 과정에서 copy 노드의 개수 `num_device_copy_ops_`를 기록하고, copy 노드의 개수가 0보다 크면 `PropagateDeviceId`를 실행한다.
    
    
    void PropagateDeviceId() {  
        int out_dev_type = post_visitor_.out_dev_type_;  
        for (auto& it : post_visitor_.device_tag_) {  
          if (it.second != -1) {  
            device_map_.Set(GetRef<Expr>(it.first), it.second);  
          } else {  
            device_map_.Set(GetRef<Expr>(it.first), out_dev_type);  
          }  
        }  
      }  
    

이 함수를 통해 copy 이후 노드의 디바이스 정보를 얻을 수 있다. 노드의 디바이스 정보를 모두 수집한 뒤에는 `device_info.device_map_`을 반환해 TokenMap을 만드는 데 사용한다. TokenMap에는 노드의 `ttype`과 `device_type` 정보가 담겨 있다. 구체적인 구현은 아래와 같다. 여기서 `CreateToken` 함수는 `StorageAllocaInit`이 기반 클래스 `StorageAllocaBaseVisitor`의 `CreateToken` 가상 함수를 오버라이드한 것이며, 노드의 `ttype`과 `device_type` 정보만 처리한다.
    
    
    void CreateToken(const ExprNode* op, bool can_realloc) final {  
        ICHECK(!token_map_.count(op));  
        std::vector<StorageToken*> tokens;  
        int device_type =  
            node_device_map_.count(GetRef<Expr>(op)) ? node_device_map_[GetRef<Expr>(op)]->value : 0;  
        if (const auto* tuple_type = op->checked_type().as<TupleTypeNode>()) {  
          for (Type t : tuple_type->fields) {  
            const auto* ttype = t.as<TensorTypeNode>();  
            ICHECK(ttype);  
            StorageToken* token = arena_->make<StorageToken>();  
            token->ttype = ttype;  
            token->device_type = device_type;  
            tokens.push_back(token);  
          }  
        } else {  
          const auto* ttype = op->checked_type().as<TensorTypeNode>();  
          ICHECK(ttype);  
          StorageToken* token = arena_->make<StorageToken>();  
          token->ttype = ttype;  
          token->device_type = device_type;  
          tokens.push_back(token);  
        }  
        token_map_[op] = tokens;  
      }  
    

여기까지 진행되면 TokenMap의 초기화가 완료된다. TokenMap 초기화가 끝나면 StorageToken 리스트를 반환하며, 그 안에는 Expr에 대응하는 Token이 만들어지고 참조 상황도 카운트되며, 각 op의 `ttype`이 기록된다. 이때 `ttype`은 op의 `checked_type_` attribute에 대응한다. 간단히 말해 `checked_type_`은 op의 일부 attribute 집합이라고 볼 수 있으며, shape, 데이터 타입 등을 포함한다(위 코드의 `if (const auto* tuple_type = op->checked_type().as<TupleTypeNode>())` 부분에 해당).

이어서 `StorageAllocator` 클래스 자체도 자신의 `CreateToken` 함수를 실행한다. 이 함수는 각 노드에 메모리를 할당한다. TVM은 메모리를 재사용하여 메모리 할당을 최적화한다. 구체적으로 두 가지 경우로 나뉜다.

`can_realloc`이 참일 때는 메모리를 재사용할 수 있으므로 Request 함수를 호출해 tok의 크기를 다시 계산한 뒤 tokens 리스트에 `push_back`한다.
    
    
    // override create token by getting token as prototype requirements.  
      void CreateToken(const ExprNode* op, bool can_realloc) final {  
        ICHECK(!token_map_.count(op));  
        auto it = prototype_.find(op);  
        ICHECK(it != prototype_.end());  
        std::vector<StorageToken*> tokens;  
        for (StorageToken* tok : it->second) {  
          if (can_realloc) {  
            tokens.push_back(Request(tok));  
          } else {  
            // Allocate a new token,  
            StorageToken* allocated_tok = Alloc(tok, GetMemorySize(tok));  
            allocated_tok->device_type = tok->device_type;  
            // ensure it never get de-allocated.  
            allocated_tok->ref_counter += 1;  
            tokens.push_back(allocated_tok);  
          }  
        }  
        token_map_[op] = tokens;  
      }  
    

여기서 `prototype_`은 TokenMap이 만들어질 때 얻어진 `ExprNode`와 `std::vector<StorageToken*>` 사이의 매핑 테이블이다. `StorageAllocaInit`에서는 CallNode가 Token을 만들 때만 `can_realloc`이 True로 설정된다. 먼저 `can_realloc`이 False일 때의 분기 처리를 보자. 주로 `StorageToken* allocated_tok = Alloc(tok, GetMemorySize(tok));`를 통해 새 StorageToken을 생성한다. `GetMemorySize` 함수는 Token이 차지할 메모리 용량을 계산한다. 코드 구현은 다음과 같다.
    
    
    size_t GetMemorySize(StorageToken* prototype) {  
        const TensorTypeNode* ttype = prototype->ttype;  
        ICHECK(ttype != nullptr);  
        size_t size = 1;  
        for (IndexExpr dim : ttype->shape) {  
          const int64_t* pval = tir::as_const_int(dim);  
          ICHECK(pval != nullptr) << "Cannot allocate memory symbolic tensor shape " << ttype->shape;  
          ICHECK_GE(*pval, 0) << "Cannot allocate memory for tensor with negative shape" << *pval;  
          size *= static_cast<size_t>(pval[0]);  
        }  
        size *= DivRoundUp(ttype->dtype.bits() * ttype->dtype.lanes(), 8);  
        return size;  
      }  
    

이 함수는 먼저 원소 개수를 계산한 뒤, 원소가 차지하는 공간을 계산한다(8바이트 정렬). 이어서 Alloc의 구현을 살펴보자.
    
    
    StorageToken* Alloc(StorageToken* prototype, size_t size) {  
        prototype->max_bytes = size;  
        prototype->storage_id = static_cast<int64_t>(data_.size());  
        data_.push_back(prototype);  
        return prototype;  
      }  
    

Alloc 함수는 할당받을 공간 크기를 `max_bytes` 필드에 넣어둔다. 그리고 StorageToken 안에는 처리된 StorageToken을 기록하는 `data_` 필드도 있다. Alloc이 끝난 뒤에는 StorageToken의 `ref_counter` 필드를 갱신해 동일 Token에 대해 다시 메모리를 할당하지 않도록 한다.

이어서 `can_realloc`이 True인 분기를 살펴보자. 여기서는 `Request` 함수를 실행하며, `std::multimap<size_t, StorageToken*> free_;`를 사용한다. 기본값은 빈 map이다. 다음 함수를 통해 이 map에 원소를 삽입하는데, 현재로서는 CallNode에서만 호출된다. CallNode만이 `can_realloc`을 True로 설정하기 때문이다.
    
    
    // The call map  
      void VisitExpr_(const CallNode* op) final {  
        std::vector<StorageToken*> args;  
        // for each input, visit argument token.  
        for (Expr arg : op->args) {  
          for (StorageToken* tok : GetToken(arg)) {  
            args.push_back(tok);  
          }  
        }  
        // create token for the call node.  
        CreateToken(op, true);  
        // check if there is orphaned output that can be released immediately.  
        for (StorageToken* tok : token_map_.at(op)) {  
          CheckForRelease(tok);  
        }  
        for (StorageToken* tok : args) {  
          tok->ref_counter -= 1;  
          CheckForRelease(tok);  
        }  
      }  
    

이 함수의 마지막 for 루프는 `ref_counter`를 감소시키는데, 이 플래그 변수 `ref_counter`가 0이 되면 해당 `StorageToken`을 `free_` 리스트에 추가한다. 이제 `Request` 함수의 구현을 자세히 보자.
    
    
    StorageToken* Request(StorageToken* prototype) {  
        // calculate the size;  
        size_t size = GetMemorySize(prototype);  
        // search memory block in [size / match_range_, size * match_range_)  
        if (match_range_ == 0) {  
          return this->Alloc(prototype, size);  
        }  
        auto begin = free_.lower_bound(size / match_range_);  
        auto mid = free_.lower_bound(size);  
        auto end = free_.upper_bound(size * match_range_);  
        // search for memory blocks larger than requested  
        for (auto it = mid; it != end; ++it) {  
          StorageToken* tok = it->second;  
          if (tok->device_type != prototype->device_type) continue;  
          ICHECK_EQ(tok->ref_counter, 0);  
          // Use exect matching strategy  
          tok->max_bytes = std::max(size, tok->max_bytes);  
          tok->ref_counter = prototype->ref_counter;  
          // find a exact match, erase from map and return  
          free_.erase(it);  
          return tok;  
        }  
        // then search for memory blocks smaller than requested space  
        for (auto it = mid; it != begin;) {  
          --it;  
          StorageToken* tok = it->second;  
          if (tok->device_type != prototype->device_type) continue;  
          ICHECK_EQ(tok->ref_counter, 0);  
          // Use exect matching strategy  
          tok->max_bytes = std::max(size, tok->max_bytes);  
          tok->ref_counter = prototype->ref_counter;  
          // erase from map and return  
          free_.erase(it);  
          return tok;  
        }  
        // cannot find anything return a new one.  
        return this->Alloc(prototype, size);  
      }  
    

여기서 `match_range_`의 기본값은 16이며, 먼저 `free_` 리스트에서 `size`가 요청 크기보다 큰 StorageToken을 찾는다. 만약 요청 크기보다 큰 가용 블록이 없다면 반대로 찾아 가장 큰 가용 메모리 블록을 사용한다. 개인적인 이해로는 여기서의 TokenMap 처리 흐름은 TokenMap이 생성한 StorageToken을 재사용하여 메모리 할당 최적화를 구현하는 것이다. 최종적으로는 할당이 필요한 StorageToken을 기록한 `data_` 리스트를 반환한다.

Token 처리가 끝난 뒤 다시 0x2절 시작 부분의 `auto pf = GetPackedFunc("relay.backend.GraphPlanMemory"); storage_device_map_ = (*pf)(func);`로 돌아가면, GraphCodegen 전체의 메모리 할당 흐름이 정리된다. 다시 한번 `GraphPlanMemory`의 `Plan` 함수 구현을 돌아보면, `Map<Expr, Array<IntegerArray> > smap;` 줄부터 끝까지 주로 데이터 Copy를 수행한다는 것을 알 수 있다. Plan 함수는 마지막에 Expr과 배열의 매핑 테이블을 반환한다. 배열에는 `data_`에 대응하는 StorageToken 인덱스와 그 디바이스 type이 담겨 있다. 최종적으로 `LoweredOutput Codegen(relay::Function func)`의 시작 부분에서 `storage_device_map_`을 사용해 메모리 할당 결과를 기록한다.
    
    
    // Run storage allocation for a function.  
      Map<Expr, Array<IntegerArray> > Plan(const Function& func) {  
        prototype_ = StorageAllocaInit(&arena_).GetInitTokenMap(func);  
        this->Run(func);  
      
        // The value of smap contains two integer arrays where the first array  
        // contains the planned storage ids and the second holds the device types.  
        Map<Expr, Array<IntegerArray> > smap;  
        int num_annotated_nodes = 0;  
        int num_nodes = 0;  
      
        for (const auto& kv : token_map_) {  
          std::vector<Integer> storage_ids;  
          std::vector<Integer> device_types;  
          for (StorageToken* tok : kv.second) {  
            if (tok->device_type) {  
              num_annotated_nodes++;  
            }  
            num_nodes++;  
            storage_ids.push_back(tok->storage_id);  
            device_types.push_back(tok->device_type);  
          }  
          smap.Set(GetRef<Expr>(kv.first), Array<IntegerArray>({storage_ids, device_types}));  
        }  
        // Either all or none of the nodes should be annotated.  
        if (num_annotated_nodes != 0 && num_annotated_nodes != num_nodes) {  
          LOG(FATAL) << num_annotated_nodes << " out of " << num_nodes  
                     << "expressions are assigned with virtual device types. Either all "  
                        "or none of the expressions are expected to be annotated.";  
        }  
        return smap;  
      }  
    

# 0x3. Graph Codegen

Relay Func 안의 노드에 대한 메모리 할당 결과를 얻은 뒤, Graph Codegen을 살펴보자. 즉 `LoweredOutput Codegen(relay::Function func)` 함수의 나머지 부분이다. 먼저 IR의 parameter들을 입력 노드로 변환한다.
    
    
    // First we convert all the parameters into input nodes.  
     for (auto param : func->params) {  
       auto node_ptr = GraphInputNode::make_node_ptr(param->name_hint(), GraphAttrs());  
       var_map_[param.get()] = AddNode(node_ptr, param);  
     }  
    

여기서는 먼저 `GraphInputNode` 객체를 만든다. 이 객체는 `GraphNode` 객체를 상속받았으며, `GraphNode`의 정의는 다음과 같이 name, num_outputs 등의 노드 attribute를 가진다.
    
    
    /*! \brief Base Node class */  
    class GraphNode {  
     public:  
      GraphNode() {}  
      virtual void Save(dmlc::JSONWriter* writer) const {}  
      virtual void Load(dmlc::JSONReader* reader) {}  
      virtual GraphNodeType Type() const { return kGraphNop; }  
      virtual ~GraphNode() {}  
      
     public:  
      int num_outputs_{1};  
      std::string name_;  
      GraphAttrs attrs_;  
    };  
      
    

이때 `GraphAttrs attrs_;`의 정의는 `using GraphAttrs = std::unordered_map<std::string, dmlc::any>;`이다.

이어서 func의 parameter들을 순회하며 GraphInputNode로 변환한다. 구체적인 흐름은 먼저 각 parameter에 대해 GraphInputNode 노드를 만들고, `make_node_ptr` 함수를 호출해 메모리를 할당한 뒤, AddNode 함수를 통해 parameter를 GraphInputNode로 변환해 노드 리스트 `nodes_`에 추가하는 것이다.
    
    
    static std::shared_ptr<GraphNode> make_node_ptr(const std::string& name,  
                                                      const GraphAttrs& attrs) {  
        auto ptr = std::make_shared<GraphInputNode>(name, attrs);  
        return std::dynamic_pointer_cast<GraphNode>(ptr);  
      }  
    

`nodes_`의 정의는 `std::vector<GraphObjectPtr> nodes_;`이며, 이는 `GraphExecutorCodegen` 클래스의 멤버 변수이다. `AddNode`는 주로 node(여기서 말하는 node는 GraphInputNode)에 `attrs_` attribute를 설정한다. 구체적으로는 다음을 포함한다.

  * 메모리 할당 시 분배된 StorageToken의 id로, `node->attrs_["storage_id"]`에 대응한다.
  * node의 device 타입을 기록하며, 이 값은 StorageToken 계산 과정에서 결정되어 `node->attrs_["device_index"]`에 들어간다.
  * node의 `shape`와 `dtype`을 기록하며, 각각 `_ShapeToJSON`과 `DType2String`을 사용해 얻는다.



마지막으로 AddNode는 node의 참조를 반환해 `var_map_`에 추가한다. `var_map_`의 정의는 `std::unordered_map<const Object*, std::vector<GraphNodeRef>> var_map_;`이다. 여기까지가 Relay Func의 parameter를 GraphInputNode로 변환하는 과정이다.

다음은 노드 순회이다. `std::vector<GraphNodeRef> heads_;`를 사용해 Graph 노드를 기록한다. 여기서는 `VisitExpr` 함수를 호출해 func의 순회를 완료하며, 순회 과정에서 func를 graphNode로 변환한다. VarNode의 경우 이미 `var_map_`에 기록되어 있으므로 `VarNode`의 `VisitExpr_` 함수는 그 참조를 그대로 반환하기만 한다. ConstantNode는 GraphInputNode로 변환되며, TupleNode는 각 필드의 graphNode를 반환한다. 노드 순회 과정에서 graphNode들은 모두 `nodes_`에 추가된다. 특히 CallNode의 `VisitExpr_`을 주의 깊게 보자. 우선 앞부분 코드는 다음과 같다.
    
    
    Expr expr = GetRef<Expr>(op);  
        Function func;  
        if (op->op.as<OpNode>()) {  
          LOG(FATAL) << "Operators should be transformed away; try applying"  
                     << "the fuse_ops transformation to the expression.";  
        } else if (op->op.as<GlobalVarNode>()) {  
          LOG(FATAL) << "Not implemented";  
        } else if (op->op.as<FunctionNode>()) {  
          func = GetRef<Function>(op->op.as<FunctionNode>());  
        } else {  
          LOG(FATAL) << "TVM runtime does not support calls to " << op->op->GetTypeKey();  
        }  
        if (!func->HasNonzeroAttr(attr::kPrimitive)) {  
          LOG(FATAL) << "TVM only support calls to primitive functions "  
                     << "(i.e functions composed of fusable operator invocations)";  
        }  
    

여기서 보면 CallNode가 컴파일까지 진행되려면 op가 FunctionNode 타입이어야 하고, 반드시 융합된 상태여야 한다. 이전에 다룬 TVM의 [op fusion](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247495225&idx=1&sn=5819431f3b1bca6687cb171bd4243bb6&scene=21#wechat_redirect>) 글에서도 이 점을 언급했고, 이에 대한 보호 처리를 했다. op fusion에서 FunctionNode를 보호하는 코드는 다음과 같다.
    
    
    Expr MakeNewFunction(GraphPartitioner::Group* group, Type ret_type, Expr body) {  
        // If the function has no call, it is not a primitive function.  
        struct HasCallVisitor : ExprVisitor {  
          bool has_call = false;  
          void VisitExpr_(const CallNode* op) final { has_call = true; }  
        } visitor;  
        visitor(body);  
        const GroupInfo& ginfo = ginfo_[group];  
        auto func = Function(ginfo.params, body, ret_type, {});  
        func = WithAttr(std::move(func), attr::kPrimitive, tvm::Integer(visitor.has_call));  
        return Call(func, ginfo.arguments, Attrs());  
      }  
    

그 다음은 CallNode의 function 생성 부분이다.
    
    
    auto pf0 = GetPackedFunc("relay.backend._make_CCacheKey");  
        auto pf1 = GetPackedFunc("relay.backend._CompileEngineLower");  
        Target target;  
        // Handle external function  
        if (func->GetAttr<String>(attr::kCompiler).defined()) {  
          target = Target("ext_dev");  
          CCacheKey key = (*pf0)(func, target);  
          CachedFunc ext_func = (*pf1)(compile_engine_, key);  
          ICHECK(ext_func.defined()) << "External function is not defined.";  
          UpdateConstants(func, &params_);  
          return GraphAddCallNode(op, ext_func->func_name, ext_func->func_name);  
        }  
    

function 생성은 두 가지 분기로 나뉜다. 하나는 외부 Function의 codegen이고, 다른 하나는 일반 Function의 codegen이다. 외부 codegen인지의 판단은 `func`의 `attr::kCompiler`로 결정한다.

먼저 외부 Function 처리부터 살펴보자. 여기서 `relay.backend._make_CCacheKey`와 `relay.backend._CompileEngineLower`의 정의는 아래와 같으며, 둘 다 `tvm/src/relay/backend/compile_engine.cc` 파일에 구현되어 있다.
    
    
    CCacheKey::CCacheKey(Function source_func, Target target) {  
      auto n = make_object<CCacheKeyNode>();  
      n->source_func = std::move(source_func);  
      n->target = std::move(target);  
      data_ = std::move(n);  
    }  
      
    CachedFunc Lower(const CCacheKey& key) { return LowerInternal(key)->cached_func; }  
    

여기서는 먼저 `CCacheKey` 타입을 만들어 `_CompileEngineLower` 함수의 인자로 넘긴다. `_CompileEngineLower` 함수의 구현은 `tvm/src/relay/backend/compile_engine.cc`에 있다. 호출 체인은 Lower -> LowerInternal(key) -> cached_func이며, LowerInternal 구현에서 해당하는 코드 블록은 다음과 같다.
    
    
    // No need to lower external functions for now. We will invoke the external  
        // codegen tool once and lower all functions together.  
        if (key->source_func->GetAttr<String>(attr::kCompiler).defined()) {  
          auto cache_node = make_object<CachedFuncNode>();  
          const auto name_node = key->source_func->GetAttr<String>(tvm::attr::kGlobalSymbol);  
          ICHECK(name_node.defined()) << "External function has not been attached a name yet.";  
          cache_node->func_name = std::string(name_node.value());  
          cache_node->target = Target("ext_dev");  
          cache_node->funcs->Add(GlobalVar(cache_node->func_name), key->source_func);  
          value->cached_func = CachedFunc(cache_node);  
          return value;  
        }  
    

외부 Function의 Codegen이라면, cache_node를 정의하고 이를 CachedFunc으로 감싸 반환한다. 그 다음 GraphAddCallNode를 통해 `nodes_`에 추가한다. GraphAddCallNode 안에서는 op->args에 대한 깊이 우선 순회도 함께 수행된다.
    
    
    std::vector<GraphNodeRef> GraphAddCallNode(const CallNode* op, const std::string& op_name,  
                                                 const std::string& func_name) {  
        std::vector<GraphNodeRef> inputs;  
        for (auto arg : op->args) {  
          auto res = VisitExpr(arg);  
          for (auto nr : res) {  
            inputs.push_back(nr);  
          }  
        }  
        auto node = GraphOpNode::make_node_ptr(op_name, GraphAttrs(), func_name, inputs, GraphAttrs());  
        return AddNode(node, GetRef<Expr>(op));  
      }  
    

이렇게 VisitExpr 순회를 통해 각 노드들을 대응되는 GraphNode로 변환해 `nodes_` 리스트에 추가한다.

이어서 내부 Function의 Codegen을 살펴보자. 여기서는 target 판단 부분을 생략해 코드를 단순화했다.
    
    
        CCacheKey key = (*pf0)(func, target);  
        CachedFunc lowered_func = (*pf1)(compile_engine_, key);  
        if (!lowered_funcs_.count(target->str())) {  
          lowered_funcs_[target->str()] = IRModule(Map<GlobalVar, BaseFunc>({}));  
        }  
        lowered_funcs_[target->str()]->Update(lowered_func->funcs);  
        return GraphAddCallNode(op, _GetUniqueName(lowered_func->func_name), lowered_func->func_name);  
    

여기서도 동일한 pf0, pf1 함수를 사용한다. CCacheKey의 생성 과정은 같지만, lowerInternal에서의 처리 과정이 다르다. 내부 Function의 처리는 LowerInternal 함수 구현 코드에서 다음 부분에 해당한다.
    
    
    // Enforce use the target.  
        With<Target> target_scope(key->target);  
      
        ICHECK(!value->cached_func.defined());  
        auto cfunc = CreateSchedule(key->source_func, key->target);  
        auto cache_node = make_object<CachedFuncNode>(*(cfunc.operator->()));  
      
        // Skip lowering for device copy node.  
        const Expr body = (key->source_func)->body;  
        if (const CallNode* call_node = body.as<CallNode>()) {  
          if (call_node->attrs.as<DeviceCopyAttrs>()) {  
            value->cached_func = CachedFunc(cache_node);  
            return value;  
          }  
        }  
      
        cache_node->func_name = GetUniqueName(cache_node->func_name);  
        // NOTE: array will copy on write.  
        Array<te::Tensor> all_args = cache_node->inputs;  
        for (te::Tensor arg : cache_node->outputs) {  
          all_args.push_back(arg);  
        }  
        // lower the function  
        if (const auto* f = runtime::Registry::Get("relay.backend.lower")) {  
          cache_node->funcs = (*f)(cfunc->schedule, all_args, cache_node->func_name, key->source_func);  
        } else {  
          using tvm::transform::PassContext;  
          With<PassContext> fresh_pass_ctx_scope(PassContext::Create());  
      
          std::unordered_map<te::Tensor, tir::Buffer> binds;  
          cache_node->funcs = tvm::lower(cfunc->schedule, all_args, cache_node->func_name, binds);  
        }  
        value->cached_func = CachedFunc(cache_node);  
        return value;  
    

TVM의 op는 schedule과 compute의 분리 원칙을 따른다. 여기서 먼저 `CreateSchedule`의 정의를 보자.
    
    
    CachedFunc CreateSchedule(const Function& source_func, const Target& target) {  
        return ScheduleGetter(target).Create(source_func);  
      }  
    

여기서는 ScheduleGetter를 정의해 지정한 target의 schedule을 얻는다. 이 Create 함수는 IR에서 TIR 노드로의 변환과 schedule 최적화를 모두 수행한다. 이 함수는 매우 복잡해 능력의 한계로 지금은 더 깊이 읽기 어려웠다. Create 함수에는 또한 Lower function의 생성 과정이 있으며, 구체적으로는 `runtime::Registry::Get("relay.backend.lower"))` 함수를 호출해 동작을 수행한다.

# 0x4. 정리

여기까지 읽었다면 이 글은 마무리할 수 있다. 이 글에서는 TVM의 Codegen 흐름을 살펴보았다. Relay frontend부터 시작해 Graph 노드의 메모리 할당, Relay IR에서 TIR 노드로의 변환, TIR 그래프 노드의 Schedule 최적화, 그리고 Lower function이 발생하는 위치까지 정리하였다. 이 글은 호출 체인에 초점을 맞추었으며, Schedule 최적화, IR에서 TIR 노드로의 변환, Lower Function 같은 구체적인 동작은 더 깊이 다루지 않았다.

# 0x00. 머리말

이전 글에서는 TVM Pass Infra를 체계적으로 소개하고, Relay 트리 구조와 Constant Folding Pass의 구체적인 동작을 분석했습니다. 이번 글에서는 먼저 TVM Pass Infra의 사용법을 보충 설명하고, 이어서 TVM의 operator fusion에 사용되는 dominator tree라는 자료구조를 소개한 뒤, 마지막으로 TVM에서 매우 중요한 Pass 중 하나인 operator fusion을 깊이 있게 분석하겠습니다.

# 0x01. TVM Pass Infra 사용 방법

TVM Pass Infra에 대한 소개는 [밑바닥부터 배우는 딥러닝 컴파일러] 7. TVM Pass에 입문하는 만 자 분량의 글을 참고하시기 바랍니다. 여기서는 TVM Pass Infra의 사용 방법을 소개하며, 내용은 https://tvm.apache.org/docs/tutorials/arch/use_pass_infra.html 에서 번역하면서 저의 이해를 추가했습니다.

Relay/tir에서 최적화 Pass의 횟수가 늘어남에 따라, 수동으로 그것들을 실행하고 의존 관계를 관리하는 것이 번거로워집니다. 그래서 우리는 최적화 Pass들을 관리하고 TVM 스택의 서로 다른 계층의 IR에 적용할 수 있도록 Pass 인프라를 도입했습니다.

Relay/tir 프로그램의 최적화 Pass는 다양한 단위로 적용할 수 있습니다. 즉, `tvm.relay.transform.FunctionPass`/`tvm.tir.transform.PrimFuncPass`와 `tvm.transform.ModulePass`를 각각 사용해 function-level과 module-level 단위의 최적화 Pass를 실행할 수 있습니다. 또는 사용자는 `tvm.transform.Sequential`을 사용해 Relay/tir 프로그램에 일련의 Pass들을 적용할 수 있고, 이때 Pass들 사이의 의존 관계는 Pass Infra가 해결해 줍니다.

여기서는 주로 개발자가 Pass Infra를 사용해 어떤 최적화를 수행하고 Relay 프로그램의 최적화 파이프라인을 만드는 방법을 보여줍니다. 여기서 다루는 방법은 tir에도 동일하게 적용됩니다. 먼저 필요한 패키지를 import 합니다.


    import numpy as np  
    import tvm  
    from tvm import te  
    import tvm.relay as relay  


다음으로, 다양한 예시 Pass를 실행하는 데 사용할 간단한 Relay 프로그램을 보여 드립니다. 마찬가지로 사용자는 tir 원시 함수를 작성해서 Pass를 적용할 수도 있습니다.

**Relay 프로그램 예시 만들기**


     def example():  
        shape = (1, 64, 54, 54)  
        c_data = np.empty(shape).astype("float32")  
        c = relay.const(c_data)  
        weight = relay.var("weight", shape=(64, 64, 3, 3))  
        x = relay.var("x", relay.TensorType((1, 64, 56, 56), "float32"))  
        conv = relay.nn.conv2d(x, weight)  
        y = relay.add(c, c)  
        y = relay.multiply(y, relay.const(2, "float32"))  
        y = relay.add(conv, y)  
        z = relay.add(y, c)  
        z1 = relay.add(y, c)  
        z2 = relay.add(z, z1)  
        return relay.Function([x, weight], z2)  


그 다음으로 conv op에 대해 출력 데이터 layout을 변경하는 Pass를 등록합니다. 이 Pass는 convolution 레이어의 NCHW 데이터 layout을 `NCHW16c` layout으로 바꿉니다.


    @relay.op.register_alter_op_layout("nn.conv2d", level=101)  
    def alter_conv2d(attrs, inputs, tinfos, out_type):  
        data, weight = inputs  
        new_attrs = dict(attrs)  
        new_attrs["data_layout"] = "NCHW16c"  
        return relay.nn.conv2d(data, weight, **new_attrs)  


**프로그램 최적화**

Pass를 적용하기 전에 Relay 프로그램이 어떻게 생겼는지 살펴봅시다.


    def @main(%x: Tensor[(1, 64, 56, 56), float32], %weight: Tensor[(64, 64, 3, 3), float32]) {  
      %0 = add(meta[relay.Constant][0], meta[relay.Constant][0]);  
      %1 = nn.conv2d(%x, %weight, padding=[0, 0, 0, 0]);  
      %2 = multiply(%0, 2f);  
      %3 = add(%1, %2);  
      %4 = add(%3, meta[relay.Constant][0]);  
      %5 = add(%3, meta[relay.Constant][0]);  
      add(%4, %5)  
    }  


이제 프로그램을 최적화하려고 합니다. Relay에는 많은 최적화 기능이 있습니다. 그 중 일부를 골라 이 예시 프로그램에 적용해 보겠습니다.

수동으로 최적화 Pass들을 적용하되, 여기서는 FoldConstant Pass를 사용합니다.


    # Let's first create a relay Module which contains one or multiple Relay  
    # functions for optimization.  
    f = example()  
    mod = tvm.IRModule.from_expr(f)  
      
    # Now we can apply constant folding on the module.  
    # fold_const here is a callback that doesn't take any parameters.  
    fold_const = relay.transform.FoldConstant()  
    # Then, we can invoke the pass on the given module. Note that the constant  
    # folding pass works at the function-level. That being said, each function in  
    # the module will be applied with the optimization. Users don't need to iterate  
    # through individual functions manually to apply this pass.  
    mod = fold_const(mod)  
    # We can see from the updated program that the constants are folded.  
    print(mod)  


FoldConstant Pass를 적용한 뒤 Relay 프로그램은 다음과 같이 됩니다.


    def @main(%x: Tensor[(1, 64, 56, 56), float32], %weight: Tensor[(64, 64, 3, 3), float32]) -> Tensor[(1, 64, 54, 54), float32] {  
      %0 = nn.conv2d(%x, %weight, padding=[0, 0, 0, 0]) /* ty=Tensor[(1, 64, 54, 54), float32] */;  
      %1 = add(%0, meta[relay.Constant][0] /* ty=Tensor[(1, 64, 54, 54), float32] */) /* ty=Tensor[(1, 64, 54, 54), float32] */;  
      %2 = add(%1, meta[relay.Constant][1] /* ty=Tensor[(1, 64, 54, 54), float32] */) /* ty=Tensor[(1, 64, 54, 54), float32] */;  
      %3 = add(%1, meta[relay.Constant][1] /* ty=Tensor[(1, 64, 54, 54), float32] */ /* ty=Tensor[(1, 64, 54, 54), float32] */) /* ty=Tensor[(1, 64, 54, 54), float32] */;  
      add(%2, %3) /* ty=Tensor[(1, 64, 54, 54), float32] */  
    }  


최적화 전 IR과 비교하면, FoldConstant Pass를 적용한 결과 초기 IR의 `%2 = multiply(%0, 2f);`가 상수이기 때문에 직접 fold되어 `relay.Constant][1]`로 변환된 것을 볼 수 있습니다. 이어서 비슷한 방식으로 더 많은 최적화를 적용할 수 있습니다. 예를 들어, z와 z1이 사용하는 공통 표현식을 제거할 수 있는데, 이는 `EliminateCommonSubexpr` Pass를 사용해 처리합니다.


    mod = relay.transform.EliminateCommonSubexpr()(mod)  
    print(mod)

**Sequential을 사용해 일련의 Pass 적용**

위처럼 Pass를 적용하는 것은 사실 매우 번거롭고, 사용자가 Pass들 사이의 의존 관계를 잘 이해해야 할 수도 있습니다. 예를 들어, 현재 fusion은 let bindings에서 잘 동작하지 않습니다. 따라서 fusion 이전에 `relay.transform.ToANormalForm()`을 적용한다면, 이 Pass가 Relay 프로그램을 정규화하기 위해 모든 표현식에 대해 let bindings를 생성하기 때문에 **융합 가능한 operator** 들을 fuse할 수 없게 됩니다.

그래서 Relay는 `tvm.transform.Sequential`을 제공해, 각 Pass에 필요한 Pass들을 명시하고 그것들을 하나의 묶음으로 패키징하여 실행함으로써 개발자가 이런 문제들을 명시적으로 처리하는 부담을 줄여 줍니다. 예를 들어 이제 sequential 스타일로 동일한 Pass들을 다음과 같이 적용할 수 있습니다. `tvm.transform.Sequential`은 `torch.nn.sequential`이나 `mxnet.gluon.block`과 비슷합니다. 예를 들어 `torch.nn.sequential`은 네트워크를 구성하기 위해 추가될 일련의 PyTorch Module을 담는 데 사용되며, 네트워크 레이어에 초점이 맞춰져 있습니다. 반면, 우리 Pass Infra의 `tvm.transform.Sequential`은 최적화 Pass에 사용됩니다.


    # Now let's execute some passes through :py:class:`tvm.transform.Sequential`  
    f = example()  
    mod = tvm.IRModule.from_expr(f)  
    # Glob the interested passes.  
    seq = tvm.transform.Sequential(  
        [  
            relay.transform.FoldConstant(),  
            relay.transform.EliminateCommonSubexpr(),  
            relay.transform.FuseOps(fuse_opt_level=2),  
        ]  
    )  
    mod1 = seq(mod)  
    print(mod1)

**Python 데코레이터로 Pass 구현하기**

이 예시는 Python 데코레이터를 사용해 Pass Infra를 통해 커스텀 최적화 Pass를 어떻게 만드는지 보여 줍니다. 이 기능은 Pass를 구현하는 난이도를 크게 줄여 줍니다. 예를 들어 사용자는 함수 단위의 최적화를 구현하기 위해 데코레이터 클래스를 단순하게 정의할 수 있습니다. 다음 예시처럼 transform_function은 모든 상수에 c를 곱하는 클래스를 감쌉니다. 그런 뒤 이 커스텀 Pass를 호출하면 주어진 Module의 모든 함수가 방문되고, 함수 안의 모든 상수가 치환됩니다.


    @relay.transform.function_pass(opt_level=1)  
    class CustomPipeline:  
        """Simple test function to replace one argument to another."""  
      
        def __init__(self, multiplier):  
            self.multiplier = multiplier  
      
        # This function can define a pass.  
        def transform_function(self, func, mod, ctx):  
            obj = self  
      
            class ReplaceConstant(tvm.relay.ExprMutator):  
                def visit_constant(self, c):  
                    return relay.multiply(obj.multiplier, c)  
      
            return ReplaceConstant().visit(func)  
      
      
    f = example()  
    mod = tvm.IRModule.from_expr(f)  
    custom_pass = CustomPipeline(multiplier=relay.const(3, "float32"))  
    assert custom_pass.info.name == "CustomPipeline"  
    mod3 = custom_pass(mod)  
    print(mod3)

**Pass 디버깅하기**

TVM은 사용자에게 plug-and-play 스타일의 Pass 디버깅 기능을 제공합니다. 특수 Pass(`PrintIR`)를 사용해, 특정 Pass가 끝난 뒤 IR을 출력해서 모듈 전체의 IR을 덤프할 수 있습니다. 다음과 같이 사용합니다.


    f = example()  
    mod = tvm.IRModule.from_expr(f)  
    seq = tvm.transform.Sequential(  
        [  
            relay.transform.FoldConstant(),  
            tvm.transform.PrintIR(),  
            relay.transform.EliminateCommonSubexpr(),  
            relay.transform.FuseOps(),  
            relay.transform.AlterOpLayout(),  
        ]  
    )  
      
    # By inserting the ``PrintIR`` pass after ``FoldConstant``, the pass infra will  
    # dump out the module IR when ``FoldConstant`` is done. Users can plug in this  
    # pass after any pass they want to debug for viewing the optimization effect.  
    #  
    # There is a more flexible debugging mechanism also exposed by the build configuration  
    # object. One can pass a tracing function which can be used to execute arbitrary code  
    # before and/or after each pass. A tracing function will receive a :py::class:`tvm.IRModule`,  
    # a :py:class:`tvm.transform.PassInfo` object,  
    # and a boolean indicating whether you are executing before, or after a pass.  
    # An example is below.  
      
      
    @tvm.instrument.pass_instrument  
    class PrintIR:  
        """Print the name of the pass, the IR, only before passes execute."""  
      
        def run_before_pass(self, mod, info):  
            print("Running pass: {}", info)  
            print(mod)  
      
      
    with tvm.transform.PassContext(opt_level=3, instruments=[PrintIR()]):  
        with tvm.target.Target("llvm"):  
            # Perform the optimizations.  
            mod = seq(mod)  
    print(mod)  
      
    print("done")  


여기서는 sequential pass의 각 Pass 뒤에 이 `PrintIR` Pass를 삽입했기 때문에 출력 결과가 매우 길어집니다. 그래서 여기서는 따로 붙이지 않겠습니다. 관심 있는 분들은 직접 프로그램을 실행해 보세요: `tvm/tutorials/dev/use_pass_infra.py`.

# 0x02. dominator tree

operator fusion 코드를 설명하기 전에 먼저 dominator tree(지배 트리)라는 것을 짚고 넘어가야 합니다. 그렇지 않으면 코드가 이해되기 어렵습니다. 이해를 위해 다음 블로그를 참고하시면 좋습니다: https://www.cnblogs.com/ZeonfaiHo/p/6594642.html. 다만 TVM operator fusion Pass에서는 dominator tree의 비교적 초보적인 부분만 사용하므로, 여기서는 이 블로그의 일부 설명을 빌려 간단히 짚고 넘어가겠습니다.

## dominator tree란 무엇인가?

방향 그래프(사이클이 있을 수 있음)에 시작점을 하나 지정하면, 어떤 점에서 그래프 위의 다른 점까지 여러 개의 경로가 존재할 수 있습니다(이하 줄여서 경로라고 부르겠습니다).

특정 점의 모든 경로 위에 어떤 점이 반드시 존재한다면, 그 점을 해당 점의 dominator(필수 통과점, 지배점)라고 부릅니다. 시작점에 대해서는 dominator를 따지지 않습니다. 어떤 점에서 가장 가까운 dominator를 idom이라고 표시합니다.

원래 그래프의 시작점을 제외한 모든 점에 대해, 그 점에서 idom으로 향하는 간선 하나씩을 만들면, 결국 시작점을 root로 하는 트리를 얻을 수 있습니다. 이 트리를 "dominator tree"라고 부릅니다.

## 문제 단순화

  1. **트리**



트리에 대해, root 노드를 r, 트리 위의 어떤 비root 노드를 v라 합시다. r에서 v까지의 경로 위의 모든 점이 v의 dominator이며, idom(v)는 v의 부모 노드라는 것을 쉽게 알 수 있습니다.

이는 O(n) 시간 안에 구현할 수 있습니다.

  2. **DAG(방향 비순환 그래프)**



방향 비순환 그래프이므로, 위상 순서(topological order)에 따라 dominator tree를 구성할 수 있습니다.

현재 위상 순서상 i번째 노드(번호 v)를 처리한다고 가정하면, 위상 순서상 i 이전의 노드들은 이미 처리가 끝났다는 뜻입니다. v로 직접 도달할 수 있는 모든 노드들을 고려하여, 그 노드들의 dominator tree 상에서의 최소 공통 조상(LCA)을 구하면, 이 점이 v의 dominator tree에서의 부모 노드가 됩니다.

배가 LCA(binary lifting)를 사용하면, 이 문제는 $O(n \log n)$ 시간에 구현할 수 있습니다.

이 두 가지 문제에 대해서는 아주 간단하게 dominator tree를 구할 수 있습니다.

TVM Relay가 표현하는 계산 그래프는 DAG이므로, 사실 여기까지만 알아도 충분하고 더 깊이 들어갈 필요는 없습니다.

![이미지](images/img_02.png)TVM 코드에서도 이 점을 언급하고 있다

**다만 operator fusion Pass에서 실제로 만드는 것은 post-dominator tree(후지배 트리)이며, 이 경우 위상 순서가 아니라 DFS 순서로 처리합니다.**

왜 이렇게 할까요? `fuse_ops.cc`에 있는 예시를 보면 알 수 있습니다.


                conv2d  
                /  |  \  
               /   |   \  
             op    op   op  
              \    |    /  
               \   |   /  
              elemwise add  
                   |  


conv2d와 elemwise add가 fuse되려면, 그 사이의 세 op가 elemwise add와 fuse될 수 있는지를 모두 판단해야 한다는 것을 알 수 있습니다.

# 0x03. TVM의 operator fusion(연산자 융합)

TVM 논문에서는, GPU 및 특정 가속기의 경우 여러 연산을 하나로 묶어 fuse하는 최적화 방법이 실행 시간을 비교적 명확하게 줄일 수 있다고 언급하고 있습니다. operator fusion의 아이디어는, 단일 Kernel 함수가 중간 결과를 글로벌 메모리에 다시 쓰는 시간 비용을 절약할 수 있다는 데서 나옵니다. 구체적으로 분석해 보면, 우리는 그래프 operator를 다음 네 가지 부류로 정리할 수 있습니다.

  * injective(one-to-one map): 매핑 함수, 예를 들어 덧셈, 점곱(dot product) 등.
  * reduction: 축소(약분), 예를 들어 sum/max/min처럼 입력에서 출력으로 갈 때 차원이 줄어드는 성격을 가지는 것. 예: sum.
  * complex-out-fusable(can fuse element-wise map to output): 계산이 비교적 복잡한 것, 예를 들어 conv2d.
  * opaque(cannot be fused) 융합할 수 없는 operator, 예를 들어 sort.



위와 같은 operator의 종류 구분을 바탕으로, TVM은 세 가지 융합 규칙을 제공합니다.

![이미지](images/img_03.png)TVM의 operator fusion Pass 규칙과 그에 대응하는 예시

다음 그림은 conv-bn-relu fusion 전후의 성능 비교를 보여 줍니다.

![이미지](images/img_04.png)operator fusion 후 명확한 가속이 보임

TVM에서 operator fusion Pass를 구현한 코드는 `tvm/src/relay/transforms/fuse_ops.cc`에 있습니다. TVM의 operator fusion은 주로 다음 세 단계로 구성됩니다.

  * Relay 트리를 순회하여 post-dominator tree 분석을 위한 DAG를 만든다
  * post-dominator tree를 만든다
  * operator fusion 알고리즘을 적용한다



## DAG 만들기

먼저 Pass 등록 인터페이스를 살펴보겠습니다.


    namespace transform {  
      
    Pass FuseOps(int fuse_opt_level) {  
      runtime::TypedPackedFunc<Function(Function, IRModule, PassContext)> pass_func =  
          [=](Function f, IRModule m, PassContext pc) {  
            int opt_level = fuse_opt_level == -1 ? pc->opt_level : fuse_opt_level;  
            auto max_fuse_depth = pc->GetConfig("relay.FuseOps.max_depth", Integer(kMaxFusedOps));  
            return Downcast<Function>(FuseOps(f, opt_level, max_fuse_depth.value(), m));  
          };  
      return CreateFunctionPass(pass_func, 1, "FuseOps", {"InferType"});  
    }  
      
    TVM_REGISTER_GLOBAL("relay._transform.FuseOps").set_body_typed(FuseOps);  


FuseOps Pass가 InferType Pass에 의존하고 있고, 여기서는 PassContext를 통해 필요한 설정 정보를 얻으며, 마지막으로 이 Pass에 대해 Python 인터페이스도 등록한다는 것을 볼 수 있습니다.

이제 DAG 생성 흐름을 살펴보겠습니다. 먼저 다음 위치를 봅니다.


    IndexedForwardGraph IndexedForwardGraph::Create(support::Arena* arena, const Expr& body) {  
      return Creator(arena).Prepare(body);  
    }  


여기서 Creator 클래스가 핵심입니다.


    // Creator of post dominator tree of the dataflow  
    class IndexedForwardGraph::Creator : private ExprVisitor {  
     public:  
      explicit Creator(support::Arena* arena) : arena_(arena) {}  
      
      IndexedForwardGraph Prepare(const Expr& body) {  
        this->Update(body, nullptr, kOpaque);  
        this->VisitExpr(body);  
        return std::move(graph_);  
      }  
      // 여기서는 멤버 변수와 함수를 생략했습니다  
      .....  
    }  


이전에 Pass Infra에 대해 소개한 내용을 통해, `VisitExpr`이 실제로는 IndexedforwardGraph 클래스에서 정의된 `VisitExpr_` 함수를 호출한다는 것을 알 수 있습니다. 깊이 우선 탐색(DFS)으로 Relay 트리를 순회하면서 DAG를 만듭니다. 위 함수에서는 `VisitExpr`을 호출하기 전에 `Update` 함수를 호출하여 DAG를 초기화하고 root 노드를 삽입합니다.

DFS는 후위 탐색 트리(post-order search tree)를 얻을 수 있고, 탐색 과정에서 `Update`라는 멤버 함수를 사용해 간선을 만듭니다. 이 함수의 인터페이스를 살펴봅시다.


    // Update the message stored at the node.  
      void Update(const Expr& node, IndexedForwardGraph::Node* parent, OpPatternKind pattern) {  
        const tvm::Object* key = node.get();  
        IndexedForwardGraph::Node* current;  
        auto it = graph_.node_map.find(key);  
        if (it != graph_.node_map.end()) {  
          current = it->second;  
        } else {  
          current = arena_->make<IndexedForwardGraph::Node>();  
          graph_.node_map[key] = current;  
        }  
        if (parent != nullptr) {  
          auto* link = arena_->make<LinkNode<IndexedForwardGraph::Edge> >();  
          link->value.node = parent;  
          link->value.pattern = pattern;  
          current->outputs.Push(link);  
        } else {  
          current->extern_ref = true;  
        }  
      }  


먼저 입력 파라미터 중 `OpPatternKind pattern`이 있습니다. 이 데이터 타입은 `tvm/include/tvm/relay/op_attr_types.h`에 정의되어 있고, 노드와 간선(간선도 일종의 노드)의 Operator 타입을 지정할 수 있습니다. 그리고 위 함수에서 노드에는 `outputs`라는 멤버 변수가 있는데, 이는 **노드의 입력 간선들을 저장하며, post-dominator tree를 만들 때 이 입력 간선들을 통해 LCA를 구합니다**. 즉, 이 탐색 트리(DAG) 위에서 dominator tree 알고리즘을 적용하면 post-dominator tree를 얻을 수 있습니다.


    /*! \brief operator pattern used in graph fusion */  
    enum OpPatternKind {  
      // Elementwise operation  
      kElemWise = 0,  
      // Broadcasting operator, can always map output axis to the input in order.  
      // for example :code:`out[i, ax1, j, ax2] = input[i, j]`.  
      // Note that the axis need to be in order so transpose is not a bcast operator.  
      kBroadcast = 1,  
      // Injective operator, can always injectively map output axis to a single input axis.  
      // All injective operator can still be safely fused to injective and reduction.  
      kInjective = 2,  
      // Communicative reduction operator.  
      kCommReduce = 3,  
      // Complex operation, can still fuse elemwise operations into its output.  
      // but cannot chain another complex op  
      kOutEWiseFusable = 4,  
      // The pattern for tuple nodes. Can fuse into subsequent injective ops,  
      // but treated specially  
      kTuple = 7,  
      // Opaque operation, cannot fuse anything.  
      kOpaque = 8  
    };  
      


`IndexedForwardGraph::Creator` 클래스에서는 서로 다른 노드 타입에 대해 `visitExpr_` 함수를 오버라이드합니다. 노드 타입에는 FunctionNode, ConstantNode, CallNode, TuppleNode 등이 있습니다. 예를 들어 CallNode의 `visitExpr_` 함수 구현은 다음과 같습니다.


    void VisitExpr_(const CallNode* call) final {  
        ICHECK(graph_.node_map.count(call));  
        Node* node = graph_.node_map.at(call);  
        static auto fpattern = Op::GetAttrMap<TOpPattern>("TOpPattern");  
        // Now we set the pattern of this call.  
        //  
        // If we see a call mentioning an operator we should mark it with its  
        // annotated pattern.  
        //  
        // If the pattern is not annotated we will default to opaque.  
        //  
        // Finally if the operator position is not a call node we will  
        // need to call Update, as it may be an arbitrary expression.  
        OpPatternKind op_pattern = kOpaque;  
        if (const OpNode* opnode = call->op.as<OpNode>()) {  
          auto op = GetRef<Op>(opnode);  
          if (IsDynamic(call->checked_type()) && IsDataDependent(call)) {  
            // output of a shape func can't be fed to a data-dependent shape func  
            op_pattern = kOpaque;  
          } else {  
            op_pattern = static_cast<OpPatternKind>(fpattern[op]);  
          }  
        } else {  
          this->Update(call->op, node, kOpaque);  
        }  
      
        node->pattern = op_pattern;  
        this->Update(call->op, nullptr, kOpaque);  
        const auto* rtype = call->checked_type().as<TensorTypeNode>();  
        // pass the analysis back to all the children it references.  
        for (size_t i = 0; i < call->args.size(); ++i) {  
          const auto* arg_type = call->args[i]->checked_type().as<TensorTypeNode>();  
          // specifically check if result type is the same as arguments type  
          OpPatternKind edge_pattern = op_pattern;  
          if (edge_pattern == kBroadcast && arg_type != nullptr && rtype != nullptr &&  
              attr_equal_(rtype->shape, arg_type->shape)) {  
            edge_pattern = kElemWise;  
          }  
          this->Update(call->args[i], node, edge_pattern);  
        }  
        ExprVisitor::VisitExpr_(call);  
        this->AddNode(call);  
      }  


함수의 마지막에 `VisitExpr_` 함수를 다시 재귀 호출하는 것을 볼 수 있습니다. 이를 통해 깊이 우선 탐색으로 발견된 노드들을 leaf 노드를 시작점으로 하여 차례로 DAG에 추가합니다. 이것이 후위 순회이기 때문입니다. `ConstantNode`의 `VisitExpr_` 함수만 더 이상 재귀 호출을 하지 않는데, 이는 상수 노드에는 leaf 노드가 존재하지 않기 때문입니다.

CallNode의 `VisitExpr_` 구현에서, 먼저 입력을 DAG에 추가하고, 동시에 입력 및 입력과 관련된 Edge(간선도 특수한 노드)를 순회하면서 DAG를 갱신합니다. 베이스 클래스 `ExprVisitor`의 CallNode 방문 함수는 다음과 같이 정의되어 있습니다.


    void ExprVisitor::VisitExpr_(const CallNode* op) {  
      this->VisitSpan(op->span);  
      this->VisitExpr(op->op);  
      
      for (auto ty_arg : op->type_args) {  
        this->VisitType(ty_arg);  
      }  
      
      for (auto arg : op->args) {  
        this->VisitExpr(arg);  
      }  
    }  


그리고 IndexForwardGraph 클래스가 ExprVisitor 클래스를 상속하며, `VisitExpr_`은 가상 함수이므로, this는 IndexForwardGraph 인스턴스를 가리키게 되고, 결국 이 클래스에서 정의된 VisitExpr_ 함수가 호출되어 Relay 트리를 재귀적으로 순회하게 됩니다.

## post-dominator tree 만들기

위에서 우리는 Relay 트리를 사용해 post-dominator tree 구축을 위한 DAG를 만드는 작업을 끝냈습니다(이 DAG는 DFS 후위 순회로 만들어진 것임에 주의하세요). 이어서 post-dominator tree를 어떻게 만드는지 살펴보겠습니다. dominator tree의 구축은 `DominatorTree` 클래스의 `PostDom` 멤버 함수로 수행됩니다. Relay 트리를 후위 순회할 때 마지막 노드가 곧 post-dominator tree의 root 노드이기 때문에, root 노드부터 시작해서 각 노드와 연결된 노드들의 LCA를 찾습니다. 이 LCA가 바로 post-dominator point가 됩니다. 코드 구현은 다음과 같습니다.


    DominatorTree DominatorTree::PostDom(support::Arena* arena, const IndexedForwardGraph& graph) {  
      DominatorTree tree;  
      tree.nodes.resize(graph.post_dfs_order.size(), nullptr);  
      // reverse topo order  
      for (size_t i = graph.post_dfs_order.size(); i != 0; --i) {  
        size_t index = i - 1;  
        tree.nodes[index] = tree.GetNode(arena, graph.post_dfs_order[index]);  
      }  
      return tree;  
    }  


여기서 `GetNode` 함수는 dominator point를 얻어 post-dominator tree를 구성하는 데 사용됩니다. 코드 구현은 다음과 같습니다.


    /*!  
       * \brief Convert the Node from an IndexedForwardGraph Node into DomaintorTree Node.  
       * \param arena The Arena.  
       * \param gnode An IndexedForwardGraph Node.  
       * \return The DominatorTree Node.  
       */  
      Node* GetNode(support::Arena* arena, IndexedForwardGraph::Node* gnode) {  
        Node* tnode = arena->make<Node>();  
        tnode->gnode = gnode;  
        if (gnode->extern_ref) {  
          tnode->depth = 1;  
          tnode->parent = nullptr;  
          tnode->pattern = kOpaque;  
        } else {  
          // find the LCAs of all outputs.  
          OpPatternKind pattern = kElemWise;  
          Node* parent = LeastCommonAncestor(gnode->outputs, &pattern);  
          tnode->depth = parent ? parent->depth + 1 : 1;  
          tnode->parent = parent;  
          tnode->pattern = pattern;  
        }  
        return tnode;  
      }  


이 함수는 먼저 post-dominator tree의 root 노드를 초기화한 뒤, 각 노드의 입력 노드들의 LCA, 즉 그 노드의 dominator point를 구합니다. 이 함수에 있는 `LeastCommonAncestor`는 현재 노드의 모든 입력 노드(이 노드와 연결된 모든 간선)의 LCA를 구하는 데 사용됩니다. 코드 구현은 다음과 같습니다.


    /*!  
       * \brief Find the least common ancestor of a list of nodes.  
       * \param nodes the nodes.  
       * \param edge_pattern  
       *        The combined edge pattern across all the parents.  
       * \return The least common ancestor of all nodes.  
       */  
      Node* LeastCommonAncestor(const LinkedList<IndexedForwardGraph::Edge>& input_nodes,  
                                OpPatternKind* edge_pattern) {  
        auto link = input_nodes.head;  
        if (link == nullptr) {  
          return nullptr;  
        }  
        auto get_node = [&](const IndexedForwardGraph::Edge& edge) {  
          size_t oindex = edge.node->index;  
          ICHECK_LT(oindex, nodes.size());  
          Node* onode = nodes[oindex];  
          ICHECK(onode != nullptr);  
          return onode;  
        };  
        Node* parent = get_node(link->value);  
        *edge_pattern = CombinePattern(*edge_pattern, link->value.pattern);  
        link = link->next;  
        for (; link != nullptr; link = link->next) {  
          parent = LeastCommonAncestor(parent, get_node(link->value), edge_pattern);  
          *edge_pattern = CombinePattern(*edge_pattern, link->value.pattern);  
        }  
        return parent;  
      }  


이 함수는 사실상 for 루프로, 첫 번째 노드를 고정하고 두 번째 노드와 첫 번째 노드의 LCA를 구한 뒤, 그 결과로 현재 `parent`를 갱신하고, 이런 식으로 모든 노드를 순회하면 모든 노드의 LCA를 얻을 수 있습니다. DAG에서 LCA를 구하는 구체적인 코드 구현은 다음과 같습니다.


    /*!  
       * \brief Find the least common ancestor of the two nodes.  
       * \param lhs The left node.  
       * \param rhs The right node.  
       * \param edge_pattern  
       *        The combined edge pattern across all the parents.  
       * \return The least common ancestor of the two.  
       */  
      static Node* LeastCommonAncestor(Node* lhs, Node* rhs, OpPatternKind* edge_pattern) {  
        while (lhs != rhs) {  
          if (lhs == nullptr) return nullptr;  
          if (rhs == nullptr) return nullptr;  
          if (lhs->depth < rhs->depth) {  
            edge_pattern[0] = CombinePattern(edge_pattern[0], rhs->pattern);  
            rhs = rhs->parent;  
          } else if (rhs->depth < lhs->depth) {  
            edge_pattern[0] = CombinePattern(edge_pattern[0], lhs->pattern);  
            lhs = lhs->parent;  
          } else {  
            edge_pattern[0] = CombinePattern(edge_pattern[0], lhs->pattern);  
            edge_pattern[0] = CombinePattern(edge_pattern[0], rhs->pattern);  
            lhs = lhs->parent;  
            rhs = rhs->parent;  
          }  
        }  
        return lhs;  
      }  


이 함수는 LCA를 구하는 가장 무차별 대입 방식의 구현으로, 두 노드의 깊이가 다를 때 부모 노드를 따라 위로 올라가며, 깊이가 같아질 때까지 진행하여 두 노드의 LCA 노드를 찾습니다. dominator point의 pattern을 계산할 때는 pattern의 정의를 따라 pattern 값이 가장 큰 것을 LCA의 pattern으로 선택합니다. 이는 추측건대, pattern이 작은 값에서 큰 값으로 갈수록 아래쪽으로 fuse가 가능하기 때문일 것입니다. 예를 들어 `kElemWise=0`, `kInjective=2`라면, 전자는 `KInjective` 안으로 fuse될 수 있는 식입니다.

## fusion

이제 operator fusion의 마지막 단계인 fusion을 소개합니다. DAG와 post-dominator tree(`postDominator Tree`) 구축이 끝나면 operator fusion을 수행할 수 있습니다. fusion에 대한 `fuse_ops.cc`의 개요는 다음과 같습니다.

The fusion algorithm traverses from each node and checks if it can be fused to its immediate post dominator. It has to check the following things:

  * CheckPath: check all the path between a node and its immediate post-dominator satisfies the fuse condition.
  * Note that these intermediate node can already be fused with another nodes, the algorithm will still run correctly.
  * CommitFuse: mark all the nodes between source and post-dominator as the same group.
  * We use an Union-Find data structure to manage the groups.



먼저 fusion 동작의 인터페이스는 다음 함수에 있습니다.


    std::vector<GraphPartitioner::Group*> GraphPartitioner::Partition(  
        const IndexedForwardGraph& graph) {  
      this->InitGroups(graph);  
      if (opt_level_ == 0) return std::move(groups_);  
      // get post dominator tree  
      auto post_dom_tree = DominatorTree::PostDom(arena_, graph);  
      // run fusion algorithm.  
      for (int phase = 0; phase < 3; ++phase) {  
        this->RunFuse(graph, post_dom_tree, phase);  
      }  
      return std::move(groups_);  
    }  


여기서 `IndexedForwardGraph`와 `DominatorTree`를 얻은 뒤, 세 개의 phase를 사용해 operator fusion을 완료합니다. 다음으로 phase 0 과정을 따라가 봅시다. 그 전에 `this->InitGroups` 함수가 있는데, 이는 `groups_`를 초기화하는 기능을 수행합니다. 여기서 `groups_`는 `GraphPartitioner` 클래스에 정의된 구조체 변수로, fuse 후의 그래프 구조를 표현합니다. 어떤 operator들이 fuse될 수 있다면, 이 구조체의 `parent`, `master_ref`를 통해 그 노드들 사이에 연결 관계를 구축합니다. Group은 처음에 DAG와 동일한 그래프로 초기화되며, 이것이 바로 여기의 `InitGroups` 함수입니다. 그 후 DAG, `postDominator tree`, 그리고 `group` 그래프의 노드들을 각각 순회하면서, 어떤 op들이 fuse될 수 있는지 판단합니다. `Group` 구조체의 정의와 `InitGroups`의 코드 구현은 다음과 같습니다.


      
    struct Group {  
        /*! \brief The parent in the union find data structure. */  
        Group* parent{nullptr};  
        /*! \brief The pattern of the group */  
        OpPatternKind pattern;  
        /*! \brief reference to the root node. */  
        const tvm::Object* root_ref{nullptr};  
        /*!  
         * \brief Reference to the anchor node,  
         * this field is not nullptr only if pattern is kOutEWiseFusable.  
         */  
        const tvm::Object* anchor_ref{nullptr};  
        /*!  
         * \brief Find the group root, perform path compression  
         * \return The root type node.  
         */  
        Group* FindRoot() {  
          // fast path  
          if (this->parent == nullptr) return this;  
          // slow path with path compression.  
          Group* root = this;  
          while (root->parent != nullptr) {  
            root = root->parent;  
          }  
          for (Group* p = this; p != root;) {  
            Group* parent = p->parent;  
            p->parent = root;  
            p = parent;  
          }  
          return root;  
        }  
      
        /*!  
         * \brief The number of nodes belonging to this group  
         */  
        uint32_t num_nodes{1};  
      };  
      
    // Initialize the groups.  
      void InitGroups(const IndexedForwardGraph& graph) {  
        groups_.resize(graph.post_dfs_order.size());  
        for (size_t nid = 0; nid < groups_.size(); ++nid) {  
          const auto* graph_node = graph.post_dfs_order[nid];  
          auto* group_node = arena_->make<Group>();  
          group_node->pattern = graph_node->pattern;  
          group_node->root_ref = graph_node->ref;  
          // set anchor ref if necessary.  
          if (group_node->pattern == kOutEWiseFusable) {  
            group_node->anchor_ref = graph_node->ref;  
          }  
          groups_[nid] = group_node;  
        }  
      }  


이어서 operator fusion의 핵심 함수 `RunFuse`를 살펴봅시다.


    // execute the fusion algorithm.  
      void RunFuse(const IndexedForwardGraph& graph, const DominatorTree& post_dom_tree, int phase) {  
        // 계산 그래프의 상단부터 순회를 시작한다. 초기 상태에서 groups_의 노드는 IndexedForwardGraph의 노드와 동일하다.  
        for (size_t nid = 0; nid < groups_.size(); ++nid) {  
          // the group of current node has been specified already.  
          auto* graph_node = graph.post_dfs_order[nid];  
          auto* dom_node = post_dom_tree.nodes[nid];  
          Group* group_node = groups_[nid];  
          ICHECK(group_node != nullptr);  
          // no actions for opaque nodes  
          if (group_node->pattern == kOpaque) continue;  
          // no actions needed if the current node have no dominator  
          if (dom_node->parent == nullptr) continue;  
          ICHECK(!graph_node->extern_ref);  
          size_t dom_parent_gindex = dom_node->parent->gnode->index;  
         
       //  fuse 가능한 노드 수가 일정 한도(max_fuse_depth_)를 넘으면 fuse하지 않는다  
          if (CountFusedNodesWithNewChild(graph_node, dom_node->parent->gnode) > max_fuse_depth_)  
            continue;  
      
          if (phase == 2) {  
            // Fuse injective ops into intermediate tuples, if any  
            if (group_node->pattern > kInjective) continue;  
            Group* dom_parent_group = groups_[dom_parent_gindex];  
            Group* dom_root_group = dom_parent_group->FindRoot();  
            // If dom node group has a tuple as its root, we do not fuse tuple fields into it  
            if (dom_root_group->pattern == kTuple) continue;  
            if (dom_parent_group->pattern == kTuple && dom_root_group->pattern <= kInjective) {  
              // Now we know the tuple has been fused into subsequent injective ops  
              auto fcond = [](OpPatternKind kind, bool is_sink) { return kind <= kInjective; };  
              // dom_root_group can also be tuple, as in inception layers  
              // CheckPath is needed to avoid fusing two intermediate tuples  
              if (CheckPath(graph_node, dom_node->parent->gnode, fcond)) {  
                CommitFuse(graph_node, dom_node->parent->gnode);  
              }  
            }  
            continue;  
          }  
      
          // 현재 노드와 부모 노드가 이미 fuse된 경우, 현재 노드를 건너뛰어 중복 fusion을 방지한다  
          if (groups_[dom_parent_gindex] != nullptr &&  
              group_node->FindRoot() == groups_[dom_parent_gindex]->FindRoot()) {  
            continue;  
          }  
          // Do not fuse into tuple for now  
          if (groups_[dom_parent_gindex]->pattern == kTuple) continue;  
          // Try to fuse current node to its post-dominator.  
          if (group_node->pattern == kOutEWiseFusable) {  
            if (phase != 0) continue;  
            // Path for OutEWiseFusable: conv2d  
            // Check if the dominator relation is elemwise.  
            if (dom_node->parent != nullptr && dom_node->pattern == kElemWise) {  
              ICHECK(dom_node->parent->gnode != nullptr);  
              // The fuse can be executed if all the intermediate ops are still broadcast.  
              auto fcond = [](OpPatternKind kind, bool is_sink) { return kind <= kBroadcast; };  
              if (CheckPath(graph_node, dom_node->parent->gnode, fcond)) {  
                CommitFuse(graph_node, dom_node->parent->gnode);  
              }  
            }  
          } else if (group_node->pattern <= kBroadcast) {  
            // Pre-condition: can only be fused to parent which is injective or reduction.  
            if (dom_node->parent != nullptr &&  
                (dom_node->pattern <= kInjective || dom_node->pattern == kCommReduce)) {  
              // Check if all the intermediate ops are still broadcast.  
              // The final terminal node can already be fused to a OutEWiseFusable group.  
              auto fcond = [](OpPatternKind kind, bool is_sink) {  
                if (!is_sink) {  
                  // Elemwise, broadcast, and injective ops on the parallel branches  
                  // are allowed be fused to the elemwise/broadcast anchor.  
                  return kind <= kInjective;  
                } else {  
                  return (kind <= kBroadcast || kind == kCommReduce || kind == kInjective ||  
                          kind == kOutEWiseFusable);  
                }  
              };  
              if (CheckPath(graph_node, dom_node->parent->gnode, fcond)) {  
                CommitFuse(graph_node, dom_node->parent->gnode);  
              }  
            }  
          } else if (group_node->pattern == kInjective || group_node->pattern == kTuple) {  
            // defer injective fusion to second phase.  
            // so conv2d always finishes fusing.  
            if (phase != 1) continue;  
            // Check if all path are injective.  
            auto fcond = [](OpPatternKind kind, bool is_sink) { return kind <= kInjective; };  
            if (CheckPath(graph_node, dom_node->parent->gnode, fcond)) {  
              CommitFuse(graph_node, dom_node->parent->gnode);  
            }  
          } else {  
            // do nothing.  
            ICHECK(group_node->pattern == kCommReduce);  
          }  
        }  
      }  
    };  


`phase=0`일 때는 kElemWise, kBroadcast의 fusion만 처리한다는 것을 어렵지 않게 알 수 있습니다. 앞서 소개한 op fusion 우선순위 관계로 보면, ElemWise/Broadcast는 ElemWise/Broadcast/Injective/CommReduce와 fusion을 시도할 수 있습니다.

이어서 다른 두 핵심 함수, 즉 `CheckPath`와 `CommitFuse`를 살펴봅시다. `CheckPath`는 src에서 sink 노드까지의 경로가 fuse될 수 있는지 검사하는 것이고, 판단 조건은 일종의 정규 표현식, 즉 위의 `auto fcond = [](OpPatternKind kind, bool is_sink) ...`를 사용합니다. 그러면 `CheckPath` 함수의 코드 구현을 살펴보겠습니다.


    /*!  
       * \brief Check all the node and edge pattern  
       *  between src and sink satisfies fcond.  
       *  
       * src is not checked.  
       *  
       * \param src The source node.  
       * \param sink The termination node.  
       * \param fcond The condition to be checked.  
       * \tparam F the condition function, with signature  
       * \note sink must be a post-dominator of src.  
       */  
      template <typename F>  
      bool CheckPath(IndexedForwardGraph::Node* src, IndexedForwardGraph::Node* sink, F fcond) {  
        ICHECK(!src->extern_ref);  
        visited_.clear();  
        ICHECK(src != sink);  
        for (auto link = src->outputs.head; link != nullptr; link = link->next) {  
          if (!CheckPath_(link->value.node, sink, fcond)) return false;  
        }  
        return true;  
      }  


CheckPath 함수에서는 `CheckPath_`라는 함수를 재귀 호출합니다. 이를 통해 현재 노드에서 LCA 노드까지의 모든 경로를 순회하고, `fcond(gnode->pattern, src == sink)`를 사용해 fusion 규칙에 부합하는지 판단할 수 있습니다.

**종단 노드** 만이 kCommReduce와 kOutEWiseFusable을 지원하고, **path 경로 위의 노드** 는 path 경로에서 ElemWise/Broadcast/Injective의 다양한 operator fusion만 지원합니다.

그런 다음 `CommitFuse`라는 핵심 함수를 살펴봅시다. dominator tree의 앞뒤 노드가 fuse될 수 있다고 판단되면 fusion 동작을 수행합니다. 함수 구현은 다음과 같습니다.


    /*!  
       * \brief Commit fusion operation.  
       * \param src The source node.  
       * \param sink The termination node.  
       * \note sink must be a post-dominator of src.  
       */  
      void CommitFuse(IndexedForwardGraph::Node* src, IndexedForwardGraph::Node* sink) {  
        Group* target = groups_[sink->index];  
        visited_.clear();  
        ICHECK(src != sink);  
        CommitFuse_(src, sink, target);  
      }  


여기서는 fuse 대상 노드를 지정하기 위해 `Group* target` 포인터가 새로 추가되었습니다. 그런 다음 `CommitFuse_`라는 재귀 함수를 호출하여 fusion을 완료합니다.


      // Internal implelementation of CommitFuse  
      void CommitFuse_(IndexedForwardGraph::Node* src, IndexedForwardGraph::Node* sink, Group* target) {  
        if (src == sink) return;  
        if (visited_.count(src)) return;  
        visited_.insert(src);  
        Group* gnode = groups_[src->index];  
        ICHECK(gnode != nullptr);  
        // merge the current group to the parent if possible.  
        MergeFromTo(gnode, target);  
        for (auto link = src->outputs.head; link != nullptr; link = link->next) {  
          CommitFuse_(link->value.node, sink, target);  
        }  
      }  


여기의 `MergeFromTo(gnode, target);` 노드 부분이 fusion의 실제 구현입니다. 코드는 다음과 같습니다.


    /*!  
       * \brief Merge the child group to the parent.  
       * \param child The child group.  
       * \param parent The parent group.  
       */  
      void MergeFromTo(Group* child, Group* parent) {  
        child = child->FindRoot();  
        parent = parent->FindRoot();  
        if (child == parent) return;  
        // update the number of nodes of the parent group  
        parent->num_nodes += child->num_nodes;  
        child->parent = parent;  
        // update anchor ref and pattern  
        if (child->anchor_ref != nullptr) {  
          ICHECK(parent->anchor_ref == nullptr);  
          parent->anchor_ref = child->anchor_ref;  
          parent->pattern = CombinePattern(child->pattern, parent->pattern);  
        }  
      }  


먼저 이 함수 안의 `child->FindRoot()` 함수를 살펴봅시다. 위에서 `Group` 정의를 소개할 때 코드를 한 번 붙였는데, 이는 현재 노드의 부모 노드를 찾는 데 사용됩니다. 왜 부모 노드를 찾을까요? parent가 계산 그래프 안에서 fuse 가능한 노드들 중 가장 아래쪽에 있는 노드이기 때문입니다. 예를 들어 A-B-C가 fuse 가능하다면, B의 parent는 결국 C가 되고, A의 parent도 C가 됩니다. 따라서 위 함수의 첫 네 줄 동작 후, 중간 노드들의 parent는 모두 target이나 target의 root를 가리키게 됩니다. 이는 union-find와 매우 비슷합니다. 이 과정이 잘 이해되지 않는다면 union-find의 합치기 함수를 한 번 보시면 됩니다.

이런 동작을 마치고 나면, `RunFuse`는 `graph_`라는 그래프를 반환합니다. 이 그래프는 DAG이고, 그래프 안의 각 노드의 parent는 그 노드가 fuse될 수 있는 목표 노드를 나타냅니다. 마지막으로 `std::vector<GraphPartitioner::Group*>`이라는 자료구조에 저장됩니다.

그 후 마지막 동작이 하나 더 있습니다. 위의 출력 그래프를 순회하면서 실제 fusion을 완료하고, fuse된 Expr을 생성하는 일입니다. 이것은 `FuseMutator`라는 클래스를 통해 수행됩니다. 이 클래스는 노드와 그에 대응하는 fuse 가능 목표 노드를 매핑하기 위해 `std::unordered_map<const Object*, GraphPartitioner::Group*> gmap_;`을 정의합니다. `FuseMutator` 클래스의 멤버 함수 `Transform`이 보여주듯이 다음과 같습니다.


    // Run the transform  
      Expr Transform(const Expr& body, int fuse_opt_level, size_t max_fuse_depth) {  
        // setup the group map.  
        auto graph = IndexedForwardGraph::Create(&arena_, body);  
        auto groups = GraphPartitioner(&arena_, fuse_opt_level, max_fuse_depth).Partition(graph);  
        for (size_t nid = 0; nid < graph.post_dfs_order.size(); ++nid) {  
          ICHECK(graph.post_dfs_order[nid]->ref != nullptr);  
          gmap_[graph.post_dfs_order[nid]->ref] = groups[nid];  
        }  
        // The following line can be used for debug.  
        // this->DebugDumpGroup(body);  
        return this->Mutate(body);  
      }  


이 함수의 구체적인 세부 사항은 더 이상 펼쳐 설명하지 않겠습니다. 관심 있는 독자는 직접 이 클래스의 세부 구현을 확인하시고, 이해되지 않는 부분이 있으면 저와 논의하셔도 좋습니다. 이 클래스가 fuse될 수 있는 일련의 노드들을 Function Expr로 변환해 반환하는 일을 하며, 이 과정이 있어야 비로소 operator fusion Pass가 완성된다는 점만 알면 충분합니다.

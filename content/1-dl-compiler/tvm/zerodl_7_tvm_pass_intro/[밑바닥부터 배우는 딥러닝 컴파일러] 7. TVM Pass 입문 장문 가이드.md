# 【밑바닥부터 배우는 딥러닝 컴파일러】 7. TVM Pass 입문 장문 가이드

# 0x0. 머리말

이 글은 TVM 0.8.0.dev 버전을 기반으로 합니다. 【밑바닥부터 배우는 딥러닝 컴파일러】 5. TVM Relay 및 Pass 소개 글에서는 Relay와 Pass 메커니즘을 간단히 소개했습니다. 하지만 Pass의 기반 구조(Pass Infrastructure)와 Relay 트리 구조에 대해서는 자세히 다루지 않았기 때문에, 이번 글에서는 Pass Infrastructure와 Relay 트리 구조를 주로 소개하고, 이러한 핵심 기초 지식을 바탕으로 Constant Folding Pass에 대해 자세히 알아보고자 합니다. 이 글을 끝까지 읽으면 독자가 TVM의 Pass에 대해 더 깊이 이해할 수 있고, 다른 Pass를 읽거나 커스텀 Pass를 구현할 때에도 매우 편안하게 할 수 있을 것입니다.

# 0x1. Pass Infrastructure

먼저 Pass Infrastructure를 공식 문서를 바탕으로 소개하겠습니다.

Pass의 일반적인 등록 및 실행 흐름을 설명하기 전에, TVM의 Pass Infrastructure를 먼저 소개하겠습니다. 공식 문서를 참고하세요:https://tvm.apache.org/docs/arch/pass_infra.html

Relay와 TVM IR은 모델의 성능 지표(예: 평균 추론 속도, 메모리 점유율, 특정 디바이스의 전력 소비 등)를 향상시키기 위한 일련의 최적화 pass들을 포함합니다. TVM에는 표준 최적화 방법뿐만 아니라 머신러닝에 특화된 최적화 방법들이 있으며, 여기에는 상수 폴딩(constant folding), 죽은 코드 제거(dead-code-elimination), operator 레이아웃 변경, operator fusion, 버퍼 처리, 루프 변환 등이 포함됩니다. 각 Pass는 traversal 중 그리고/또는 그 이전에 수집된 분석 결과를 사용해 ir-to-ir pass를 구성합니다.

그러나 TVM이 빠르게 발전함에 따라, 이러한 pass들을 더 체계적이고 효율적으로 관리할 방법이 필요해졌습니다. 또한 Relay와 tir 같은 TVM 스택의 다양한 계층에 걸친 pass들을 관리할 수 있는 범용 프레임워크는, 개발자들이 빠르게 프로토타입을 만들고 구현한 pass를 시스템에 끼워 넣는 길을 열어줍니다.

예를 들어 GCC와 LLVM 같은 많은 기존 프로덕션 컴파일러들은 pass의 실행을 효과적으로 관리하기 위해 pass manager를 채택하고 있습니다. 처음에는 pass의 수가 적기 때문에 pass 관리가 단순했지만, 성숙한 컴파일러는 수백 개의 개별 pass를 포함하게 됩니다. 외부 사용자들은 종종 직접 만든 단일 pass 순서를 수정하지 않고도 커스텀 pass가 올바르게 스케줄되기를 원합니다.

마찬가지로 Pytorch와 MXNet Gluon 같은 최신 딥러닝 프레임워크들도 각각 Sequential과 Block을 통해 pass-style의 레이어 구성 방식을 활성화하는 추세입니다. 이러한 구조 덕분에 이런 최신 프레임워크들은 모듈/레이어를 컨테이너에 편리하게 추가하고 신경망을 쉽게 구축할 수 있게 됩니다.

Relay pass infra의 설계는 LLVM에서 사용되는 계층적 pass manager와 유명한 딥러닝 프레임워크에서 사용되는 block-style 컨테이너에서 큰 영향을 받았습니다. pass infra의 주요 목표는 다음과 같습니다:

  * 더 나은 optimizer 프로그래밍 오케스트레이션을 구현합니다. 이는 사용자가 자신의 최적화 파이프라인을 유연하게 커스터마이즈하고 구축할 수 있게 해줍니다.
  * pass를 디버깅할 수 있는 사용자 친화적인 방법을 제공합니다.
  * 개발자가 pass 간의 의존성을 수동으로 별도로 해결해야 하는 부담을 줄입니다.
  * 개발자가 새로운 pass를 구현하는 난이도를 단순화합니다. 예를 들어, 사용자가 Python에서 pass를 구현하고 pass infra가 그 실행을 다루도록 허용합니다.



**The Design**

우리는 사용자에게 쉽게 확장 가능한 기능을 제공하는 데 집중하여, 사용자가 하위 호환성을 잃지 않고 빠르게 새로운 pass를 추가할 수 있도록 합니다. 설계는 backend와 frontend로 구성됩니다. 전자는 pass infra의 주요 로직을 구현합니다. 후자는 사용자가 상호작용할 수 있는 간단한 API를 제공하며, 즉 사용자가 자신만의 최적화 파이프라인을 빠르게 만들 수 있도록 해줍니다.

**C++ Backend**

우리는 pass에 필요한 기본 정보를 담는 `PassInfo` 객체를 제공합니다. `name`은 pass 이름, `opt_level`은 pass가 활성화될 최적화 레벨을 나타내며, `required`는 어떤 pass를 실행하기 위해 필요한 pass들을 나타냅니다(자세한 내용은 `include/tvm/ir/transform.h` 참고). 예를 들어, pass를 등록할 때(뒤에서 소개) pass 개발자는 pass의 이름, 실행될 최적화 레벨, 그리고/또는 필요한 pass를 지정할 수 있습니다. `opt_level`은 사용자가 제공한 최적화 레벨에서 어떤 pass를 실행해야 하는지를 pass infra가 식별하는 데 도움을 줍니다. `required` 필드는 pass infra가 pass 의존성을 해결하는 데 사용할 수 있습니다.
    
    
    class PassInfoNode : public Object {  
      String name;  
      int opt_level;  
      Array<String> required;  
    };  
    

**PassContext**

`PassContext`는 최적화 pass에 유용한 정보를 담고 있습니다. 예를 들어, 오류 보고 시스템을 포함하여 pass 작성자가 최적화 실패 원인에 대한 주석을 제공할 수 있습니다. `PassContext`는 또한 기존의 `BuildConfig`를 대체하기 위해 설계되었으며, 사용자가 최적화 레벨이나 필수/비활성 pass 등 컴파일 옵션을 구성하는 데 도움을 줍니다. 예를 들어, 우리는 `opt_level=3`에서 모든 pass를 실행하되, `PassContext`가 제공하는 `disabled_pass=xx`로 일부 pass를 비활성화하는 구성을 가질 수 있습니다. 이제 `opt_level=3`에서 모든 pass를 전역적으로 처리하면서 비활성 pass 목록에 있는 것들은 제외할 수 있습니다.

이 클래스는 사용자가 Python으로 작성하기 편하도록 설계되었으며, 그 문법은 특정 구성 아래에서 최적화를 실행할 수 있게 합니다. 또한 사용자는 `PassContext::Current()`를 통해 어떤 프로그램 범위 내에서 사용할 수 있는 context를 스레드 안전한 방식으로 가져올 수 있는데, 이는 ThreadLocalStore가 생성된 pass context 객체를 보관하는 데 사용되기 때문입니다. ThreadLocalStore에 관해서는 다음 글을 추천합니다: https://zhuanlan.zhihu.com/p/61587053. TVM은 Java의 ThreadLocalStore를 모방하여 C++ 레이어에서 직접 구현해 스레드를 관리합니다. 나중에 C++와 Python API를 사용해 pass context를 활용하는 컴파일 파이프라인을 만드는 예시를 보여드리겠습니다.
    
    
    class PassContextNode : public Object {  
     public:  
      ErrorReporter err_reporter;

      tvm::Array<tvm::Expr> disabled_pass;  
    };

**Pass Constructs**

pass infra는 계층적인 방식으로 설계되어 있어, 다양한 단위의 Relay/tir 프로그램에서 작동할 수 있습니다. 다양한 최적화 pass의 기반으로 순수 가상 클래스 `PassNode`가 도입되었습니다. 이 클래스는 modules, functions, sequences of passes에서 서브클래스가 반드시 구현해야 하는 몇 가지 가상 메서드를 포함합니다.
    
    
    class PassNode : Object {  
      virtual PassInfo Info() const = 0;  
      virtual Module operator()(const IRModule& mod  
                                const PassContext& pass_ctx) const = 0;  
    };  
    

멤버 함수는 pass가 어떻게 구현되어야 하는지 보여줍니다. 예를 들어 항상 특정 context에서 `IRModule`에 대해 작동하며, 모든 pass는 `Module` to `Module`의 매니저 안에서 동작하도록 설계되었습니다. 따라서 pass infra가 통제하는 최적화는 항상 전체 module을 갱신합니다.

다양한 종류의 최적화 pass를 구현하기 위해 여러 서브클래스가 만들어졌습니다. 예를 들어 function-level passes, module-level passes, sequential passes 등입니다. 각 서브클래스 자체가 pass 매니저 역할을 할 수 있습니다. 예를 들어, 필요한 pass들을 모아 실행하거나, 주어진 메타데이터를 기반으로 의존성 그래프를 구축할 수 있습니다. 그 전체 정의는 `src/relay/ir/transform.cc`와 `src/ir/transform.cc`에서 찾을 수 있습니다.

**Module-Level Passes**

Module Level Passes는 주로 전역 및 프로시저 간 최적화(IPO)에 사용되며, LLVM에서 사용되는 module pass와 유사합니다. Relay에서 module의 global picture가 필요한 일부 일반적인 pass들, 예를 들어 A-normal form conversion, lambda lifting 등이 이 집합에 속합니다. 이 레벨에서는 사용자가 module에 function을 추가하거나 삭제할 수도 있습니다.
    
    
    class ModulePassNode : PassNode {  
      PassInfo pass_info;  
      runtime::TypedPackedFunc<Module(Module, PassContext)> pass_func;  
      Module operator()(const Module& mod, const PassContext& pass_ctx) const final;  
      // Other members/methods are omitted  
    };  
    

`pass_info`는 module-level pass에 필요한 정보를 유지합니다. `pass_func`는 실제 최적화를 구현합니다. 예를 들어, 우리는 module에 대해 dead-code-elimination을 수행해야 할 수 있습니다. `pass_func` 안에 알고리즘을 구현하고 module에서 실행되도록 할 수 있습니다. 그러면 module 내에서 사용되지 않는 함수를 포함한 죽은 코드를 제거합니다. 이 필드는 packed function으로 설계되어, 이 최적화는 C++뿐만 아니라 Python으로도 구현할 수 있습니다.

**Function-Level Passes**

Function-level passes는 주어진 Relay/tir module에 대한 다양한 함수 내부 함수 레벨 최적화를 구현하는 데 사용됩니다. module의 함수 목록에서 한 번에 하나의 함수를 가져와 최적화하고, 다시 작성된 `Relay Function` 또는 `tir PrimFunc`를 생성합니다. Relay에서의 공통 부분식 제거(EliminateCommonSubexpr) 및 추론 단순화, tir에서의 벡터화 및 flattening storage 등 대부분의 pass가 이 범주에 속합니다.

이 레벨의 pass의 범위는 Relay Function 또는 tir PrimFunc임에 유의하세요. 따라서 이 pass들로는 함수를 추가하거나 삭제할 수 없는데, 이는 전역 정보를 알지 못하기 때문입니다.
    
    
    class FunctionPassNode : PassNode {  
      PassInfo pass_info;  
      runtime::TypedPackedFunc<Function(Function, Module, PassContext)> pass_func;  
      Module operator()(const Module& mod, const PassContext& pass_ctx) const final;  
      bool SkipFunction(const Function& func) const;  
      // Other members/methods are omitted...  
    };  
    

`pass_info`는 방금 Module pass에서 설명한 것과 동일합니다. `pass_func`는 최적화할 함수를 가져오며, 오류 보고에 사용할 수 있도록 Module도 필요합니다. 함수에는 "SkipOptimization"으로 주석을 달아 최적화 과정에서 무시할 수 있습니다.

**Sequential Passes**

`SequentialPass`는 Pytorch의 `nn.Sequential`과 유사하며, 실행할 여러 pass를 포함합니다.
    
    
    class SequentialPassNode : PassNode {  
      PassInfo pass_info;  
      // Passes need to be executed.  
      Array<Pass> passes;  
      bool PassEnabled(const PassInfo& info) const;  
      Module operator()(const Module& mod, const PassContext& pass_ctx) const final;  
    };  
    

현재 Relay에서는 일부 pass만 이 그룹에 속합니다. 예를 들어, `FoldScaleAxis`는 내부적으로 `ForwardFoldScaleAxis`와 `BackwardFoldScaleAxis`를 스케줄해야 합니다. 또한 `BackwardFoldScaleAxis`를 먼저 완료하는 것이 권장됩니다. 따라서 이 pass는 `SequentialPass`의 이상적인 후보입니다.

다음 코드는 sequential pass 안에서 각각의 pass가 어떻게 호출되는지 보여줍니다.
    
    
    Module SequentialNode::operator()(const Module& module,  
                                      const PassContext& pass_ctx) const {  
      Module mod = module;  
      for (const Pass& pass : passes) {  
        ICHECK(pass.defined()) << "Found undefined pass for optimization.";  
        const PassInfo& pass_info = pass->Info();  
        if (!PassEnabled(pass_info))  continue;  
        for (const auto& it : pass_info->required) {  
          const auto* name = it.as<tvm::ir::StringImm>();  
          ICHECK(name);  
          mod = GetPass(name->value)(mod, pass_ctx);  
        }  
        mod = pass(mod, pass_ctx);  
      }  
      return mod;  
    }  
    

pass를 호출할 때, 먼저 이 pass가 활성화되어 있는지 확인합니다. 이는 사용자가 명시적으로 이 pass를 비활성화했는지 먼저 확인한 다음, 사용자가 필수 pass로 지정했는지 확인하는 방식으로 이루어집니다. 그래도 이 pass가 활성화되어 있는지 확실하지 않다면, 그 `opt_level`을 확인합니다. 그 `opt_level`이 pass context에 구성된 최적화 레벨보다 낮지 않을 때만 이 pass가 활성화되고 따라서 실행됩니다.

pass를 실행하려면, 먼저 pass name으로 TVM packed function 레지스트리에 이미 등록된 pass를 사용해야 합니다. 이는 각 pass가 API 인터페이스를 등록하기 때문에 가능하며, 이는 뒤에서 보여드리겠습니다.
    
    
    Pass GetPass(const std::string& pass_name) {  
      using tvm::runtime::Registry;  
      std::string fpass_name = "relay._transform." + pass_name;  
      const auto* f = Registry::Get(fpass_name);  
      ICHECK(f != nullptr) << "Cannot find " << fpass_name  
                          << "to create the pass " << pass_name;  
      return (*f)();  
    }  
    

위에서 언급한 각 종류의 Pass를 만들기 위한 helper function들이 제공됩니다. 이 helper function들은 Python 프론트엔드에도 노출되어 있어, 사용자가 Python API를 사용해 특정 pass 객체를 편리하게 만들 수 있습니다.

**Pass Registration**

지금까지 다양한 레벨의 pass 개념과 컴파일에 사용되는 context를 소개했습니다. 사용자가 얼마나 쉽게 pass를 등록할 수 있는지는 의미 있는 일입니다. 예를 들어 constant folding을 예로 들겠습니다. 이 pass는 Relay Function 안의 상수를 폴딩하기 위해 이미 구현되어 있습니다(`tvm/src/relay/transforms/fold_constant.cc`에서 찾을 수 있습니다).

`Expr`을 `Expr`로 변환하는 API가 제공됩니다.
    
    
    Expr FoldConstant(const Expr& expr);  
    

이 pass를 pass infra에 등록하려면, 먼저 이 pass가 어떤 레벨에서 실행될지 결정해야 합니다. constant folding은 단일 함수에서 일어나므로, 직관적으로 `CreateFunctionPass`를 통해 `FunctionPass`를 만들어야 합니다. `pass_func`는 packed function으로 반환되며, 이 함수는 `IRModule`의 각 function에서 `Expr` to `Expr` API를 호출합니다. `{}`는 이 pass에 전제 조건이 필요하지 않음을 나타냅니다. 그렇지 않다면 pass 개발자가 그것들을 식별하고 나열해야 합니다.
    
    
    namespace transform {  
      
    Pass FoldConstant() {  
      runtime::TypedPackedFunc<Function(Function, IRModule, PassContext)> pass_func =  
        [=](Function f, IRModule m, PassContext pc) {  
          return Downcast<Function>(FoldConstant(f));  
      };  
      return CreateFunctionPass(pass_func, 2, "FoldConstant", {});  
    }  
      
    TVM_REGISTER_GLOBAL("relay._transform.FoldConstant")  
    .set_body_typed(FoldConstant);  
      
    }  // namespace transform  
    

다른 C++ 모듈에서 이 pass를 적용할 수 있도록, `include/tvm/relay/transform.h`에 다음과 같이 free function을 선언합니다:
    
    
    TVM_DLL Pass FoldConstant();  
    

**Python Frontend**

Python 프론트엔드는 몇 가지 간단한 API만 필요합니다. 예를 들어, 다음과 같이 사용자가 pass를 만들고 실행할 수 있는 API를 제공할 수 있습니다(완전한 구현은 `python/tvm/relay/transform.py`와 `python/tvm/ir/transform.py`에 있습니다). backend는 정보를 받아 어떤 함수를 사용해 Pass 객체를 만들지 결정합니다.

**PassContext**

Python 프론트엔드는 `__enter__`와 `__exit__`를 오버라이드해 `with` 문법을 활성화하기 위한 `PassContext`의 래퍼를 제공합니다. 사용자에게 특정 범위 내에서 사용 중인 context를 가져오기 위한 `current` 정적 메서드를 제공합니다.
    
    
    @tvm._ffi.register_object("transform.PassContext")  
    class PassContext(tvm.runtime.Object):  
        def __enter__(self):  
            _transform.EnterPassContext(self)  
            return self  
      
        def __exit__(self, ptype, value, trace, config):  
            _transform.ExitPassContext(self)  
      
        @staticmethod  
        def current():  
            """Return the current pass context."""  
            return _transform.GetCurrentPassContext()  
    

PassContext는 최적화 레벨이나 필수/비활성 pass를 포함한 컴파일 옵션을 구성하는 데 사용됩니다. 또한 구성 사전을 가져올 수 있어, 다양한 pass가 fallback 디바이스 정보나 루프 언롤링의 step/depth 등 전달된 데이터를 편리하게 가져올 수 있습니다. 필요한 구성을 가져오려면, `TVM_REGISTER_PASS_CONFIG_OPTION`을 통해 키워드를 등록해야 합니다. 예를 들어 loop unrolling pass는 다음을 사용합니다:
    
    
    TVM_REGISTER_PASS_CONFIG_OPTION("tir.UnrollLoop", UnrollLoopConfig);  
    

자세한 내용은 `src/tir/transforms/unroll_loop.cc`를 참고하세요.

**Pass Objects**

`Pass`는 모든 pass 객체의 베이스 클래스입니다. 여기 있는 모든 메서드는 backend에서 구현된 것의 단순 래퍼일 뿐입니다. 사용자가 Python에서 베이스 클래스와 편리하게 상호작용하기 위해 정의되었습니다. pass 베이스 클래스에는 서브클래스를 호출 가능 객체로 만들기 위한 `__call__`만 정의되어 있어, 쉽게 호출(예: `pass_xx(arg)`)되어 실행될 수 있습니다.
    
    
    @register_relay_node  
    class Pass(RelayNode):  
       def __call__(self, mod):  
           return _transform.RunPass(self, mod)  
    

Python 프론트엔드에서 쉽게 pass를 만들고 pass infra가 실행을 다룰 수 있도록 일부 helper API가 제공됩니다. 예를 들어 사용자가 자체 pass나 pass 파이프라인을 커스터마이즈할 수 있도록 `module_pass`, `function_pass`, `sequential`을 제공합니다.

C++ backend에서 구현된 모든 pass에 대해, `python/tvm/ir/transform.py`와 `python/tvm/relay/transform.py`에 각각 대응하는 Python API를 제공합니다. 예를 들어, constant folding은 다음과 같은 Python API를 가지고 있습니다:
    
    
    def FoldConstant():  
        return _transform.FoldConstant()  
    

사용자는 데코레이터를 통해 다음과 같이 pass를 구성할 수 있습니다:
    
    
     @relay.transform.module_pass(opt_level=2)  
     def transform(mod, ctx):  
        tp = relay.TensorType((10,), "float32")  
        x = relay.var("x", tp)  
        gv = relay.GlobalVar("abs")  
        func = relay.Function([x], relay.abs(x))  
        new_mod = relay.Module({gv: func})  
        new_mod.update(mod)  
        return new_mod  
      
    module_pass = transform  
    assert isinstance(module_pass, transform.ModulePass)  
    assert module_pass.info.opt_level == 2  
    

여기서 `transform` 함수는 입력 module에 `abs` 함수를 추가하지만, 이는 module level의 어떤 커스텀 pass라도 될 수 있습니다. 이 `module_pass`를 만든 후, 사용자는 어떤 `Relay` 모듈에든 적용할 수 있습니다. 예를 들어, empty module을 만들고 이 pass를 적용해 `abs` 함수를 추가할 수 있습니다.
    
    
    mod = relay.Module()  
    mod = module_pass(mod)  
    

대응되게, `function_pass`에도 이러한 기능을 제공합니다. 예를 들어, function-level pass의 예시는 다음과 같이 작성할 수 있습니다:
    
    
    @relay.transform.function_pass(opt_level=1)  
    class TestReplaceFunc:  
       def __init__(self, new_func):  
          self.new_func = new_func  
          def transform_function(self, func, mod, ctx):  
             # Just for demo purposes  
             # Transform func to new_func  
             return self.new_func  
      
    x = relay.var("x", shape=(10, 20))  
    f1 = relay.Function([x], x)  
    f2 = relay.Function([x], relay.log(x))  
    # fpass is now a special pass that replaces every  
    # function to f1  
    fpass = TestReplaceFunc(f1)  
    # Now every function in input_mod is replaced by f1  
    res_mod = fpass(input_mod)  
    

또는, 사용자는 데코레이터를 사용하지 않고 직접 pass를 등록한 다음 호출할 수도 있습니다. 자체 최적화 파이프라인을 어떻게 커스터마이즈하고 Relay 및 tir pass를 디버깅할 수 있는지에 대한 더 많은 예시는 use pass infra 튜토리얼(`https://github.com/apache/tvm/blob/main/tutorials/dev/use_pass_infra.py`)을 참고하세요.

# 0x2. TVM Relay 트리 구조

## AST

> 위키피디아에서 발췌. 컴퓨터 과학에서 추상 구문 트리(Abstract Syntax Tree, AST), 또는 줄여서 구문 트리(Syntax tree)는 소스 코드의 구문 구조에 대한 추상적 표현이다. 이는 트리 형태로 프로그래밍 언어의 구문 구조를 표현하며, 트리의 각 노드는 소스 코드 안의 어떤 구조를 나타낸다. 구문이 "추상적"이라고 말하는 이유는, 여기서의 구문이 실제 구문에 등장하는 모든 세부사항을 표현하지 않기 때문이다. 예를 들어, 중첩 괄호는 트리 구조 안에 암시적으로 포함되어 노드 형태로 나타나지 않으며, if-condition-then 같은 조건 분기 명령문은 세 분기를 가진 노드로 표현될 수 있다. 추상 구문 트리에 대응되는 것은 구체적인 구문 트리(보통 파스 트리라고 부름)이다. 일반적으로 소스 코드의 번역 및 컴파일 과정에서 구문 분석기가 파스 트리를 만들고, 거기서 AST를 생성한다. AST가 만들어지면, 의미 분석 단계 등 후속 처리 과정에서 일부 정보가 추가된다.

이전에 TVM Relay의 ONNX frontend를 분석할 때, 각 OP 변환을 완료한 후 `IRModule.from_expr`을 사용해 변환된 모든 Relay Function을 감싸 반환해야 한다고 언급했습니다. 그 과정은 다음과 같으며, 마지막 줄의 코드만 보면 됩니다:
    
    
    def from_onnx(self, graph, opset, get_output_expr=False):  
            """ONNX 모델을 기반으로 Relay IR을 구축한다.  
      
            매개변수  
            ----------  
            graph : onnx protobuf 객체  
               로드된 ONNX Graph  
      
            opset : 연산 셋 버전  
      
            get_output_expr: bool  
                true로 설정되면, 이 변환은 packed module 대신 각 출력 표현식을 반환한다.   
                서브그래프를 Relay로 변환할 때 유용할 수 있다.   
      
            Returns  
            -------  
            mod : tvm.IRModule  
                The returned relay module  
      
            params : dict  
                A dict of name: tvm.nd.array pairs, used as pretrained weights  
            """  
            self.opset = opset  
            # 네트워크의 입력을 relay로 파싱한다. 매개변수라고도 하며, onnx의 initializer는 모델 매개변수를 저장하는 데 사용된다.  
            for init_tensor in graph.initializer:  
                if not init_tensor.name.strip():  
                    raise ValueError("Tensor's name is required.")  
                # 구체적인 구현은 먼저 이 TensorProto에 get_numpy 함수를 사용해 값을 얻고, 특정 형태로 reshape한 다음, 이 numpy를 기반으로 tvm.nd.array를 만든다.  
                array = self._parse_array(init_tensor)  
                # 앞서 설명했듯이, 매개변수 freeze가 설정되어 있으면 이 매개변수를 Relay의 상수 OP로 설정한다  
                if self._freeze_params:  
                      
                    self._nodes[init_tensor.name] = _expr.const(array)  
                else:  
                    self._params[init_tensor.name] = array  
                    self._nodes[init_tensor.name] = new_var(  
                        init_tensor.name,  
                        shape=self._params[init_tensor.name].shape,  
                        dtype=self._params[init_tensor.name].dtype,  
                    )  
            # ONNX 모델의 입력을 파싱한다  
            for i in graph.input:  
                # from onnx v0.2, GraphProto.input has type ValueInfoProto,  
                #  and the name is 'i.name'  
                # i 입력의 이름, shape, 데이터 타입, 그리고 shape의 각 차원에 대응되는 이름을 가져온다  
                i_name, i_shape, d_type, i_shape_name = get_info(i)  
                # i 입력이 가중치 매개변수인지 입력인지 판단한다  
                if i_name in self._params:  
                    # i is a param instead of input  
                    self._num_param += 1  
                    self._params[i_name] = self._params.pop(i_name)  
                    self._nodes[i_name] = new_var(  
                        i_name, shape=self._params[i_name].shape, dtype=self._params[i_name].dtype  
                    )  
                # 입력 노드가 이미 Relay IR에 있다면 처리할 필요가 없다  
                elif i_name in self._nodes:  
                    continue  
                else:  
                    # 진짜 입력 노드는 사용자의 지정에 의존한다  
                    self._num_input += 1  
                    self._input_names.append(i_name)  
                    if i_name in self._shape:  
                        i_shape = self._shape[i_name]  
                    else:  
                        if "?" in str(i_shape):  
                            warning_msg = (  
                                "Input %s has unknown dimension shapes: %s. "  
                                "Specifying static values may improve performance"  
                                % (i_name, str(i_shape_name))  
                            )  
                            warnings.warn(warning_msg)  
                    if isinstance(self._dtype, dict):  
                        dtype = self._dtype[i_name] if i_name in self._dtype else d_type  
                    else:  
                        dtype = d_type  
                    self._nodes[i_name] = new_var(i_name, shape=i_shape, dtype=dtype)  
                self._inputs[i_name] = self._nodes[i_name]  
            # Only check user inputs in the outer-most graph scope.  
            if self._old_manager is None:  
                assert all(  
                    [name in self._input_names for name in self._shape.keys()]  
                ), "User specified the shape for inputs that weren't found in the graph: " + str(  
                    self._shape  
                )  
            # 지원되지 않는 operator 목록을 가져온다  
            convert_map = _get_convert_map(opset)  
            unsupported_ops = set()  
            for node in graph.node:  
                op_name = node.op_type  
                if (  
                    op_name not in convert_map  
                    and op_name != "Constant"  
                    and op_name not in _identity_list  
                ):  
                    unsupported_ops.add(op_name)  
            # 지원되지 않는 operator 집합을 출력한다  
            if unsupported_ops:  
                msg = "The following operators are not supported for frontend ONNX: "  
                msg += ", ".join(unsupported_ops)  
                raise tvm.error.OpNotImplemented(msg)  
            # 여기까지 왔다는 것은 이 ONNX 모델의 모든 operator가 Relay에 의해 지원되며, 정상적으로 변환을 진행할 수 있음을 의미한다  
            for node in graph.node:  
                op_name = node.op_type  
                # attribute 매개변수를 파싱한다  
                attr = self._parse_attr(node.attribute)  
                # onnx 입력 객체를 만들고 채운다.  
                inputs = onnx_input()  
                for i in node.input:  
                    if i != "":  
                        # self._renames.get(i, i)는 ONNX Graph의 각 노드의 입력을 가져오는 데 사용된다  
                        inputs[i] = self._nodes[self._renames.get(i, i)]  
                    else:  
                        inputs[i] = None  
                i_name = self._parse_value_proto(node)  
                node_output = self._fix_outputs(op_name, node.output)  
                attr["tvm_custom"] = {}  
                attr["tvm_custom"]["name"] = i_name  
                attr["tvm_custom"]["num_outputs"] = len(node_output)  
       # 변환 작업을 수행한다  
                op = self._convert_operator(op_name, inputs, attr, opset)  
                # op의 출력은 하나일 수도 있고 여러 개일 수도 있다  
                if not isinstance(op, _expr.TupleWrapper):  
                    outputs_num = 1  
                else:  
                    outputs_num = len(op)  
                if outputs_num > 1:  
                    # ONNX의 일부 노드는 선택적 출력을 지원한다  
                    # 이 부분은 ONNX Graph에서 누락된 출력을 검색하고 필요 없는 노드를 제거한다  
                    valid_outputs = [False] * outputs_num  
                    for i, output in enumerate(node_output):  
                        if output != "":  
                            valid_outputs[i] = True  
                    # If we have outputs ONNX isn't expecting, we need to drop them  
                    # ONNX가 예상하지 않은 출력이 있으면 제거해야 한다  
                    if not all(valid_outputs):  
                        tup = op.astuple()  
                        # TupleWrapper can also wrap ops with TupleType outputs  
                        if isinstance(tup, _expr.Tuple):  
                            # For tuples, we extract the fields instead of using GetTupleItem  
                            outputs = [tup.fields[i] for i, valid in enumerate(valid_outputs) if valid]  
                        else:  
                            # For call nodes, we need to GetTupleItem  
                            outputs = [op[i] for i, valid in enumerate(valid_outputs) if valid]  
                        # Create the new op with valid outputs  
                        if len(outputs) == 1:  
                            op = outputs[0]  
                        else:  
                            op = _expr.TupleWrapper(outputs, len(outputs))  
                        # Drop invalid outputs for the onnx node  
                        outputs_num = len(outputs)  
                        node_output = [output for output in node_output if output != ""]  
                assert (  
                    len(node_output) == outputs_num  
                ), "Number of output mismatch {} vs {} in {}.".format(  
                    len(node_output), outputs_num, op_name  
                )  
                # 출력이 하나인 경우 상수 OP일 가능성이 있으며, 한 번의 constant folding 기능을 실행할 수 있다  
                if outputs_num == 1:  
                    self._nodes[node_output[0]] = fold_constant(op)  
                else:  
                    op = _expr.TupleWrapper(fold_constant(op.astuple()), len(op))  
                    for k, i in zip(list(node_output), range(len(node_output))):  
                        self._nodes[k] = op[i]  
      
            # ONNX 모델의 출력을 파싱한다  
            outputs = [self._nodes[self._parse_value_proto(i)] for i in graph.output]  
            outputs = outputs[0] if len(outputs) == 1 else _expr.Tuple(outputs)  
            # 변환된 표현식을 직접 반환해야 한다면 여기서 return  
            if get_output_expr:  
                return outputs  
            # ONNX Graph로부터의 입력과 매개변수의 순서를 유지하되, Relay로 변환되어야 하는 노드들만 포함한다  
            free_vars = analysis.free_vars(outputs)  
            nodes = {v: k for k, v in self._nodes.items()}  
            free_vars = [nodes[var] for var in free_vars]  
            for i_name in self._params:  
                if i_name in free_vars and i_name not in self._inputs:  
                    self._inputs[i_name] = self._nodes[i_name]  
            # 우리의 출력 표현식과 모든 입력 변수에 따라 함수를 만든다.   
            func = _function.Function([v for k, v in self._inputs.items()], outputs)  
            # 이 함수를 IRModule로 감싸 반환하고 동시에 가중치 매개변수도 반환한다  
            return IRModule.from_expr(func), self._params  
    

여기서 `IRModule.from_expr(func)`가 Relay 추상 구문 트리 구조의 구축을 완료합니다. TVM은 이 트리 구조를 `tvm.IRModule` 클래스로 정의하며, 이것이 곧 Relay IR입니다.

## Relay 트리 구조

이제 Relay 추상 구문 트리, 즉 `tvm.IRModule` 관련 데이터 구조를 학습해 봅시다.

### 노드 정의

트리의 노드 정의는 `/include/tvm/relay/expr.h`에 있으며, 주로 다음과 같은 종류가 있습니다: ConstantNode, VarNode, TupleNode, CallNode, LetNode, IfNode.

이 노드들은 모두 `include/tvm/ir/expr.h`에 정의된 `RelayExprNode`를 상속하고, `RelayExprNode`는 또한 `BaseExprNode`를 상속합니다. `RelayExprNode`가 무엇을 할 수 있는지는 다음 주석들을 참고할 수 있습니다:
    
    
    /*!  
     * \brief Base node of all non-primitive expressions.  
     *  
     * RelayExpr supports tensor types, functions and ADT as  
     * first class citizens. The life-cycle of the corresponding  
     * objects are implicitly managed by the language.  
     *  
     * \sa RelayExpr  
     */  
      
    /*!  
      * \brief 모든 비-원시(non-primitive) 표현식의 기본 노드.  
      *  
      * RelayExpr는 텐서 타입, 함수 및 ADT를 일등 시민으로 지원한다.  
      * 대응되는 객체의 라이프사이클은  
      * 언어에 의해 암묵적으로 관리된다.  
      *  
      * \sa RelayExpr  
      */   
    

그런 다음 IfNode와 CallNode를 예로 들어 그들의 구현을 살펴보겠습니다:
    
    
    class IfNode : public ExprNode {  
     public:  
      /*! \brief The condition */  
      Expr cond;  
      /*! \brief The expression evaluated when condition is true. */  
      Expr true_branch;  
      /*! \brief The expression evaluated when condition is false */  
      Expr false_branch;  
      
    };  
      
    class CallNode : public ExprNode {  
     public:  
      /*!  
       * \brief The operator(function) being invoked  
       *  
       *  - It can be tvm::Op which corresponds to the primitive operators.  
       *  - It can also be user defined functions (Function, GlobalVar, Var).  
       */  
      Expr op;  
      
      /*! \brief The arguments(inputs) of the call */  
      tvm::Array<relay::Expr> args;

여기서는 이 노드들의 멤버 변수를 보여주며, 이 노드들의 내부 구조를 대략 이해할 수 있습니다.

### 노드의 데이터 접근

Relay 모델의 트리 노드를 이해한 후, 우리는 TVM이 어떻게 이 노드들의 데이터를 접근하는지 알아야 합니다. 공식 문서에서 다음 문장을 찾을 수 있습니다: `ExprVisitor`는 프로그램을 수정하지 않고 프로그램 분석 및 정보 수집을 수행하는 pass에 사용됩니다. 그리고 `ExprVisitor`는 `ExprFunctor`(`tvm/include/tvm/relay/expr_functor.h`에 정의됨)를 상속하며, `ExprFunctor`는 `VisitExpr_`라는 가상 함수를 설정하여, 파싱 시 다시 `ExprVisitor`로 돌아와 노드를 파싱합니다. `ExprFunctor`는 `public` 인터페이스 메서드 `VisitExpr`를 제공하며, 이는 표현식과 0개 이상의 매개변수를 받아 어떤 타입의 인스턴스를 반환합니다. 이 클래스를 확장할 때, 각 표현식 타입에 대해 `VisitExpr_` 구현을 오버라이드하여 AST 순회 패턴을 정의합니다.

`VisitExpr`와 `VisitExpr_` 사이의 관계는 디스패치와 관련이 있습니다. 각 `VisitExpr_` 정의는 특정 타입의 표현식을 위한 것이지만, 방문할 노드가 어떤 타입인지 항상 알 수 있는 것은 아닙니다. 이 문제를 해결하기 위해, `ExprFunctor`는 주어진 표현식에서 그것을 처리하는 `VisitExpr_` 케이스로 라우팅하는 `VisitExpr` 함수를 제공합니다. C++가 이미 동적 디스패치를 제공하지만, `ExprFunctor`는 자신만의 vtable을 정의하며, `VisitExpr`가 이를 사용합니다. 자체 vtable을 정의함으로써 디스패치를 더 잘 제어할 수 있습니다. 예를 들어, 매번 방문 전에 "Here"를 출력하는 `PrintVisitor` 순회기를 정의하고 싶다면, `VisitExpr`를 오버라이드할 수 있습니다:
    
    
    void PrintVisitor::VisitExpr(const Expr& expr) {  
      std::cout << "Here" << std::endl;  
      ExprFunctor::VisitExpr(expr);  
    }  
    

`ExprFunctor` 자체는 매우 일반적인 클래스이므로, 보통은 `ExprVisitor`나 `ExprMutator`를 확장하게 됩니다. 이 클래스들은 `ExprFunctor`를 확장하고 각 표현식 타입의 일반적인 순회 패턴을 캡처하는 `VisitExpr_`의 기본 구현을 제공합니다. 이러한 기본 구현이 있다는 것은, 다른 동작이 필요한 표현식 타입에 대해서만 `VisitExpr_` 메서드를 오버라이드하면 된다는 의미입니다.

예를 들어 `tvm/src/relay/transforms/fold_constant.cc`의 `ConstantChecker` 클래스는 `ExprVisitor`를 상속하고, `VisitExpr(expr)`을 통해 데이터를 접근합니다. `ExprVisitor`의 `VisitExpr` 멤버 함수 구현은 다음과 같습니다:
    
    
    void ExprVisitor::VisitExpr(const Expr& expr) {  
      auto it = visit_counter_.find(expr.get());  
      if (it != visit_counter_.end()) {  
        ++it->second;  
      } else {  
        using TParent = ExprFunctor<void(const Expr&)>;  
        TParent::VisitExpr(expr);  
        visit_counter_.insert({expr.get(), 1});  
      }  
    }  
      

이 클래스는 실제로 부모 클래스(`ExprFunctor`)의 `VisitExpr`를 호출하는 것을 볼 수 있고, `ExprFunctor`의 `VisitExpr` 구현은 다음과 같습니다:
    
    
    virtual R VisitExpr(const Expr& n, Args... args) {  
        ICHECK(n.defined()) << "Found null pointer node while traversing AST. The previous pass may "  
                               "have generated invalid data.";  
        static FType vtable = InitVTable();  
        return vtable(n, this, std::forward<Args>(args)...);  
      }  
    

`ExprFunctor`가 `VisitExpr` 가상 함수를 설정하고, 파싱 시 다시 `ExprVisitor`로 돌아와 노드를 파싱하는 것을 볼 수 있습니다. `ConstantChecker` 클래스는 `ExprVisitor`를 상속했으므로, `ConstantChecker` 클래스에서 `VisitExpr_`만 오버라이드하면 됩니다.

`ExprFunctor`의 `VisitExpr` 구현에는 `RELAY_EXPR_FUNCTOR_DISPATCH` 매크로가 있는데, 이 매크로의 정의는 다음과 같습니다:
    
    
    #define RELAY_EXPR_FUNCTOR_DISPATCH(OP)                                                    \  
      vtable.template set_dispatch<OP>([](const ObjectRef& n, TSelf* self, Args... args) {     \  
        return self->VisitExpr_(static_cast<const OP*>(n.get()), std::forward<Args>(args)...); \  
      });  
      
    

여기서 `self`는 `ExprFunctor`의 `VisitExpr` 구현 안의 `vtable(n, this, std::forward<Args>(args)...)`이며, `this`는 `ExprFunctor`를 가리킵니다. 또한 `ExprVisitor::VisitExpr` 메서드는 `ExprFunctor`의 함수를 호출하므로, 여기서의 `this`는 `ExprVisitor` 인스턴스를 가리킵니다.

IfNode를 예로 들어 `ExprVisitor`의 `VisitExpr_` 구현을 살펴보겠습니다. `this`가 `ExprVisitor` 인스턴스를 가리키므로, 결국 `ExprVisitor` 인스턴스에서 `visit_counter_` 리스트가 생성됩니다.
    
    
    void ExprVisitor::VisitExpr_(const IfNode* op) {  
      this->VisitSpan(op->span);  
      this->VisitExpr(op->cond);  
      this->VisitExpr(op->true_branch);  
      this->VisitExpr(op->false_branch);  
    }  
    

`visit_counter_`는 `ExprVisitor`에 정의된 `unordered_map`으로, Relay AST를 순회할 때 어떤 종류의 Expr이 등장했는지 표시하고 동시에 등장 횟수를 기록합니다.
    
    
    // Internal visiting counter  
      std::unordered_map<const Object*, size_t> visit_counter_;  
    

### 노드 수정

pass는 Relay 트리 구조, 즉 계산 그래프를 최적화하는 것이며, 최적화는 필연적으로 그래프 구조의 수정을 수반합니다. 이것이 위에서 언급한 `ExprMutator` 서브클래스이며, `ExprVisitor`와 마찬가지로 `ExprFunctor`를 상속합니다. 클래스 정의는 다음과 같습니다:
    
    
    class ExprMutator : public ::tvm::relay::ExprFunctor<Expr(const Expr&)> {  
     public:  
      /*!  
       * \brief Mutate is alias for VisitExpr  
       * \return expr.  
       */  
      Expr Mutate(const Expr& expr) { return this->VisitExpr(expr); }  
      Expr VisitExpr(const Expr& expr) override;  
      Expr VisitExpr_(const VarNode* op) override;  
      Expr VisitExpr_(const ConstantNode* op) override;  
      Expr VisitExpr_(const GlobalVarNode* op) override;  
      Expr VisitExpr_(const OpNode* op) override;  
      Expr VisitExpr_(const TupleNode* op) override;  
      Expr VisitExpr_(const FunctionNode* op) override;  
      Expr VisitExpr_(const CallNode* call_node) override;  
      Expr VisitExpr_(const LetNode* op) override;  
      Expr VisitExpr_(const IfNode* op) override;  
      Expr VisitExpr_(const TupleGetItemNode* op) override;  
      Expr VisitExpr_(const RefCreateNode* op) override;  
      Expr VisitExpr_(const RefReadNode* op) override;  
      Expr VisitExpr_(const RefWriteNode* op) override;  
      Expr VisitExpr_(const ConstructorNode* op) override;  
      Expr VisitExpr_(const MatchNode* op) override;  
      
      /*!  
       * \brief Used to visit the types inside of expressions.  
       *  
       * Can be overloaded to transform the types in arbitrary  
       * ways, one way would be to define a sub-class of type  
       * visitor for types which transform them appropriately.  
       */  
      virtual Type VisitType(const Type& t);  
      virtual Clause VisitClause(const Clause& c);  
      virtual Pattern VisitPattern(const Pattern& c);  
      
     protected:  
      /*! \brief Internal map used for memoization. */  
      std::unordered_map<Expr, Expr, ObjectPtrHash, ObjectPtrEqual> memo_;  
    };  
      
    

우리가 주목해야 할 것은 `memo_`라는 멤버 변수이며, 이 클래스의 `VisitExpr` 구현을 살펴보겠습니다:
    
    
    Expr ExprMutator::VisitExpr(const Expr& expr) {  
      auto it = this->memo_.find(expr);  
      if (it != this->memo_.end()) {  
        return it->second;  
      } else {  
        Expr new_expr = ExprFunctor::VisitExpr(expr);  
        memo_[expr] = new_expr;  
        return new_expr;  
      }  
    }  
    

`memo_`가 그래프 안의 각 노드를 저장하는 것을 볼 수 있습니다. IfNode의 구현을 참고하세요:
    
    
    Expr ExprMutator::VisitExpr_(const IfNode* op) {  
      auto guard = this->Mutate(op->cond);  
      auto true_b = this->Mutate(op->true_branch);  
      auto false_b = this->Mutate(op->false_branch);  
      if (op->cond.same_as(guard) && op->true_branch.same_as(true_b) &&  
          op->false_branch.same_as(false_b)) {  
        return GetRef<Expr>(op);  
      } else {  
        return If(guard, true_b, false_b, op->span);  
      }  
    }  
    

만약 IfNode의 자식 노드들이 모두 수정되지 않았다면, 이 노드 자체를 반환합니다. 그렇지 않다면 새로운 노드 `If(guard, true_b, false_b, op->span);`을 만들어 반환합니다. 여기서 새로운 노드를 만드는 클래스 `If`의 정의와 구현은 각각 `tvm/src/relay/ir/expr.h`와 `tvm/src/relay/ir/expr.cc`에 있습니다:
    
    
    class If : public Expr {  
     public:  
      /*!  
       * \brief The constructor  
       * \param cond The condition of a if node.  
       * \param true_branch The fall through branch  
       * \param false_branch The branch for execution when condition is false.  
       * \param span The source span of the expression.  
       */  
      TVM_DLL If(Expr cond, Expr true_branch, Expr false_branch, Span span = Span());  
      
      TVM_DEFINE_OBJECT_REF_METHODS(If, RelayExpr, IfNode);  
    };  
      
    If::If(Expr cond, Expr true_branch, Expr false_branch, Span span) {  
      ObjectPtr<IfNode> n = make_object<IfNode>();  
      n->cond = std::move(cond);  
      n->true_branch = std::move(true_branch);  
      n->false_branch = std::move(false_branch);  
      n->span = std::move(span);  
      data_ = std::move(n);  
    }  
    

### 정리

이 절에서는 주로 Relay 표현식 트리의 데이터 구조를 분석했습니다. TVM의 모든 pass는 `tvm/include/tvm/relay/expr.h`에 정의된 다양한 Node로 구성된 표현식 트리, 즉 계산 그래프를 기반으로 완성됩니다. 또한 TVM이 이 표현식 노드들에 대한 접근 및 조작을 편리하게 하기 위해 `ExprFunctor` 클래스를 추상화했고, `ExprFunctor` 클래스를 기반으로 `ExprVisitor`나 `ExprMutator`를 확장한 것을 설명했는데, 이는 각 Pass의 C++ backend 코드를 구현할 때 매우 유용합니다. 마지막으로 IfNode의 구현과 constant folding Pass의 `ConstantChecker` 클래스 구현을 예로 들어 이 클래스들의 구체적인 사용법을 보여주었습니다.

# 0x3. Function Pass의 C++ backend 일반 생성 흐름

여기서는 먼저 Constant Folding Pass를 기반으로 Function Pass의 C++ backend 일반 생성 흐름을 설명하겠습니다. 먼저 FoldConstant의 정의를 살펴보겠습니다:
    
    
    namespace transform {  
      
    Pass FoldConstant() {  
      runtime::TypedPackedFunc<Function(Function, IRModule, PassContext)> pass_func =  
          [=](Function f, IRModule m, PassContext pc) {  
            return Downcast<Function>(FoldConstant(f, m));  
          };  
      return CreateFunctionPass(pass_func, 2, "FoldConstant", {});  
    }  
      
    TVM_REGISTER_GLOBAL("relay._transform.FoldConstant").set_body_typed(FoldConstant);  
      
    }  // namespace transform  
    

`CreateFunctionPass` 함수는 FunctionPass를 만드는 데 사용되며, 관련 코드는 다음과 같습니다:
    
    
      
    Pass CreateFunctionPass(  
        const runtime::TypedPackedFunc<Function(Function, IRModule, PassContext)>& pass_func,  
        int opt_level, String name, tvm::Array<String> required) {  
      PassInfo pass_info = PassInfo(opt_level, name, required);  
      return FunctionPass(pass_func, pass_info);  
    }  
      
    FunctionPass::FunctionPass(  
        runtime::TypedPackedFunc<Function(Function, IRModule, PassContext)> pass_func,  
        PassInfo pass_info) {  
      auto n = make_object<FunctionPassNode>();  
      n->pass_func = std::move(pass_func);  
      n->pass_info = std::move(pass_info);  
      data_ = std::move(n);  
    }  
    

`FunctionPass`에서 `FunctionPassNode` 인스턴스를 만들고 그것을 `data_`에 넣는 것을 볼 수 있습니다. `data_`는 `ObjectRef` 클래스의 멤버 변수에서 비롯되며, 여기서 `FunctionPass->Pass->ObjectRef`입니다.

위 코드에서 생성된 `Pass` 객체를 Pass Infrastructure에 제공하면, 이는 주어진 Relay Module의 각 Function에 AST 순회를 적용하도록 보장합니다. 이는 Constant Folding Pass에 대해 우리가 기대하는 동작입니다(가능한 한 모든 상수를 폴딩해야 합니다).

`CreateFunctionPass` 함수는 전달되는 pass의 최적화 레벨(이 예시에서는 2)을 등록할 수 있게 해주며, 이는 pass의 일반적인 효용성, pass의 이름, pass의 모든 의존성을 기반으로 pass를 함께 그룹화하는 데 사용될 수 있습니다. 한 pass의 의존성은 그 pass의 결과에 영향을 줄 수 있는 일련의 pass입니다. `FoldConstant`는 의존성이 없습니다. 그러나 많은 Relay pass는 실제로 타입 정보에 의존하므로, `InferType`은 흔한 의존성입니다. 다른 pass들은 ToANormalForm pass를 통해 프로그램이 A-normal form인 것에 의존할 수 있습니다.

`PassContext` 객체는 오류 보고와 구성 옵션에 사용되는 정보를 담고 있다는 점에 유의하세요. `FoldConstant`는 이 정보가 필요하지 않지만, 다른 Pass는 자신의 `PassContext` 객체를 참조할 수 있습니다.

이제 Pass Infrastructure를 통해 pass를 호출할 수 있지만, 다음 코드 스니펫처럼 pass에 Python 바인딩을 추가하는 것이 가장 좋습니다:
    
    
    TVM_REGISTER_GLOBAL("relay._transform.FoldConstant")  
    .set_body_typed(FoldConstant);  
    

위와 같은 방식으로 Pass 객체가 정의되면, Pass Infrastructure의 Sequential 구조를 사용해 호출할 수 있습니다. 이 구조는 pass 리스트를 받아 Relay 모듈에 순차적으로 적용해 변환된 Module을 얻습니다. 예를 들어, 다음 코드는 FoldConstant와 ToANormalForm Pass(차례로)를 `mod`의 각 함수에 적용해 새로운 Module을 얻습니다.
    
    
    seq = transform.Sequential([  
        relay.transform.FoldConstant(),  
        relay.transform.ToANormalForm()  
    ])  
    new_mod = seq(mod)  
    

`Sequential`의 호출 흐름을 살펴봅시다:
    
    
    // TODO(zhiics): we currenlty only sequentially execute each pass in  
    // a Sequential without the consideration of their orders. The phase  
    // ordering problem needs to be handled in the future.  
    IRModule SequentialNode::operator()(IRModule mod, const PassContext& pass_ctx) const {  
      for (const Pass& pass : passes) {  
        ICHECK(pass.defined()) << "Found undefined pass for optimization.";  
        const PassInfo& pass_info = pass->Info();  
        if (!pass_ctx.PassEnabled(pass_info)) continue;  
        // resolve dependencies  
        for (const auto& it : pass_info->required) {  
          mod = GetPass(it)(std::move(mod), pass_ctx);  
        }  
        mod = pass(std::move(mod), pass_ctx);  
      }  
      return mod;  
    }  
      
    

여기서 두 부분으로 나뉩니다. pass에 의존성이 있다면, 먼저 의존 pass를 실행합니다. GetPass는 `relay._transform`의 리스트에서 이름에 따라 대응하는 pass를 반환합니다. 코드 구현은 다음과 같습니다:
    
    
    Pass GetPass(const String& pass_name) {  
      using tvm::runtime::Registry;  
      const runtime::PackedFunc* f = nullptr;  
      if (pass_name.operator std::string().find("transform.") != std::string::npos) {  
        f = Registry::Get(pass_name);  
      } else if ((f = Registry::Get("transform." + pass_name))) {  
        // pass  
      } else if ((f = Registry::Get("relay._transform." + pass_name))) {  
      }  
      ICHECK(f != nullptr) << "Cannot use " << pass_name << "to create the pass";  
      return (*f)();  
    }  
      
    

이어서 `mod = pass(std::move(mod), pass_ctx);`를 따라가 보면, 코드 구현은 다음과 같습니다:
    
    
    IRModule Pass::operator()(IRModule mod, const PassContext& pass_ctx) const {  
      const PassNode* node = operator->();  
      ICHECK(node != nullptr);  
      PassProfile::EnterPass(node->Info()->name);  
      auto ret = node->operator()(std::move(mod), pass_ctx);  
      PassProfile::ExitPass();  
      return std::move(ret);  
    }  
      
    // PassNode 타입의 node 인스턴스를 만든다  
    const Object* operator->() const { return get(); }  
    // 가상 함수, 서브클래스가 오버라이드해야 한다  
    virtual IRModule operator()(IRModule mod, const PassContext& pass_ctx) const = 0;  
    

여기서 `virtual IRModule operator()(IRModule mod, const PassContext& pass_ctx)`의 인터페이스 구현에 주목하세요. 구체적으로 FunctionPassNode가 이 `operator()` 메서드를 오버라이드합니다. FunctionPassNode가 PassNode를 상속하기 때문이며, 코드 구현은 다음과 같습니다:
    
    
    // Perform Module -> Module optimizations at the Function level.  
    IRModule FunctionPassNode::operator()(IRModule mod, const PassContext& pass_ctx) const {  
      DiagnosticContext previous = DiagnosticContext::Default(mod);  
      
      if (pass_ctx->diag_ctx) {  
        DiagnosticContext tmp = pass_ctx->diag_ctx.value();  
        pass_ctx->diag_ctx = previous;  
        previous = tmp;  
      } else {  
        pass_ctx->diag_ctx = previous;  
      }  
      
      ICHECK(pass_ctx->diag_ctx)  
          << "The diagnostic context was set at the top of this block this is a bug.";  
      
      const PassInfo& pass_info = Info();  
      
      ICHECK(mod.defined());  
      
      DLOG(INFO) << "Executing function pass : " << pass_info->name  
                 << " with opt level: " << pass_info->opt_level;  
      
      pass_ctx.Trace(mod, pass_info, true);  
      
      // Execute the pass function and return a new module.  
      IRModule updated_mod =  
          IRModule(mod->functions, mod->type_definitions, mod->Imports(), mod->source_map);  
      
      std::vector<std::pair<GlobalVar, Function> > updates;  
      for (const auto& it : updated_mod->functions) {  
        // only picks up relay::Function  
        if (auto* n = it.second.as<FunctionNode>()) {  
          Function func = GetRef<Function>(n);  
          auto updated_func = SkipFunction(func) ? func : pass_func(func, updated_mod, pass_ctx);  
          updates.push_back({it.first, updated_func});  
        }  
      }  
      
      for (const auto& pair : updates) {  
        updated_mod->Add(pair.first, pair.second, true);  
      }  
      
      ICHECK(pass_ctx->diag_ctx)  
          << "The diagnostic context was set at the top of this block this is a bug.";  
      
      pass_ctx->diag_ctx.value().Render();  
      pass_ctx->diag_ctx = previous;  
      
      pass_ctx.Trace(updated_mod, pass_info, false);  
      
      // TODO(@jroesch): move away from eager type checking for performance reasons  
      // make issue.  
      return transform::InferType()(updated_mod);  
    }  
    

이 구현은 비교적 복잡하지만, 우리가 신경 써야 하는 부분은 `auto updated_func = SkipFunction(func) ? func : pass_func(func, updated_mod, pass_ctx);` 이 한 줄입니다. 이것이 Pass를 실행하는 핵심 동작입니다.

이상으로 Function Pass의 C++ backend 일반 생성 흐름이었습니다.

# 0x4. Constant Folding Pass

이제 Constant Folding Pass의 C++ backend 코드 구현에서 어떤 점에 주의해야 하는지 살펴봅시다. 먼저 Constant Folding Pass는 Function-level Pass에 속합니다. 진입점은 여전히 Function Pass의 등록 인터페이스입니다:
    
    
    namespace transform {  
      
    Pass FoldConstant() {  
      runtime::TypedPackedFunc<Function(Function, IRModule, PassContext)> pass_func =  
          [=](Function f, IRModule m, PassContext pc) {  
            return Downcast<Function>(FoldConstant(f, m));  
          };  
      return CreateFunctionPass(pass_func, 2, "FoldConstant", {});  
    }  
      
    TVM_REGISTER_GLOBAL("relay._transform.FoldConstant").set_body_typed(FoldConstant);  
      
    }  // namespace transform  
    

`FoldConstant`의 구체적인 구현을 살펴봅시다:
    
    
    Expr FoldConstant(const Expr& expr, const IRModule& mod) {  
      return ConstantFolder(mod).Mutate(expr);  
    }  
      
    

constant folding이 주로 `ConstantFolder` 클래스의 `Mutate` 함수를 호출하는 것을 볼 수 있습니다. 그리고 ConstantFolder는 MixedModeMutator 클래스를 상속합니다. MixedModeMutator 클래스는 꽤 흥미로우며, 정의는 다음과 같습니다:
    
    
    /*! \brief 커스텀 재작성 Pass를 위한 비재귀 DFS 그래프 순회  
     *  
     * MixedModeMutator는 Expr을 dataflow 그래프로 다루며, 각 Expr은 한 번만 재작성된다.  
     * mutated 결과는 map에 기록되어 재사용되며, dataflow 위의 로컬 transformation이  
     * 그래프 구조를 보존하도록 한다  
     *  
    * MixedModeMutator는 ExprMutator와 동일한 재귀 API를 제공하며,  
    * 대부분의 IR 형식에 대해 재귀 순회를 사용하지만, 내부적으로는 중첩된 dataflow 영역을  
    * 확장하여 반복적으로 처리해 스택 오버플로를 방지한다  
     *  
     * Uses Rewrite_ API of ExprRewriter for a cleaner split between recrusive and non-recursive  
     * behavior.  
     */  
    class MixedModeMutator : public ::tvm::relay::ExprMutator {  
     public:  
      MixedModeMutator(bool pre = false) : pre_{pre} {};  
      Expr VisitExpr(const Expr& expr) final;  
      
      virtual Expr DispatchVisitExpr(const Expr& expr);  
      Expr VisitExpr_(const TupleNode* op) final { return Rewrite(op); };  
      Expr VisitExpr_(const CallNode* call_node) final { return Rewrite(call_node); };  
      Expr VisitExpr_(const TupleGetItemNode* op) final { return Rewrite(op); };  
      /*!  
       * \brief 사용자는 Rewrite_ 메서드를 오버라이드해 자신의 Pass를 구현해야 한다. Rewrite_ functions will be able to rewrite  
       *  the op only with data about the original node `pre` and the same node with modified   
       * inputs `post` and should not recurse.  
       * \param pre 재작성 전의 표현식 노드.  
       * \param post 재작성된 입력을 가진 표현식.  
       */  
      virtual Expr Rewrite_(const TupleNode* pre, const Expr& post) { return post; }  
      virtual Expr Rewrite_(const CallNode* pre, const Expr& post) { return post; }  
      virtual Expr Rewrite_(const TupleGetItemNode* pre, const Expr& post) { return post; }  
      
     protected:  
      bool pre_;  
      /*! \brief Implement Rewrite API by calling ExprMutator's VisitExpr_(op) to get a `post` node with  
       * changed inputs.  
       */  
      template <typename T>  
      Expr Rewrite(const T* op) {  
        Expr post = ExprMutator::VisitExpr_(op);  
        return Rewrite_(op, post);  
      }  
      
      virtual void VisitLeaf(const Expr& expr);  
      virtual bool CheckVisited(const Expr& expr);  
    };  
    

0x2 절에서 pass는 Relay 트리 구조, 즉 계산 그래프를 최적화하는 것이며, 최적화는 필연적으로 그래프 구조의 수정을 수반한다고 했습니다. 이것이 위에서 언급한 `ExprMutator` 서브클래스이며, `ExprVisitor`와 마찬가지로 `ExprFunctor`를 상속합니다. Pass를 구현한다는 것은 사실 그 클래스에 대해 `VisitExpr_` 멤버 함수를 오버라이드하는 것입니다. 그러나 이 `MixedModeMutator` 클래스에서는 `VisitExpr_` 멤버 함수가 실제로 다시 `Rewrite_`를 호출합니다. 따라서 Constant Folding이 노드를 수정할 때는 이 `Rewrite_` 멤버 함수만 오버라이드하면 됩니다. (이리저리 헷갈리는데, 잘 살펴봐야 합니다.)

그런 다음 `ConstantChecker`의 구현을 살펴봅시다. Constant Folding은 ConstantChecker를 통해 재귀적으로 Expr의 상수 판단을 구현합니다. 코드를 보면 알 수 있듯이, Constant는 주로 요소가 ConstantNode인지, 또는 TupleNode 안의 요소들이 모두 ConstantNode인지를 판단합니다.
    
    
    class ConstantChecker : private ExprVisitor {  
     public:  
      // Check whether an expression is constant. The results are memoized.  
      bool Check(const Expr& expr) {  
        // The `ConstantNode` case is common enough that we check directly for the  
        // case here, to avoid the time overhead of dispatching through the vtable  
        // and the space overhead of memoizing always-true results.  
        if (expr.as<ConstantNode>()) {  
          return true;  
        }  
        const auto it = memo_.find(expr);  
        if (it != memo_.end()) return it->second;  
        VisitExpr(expr);  
        return memo_[expr];  // return memoized result or the default value false  
      }  
      
     private:  
      std::unordered_map<Expr, bool, ObjectPtrHash, ObjectPtrEqual> memo_;  
      
      void VisitExpr_(const TupleNode* n) final {  
        bool result = true;  
        for (const auto& field : n->fields) {  
          if (!Check(field)) {  
            result = false;  
            break;  
          }  
        }  
        memo_[GetRef<Tuple>(n)] = result;  
      }  
    };  
      
    bool ConstantCheck(const Expr& e) { return ConstantChecker().Check(e); }  
    

이어서 실제로 constant folding이 일어나는 함수, 즉 ConstantFolder에서 오버라이드된 `Rewrite_` 함수를 분석해 보겠습니다. 코드 구현은 다음과 같습니다:
    
    
    Expr Rewrite_(const CallNode* call, const Expr& post) final {  
        if (inside_primitive) {  
          return GetRef<Expr>(call);  
        }  
        static auto op_stateful = Op::GetAttrMap<TOpIsStateful>("TOpIsStateful");  
      
        auto origin_args = call->args;  
        call = post.as<CallNode>();  
        // We don't constant fold function with zero arguments.  
        // This is a heuristic that is useful.  
        // For example it is harmful to fold ones(shape=(4, 5)).  
        if (call->args.size() == 0) return post;  
        const OpNode* op = call->op.as<OpNode>();  
        if (op == nullptr) return post;  
        // skip stateful ops.  
        if (op_stateful.get(GetRef<Op>(op), false)) return post;  
        // Try to evaluate shape_of op  
        if (call->op == shape_of_op_ || call->op == vm_shape_of_op_) {  
          return EvaluateShapeOf(post, origin_args, call->attrs);  
        }  
      
        if (call->op == ndarray_size_op_) {  
          return EvaluateNdarraySize(post, origin_args, call->attrs);  
        }  
      
        // We should think about potentially constant evaluation over these ops too.  
        static auto fnoncomputational = Op::GetAttrMap<TNonComputational>("TNonComputational");  
        if (const auto* call_node = call->op.as<OpNode>()) {  
          Op op = GetRef<Op>(call_node);  
          if ((fnoncomputational.count(op) && fnoncomputational[op]) || (call->op == device_copy_op_)) {  
            return GetRef<Call>(call);  
          }  
        }  
      
        bool all_const_args = true;  
        for (Expr arg : call->args) {  
          if (!checker_.Check(arg)) {  
            all_const_args = false;  
          }  
        }  
        if (all_const_args) {  
          return ConstEvaluate(post);  
        } else {  
          return post;  
        }  
      }  
    

여기서 callnode의 타입에 따라 또 몇 가지 경우로 나뉩니다. 코드가 `return ConstEvaluate(post);` 줄까지 갔다고 가정해 봅시다. 이 함수는 constant folding을 완성하기 위해 무엇을 했을까요?
    
    
    // Constant evaluate an expression.  
      Expr ConstEvaluate(Expr expr) {  
        std::vector<transform::Pass> passes = {transform::FuseOps(0), transform::ToANormalForm(),  
                                               transform::InferType()};  
        Function func;  
        if (expr.as<FunctionNode>()) {  
          func = Downcast<Function>(expr);  
        } else {  
          // TODO(@jroesch): fix this  
          func = Function(FreeVars(expr), expr, Type(), FreeTypeVars(expr, module_), {});  
        }  
        auto mod = IRModule({}, module_->type_definitions, module_->Imports());  
        auto global = GlobalVar("main");  
        mod->Add(global, func);  
        auto seq = transform::Sequential(passes);  
        mod = seq(mod);  
        auto entry_func = Downcast<Function>(mod->Lookup("main"));  
        expr = expr.as<FunctionNode>() == nullptr ? entry_func->body : entry_func;  
      
        using tvm::transform::PassContext;  
        Device dev;  
        dev.device_type = kDLCPU;  
        dev.device_id = 0;  
        Target target = Target("llvm");  
        // use a fresh build context  
        // in case we are already in a build context.  
        // needed for both execution and creation(due to JIT)  
        With<PassContext> fresh_build_ctx(PassContext::Create());  
      
        FInterpreter executor = CreateInterpreter(mod, dev, target);  
        return ObjectToExpr(executor(expr));  
      }  
      
    

여기서 세 개의 pass가 추가된 것을 볼 수 있고, 이어서 이 세 개의 Pass를 실행해 constant folding 기능을 완성합니다. 여기서 ToANormalForm의 정의는 위키피디아에서 찾을 수 있습니다(https://en.wikipedia.org/wiki/A-normal_form):
    
    
    passes = {transform::FuseOps(0), transform::ToANormalForm(),  
    transform::InferType()};  
    

Sequential Pass와 target, 그리고 전역 module 정보를 얻은 후 ObjectToExpr을 실행해 상수 값을 표현식으로 변환하는 것을 완료합니다.
    
    
    // Convert value to expression.  
      Expr ObjectToExpr(const ObjectRef& value) {  
        if (value->IsInstance<runtime::NDArray::ContainerType>()) {  
          auto nd_array = Downcast<runtime::NDArray>(value);  
          return Constant(nd_array);  
        } else if (const auto* val = value.as<runtime::ADTObj>()) {  
          runtime::ADT adt = GetRef<runtime::ADT>(val);  
          Array<Expr> fields;  
          for (size_t i = 0; i < adt.size(); ++i) {  
            fields.push_back(ObjectToExpr(adt[i]));  
          }  
          return Tuple(fields);  
        } else {  
          LOG(FATAL) << "Cannot handle " << value->GetTypeKey();  
          return Expr();  
        }  
      }  
    

이 함수는 주로 runtime의 결과를 기반으로 새로운 Expr을 생성하여 원래 Expr을 대체하는 것을 구현합니다. `Rewrite_` 함수에서 다른 몇 가지 타입의 CallNode에 대한 constant folding 구현은 여기서 소개하지 않겠습니다. 관심이 있는 분들은 직접 살펴보시기 바랍니다.

# 0x5. 필자의 조언

저의 조언은, 만약 여러분의 C++ 능력이 그리 뛰어나지 않다면, TVM의 C++ backend Pass에 대해서는 이해 정도만 하시고 깊이 파고들 필요는 없다는 것입니다. 첫 절에서 소개한 방법대로만 하면 TVM이 이미 구현한 Pass를 능숙하게 다룰 수 있습니다. 만약 Pass를 커스터마이즈하고 싶다면, TVM이 제공하는 데코레이터를 기반으로 Python 레이어에서 직접 구현하면 됩니다. 자세한 내용은 다음을 참고하세요: https://github.com/apache/tvm/blob/main/tutorials/dev/use_pass_infra.py. 따라서 글이 너무 길어서 부담스러우시면 첫 절만 보셔도 됩니다.

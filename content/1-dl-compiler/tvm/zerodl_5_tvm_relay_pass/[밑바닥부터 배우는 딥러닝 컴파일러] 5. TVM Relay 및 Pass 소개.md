# 【밑바닥부터 배우는 딥러닝 컴파일러】 5. TVM Relay 및 Pass 소개

이 글은 TVM의 Relay를 소개하고, Relay를 기반으로 Conv+BN+ReLU의 작은 네트워크를 어떻게 구성하는지 설명한다. 그 다음 TVM의 Pass 동작 원리를 소개하며, RemoveUnusedFunctions, ToBasicBlockNormalForm, EliminateCommonSubexpr 세 가지 Pass를 비교적 자세히 설명한다. Relay 부분의 자세한 설명은 대부분 공식 문서(https://tvm.apache.org/docs/tutorials/get_started/introduction.html)에서 인용했다.

# 0x0. 소개

앞 절들에서 TVM이 ONNX frontend 모델을 IRModule로 어떻게 변환하는지 살펴보았고, 또한 TVM에서 Relay op과 TOPI op이 서로 어떻게 연결되는 과정인지 분석했으며, Relay op의 최종 계산도 결국 TOPI op 집합을 기반으로 이루어진다는 것을 알게 되었다. 그런데 ONNX 모델 구조를 기반으로 TVM의 frontend를 살펴보았던 글에서 제시했던 예제 프로그램에서 매우 중요한 디테일 하나가 자세히 설명되지 않았는데, 바로 TVM의 컴파일 과정이다. 즉, 다음 몇 줄의 코드이다.


    ######################################################################  
    # Relay Build  
    # -----------  
    # Compile the graph to llvm target with given input specification.  
    target = "llvm"  
    target_host = "llvm"  
    dev = tvm.cpu(0)  
    with tvm.transform.PassContext(opt_level=3):  
        lib = relay.build(mod, target=target, target_host=target_host, params=params)  
    

이 몇 줄의 코드는 TVM의 컴파일 과정을 보여준다. 이 컴파일 과정에는 Relay IR 기반의 최적화 전략(즉 Pass)을 사용해 불필요한 op을 제거하는 일뿐만 아니라, Relay 프로그램을 특정 backend(여기서는 llvm)에서 실행 가능한 코드로 컴파일하는 일(codegen)도 포함된다.

이 글에서는 Relay를 간단히 소개한 다음, TVM의 Pass도 함께 살펴본다. 즉, `with tvm.transform.PassContext(opt_level=3)` 이 클래스가 구체적으로 어떤 일을 하는지 설명한다. codegen과 자세한 컴파일 과정에 대해서는 TVM의 깊이가 매우 깊어 아직 충분히 파악하지 못했으니 다음에 살펴보기로 한다.

# 0x2. Relay 소개

이 절에서는 TVM 문서(`https://tvm.apache.org/docs/dev/relay_intro.html`)를 참고해 NNVM의 2세대인 Relay를 소개한다. Relay의 설계 목표는 다음과 같다.

  * 전통적인 dataflow 스타일 프로그래밍을 지원한다.
  * functional-style scoping을 지원하며, 프로그래밍 언어 분야의 지식을 결합해 새로운 특성(Let 표현식 지원, 재귀 지원 등)을 가진다.
  * dataflow 스타일과 함수형 스타일의 혼합 프로그래밍을 지원한다.



## 0x2.1 Relay로 계산 그래프 만들기

전통적인 딥러닝 프레임워크는 계산 그래프를 중간 표현으로 사용한다. 계산 그래프(또는 dataflow 그래프)는 계산 과정을 나타내는 방향성 비순환 그래프(DAG)이다. 제어 흐름이 없기 때문에 계산 능력 측면에서는 제약이 있지만, 단순함 덕분에 자동 미분을 구현하기 쉽고 이종(heterogeneous) 실행 환경에서 컴파일하기에 용이하다(예를 들어 전용 하드웨어에서 계산 그래프의 일부, 즉 서브그래프를 실행하는 등).

![이미지](images/img_01.png)Relay로 간단한 계산 그래프를 만드는 예제 코드와 그에 대응하는 텍스트 형식 및 AST 추상 구문 트리

Relay를 사용해 계산(dataflow) 그래프를 만들 수 있다. 구체적으로 위의 코드는 두 노드로 이루어진 간단한 계산 그래프를 어떻게 만드는지 보여준다. 이 예제 코드는 NNVMv1 같은 기존 Graph IR과 큰 차이가 없으며, 유일한 차이는 용어 측면에서이다.

  * 기존 프레임워크는 보통 graph와 subgraph를 사용한다.
  * Relay는 함수, 예를 들어 `fn(%x)` 와 같이 그래프를 나타낸다.



각 dataflow 노드는 Relay에서 하나의 CallNode이다. Relay의 Python DSL을 통해 계산 그래프를 빠르게 만들 수 있다. 위의 코드에서 주목할 점은 여기서 명시적으로 Add 노드를 하나 만들었으며 두 입력 모두 `%1`을 가리킨다는 점이다. 딥러닝 프레임워크가 위의 계산 그래프에 대해 추론을 수행할 때, 위상 정렬 순서로 계산을 진행하며 `%1`은 단 한 번만 계산된다. 이 사실은 딥러닝 프레임워크 개발자에게는 매우 자연스러운 것이지만, 알고리즘에만 관심 있는 연구자에게는 혼란을 일으킬 수 있다. 만약 결과를 출력하는 간단한 visitor를 구현하고 결과를 중첩된 Call 표현식으로 본다면, 이는 `log(%x) + log(%x)`가 될 것이다.

DAG에 공유 노드가 존재할 때, 이러한 모호함은 프로그램 의미론에 대한 해석의 차이에서 비롯된다. 일반적인 함수형 프로그래밍 IR에서는 중첩된 표현식을 표현식 트리로 보며, `%1`이 실제로 `%2`에서 두 번 재사용된다는 사실을 고려하지 않는다.

Relay IR은 이 차이를 인지한다. 사실 딥러닝 프레임워크 사용자들은 자주 이런 방식으로 계산 그래프를 만들며, DAG 노드 재사용은 흔하게 일어난다. 그래서 Relay 프로그램을 텍스트 형식으로 출력할 때, 한 줄에 하나의 CallNode를 출력하고, 각 CallNode에 임시 ID `(%1, %2)`를 할당해 프로그램의 후속 부분에서 각 공통 노드를 참조할 수 있도록 한다.

## 0x2.2 Module: 여러 함수(Graph) 지원

위에서 dataflow 그래프 하나를 함수로 어떻게 만드는지 설명했다. 그러면 자연스레 떠오르는 질문은, 여러 함수를 만들고 서로 호출할 수 있을까 하는 것이다. Relay는 여러 함수를 하나의 Module에 결합하는 것을 허용한다. 다음 코드는 한 함수가 다른 함수를 호출하는 예이다.


    def @muladd(%x, %y, %z) {  
      %1 = mul(%x, %y)  
      %2 = add(%1, %z)  
      %2  
    }  
    def @myfunc(%x) {  
      %1 = @muladd(%x, 1, 2)  
      %2 = @muladd(%1, 2, 3)  
      %2  
    }  
    

Module은 `Map<GlobalVar, Function>`으로 볼 수 있는데, 여기서 GlobalVar는 단지 함수 이름을 나타내는 ID이다. 위의 프로그램에서 GlobalVar는 `@muladd`와 `@myfunc`이다. CallNode가 다른 함수를 호출할 때, 해당 GlobalVar는 CallNode의 op에 저장된다. 여기에는 간접적인 계층 관계가 존재한다. 호출되는 함수의 본문을 얻으려면 해당 GlobalVar를 사용해 Module에서 찾아야 한다. 이 경우 참조된 함수를 직접 CallNode의 op에 저장할 수도 있다. 그렇다면 GlobalVar를 도입하는 이유는 무엇일까? 주된 이유는 정의와 선언을 분리하기 위함이며, 함수의 재귀와 지연 선언(lazy declaration)을 지원하기 위함이다.


    def @myfunc(%x) {  
      %1 = equal(%x, 1)  
       if (%1) {  
          %x  
       } else {  
         %2 = sub(%x, 1)  
         %3 = @myfunc(%2)  
          %4 = add(%3, %3)  
          %4  
      }  
    }  
    

위의 예에서 `@myfunc`는 자기 자신을 재귀적으로 호출한다. GlobalVar `@myfunc`를 사용해 함수를 표현함으로써 자료 구조의 순환 의존을 피할 수 있다. 여기까지 Relay의 기본 개념을 모두 소개했다. 주목할 점은 NNVM에 비해 Relay가 다음 측면에서 개선되었다는 것이다.

  * 텍스트 형식의 중간 표현이 있어 개발 및 debug에 편리하다.
  * 서브그래프 함수, 모듈 결합을 지원해 통합 최적화에 편리하다.
  * frontend 사용자 친화적이며 튜닝이 편리하다.



## 0x2.3 Let Binding과 Scope

지금까지 딥러닝 프레임워크의 기존 방법으로 계산 그래프를 만드는 방법을 설명했다. 이 절에서는 Relay의 새로운 구조인 let binding을 다룬다.

Let binding은 모든 고급 프로그래밍 언어에서 사용된다. Relay에서 그것은 `Let(var, value, body)`라는 세 개의 필드를 가진 자료 구조이다. Let 표현식을 평가할 때, 먼저 value 부분을 계산한 다음 그것을 var에 binding하고, 마지막으로 body 표현식에서 계산 결과를 반환한다.

여러 개의 Let binding을 사용해 dataflow 프로그램과 논리적으로 동등한 프로그램을 만들 수 있다. 다음 예제 코드는 그 사용법을 보여준다.

![이미지](images/img_02.png)Let 표현식으로 dataflow 프로그램과 동등한 계산 그래프를 만든 모습

중첩된 Let Binding은 A-normal 형식이라고 하며, 함수형 프로그래밍 언어의 일반적인 IR로 사용된다. 위 그림을 보면, 두 프로그램의 의미는 완전히 동등하고 텍스트 표현도 동일하지만(A-norm 형식에 let 접두사가 있다는 점만 제외) AST 추상 구문 트리는 다르다는 것을 알 수 있다.

프로그램 최적화는 이러한 AST 자료 구조를 사용해 변환을 진행하므로, 이 두 가지 다른 구조는 컴파일러가 최종 생성하는 코드에 영향을 미친다. 예를 들어 `add(log(x), y)` 패턴을 검출한다고 해보자. dataflow 프로그램에서는 먼저 add 노드로 들어가서 첫 번째 인자가 log인지 직접 확인하면 된다. 하지만 A-form 프로그램에서는 직접 어떤 것도 확인할 수 없는데, add 노드의 입력이 `%v1`이기 때문이다. 변수와 그것이 binding된 값을 매핑하는 매핑 테이블을 유지하고, 이를 조회해야만 `%v1`이 log를 나타낸다는 것을 알 수 있다.

## 0x2.4 왜 Let Binding이 필요할 수 있는가

Let Binding의 핵심 용도 중 하나는 계산의 scope를 지정할 수 있다는 점이다. Let Binding을 사용하지 않은 다음 예를 살펴보자.

![이미지](images/img_03.png)Let Binding을 사용하지 않고 프로그래밍한 예

`%1` 노드를 어디서 계산해야 하는지 결정하려고 할 때 문제가 생긴다. 특히 텍스트 형식은 `%1` 노드를 if scope 바깥에서 계산해야 한다고 시사하는 것 같지만, AST는 그렇게 권장하지 않는다. 사실 dataflow 그래프는 그 계산 scope를 정의하지 않으며, 이는 의미론적으로 모호함을 야기한다.

closure가 있을 때 이 모호함은 더 흥미로워진다. closure를 반환하는 다음 프로그램을 살펴보자. `%1`을 어디서 계산해야 할지 알 수 없다. 그것은 closure 안에서도 바깥에서도 계산될 수 있다.


    fn (%x) {  
      %1 = log(%x)  
      %2 = fn(%y) {  
        add(%y, %1)  
      }  
      %2  
    }  
    

Let Binding은 이러한 문제를 해결한다. 값의 계산은 let 노드에서 일어나기 때문이다. 이 두 프로그램에서 만약 `%1 = log(%x)`를 `let %v1 = log(%x)`로 바꾸면, 우리는 계산 위치를 if scope와 closure의 바깥으로 명확히 지정한 셈이 된다. Let Binding은 계산 측에 더 정확한 범위를 제공하며, backend 코드를 생성할 때 매우 유용하다(이 범위가 IR에 포함되어 있기 때문이다).

한편 계산 scope를 명시하지 않는 dataflow 형식도 그 자체로 장점이 있다. 코드 생성 시 let을 어디에 둘지 걱정할 필요가 없다. dataflow 형식은 또한 이후에 계산 노드를 어디에 배치할지 결정하는 Pass에 더 큰 자유도를 제공한다. 따라서 최적화 초기 단계에서 dataflow 형식이 편리하다고 판단되면, dataflow 그래프 인코딩 방식을 사용하는 것은 나쁘지 않은 선택이다. 현재 Relay에는 dataflow 그래프를 위한 다양한 최적화 방식이 구현되어 있다.

다만 IR을 실제 runtime 프로그램으로 lowering할 때는 정확한 계산 scope가 필요하다. 특히 서브 함수와 closure를 사용할 때, 계산 scope가 어디서 일어나야 하는지 명확히 지정해야 한다. 후반부의 특정 최적화에서는 Let Binding을 사용해 이 문제를 해결할 수 있다.

## 0x2.5 IR 변환에 미치는 영향

이쯤이면 두 가지 표현 형식에 익숙해졌기를 바란다. 대부분의 함수형 프로그래밍 언어는 A-normal 형식으로 분석되며, 분석자는 표현식이 DAG라는 것을 신경 쓸 필요가 없다.

Relay는 dataflow 형식과 Let Binding을 동시에 지원하기로 선택했다. TVM은 프레임워크 개발자가 익숙한 표현 형식을 선택할 수 있도록 하는 것이 중요하다고 본다. 다만 이는 범용 Pass를 작성하는 데 어느 정도 영향을 미친다. 여기서는 아직 Pass에 대해 소개하지 않았고, Pass에 대한 이해도 깊지 않으며 Let 표현식으로 네트워크를 구성해본 적이 없으므로, 구체적으로 어떤 영향이 있는지 더 이상 다루지 않는다.

자세한 내용은 https://tvm.apache.org/docs/dev/relay_intro.html#let-binding-and-scopes 를 참고하라.

# 0x3. Relay 기반의 커스텀 신경망 예제 만들기

Relay의 인터페이스로 Conv+BN+ReLU의 작은 네트워크를 정의해 Relay 인터페이스 사용 방법을 보여주자. 여기서 TVM 버전은 0.8.0.dev이며, 코드는 다음과 같다.


    #coding=utf-8  
    import tvm  
    from tvm import relay  
    import numpy as np  
    from tvm.contrib import graph_executor  
      
    # BN 구성  
    def batch_norm(data,  
                         gamma=None,  
                         beta=None,  
                         moving_mean=None,  
                         moving_var=None,  
                         **kwargs):  
        name = kwargs.get("name")  
        kwargs.pop("name")  
        if not gamma:  
            gamma = relay.var(name + "_gamma")  
        if not beta:  
            beta = relay.var(name + "_beta")  
        if not moving_mean:  
            moving_mean = relay.var(name + "_moving_mean")  
        if not moving_var:  
            moving_var = relay.var(name + "_moving_var")  
        return relay.nn.batch_norm(data,  
                                   gamma=gamma,  
                                   beta=beta,  
                                   moving_mean=moving_mean,  
                                   moving_var=moving_var,  
                                   **kwargs)[0]  
      
    # 합성곱 구성  
    def conv2d(data, weight=None, **kwargs):  
        name = kwargs.get("name")  
        kwargs.pop("name")  
        if not weight:  
            weight = relay.var(name + "_weight")  
        return relay.nn.conv2d(data, weight, **kwargs)  
      
      
    # 합성곱+BN+ReLU의 simpleNet 구성  
    def simplenet(data, name, channels, kernel_size=(3, 3), strides=(1, 1),  
                   padding=(1, 1), epsilon=1e-5):  
        conv = conv2d(  
            data=data,  
            channels=channels,  
            kernel_size=kernel_size,  
            strides=strides,  
            padding=padding,  
            data_layout='NCHW',  
            name=name+'_conv')  
        bn = batch_norm(data=conv, epsilon=epsilon, name=name + '_bn')  
        act = relay.nn.relu(data=bn)  
        return act  
      
    data_shape = (1, 3, 224, 224)  
    kernel_shape = (32, 3, 3, 3)  
    dtype = "float32"

# 0x4. Pass 첫걸음

위의 simplenet을 만드는 코드에서 `relay.build` 외부에는 `tvm.transform.PassContext` 한 겹이 다음과 같이 감싸져 있다.


    with tvm.transform.PassContext(opt_level=3):  
        lib = relay.build(func, "llvm", params=params)  
    

사실 `tvm.transform.PassContext`라는 인터페이스가 바로 Pass를 정의한다. 문서에 나온 대로이다.

![이미지](images/img_04.png)tvm.transform.PassContext는 Relay IR에 어떤 Pass를 사용해 최적화할지 제어한다

Pass는 TVM에서 Relay IR을 기반으로 수행되는 일련의 최적화이다. onnx-simplifier에서 사용되는 onnxoptimizer와 유사하며, 계산 그래프를 단순화하고 불필요한 op을 제거하여 모델 추론 효율을 높일 수 있다. TVM은 모든 Pass를 `tvm/include/tvm/ir/transform.h` 파일에 추상화했으며, 주로 PassContext, PassInfo, Pass, 그리고 Sequential을 포함한다.

여기의 PassContext는 위의 Python 인터페이스에 대응되는 C++ 구현으로, Pass 실행에 필요한 매개변수(최적화 level, 의존하는 다른 특정 Pass, 특정 Pass를 사용하지 않도록 설정하는 옵션 등)를 포함한다. PassInfo는 Pass 정보를 기록하는 클래스로, Pass의 opt_level, name, 그리고 현재 Pass가 필요로 하는 전제 Pass들을 포함한다. Pass 클래스는 Pass의 본체를 실행하는 클래스이며, 이는 기본 클래스로서 각 Pass의 구체적인 C++ 구현은 `tvm/src/relay/transforms`에 있다. 이들은 모두 Pass라는 기본 클래스를 상속한다. 마지막으로 Sequential은 모든 Pass를 담는 컨테이너이다.

한 가지 짚어둘 점은, 모든 Pass가 `tvm/src/relay/transforms`에 정의되어 있는 것은 아니라는 것이다. 예를 들어 아래의 첫 번째 예는 `tvm/src/relay/backend/vm` 폴더에 있다. 이제 몇 가지 Pass 예를 살펴보고, 그것들이 Relay IR에 무슨 일을 하는지 보자.

  * **RemoveUnusedFunctions**



먼저 `tvm/src/relay/backend/vm/removed_unused_funcs.cc`에 정의된 RemoveUnusedFunctions Pass를 살펴보자. 핵심 코드 구현은 다음과 같다.


    void VisitExpr_(const FunctionNode* func_node) final {  
        auto func = GetRef<Function>(func_node);  
        if (visiting_.find(func) == visiting_.end()) {  
          visiting_.insert(func);  
          for (auto param : func_node->params) {  
            ExprVisitor::VisitExpr(param);  
          }  
          ExprVisitor::VisitExpr(func_node->body);  
        }  
      }  
      
    IRModule RemoveUnusedFunctions(const IRModule& module, Array<runtime::String> entry_funcs) {  
      std::unordered_set<std::string> called_funcs{};  
      for (auto entry : entry_funcs) {  
        auto funcs = CallTracer(module).Trace(entry);  
        called_funcs.insert(funcs.cbegin(), funcs.cend());  
      }  
      auto existing_functions = module->functions;  
      for (auto f : existing_functions) {  
        auto it = called_funcs.find(f.first->name_hint);  
        if (it == called_funcs.end()) {  
          module->Remove(f.first);  
        }  
      }  
      return module;  
    }  
      
    

비교적 쉽게 볼 수 있듯이, 이 Pass는 Relay IR에서 불필요한 노드를 제거한다. `VisitExpr_` 함수는 그래프 순회를 수행하고, 순회되지 않은 노드를 삭제한다. 삭제는 `RemoveUnusedFunctions` 함수에서 일어난다.

  * **ToBasicBlockNormalForm**



이 Pass는 `tvm/src/relay/transforms/to_basic_block_normal_form.cc`에 구현되어 있으며, 코드 구현은 다음과 같다.


    Expr ToBasicBlockNormalFormAux(const Expr& e) {  
      // calculate all the dependency between nodes.  
      support::Arena arena;  
      DependencyGraph dg = DependencyGraph::Create(&arena, e);  
      /* The scope of the whole expr is global.  
       * The scope of any subexpr, is the lowest common ancestor of all incoming edge.  
       * We also record the set of expressions whose scope is lifted.  
       */  
      std::pair<NodeScopeMap, ExprSet> scopes = CalcScope(dg);  
      return Fill::ToBasicBlockNormalForm(e, dg, &scopes.first, &scopes.second);  
    }  
      
    IRModule ToBasicBlockNormalForm(const IRModule& mod) {  
      DLOG(INFO) << "ToBBlock:" << std::endl << mod;  
      
      tvm::Map<GlobalVar, Function> updates;  
      auto funcs = mod->functions;  
      for (const auto& it : funcs) {  
        ICHECK_EQ(FreeVars(it.second).size(), 0) << "Expected no free variables";  
        if (const auto* n = it.second.as<FunctionNode>()) {  
          if (n->GetAttr<String>(attr::kCompiler).defined()) continue;  
        }  
        Expr ret = TransformF([&](const Expr& e) { return ToBasicBlockNormalFormAux(e); }, it.second);  
        updates.Set(it.first, Downcast<Function>(ret));  
      }  
      
      for (auto pair : updates) {  
        mod->Add(pair.first, pair.second, true);  
      }  
      
      DLOG(INFO) << "ToBBlock: transformed" << std::endl << mod;  
      
      return mod;  
    }  
      
    bool BasicBlockNormalFormCheck(const Expr& e) {  
      // calculate all the dependency between nodes.  
      support::Arena arena;  
      DependencyGraph dg = DependencyGraph::Create(&arena, e);  
      std::pair<NodeScopeMap, ExprSet> scopes = CalcScope(dg);  
      for (auto expr : scopes.second) {  
        LOG(FATAL) << "The expression below violates the basic block normal form in that "  
                   << "its scope should be lifted:\n"  
                   << expr;  
      }  
      return scopes.second.size() == 0;  
    }  
    

`ToBasicBlockNormalForm` 함수는 Relay IR의 function들을 순회하며 각 function을 기본 블록 형식으로 변환한다(`ToBasicBlockNormalFormAux` 함수를 호출). `ToBasicBlockNormalFormAux` 함수는 다음 몇 부분으로 나뉜다.

  * `DependencyGraph dg = DependencyGraph::Create(&arena, e)`를 호출해 DependencyGraph를 만든다. 이 자료 구조는 표현식 간 상호 의존을 나타내는 그래프 구조이다.
  * `std::pair<NodeScopeMap, ExprSet> scopes = CalcScope(dg)`를 통해 각 노드의 scope를 계산한다. 이 scope는 IfNode, FunctionNode, LetNode와 같은 분기 명령으로 구분되는 서브그래프로 간단히 이해할 수 있다. 왜냐하면 위에서 Relay Function으로부터 DependencyGraph를 만들 때 이러한 노드를 만나면 `new_scope` 표시를 할당하기 때문이다. 그러면 CalcScope 함수가 구체적으로 어떤 일을 하는지 살펴보자.



    
    std::pair<NodeScopeMap, ExprSet> CalcScope(const DependencyGraph& dg) {  
      NodeScopeMap expr_scope;  
      ExprSet lifted_exprs;  
      std::unordered_map<DependencyGraph::Node*, Expr> node_to_expr;  
      // 먼저 각 노드를 단일 scope에 속하게 한다  
      for (auto expr_node : dg.expr_node) {  
        node_to_expr[expr_node.second] = expr_node.first;  
      }  
      bool global_scope_used = false;  
      Scope global_scope = std::make_shared<ScopeNode>();  
      // LCA 알고리즘을 사용해 각 노드의 진짜 scope를 갱신한다  
      for (auto it = dg.post_dfs_order.rbegin(); it != dg.post_dfs_order.rend(); ++it) {  
        DependencyGraph::Node* n = *it;  
        auto iit = n->parents.head;  
        Scope s;  
        if (iit == nullptr) {  
          ICHECK(!global_scope_used);  
          s = global_scope;  
          global_scope_used = true;  
        } else {  
          s = expr_scope.at(iit->value);  
          const auto original_s = s;  
          iit = iit->next;  
          for (; iit != nullptr; iit = iit->next) {  
            s = LCA(s, expr_scope.at(iit->value));  
          }  
          if (s != original_s && node_to_expr.find(n) != node_to_expr.end()) {  
            // filter out exprs whose scope do not matter  
            Expr expr = node_to_expr[n];  
            if (!expr.as<OpNode>()) {  
              lifted_exprs.insert(expr);  
            }  
          }  
        }  
        if (n->new_scope) {  
          auto child_scope = std::make_shared<ScopeNode>(s);  
          expr_scope.insert({n, child_scope});  
        } else {  
          expr_scope.insert({n, s});  
        }  
      }  
      ICHECK(global_scope_used);  
      return std::make_pair(expr_scope, lifted_exprs);  
    }  
    

이 함수는 먼저 각 노드를 단일 scope에 속하게 한 다음, LCA 알고리즘을 사용해 각 노드의 진짜 scope를 갱신한다. 여기서는 LCA 알고리즘과, 여기서 각 노드의 scope를 구체적으로 어떻게 구하는지 간단히 소개한다.

최저 공통 조상은 LCA(Lowest Common Ancestor)라고 부른다. 두 노드의 최저 공통 조상이란, 두 노드의 공통 조상 중 루트에서 가장 멀리 있는 것을 말한다. 편의상 어떤 점 집합의 최저 공통 조상을 표시한다. LCA에는 다음과 같은 성질이 있는데, OI-wiki에서 인용한다.

  1. ;
  2. 가  의 조상이려면, 그리고 그럴 때만 ;
  3.  가  의 조상이 아니고  가  의 조상이 아니라면,  는  의 두 개의 서로 다른 서브트리에 각각 속한다;
  4. 전위 순회에서  는 모든  의 원소보다 앞에 나타나며, 후위 순회에서  는 모든  의 원소보다 뒤에 나타난다;
  5. 두 점 집합의 합집합의 최저 공통 조상은 두 점 집합 각각의 최저 공통 조상의 최저 공통 조상이다, 즉 ;
  6. 두 점의 최저 공통 조상은 반드시 트리 위 두 점 사이의 최단 경로 위에 위치한다;
  7. , 여기서  는 트리 위 두 점 사이의 거리이고,  는 어떤 점에서 트리 루트까지의 거리를 나타낸다.



이 성질을 보지 않아도 괜찮다. LCA가 그래프에서 두 노드의 최저 공통 조상을 구할 수 있다는 것만 알면 된다. 그러면 CalcScope 함수의 구체적인 아이디어는 다음과 같다. 먼저 각 노드를 단일 scope로 초기화한 다음, 후위 DFS 순서로 이 노드들을 순회한다. 순회된 각 노드(여기서 `n`이라고 표기)에 대해, 그 부모 노드 `iit`가 존재하는지 확인한다. 존재하지 않으면 현재 노드는 루트 노드이며, 그 scope는 `global_scope`이어야 한다. `iit`가 존재하면 `iit`의 자식 노드들을 순회하며 이 노드들의 `scope`의 LCA 표현식을 본다. 이 LCA로 구한 표현식이 `iit` 노드의 표현식과 완전히 같으면, 이 서브그래프와 현재 노드는 같은 scope에 속한다고 본다. 그렇지 않으면 현재 노드를 `lifted_exprs`에 삽입한다. `lifted_exprs`는 이 DependencyGraph 안의 분기 명령 노드들을 저장하는 집합이다. 이것이 바로 위에서 노드를 `lifted_exprs`에 삽입하기 전에 그 노드 타입이 `OpNode`인지 판단하는 이유이다. 또한 현재 열거하는 노드에 new_scope 표시가 있다면, 현재 노드는 새로운 scope에 속한다는 의미이며, 현재 노드에 새로운 `ScopeNode` 타입의 스마트 포인터를 할당해야 한다.

위 알고리즘을 통해 DependencyGraph 안의 노드와 scope 노드의 관계가 하나의 map에 매핑되었고, scope 노드 또한 트리 구조로 만들어졌다. 마지막으로 `Fill::ToBasicBlockNormalForm(e, dg, &scopes.first, &scopes.second);`를 호출해 `Fill` 클래스를 만든다. 이 클래스는 DependencyGraph와 scope 관련 정보를 포함하며, `ToBasicBlockNormalForm` 멤버 함수를 통해 기본 블록 변환을 수행한다. 그 구현은 `tvm/src/relay/transforms/to_a_normal_form.cc` 파일에 있는데, 충분히 이해하지 못했으니 관심 있는 독자는 직접 따라가 보길 바란다. 즈후의 `moon` 블로거가 이 Pass에 대해 설명한 적이 있어서 여기에 인용한다.

> (`ToBasicBlockNormalForm`) 의 기본 로직은 VisitExpr 함수를 통해 dependency 노드들을 순회하며, 같은 scope를 갖는 노드를 동일한 let_list에 push한다. let_list는 문서에서 다음과 같이 설명한다.


    /*!  
     * \file let_list.h  
     * \brief LetList record let binding and insert let expression implicitly.  
     *  using it, one can treat AST as value instead of expression,  
     *  and pass them around freely without fear of AST explosion (or effect duplication).  
     *  for example, if one write 'b = a + a; c = b + b; d = c + c', the AST will contain 8 'a'.  
     *  if one instead write 'b = ll.Push(a + a); c = ll.Push(b + b); d = ll.Get(c + c);',  
     *  the AST will contain 2 'a', as b and c are now variables.  
    

> let_list는 추상 구문 트리를 간결하게 만들어, 변수 복제로 인한 트리 폭증이 일어나지 않게 한다. 같은 scope를 갖는 expr은 같은 let_list에 묶이며, 하나의 var로 표현된다. 이렇게 하면 표현식이 var 형태로 변환된다. 하나의 var는 하나의 기본 블록에 대응한다.

  * **EliminateCommonSubexpr**



마지막으로 공통 부분식 제거 Pass를 살펴보자. 공통 부분식이란 op 타입이 같고 인자가 같으며 인자의 순서까지 완전히 같은 표현식을 말한다. 그러면 이러한 표현식은 하나의 공통 부분식으로 합쳐질 수 있다. 예를 들어 보자.

`a = b + c``d = b + c`

이 두 표현식은 완전히 동일하다. 이 Pass를 거치고 나면 계산 그래프는 두 표현식 중 하나를 제거한다. 코드 구현은 `tvm/src/relay/transforms/eliminate_common_subexpr.cc`에 있다. 여기서는 `CommonSubexprEliminator` 클래스를 정의하며, 이 클래스는 두 개의 `Rewrite_` 함수를 오버로드해 expr을 순회하고 재작성한다. 코드 구현은 다음과 같다.


    Expr Rewrite_(const CallNode* call, const Expr& post) final {  
        static auto op_stateful = Op::GetAttrMap<TOpIsStateful>("TOpIsStateful");  
        Expr new_expr = post;  
        const CallNode* new_call = new_expr.as<CallNode>();  
        ICHECK(new_call);  
        const OpNode* op = new_call->op.as<OpNode>();  
        StructuralEqual attrs_equal;  
      
        if (new_call->args.size() == 0 || op == nullptr || op_stateful.get(GetRef<Op>(op), false)) {  
          return new_expr;  
        }  
        if (fskip_ != nullptr && fskip_(new_expr)) {  
          return new_expr;  
        }  
      
        auto it = expr_map_.find(new_call->op);  
        if (it != expr_map_.end()) {  
          for (const Expr& candidate_expr : it->second) {  
            if (const CallNode* candidate = candidate_expr.as<CallNode>()) {  
              bool is_equivalent = true;  
              // attrs 매칭  
              if (!attrs_equal(new_call->attrs, candidate->attrs)) {  
                continue;  
              }  
              // args 매칭  
              for (size_t i = 0; i < new_call->args.size(); i++) {  
                if (!new_call->args[i].same_as(candidate->args[i]) &&  
                    !IsEqualScalar(new_call->args[i], candidate->args[i])) {  
                  is_equivalent = false;  
                  break;  
                }  
              }  
              if (!is_equivalent) continue;  
              return GetRef<Call>(candidate);  
            }  
          }  
        }  
        expr_map_[new_call->op].push_back(new_expr);  
        return new_expr;  
      }  
    

대략적인 아이디어는 `expr_map_`이라는 `std::unordered_map<Expr, std::vector<Expr>, ObjectPtrHash, ObjectPtrEqual> expr_map_;`를 활용해 같은 op을 가진 순회된 expr을 매핑하는 것이다. 그리고 같은 op의 표현식을 만날 때마다 이미 기록된 expr과 매칭한다. 매칭은 op의 attrs 속성뿐만 아니라 인자 목록까지 포함하며, 둘이 완전히 같으면 두 표현식은 공통 부분식이라는 뜻이므로 새 표현식을 반환하지 않는다. 이렇게 하면 Relay Function의 공통 부분식을 제거할 수 있다.

여기까지 와서도, 처음에 로드한 simplenet의 Relay Function이 일부 Pass를 거친 후 구체적으로 어떻게 변하는지는 아직 명확하지 않을 수 있다. 사실 필자도 아직 이 문제를 분명하게 파악하지 못했다. 이 문제는 다음을 위해 남겨 두어야 할 것 같다.

# 0x5. 정리

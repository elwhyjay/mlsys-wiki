# 0x1. 왜 딥러닝 컴파일러가 필요한가?

"딥러닝 컴파일러"라는 단어는 두 부분으로 나누어 살펴볼 수 있습니다.

먼저 딥러닝 분야에 대해 이야기해 봅시다. 학습 프레임워크의 관점에서 보면, Google의 TensorFlow와 Facebook의 PyTorch가 전 세계적으로 가장 주류인 딥러닝 프레임워크이고, 그 외에도 Amazon의 MxNet, Baidu의 Paddle, Megvii의 MegEngine, Huawei의 Mindspore, OneFlow 등이 점차 많은 사용자에게 받아들여지고 사용되고 있습니다. 이렇게 많은 학습 프레임워크 중에서 우리는 도대체 무엇을 선택해야 할까요? 사용 편의성을 추구한다면 PyTorch를 선택할 수도 있고, 프로젝트 배포를 중시한다면 TensorFlow를 선택할 수도 있으며, 분산 학습 속도를 중시한다면 OneFlow를 사용해 볼 수도 있습니다. 그래서 이 선택 문제에는 정답이 없고, 개인의 취향에 달려 있습니다. 추론 프레임워크의 관점에서 보면, 어떤 학습 프레임워크로 모델을 학습시키든 우리는 결국 학습된 모델을 실제 환경에 배포해야 합니다. 모델을 배포할 때, 배포해야 할 디바이스가 매우 다양하다는 점을 발견하게 됩니다. 예를 들어 Intel CPU/Nvidia GPU/Intel GPU/Arm CPU/Arm GPU/FPGA/NPU(Huawei HiSilicon)/BPU(Horizon)/MLU(Cambricon) 등이 있는데, **모든** 가능한 배포 디바이스에서 **좋은 성능과 사용 편의성**을 동시에 만족시키는 추론 프레임워크를 직접 작성하는 것은 매우 어려운 일입니다.

일반적으로 모델을 특정 디바이스에 배포하려면, 하드웨어 제조사가 자체적으로 내놓은 추론 프레임워크를 사용하게 됩니다. 예를 들어 Intel CPU/GPU에서는 OpenVINO, Arm CPU/GPU에서는 NCNN/MNN, Nvidia GPU에서는 TensorRT를 사용하는 식입니다. 하드웨어 디바이스마다 특정한 추론 프레임워크를 사용해 배포하는 것이 최적이긴 하지만, 동시에 문제도 존재합니다. 예를 들어, 어떤 개발자가 학습한 모델 하나를 여러 종류의 디바이스에 배포해야 한다고 합시다. 그러면 개발자는 학습한 모델을 각 프레임워크가 읽을 수 있는 포맷으로 각각 변환해야 하고, 게다가 각 추론 프레임워크의 op 구현이 완전히 일치하는지도 고려해야 합니다. 그뿐 아니라 서로 다른 플랫폼에 배포할 때 자주 발생하는 문제로, 한 하드웨어에서는 효율적으로 추론되던 모델이 다른 하드웨어로 옮겨가면 성능이 급격히 떨어지는 상황도 있습니다. 또한 이전에 ONNX를 다룬 글에서 보았듯이, 프레임워크 간의 모델 변환 작업 자체가 다양한 학습 프레임워크 모델의 빠른 배포를 가로막는 큰 원인 중 하나입니다.

다음으로 컴파일러에 대해 간단히 설명하겠습니다. 사실 컴파일러 발전의 초창기에도 **다양한 딥러닝 학습 프레임워크의 모델을 다양한 하드웨어에 배포해야 하는** 상황과 비슷한 일이 있었습니다. 역사적으로 C/C++/Java 등 매우 많은 프로그래밍 언어가 등장했고, 이어서 각 하드웨어마다 특정 프로그래밍 언어가 대응되었으며, 다시 특정 컴파일러를 통해 컴파일하여 머신 코드를 생성했습니다. 하드웨어와 언어의 종류가 늘어남에 따라 컴파일러의 유지 보수가 얼마나 어려워질지 충분히 상상할 수 있습니다. 다행히도 현대의 컴파일러는 이미 이 문제를 해결했습니다. 그렇다면 컴파일러는 이 문제를 구체적으로 어떻게 해결했을까요?

위의 문제를 해결하기 위해, 과학자들은 컴파일러를 frontend, 중단(middle-end), backend 등의 개념으로 추상화하고, IR(Intermediate Representation)이라는 개념을 도입했습니다. 설명은 다음과 같습니다.

  * 컴파일러 frontend: C/C++/Java 등 다양한 언어를 입력받아 코드 생성을 수행하고 IR을 출력합니다.
  * 컴파일러 중단(middle-end): IR을 입력받아 다양한 backend가 공유할 수 있는 최적화(상수 치환, 데드 코드 제거, 루프 최적화 등)를 수행하고 최적화된 IR을 출력합니다.
  * 컴파일러 backend: 최적화된 IR을 입력받아 하드웨어별 플랫폼 관련 최적화와 하드웨어 명령어 생성을 수행하고 목적 파일을 출력합니다.

컴파일러의 해결 방식에서 영감을 받아 딥러닝 컴파일러가 제안되었습니다. **각 학습 프레임워크에서 학습한 모델을 다양한 프로그래밍 언어처럼** 보고, 이러한 모델들을 딥러닝 컴파일러에 입력한 뒤 IR을 출력하는 식입니다. 딥러닝의 IR은 사실상 계산 그래프이므로, 이를 직접 **Graph IR**이라고 부를 수도 있습니다. 이 Graph IR에 대해 그래프 최적화를 수행한 뒤 IR을 다시 출력해 다양한 하드웨어에 배포할 수 있습니다. 이렇게 하면 딥러닝 컴파일러의 과정도 전통적인 컴파일러와 비슷해지며, 앞서 언급한 여러 번거로운 문제를 해결할 수 있습니다. 이 아이디어 역시 lansediao님의 그림을 빌려 표현하면 다음과 같습니다.

# 0x02. TVM

위의 딥러닝 컴파일러 아이디어를 바탕으로, Tianqi Chen이 주도한 TVM이 등장했습니다. TVM은 컴파일 최적화 기반의 딥러닝 추론 프레임워크입니다(일단 추론용이라고 해 두지만, 학습 기능도 탐색되고 통합되기 시작한 것으로 보입니다). TVM의 아키텍처 그림을 살펴봅시다. 출처: https://tvm.apache.org/2017/10/06/nnvm-compiler-announcement

![이미지](images/img_03.png)TVM 아키텍처 그림

이 그림에서 볼 수 있듯이, TVM 아키텍처의 핵심은 NNVM 컴파일러입니다(참고로 최신 TVM은 이미 NNVM을 Relay로 업그레이드했으므로, 뒤에서 언급되는 Relay 역시 NNVM의 역할을 한다고 볼 수 있습니다). NNVM 컴파일러는 TensorFlow/PyTorch/Caffe/MxNet 등 딥러닝 프레임워크의 모델을 직접 입력받을 수 있고, ONNX, CoreML 같은 일부 모델 중간 포맷도 지원합니다. 이러한 모델들은 NNVM에 의해 직접 Graph IR로 컴파일되고, 이 Graph IR은 다시 한 번 최적화되어 최적화된 Graph IR이 출력됩니다. 마지막으로 서로 다른 backend에 대해 이 Graph IR은 각 backend가 인식할 수 있는 머신 코드로 컴파일되어 모델 추론을 완성합니다. 예를 들어 CPU의 경우, NNVM은 LLVM이 인식할 수 있는 IR을 출력하고, 이는 다시 LLVM 컴파일러를 통해 머신 코드로 컴파일되어 CPU에서 실행됩니다.

# 0x03. 환경 구성

도구를 잘 쓰려면 먼저 도구를 잘 갖춰야 한다고 했듯이, TVM을 본격적으로 살펴보기 전에 TVM의 설치 절차를 알아봅시다. 여기서는 공식 설치 문서를 참고해 두 가지 방법을 제공합니다.

## 0x03.1 Docker 기반 방식

TVM이 설치 및 구성된 docker 이미지를 직접 가져와 docker 안에서 TVM을 사용할 수 있는데, 이것이 가장 빠르고 편리한 방법입니다. 예를 들어 cuda backend 지원이 컴파일된 TVM 이미지를 가져와 컨테이너를 시작하는 예시는 다음과 같습니다.
    
    
    docker pull tvmai/demo-gpu  
    nvidia-docker run --rm -it tvmai/demo-gpu bash  
    

이렇게 하면 TVM이 구성된 컨테이너로 성공적으로 진입해 TVM을 사용할 수 있습니다.

## 0x03.2 Ubuntu를 예로 한 로컬 빌드

TVM 소스 코드를 수정하거나 TVM에 기여할 필요가 있다면 로컬에서 TVM을 빌드할 수 있습니다. Ubuntu를 예로 들어 빌드 및 구성 절차는 다음과 같습니다.
    
    
    git clone --recursive https://github.com/apache/tvm tvm  
    cd tvm  
    mkdir build  
    cp cmake/config.cmake build  
    cd build  
    cmake ..  
    make -j4  
    export TVM_HOME=/path/to/tvm  
    export PYTHONPATH=$TVM_HOME/python:${PYTHONPATH}  
    

이렇게 하면 TVM 환경 구성이 완료되어 개발과 테스트를 진행할 수 있습니다.

제 권장 사항은 로컬 개발과 디버깅에는 뒤의 방식을 사용하고, 산업 배포에는 Docker 방식을 사용하는 것입니다.

# 0x04. 예제 시연

예제를 보여드리기 전에 제 환경 구성을 말씀드리자면, pytorch1.7.0 && TVM 0.8.dev0 입니다.

여기서는 PyTorch 모델을 예로 들어, TVM이 PyTorch 모델을 Relay(NNVM의 업그레이드 버전이라고 이해하면 됩니다)를 통해 TVM의 계산 그래프로 구성하고 그래프 최적화를 수행한 뒤, 최종적으로 LLVM을 통해 Intel CPU로 컴파일하여 실행하는 과정을 보여드립니다. 마지막으로, TVM 최적화 후의 Relay Graph 추론 속도와 PyTorch 모델을 직접 사용해 추론한 속도를 비교해 보았습니다. 여기서는 torchvision의 ResNet18을 예시로 사용했고, 결과는 다음과 같습니다.
    
    
    Relay top-1 id: 282, class name: tiger cat  
    Torch top-1 id: 282, class name: tiger cat  
    Relay time:  1.1846002000000027 seconds  
    Torch time:  2.4181047000000007 seconds  
    

예측 결과는 완전히 동일한 가운데, TVM이 약 2배 정도의 속도 향상을 가져온다는 것을 볼 수 있습니다. 코드의 흐름을 간단히 소개합니다. 코드는 여기(https://github.com/BBuf/tvm_learn)에서 찾아볼 수 있습니다.

## 0x04.1 TVM과 PyTorch import 및 ResNet18 모델 로드
    
    
    import time  
    import tvm  
    from tvm import relay  
      
    import numpy as np  
      
    from tvm.contrib.download import download_testdata  
      
    # PyTorch imports  
    import torch  
    import torchvision  
      
    ######################################################################  
    # Load a pretrained PyTorch model  
    # -------------------------------  
    model_name = "resnet18"  
    model = getattr(torchvision.models, model_name)(pretrained=True)  
    model = model.eval()  
      
    # We grab the TorchScripted model via tracing  
    input_shape = [1, 3, 224, 224]  
    input_data = torch.randn(input_shape)  
    scripted_model = torch.jit.trace(model, input_data).eval()  
    

주의할 점은 Relay가 PyTorch 모델을 파싱할 때는 TorchScript 포맷의 모델을 파싱하므로, 여기서는 `torch.jit.trace`를 사용해 원본 PyTorch 모델을 한 번 실행한 뒤 TorchScript 모델을 export 한다는 것입니다.

## 0x04.2 테스트 이미지 로드

테스트 이미지 한 장을 로드하고 약간의 후처리 과정을 수행합니다.
    
    
    from PIL import Image  
      
    img_url = "https://github.com/dmlc/mxnet.js/blob/main/data/cat.png?raw=true"  
    img_path = download_testdata(img_url, "cat.png", module="data")  
    img = Image.open(img_path).resize((224, 224))  
      
    # Preprocess the image and convert to tensor  
    from torchvision import transforms  
      
    my_preprocess = transforms.Compose(  
        [  
            transforms.Resize(256),  
            transforms.CenterCrop(224),  
            transforms.ToTensor(),  
            transforms.Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225]),  
        ]  
    )  
    img = my_preprocess(img)  
    # Batch 차원 추가  
    img = np.expand_dims(img, 0)  
    

## 0x04.3 Relay로 TorchScript 모델 import 및 LLVM backend로 컴파일

이제 PyTorch의 graph를 Relay로 import 하여 Relay Graph로 만듭니다. 입력 레이어의 이름은 임의로 지정할 수 있습니다. 그런 다음 주어진 설정으로 graph를 LLVM 타겟 하드웨어에 컴파일합니다.
    
    
    ######################################################################  
    # Import the graph to Relay  
    # -------------------------  
    # Convert PyTorch graph to Relay graph. The input name can be arbitrary.  
    input_name = "input0"  
    shape_list = [(input_name, img.shape)]  
    mod, params = relay.frontend.from_pytorch(scripted_model, shape_list)  
      
    ######################################################################  
    # Relay Build  
    # -----------  
    # Compile the graph to llvm target with given input specification.  
    target = "llvm"  
    target_host = "llvm"  
    ctx = tvm.cpu(0)  
    with tvm.transform.PassContext(opt_level=3):  
        lib = relay.build(mod, target=target, target_host=target_host, params=params)  
      
    

## 0x04.4 타겟 하드웨어에서 추론 수행 및 분류 결과 출력

여기서는 추론에 걸린 시간을 측정하기 위해 타이머 함수를 추가했습니다.
    
    
    ######################################################################  
    # Execute the portable graph on TVM  
    # ---------------------------------  
    # Now we can try deploying the compiled model on target.  
    from tvm.contrib import graph_runtime  
      
    tvm_t0 = time.clock()  
    for i in range(10):  
        dtype = "float32"  
        m = graph_runtime.GraphModule(lib["default"](ctx))  
        # Set inputs  
        m.set_input(input_name, tvm.nd.array(img.astype(dtype)))  
        # Execute  
        m.run()  
        # Get outputs  
        tvm_output = m.get_output(0)  
    tvm_t1 = time.clock()  
    

다음으로 1000개 클래스의 사전에서 Top1 확률에 해당하는 클래스를 찾아 출력하고, 동시에 PyTorch로 원본 모델도 실행해 두 결과가 일치하는지와 추론 시간이 어떻게 되는지 살펴봅니다.
    
    
    #####################################################################  
    # Look up synset name  
    # -------------------  
    # Look up prediction top 1 index in 1000 class synset.  
    synset_url = "".join(  
        [  
            "https://raw.githubusercontent.com/Cadene/",  
            "pretrained-models.pytorch/master/data/",  
            "imagenet_synsets.txt",  
        ]  
    )  
    synset_name = "imagenet_synsets.txt"  
    synset_path = download_testdata(synset_url, synset_name, module="data")  
    with open(synset_path) as f:  
        synsets = f.readlines()  
      
    synsets = [x.strip() for x in synsets]  
    splits = [line.split(" ") for line in synsets]  
    key_to_classname = {spl[0]: " ".join(spl[1:]) for spl in splits}  
      
    class_url = "".join(  
        [  
            "https://raw.githubusercontent.com/Cadene/",  
            "pretrained-models.pytorch/master/data/",  
            "imagenet_classes.txt",

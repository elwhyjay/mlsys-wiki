# 0x0. 머리말

[【밑바닥부터 배우는 딥러닝 컴파일러】1. 딥러닝 컴파일러 및 TVM 소개](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247493461&idx=1&sn=f2580a4a795fbbee89c8f7da6cca4ae4&scene=21#wechat_redirect>) 글에서 우리는 TVM이 다양한 딥러닝 학습 프레임워크의 모델(계산 그래프)을 내부의 Graph IR(Relay)로 변환하고, TVM이 제공하는 명령 생성 모듈을 통해 Graph IR을 특정 하드웨어에서 실행 가능한 명령이나 코드로 번역한다는 사실을 이미 알고 있습니다. 전체적으로 TVM의 사상은 표현(representation)과 스케줄링(scheduling)의 분리로 요약할 수 있습니다. 표현은 IR을 의미하고, 스케줄링은 scheduler를 의미합니다. 동시에, 고성능 컴퓨팅 측면에서 TVM은 여러 가지 schedule primitive(**scheduler**)를 제공하는데, 여기에는 op 융합, 읽기/쓰기 캐시, 블록 분할 계산, 병렬 계산 등 대부분의 일반적인 최적화 기법이 포함되어 있으며, 이러한 계산 방법들은 모두 scheduler를 통해 구현할 수 있습니다. 그래서 이번 절에서는 TVM의 scheduler를 함께 살펴보겠습니다.

# 0x01. 개요

우리는 TVM의 핵심이 자동 코드 생성 기술이며, scheduler는 이 자동 코드 생성 기술의 핵심 개념이라는 것을 알고 있습니다. scheduler는 간단히 말해 일련의 최적화 선택지의 집합으로 이해할 수 있는데, 이러한 선택은 전체 계산 결과에는 영향을 미치지 않지만 계산 성능에는 매우 중요한 영향을 줍니다. 흔한 예시 하나가 **행렬 곱(MatMul)** 입니다. 입력 행렬 A와 B가 주어지고, 차원이 각각 일정하게 정해져 있을 때, 결과 행렬 C를 얻게 됩니다. 제가 이전에 작성한 [길은 멀고 험하다_행렬 곱 최적화 재탐구](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247491392&idx=1&sn=ddb072d2ad5eeaa6c1dba0eb440366bf&scene=21#wechat_redirect>) 글에서 이 계산을 가속하기 위한 일련의 최적화 방법을 자세히 정리해 두었는데, 여기서는 Arm 단을 예로 들었습니다. 구체적인 내용은 다음과 같습니다.

이러한 구현들이 모두 행렬 곱이라는 동일한 계산 작업을 완수했고, 즉 입력과 출력은 완전히 동일하지만, 계산 과정에서는 **일련의 서로 다른 최적화 기법**을 사용했음을 알 수 있습니다. 이러한 최적화 알고리즘의 집합을 통틀어 **scheduler**라고 부를 수 있습니다.

다음으로 **scheduler**가 전체 TVM 소프트웨어 스택에서 어떤 위치에 있는지 명확히 해 봅시다. 최근 이런 그림을 계속 찾고 있었는데, OpenMMLab에서 최근에 공개한 Ansor 소개 글에 있는 그림이 마침 이 문제를 완벽히 설명해 주어, 여기 가져왔습니다. 딥러닝에서 흔히 볼 수 있는 MatMul+Add+Relu 계산 그래프를 예로 TVM의 코드 생성 과정을 살펴봅시다. 먼저 TVM은 입력으로 받은 계산 그래프를 TVM 내의 도메인 특화 언어인 Tensor Expression으로 변환하는데, 그림에서 노란색 부분에 해당합니다. 그 다음 사용자는 수동으로 계산 전략, 즉 scheduler를 지정할 수 있고, 그러면 TVM이 자동으로 특정 백엔드의 코드를 생성해 줍니다. 그림 안의 tiling과 binding은 각각 분할과 바인딩을 의미하며, 이들 또한 scheduler입니다. 이제 우리는 TVM 소프트웨어 스택에서 scheduler가 어디에 위치하는지를 분명히 하였고, TVM이 고성능 코드를 생성할 수 있는지의 관건이 scheduler가 합리적으로 지정되었는지, 즉 최적화 알고리즘이 지정한 백엔드에서 동작하고 효율적인지에 달려 있다는 것도 알게 되었습니다.

# 0x02. Tensor Expression부터 시작해서 TVM이 어떻게 CUDA 코드를 생성하는지 살펴보기

천치(chentianqi) 선배가 TVM 문서에서 소개한 Tensor Expression 예제를 통해 위 그림에 묘사된 TVM 코드 생성 과정을 처음 느껴 봅시다. 이 안에는 scheduler도 포함되어 있습니다. 이번 절 이후에는 scheduler의 더 많은 사용법을 느껴 볼 수 있도록 다른 예제도 몇 가지 더 들겠습니다. 이제 Tensor Expression부터 시작해서 TVM이 어떻게 코드를 생성하는지, 그리고 scheduler를 구체적으로 어떻게 지정하는지 살펴봅시다. 먼저 사용할 패키지들을 import 합니다.
    
    
    import tvm  
    import tvm.testing  
    from tvm import te  
    import numpy as np  
      
    # 전역 환경 정의  
      
    tgt_host = "llvm"  
    # GPU가 활성화되어 있다면, 해당 GPU로 변경합니다. 예: cuda, opencl, rocm  
    tgt = "cuda"  
    

그 다음 벡터 덧셈을 사용해 TVM의 워크플로를 시연하겠습니다. 첫 번째 단계로, 우리는 계산을 기술해야 합니다. TVM은 Tensor Expression을 채택하며, 각 중간 결과는 다차원 배열로 표현됩니다. 사용자는 결과 tensor를 생성하는 계산 규칙을 기술해야 합니다. 먼저 형상을 표현하기 위한 심볼 변수 n을 정의합니다. 그런 다음 주어진 형상을 가진 두 개의 placeholder tensor A와 B를 정의합니다. 이어서 compute 함수를 사용해 결과 tensor C를 기술합니다. compute 함수는 tensor 형식과, tensor의 각 위치의 계산 규칙을 기술하는 lambda 함수를 인자로 받습니다. 이 단계에서는 실제 계산이 일어나지 않으며, 단지 어떻게 계산해야 하는지를 선언할 뿐입니다. 코드는 다음과 같습니다.
    
    
    n = te.var("n")  
    A = te.placeholder((n,), name="A")  
    B = te.placeholder((n,), name="B")  
    C = te.compute(A.shape, lambda i: A[i] + B[i], name="C")  
    print(type(C))  
    

출력 정보는 `<class 'tvm.te.tensor.Tensor'>` 입니다.

이어서, 위의 몇 줄이 계산 규칙을 기술하긴 했지만, 우리는 C를 계산할 수 있는 여러 방법이 있습니다. 왜냐하면 C는 axis 상에서 데이터 병렬 방식으로 계산할 수 있기 때문입니다. TVM은 사용자가 schedule이라고 불리는 계산 기술을 제공하도록 요구하는데, 이는 다음 코드와 동등합니다.
    
    
    for (int i = 0; i < n; ++i) {  
      C[i] = A[i] + B[i];  
    }  
    

`te.create_schedule`을 호출하여 scheduler를 생성한 다음, split 구조를 사용해 C의 axis를 분할합니다. 이렇게 하면 원래 하나의 반복 axis가 두 개의 반복 axis의 곱으로 쪼개집니다. 코드는 다음과 같습니다.
    
    
    s = te.create_schedule(C.op)  
    bx, tx = s[C].split(C.op.axis[0], factor=64)  
    

이는 다음 코드와 동등합니다.
    
    
    for (int bx = 0; bx < ceil(n / 64); ++bx) {  
      for (int tx = 0; tx < 64; ++tx) {  
        int i = bx * 64 + tx;  
        if (i < n) {  
          C[i] = A[i] + B[i];  
        }  
      }  
    }  
    

마지막으로, 반복 axis bx와 tx를 GPU의 계산 grid 안의 thread에 bind 합니다. 이것은 GPU 전용 구조로, GPU에서 실행되는 코드를 생성할 수 있게 해 줍니다.
    
    
    if tgt == "cuda" or tgt == "rocm" or tgt.startswith("opencl"):  
        s[C].bind(bx, te.thread_axis("blockIdx.x"))  
        s[C].bind(tx, te.thread_axis("threadIdx.x"))  
    

이로써 scheduler 지정이 끝났고, 이제 위의 모든 코드를 컴파일하여 하나의 TVM 함수로 만들 수 있습니다. 기본적으로 TVM은 이를 type-erased 함수로 컴파일하며, Python 단에서 직접 호출할 수 있습니다. 아래에서는 `tvm.build`를 사용하여 컴파일된 함수를 생성합니다. 컴파일 함수는 scheduler, 함수 시그니처(입력과 출력 포함), 그리고 우리가 컴파일하려는 대상 언어를 인자로 받습니다. `fadd`의 컴파일 결과는 GPU 디바이스 함수(GPU가 관여될 때)와 그 GPU 함수를 호출하는 host 측 wrapper입니다. `fadd`는 생성된 host 측 wrapper 함수이며, 내부적으로 생성된 디바이스 함수에 대한 참조를 포함하고 있습니다. 코드는 다음과 같습니다.
    
    
    fadd = tvm.build(s, [A, B, C], tgt, target_host=tgt_host, name="myadd")  
    

컴파일된 TVM 함수는 어떤 언어에서도 호출할 수 있는 간결한 C API를 생성합니다. TVM은 빠른 테스트와 프로토타입 개발을 돕기 위해 Python에서 최소한의 array API를 제공합니다. 이 array API는 DLPack(https://github.com/dmlc/dlpack) 표준을 기반으로 합니다. 이 함수를 실행하려면 먼저 GPU context를 만들고, `tvm.nd.array`로 데이터를 GPU로 복사한 다음, 컴파일된 함수 `fadd`로 계산을 실행하고, 마지막으로 `asnumpy()`로 GPU 측 array를 CPU로 복사해 numpy로 계산하여 두 결과의 차이를 비교합니다. 이 부분의 코드는 다음과 같습니다.
    
    
    ctx = tvm.context(tgt, 0)  
      
    n = 1024  
    a = tvm.nd.array(np.random.uniform(size=n).astype(A.dtype), ctx)  
    b = tvm.nd.array(np.random.uniform(size=n).astype(B.dtype), ctx)  
    c = tvm.nd.array(np.zeros(n, dtype=C.dtype), ctx)  
    fadd(a, b, c)  
    tvm.testing.assert_allclose(c.asnumpy(), a.asnumpy() + b.asnumpy())  
    

여기까지로 전체 계산 과정은 끝났지만, 다들 TVM이 생성한 코드가 어떻게 생겼는지 매우 궁금하실 거라 생각합니다. TVM은 사용자가 생성된 코드를 확인할 수 있는 인터페이스도 제공합니다. `tvm.build`의 결과는 하나의 TVM Module입니다. `fadd`는 host wrapper를 포함하는 모듈이며, 동시에 CUDA(GPU) 디바이스용 기능 모듈도 포함하고 있습니다. 다음 코드를 사용하여 생성된 코드를 출력해 봅시다.
    
    
    if tgt == "cuda" or tgt == "rocm" or tgt.startswith("opencl"):  
        dev_module = fadd.imported_modules[0]  
        print("-----GPU code-----")  
        print(dev_module.get_source())  
    else:  
        print(fadd.get_source())  
    

출력은 다음과 같습니다.
    
    
    -----GPU code-----  
      
    #ifdef _WIN32  
      using uint = unsigned int;  
      using uchar = unsigned char;  
      using ushort = unsigned short;  
      using int64_t = long long;  
      using uint64_t = unsigned long long;  
    #else  
      #define uint unsigned int  
      #define uchar unsigned char  
      #define ushort unsigned short  
      #define int64_t long  
      #define uint64_t ulong  
    #endif  
    extern "C" __global__ void myadd_kernel0(float* __restrict__ C, float* __restrict__ A, float* __restrict__ B, int n, int stride, int stride1, int stride2) {  
      if (((int)blockIdx.x) < (n >> 6)) {  
        C[((((((int)blockIdx.x) * 64) + ((int)threadIdx.x)) * stride2))] = (A[((((((int)blockIdx.x) * 64) + ((int)threadIdx.x)) * stride))] + B[((((((int)blockIdx.x) * 64) + ((int)threadIdx.x)) * stride1))]);  
      } else {  
        if (((((int)blockIdx.x) * 64) + ((int)threadIdx.x)) < n) {  
          C[((((((int)blockIdx.x) * 64) + ((int)threadIdx.x)) * stride2))] = (A[((((((int)blockIdx.x) * 64) + ((int)threadIdx.x)) * stride))] + B[((((((int)blockIdx.x) * 64) + ((int)threadIdx.x)) * stride1))]);  
        }  
      }  
    }  
    

자, 여기까지 살펴본 것으로, 우리는 TVM에서 scheduler를 정의하고 자동으로 계산 코드를 생성하는 방법을 알게 되었습니다.

# 0x03. scheduler에 대한 더 자세한 예제

## **split**

scheduler에 대한 더 자세한 예제는 이 글에서 볼 수 있습니다: https://zhuanlan.zhihu.com/p/94846767. 여기서는 몇 가지를 간단히 예로 들어 이해해 봅시다. 예를 들어 루프 최적화에서, split을 예로 들면 코드는 다음과 같습니다.
    
    
    import tvm  
    from tvm import te  
      
    n = 1024  
    A = te.placeholder((n,), name='A')  
    k = te.reduce_axis((0, n), name='k')  
      
    B = te.compute((1,), lambda i: te.sum(A[k], axis=k), name='B')  
      
    s = te.create_schedule(B.op)  
      
    print(tvm.lower(s, [A, B], simple_mode=True))  
    print("---------cutting line---------")  
      
    ko, ki = s[B].split(B.op.reduce_axis[0], factor=32)  
      
    print(tvm.lower(s, [A, B], simple_mode=True))  
    

생성된 함수는 다음과 같습니다.
    
    
    primfn(A_1: handle, B_1: handle) -> ()  
      attr = {"global_symbol": "main", "tir.noalias": True}  
      buffers = {B: Buffer(B_2: Pointer(float32), float32, [1], []),  
                 A: Buffer(A_2: Pointer(float32), float32, [1024], [])}  
      buffer_map = {A_1: A, B_1: B} {  
      B_2[0] = 0f32  
      for (k: int32, 0, 1024) {  
        B_2[0] = ((float32*)B_2[0] + (float32*)A_2[k])  
      }  
    }  
      
      
    ---------cutting line---------  
    primfn(A_1: handle, B_1: handle) -> ()  
      attr = {"global_symbol": "main", "tir.noalias": True}  
      buffers = {B: Buffer(B_2: Pointer(float32), float32, [1], []),  
                 A: Buffer(A_2: Pointer(float32), float32, [1024], [])}

        for (k.inner: int32, 0, 32) {  
          B_2[0] = ((float32*)B_2[0] + (float32*)A_2[((k.outer*32) + k.inner)])  
        }  
      }  
    }  
    

split이 iter를 factor 간격으로 outer와 inner 두 층의 반복으로 나누어 루프 층수를 늘리고, 이를 통해 루프 연산을 더 작은 sub-task로 분할하는 데 사용된다는 것을 볼 수 있습니다. CUDA 문서에서 알 수 있듯이 gridDim과 blockDim은 모두 최대 3차원까지 가능하므로, split을 통해 새로운 차원을 만들어 grid와 block에 bind 할 수 있습니다. 이 연산은 CUDA 코드 생성에서 매우 자주 사용됩니다.

실험 코드는 https://github.com/BBuf/tvm_learn/blob/main/scheduler 에서 찾을 수 있고, 사용한 tvm 버전은 0.8.0-dev입니다.

## reorder

두 번째로 다루고 싶은 scheduler는 reorder이며, 실험 코드와 TVM이 생성한 코드를 함께 첨부합니다.
    
    
    import tvm  
    from tvm import te  
      
    n = 1024  
    A = te.placeholder((n, n), name='A')  
    B = te.placeholder((n,n), name='B')  
    C = te.compute((n, n), lambda i, j: A[i, j] + B[i, j], name='C')  
      
    s = te.create_schedule(C.op)  
      
    xo, xi = s[C].split(s[C].op.axis[0], factor=32)  
    yo, yi = s[C].split(s[C].op.axis[1], factor=32)  
      
    print(tvm.lower(s, [A, B, C], simple_mode=True))  
    print("---------cutting line---------")  
      
    s[C].reorder(xo, yo, yi, xi)  
      
    print(tvm.lower(s, [A, B, C], simple_mode=True))  
    

생성된 함수는 다음과 같습니다.
    
    
    primfn(A_1: handle, B_1: handle, C_1: handle) -> ()  
      attr = {"global_symbol": "main", "tir.noalias": True}  
      buffers = {C: Buffer(C_2: Pointer(float32), float32, [1024, 1024], []),

                 A: Buffer(A_2: Pointer(float32), float32, [1024, 1024], [])}  
      buffer_map = {A_1: A, B_1: B, C_1: C} {

            for (j.inner: int32, 0, 32) {  
              C_2[((((i.outer*32768) + (i.inner*1024)) + (j.outer*32)) + j.inner)] = ((float32*)A_2[((((i.outer*32768) + (i.inner*1024)) + (j.outer*32)) + j.inner)] + (float32*)B_2[((((i.outer*32768) + (i.inner*1024)) + (j.outer*32)) + j.inner)])  
            }  
          }  
        }  
      }  
    }  
      
      
    ---------cutting line---------  
    primfn(A_1: handle, B_1: handle, C_1: handle) -> ()

reorder가 루프 iter의 안팎 순서를 재배치한 것을 볼 수 있습니다. 지역성 원리(locality)에 따라 이렇게 하면 cache 안에 이미 들어 있는 데이터를 최대한 활용할 수 있고, 데이터를 빈번하게 로드/언로드 하는 상황을 줄여 프로그램의 성능을 높일 수 있습니다. 이는 우리가 이전에 행렬 곱을 탐구할 때 K 차원을 가장 바깥 층에 두고 M을 가장 바깥 층에 두지 않았던 이유이기도 합니다.

## tile

이어서 tile이라는 scheduler를 살펴봅시다. tile은 stage(하나의 op로 이해할 수 있고, 하나의 op는 하나의 stage에 대응됨)의 두 차원을 각자의 factor에 따라 분할하고, 고정된 순서로 두 개의 outer iter와 두 개의 inner iter를 반환하여 루프 층수를 늘리고 더 작은 계산 작업을 만들 수 있습니다. 사실 tile은 split과 reorder로 구현할 수 있으며, tile은 행렬 곱과 컨볼루션 계산의 중요한 schedule입니다. 이 글의 두 번째 절 개요 부분에서, 우리는 Arm 단에서 손으로 작성한 다양한 최적화 알고리즘으로 행렬 곱을 최적화한 사례를 첨부했는데, 거기서 분할 계산 전략을 여러 번 사용했습니다. 그것이 바로 여기의 tile scheduler이며, cache와 레지스터를 더 잘 활용하여 더 높은 성능을 얻을 수 있습니다.
    
    
    primfn(A_1: handle, B_1: handle, C_1: handle) -> ()  
      attr = {"global_symbol": "main", "tir.noalias": True}  
      buffers = {C: Buffer(C_2: Pointer(float32), float32, [1024, 1024], []),  
                 B: Buffer(B_2: Pointer(float32), float32, [1024, 1024], []),  
                 A: Buffer(A_2: Pointer(float32), float32, [1024, 1024], [])}  
      buffer_map = {A_1: A, B_1: B, C_1: C} {  
      for (i: int32, 0, 1024) {  
        for (j: int32, 0, 1024) {  
          C_2[((i*1024) + j)] = 0f32  
          for (K: int32, 0, 1024) {  
            C_2[((i*1024) + j)] = ((float32*)C_2[((i*1024) + j)] + ((float32*)A_2[((i*1024) + K)]*(float32*)B_2[((K*1024) + j)]))  
          }  
        }  
      }  
    }  
      
      
    ---------cutting line---------  
    primfn(A_1: handle, B_1: handle, C_1: handle) -> ()

      buffers = {C: Buffer(C_2: Pointer(float32), float32, [1024, 1024], []),  
                 B: Buffer(B_2: Pointer(float32), float32, [1024, 1024], []),  
                 A: Buffer(A_2: Pointer(float32), float32, [1024, 1024], [])}  
      buffer_map = {A_1: A, B_1: B, C_1: C} {

            for (j.inner: int32, 0, 32) {  
              C_2[((((i.outer*32768) + (i.inner*1024)) + (j.outer*32)) + j.inner)] = 0f32  
              for (K: int32, 0, 1024) {  
                C_2[((((i.outer*32768) + (i.inner*1024)) + (j.outer*32)) + j.inner)] = ((float32*)C_2[((((i.outer*32768) + (i.inner*1024)) + (j.outer*32)) + j.inner)] + ((float32*)A_2[(((i.outer*32768) + (i.inner*1024)) + K)]*(float32*)B_2[(((K*1024) + (j.outer*32)) + j.inner)]))  
              }

      }

## vectorize

마지막으로 또 하나의 scheduler, 즉 벡터화(vectorize)를 소개하겠습니다. 이는 공식 계정의 [AI PC 단 알고리즘 최적화] 시리즈에서 소개한 일련의 최적화 방법에 해당하며, 예를 들어 Intel CPU에서 SSE나 AVX 등의 명령어 셋을 사용해 일반 프로그램을 벡터화하여 더 좋은 성능을 얻는 것입니다. 이제 TVM 안에서는 어떻게 사용하는지 살펴봅시다. 코드는 다음과 같습니다.
    
    
    import tvm  
    import numpy  
    import timeit  
    from tvm import te  
      
    M = 1024  
    N = 1024  
    A = te.placeholder((M, N), name='A')  
    B = te.placeholder((M, N), name='B')  
    C = te.compute(  
               (M, N),  
               lambda x, y: A[x, y] + B[x, y],  
               name='C')  
      
    s = te.create_schedule(C.op)  
    xo, yo, xi, yi = s[C].tile(C.op.axis[0], C.op.axis[1], 32, 32)  
      
    print(tvm.lower(s, [A, B, C], simple_mode=True))  
    print("---------cutting line---------")  
      
    s[C].vectorize(yi)  
      
    print(tvm.lower(s, [A, B, C], simple_mode=True))  
    

생성된 함수는 다음과 같습니다.
    
    
    primfn(A_1: handle, B_1: handle, C_1: handle) -> ()

                 A: Buffer(A_2: Pointer(float32), float32, [1024, 1024], [])}  
      buffer_map = {A_1: A, B_1: B, C_1: C} {

              C_2[((((x.outer*32768) + (x.inner*1024)) + (y.outer*32)) + y.inner)] = ((float32*)A_2[((((x.outer*32768) + (x.inner*1024)) + (y.outer*32)) + y.inner)] + (float32*)B_2[((((x.outer*32768) + (x.inner*1024)) + (y.outer*32)) + y.inner)])  
            }  
          }  
        }  
      }  
    }  
      
      
    ---------cutting line---------  
    primfn(A_1: handle, B_1: handle, C_1: handle) -> ()

vectorize가 iter 방향의 루프 반복을 ramp로 치환하여 SIMD 명령어를 통해 데이터의 일괄 계산을 구현하는 것을 볼 수 있습니다. 그리고 이는 데이터 size가 상수이고 분할된 iter가 2의 거듭제곱(즉 SIMD의 계산 수량을 만족할 때)일 때만 치환이 발생합니다. 그렇지 않으면 vectorize는 효과가 없으며, 이는 SIMD 계산 디바이스(예: Intel CPU, Arm CPU)의 흔히 쓰이는 schedule입니다.

지면 관계상 일일이 다루지 못한 중요한 scheduler들이 더 있습니다. 자세한 내용은 이 글을 참고하시기 바랍니다: https://zhuanlan.zhihu.com/p/94846767. 최신 버전의 TVM scheduler 실험을 실행하고 싶다면 https://github.com/BBuf/tvm_learn 에서 코드를 찾을 수 있습니다.

# 0x04. 정리

이 글은 주로 TVM의 몇 가지 예제와 결합하여 scheduler를 소개했습니다. 사실 여기까지 쓰고 보니 자연스럽게 또 몇 가지 의문이 떠오릅니다. 예를 들어 어떤 딥러닝 모델에 대해, 전체 계산 그래프에 위에서 소개한 scheduler 기법들을 어떻게 적용해야 효율적인 특정 백엔드용 코드를 생성할 수 있을까 하는 점입니다. 이때는 계산 그래프의 scheduler를 수동으로 지정하는 것이 현실적이지 않습니다. 이는 Auto-TVM과 Auto-Scheduler(또는 Ansor라고 불리는 것)와 관련이 있습니다. 언급하지 않을 수 없는 것은 Ansor가 OSDI 학회에 발표되었으며, 현재 Auto-TVM보다 더 좋은 성능을 보인다는 것입니다. https://zhuanlan.zhihu.com/p/360041136 이 최근에 발표된 글이 Ansor의 작동 메커니즘을 잘 소개해 주고 있으므로, 독자분들께 추천 드립니다. 이후에 관련 개념들을 정리한 후, 소스 코드 관점에서 TVM에 들어가 프런트엔드와 스케줄링의 구체적인 과정을 한 번 정리해 보고자 시도할 예정입니다.

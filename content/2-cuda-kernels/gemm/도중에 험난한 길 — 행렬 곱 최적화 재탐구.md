# 도중에 험난한 길 — 행렬 곱 최적화 재탐구

**이 글은 필자가 최근에 진행한 GEMM 최적화의 아이디어와 구현을 기록한 것이다. 대부분은 공개된 방안이며, 예를 들어 how-to-optimize-gemm 프로젝트에서 가져온 최적화 기법, ncnn에서 가져온 최적화 기법 등이다. 최종적으로 필자가 현재 구현한 버전은 armv7a에서 약 50% 정도의 하드웨어 이용률을 달성할 수 있다(이 이용률은 사실 아직 높지 않다. 필자도 한 걸음씩 학습하고 시도하는 중이니 고수분들은 너그럽게 봐주시기 바란다). 이 글은 이러한 아이디어와 핵심 구현 방법을 기록한 것이다. 개선한 row-major 코드(x86 + armv7a 버전)는 `https://github.com/BBuf/how-to-optimize-gemm` 에서 직접 받을 수 있다.**

# 1\. 머리말

먼저, 한 가지 강조하고 싶은 것은 어떤 알고리즘의 가속 효과와 속도를 판단할 때는 반드시 실측해야 하며, 다른 사람이 제시한 benchmark 데이터를 전적으로 믿어서는 안 된다는 점이다. 무엇이든 차분하게 한 걸음씩 진행할 필요가 있다. 이 글은 [how-to-optimize-gemm 기반 행렬 곱 최적화 탐구](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247490755&idx=1&sn=469af40d932df5521c013e4853302db6&scene=21#wechat_redirect>)를 토대로 한층 정밀한 테스트를 진행한 것이며, NCNN의 convolution 구현 아이디어를 참고하여 최종적으로 단일 코어 A53에서 45%의 하드웨어 이용률을 얻었다. 만약 입력 데이터의 Pack을 사전에 처리해 둔다면(**NC4HW4** 입력과 유사하게), 50% 이상의 하드웨어 이용률을 얻을 수 있다. 따라서 이 글에서는 위에 소개한 각각의 최적화 포인트를 해설하며, 이 알고리즘의 최적 버전은 이미 Msnhnet(`https://github.com/msnh2012/Msnhnet`)에 통합되어 있어 독자가 그곳에서도 확인할 수 있다. 이제 바로 일련의 최적화 기법을 소개하겠다.

독자가 특정 최적화의 효과와 대응 코드 구현을 구체적으로 보고자 한다면, 아래 결과 표(armv7a 기준)를 참고하고, `https://github.com/BBuf/how-to-optimize-gemm/tree/master/armv7a/src` 에서 해당 소스 파일을 선택하여 살펴보면 된다.

# 2\. 원본 구현

매우 간단하다. 그대로 구현한 것이며, 여기서 A의 차원, B의 차원, C의 차원이 주어졌을 때 matrix multiplication의 원본 구현은 다음과 같다(여기서는 row-major임에 유의).
    
    
    #define A( i, j ) a[ (i)*lda + (j) ]  
    #define B( i, j ) b[ (i)*ldb + (j) ]  
    #define C( i, j ) c[ (i)*ldb + (j) ]  
    // gemm C = A * B + C  
    void MatrixMultiply(int m, int n, int k, float *a, int lda, float *b, int ldb, float *c, int ldc)  
    {  
        for(int i = 0; i < m; i++){  
            for (int j=0; j<n; j++ ){      
                for (int p=0; p<k; p++ ){        
                    C(i, j) = C(i, j) + A(i, p) * B(p, j);  
                }  
            }  
        }  
    }  
    

이 버전의 gflops는 0.24g에 불과하며, 하드웨어 이용률은 1.4%밖에 되지 않는다. 이제부터 점진적으로 최적화를 진행하겠다.

# 3\. 한 번에 4개 원소 계산

여기서 한 번에 4개 원소를 계산한다는 것은 결과 행렬 C의 원소 4개를 한 번에 계산한다는 의미이다. 2절의 원본 구현에서는 C 행렬의 원소를 한 번에 하나씩 계산하며, 이때 A 행렬의 한 행과 B 행렬의 한 열을 순회하면서 곱셈-덧셈 연산을 수행한다. 만약 한 번에 C 행렬의 4개 원소를 계산한다면, A 행렬의 한 행과 B 행렬의 네 열을 매번 순회할 수 있다. 코드 구현은 대체로 다음과 같다.
    
    
    void MY_MMult2( int m, int n, int k, float *a, int lda,   
                                        float *b, int ldb,  
                                        float *c, int ldc ){  
      int i, j;  
      for ( j=0; j<n; j+=4 ){  
        for ( i=0; i<m; i+=1 ){  
          AddDot( k, &A( i,0 ), lda, &B( 0,j ), &C( i,j ) );  
          AddDot( k, &A( i,0 ), lda, &B( 0,j+1 ), &C( i,j+1 ) );  
          AddDot( k, &A( i,0 ), lda, &B( 0,j+2 ), &C( i,j+2 ) );  
          AddDot( k, &A( i,0 ), lda, &B( 0,j+3 ), &C( i,j+3 ) );  
        }  
      }  
    }  
    

그러나 안타깝게도 컴파일러가 O2를 켜둔 상태에서 이 최적화 방법은 효과가 없었으며, 이 버전은 원본 구현과 비슷한 gflops를 보였다.

# 4\. 첫 번째로 어느 정도 효과 있던 최적화

처음으로 비교적 효과가 있어 보였던 방법은 register 변수를 도입하는 것이다. 컴퓨터 storage 계층 구조도(Figure3)를 보면 register 변수는 CPU에 가장 가까이 있으며, 데이터 접근 속도도 가장 빠르다는 것을 알 수 있다. 따라서 합산 시에 **누적 합** 변수와 **피승 변수** 를 명시적으로 register 변수로 선언하면, 누적 합산 시 접근 속도가 원본보다 더 빠르며 어느 정도의 성능 향상을 가져올 수 있다.

![이미지](images/img_01.png)이 부분의 코드 구현은 대체로 다음과 같다.
    
    
    void AddDot1x4( int k, float *a, int lda,  float *b, int ldb, float *c, int ldc ){  
      int p;  
      register float c_00_reg, c_01_reg, c_02_reg, c_03_reg,   a_0p_reg;  
      c_00_reg = 0.0;   
      c_01_reg = 0.0;   
      c_02_reg = 0.0;   
      c_03_reg = 0.0;  
      for ( p=0; p<k; p++ ){  
        a_0p_reg = A( 0, p );  
        c_00_reg += a_0p_reg * B( p, 0 );       
        c_01_reg += a_0p_reg * B( p, 1 );       
        c_02_reg += a_0p_reg * B( p, 2 );       
        c_03_reg += a_0p_reg * B( p, 3 );       
      }  
      C( 0, 0 ) += c_00_reg;   
      C( 0, 1 ) += c_01_reg;   
      C( 0, 2 ) += c_02_reg;   
      C( 0, 3 ) += c_03_reg;  
    }  
    

이 버전의 코드는 `https://github.com/BBuf/how-to-optimize-gemm/blob/master/armv7a/src/MMult_1x4x6.h` 에 대응하며, 0.32gflops의 성적을 거둬 원본 버전 대비 미미한 향상을 보였다.

# 5\. 첫 번째로 큰 폭의 향상이 있던 최적화

첫 번째 최적화를 기반으로, 포인터로 A의 원소를 주소 지정한다. 여기서 구현하는 것은 row-major matrix multiplication이므로, C의 원소를 하나 계산할 때마다 A의 임의의 한 행에 대한 메모리 접근은 contiguous하다. 따라서 데이터 접근 방식 대신 포인터 이동 방식으로 대체할 수 있다. 이 아이디어를 바탕으로 4절의 코드를 다음과 같이 다시 쓸 수 있다.
    
    
    void AddDot1x4( int k, float *a, int lda,  float *b, int ldb, float *c, int ldc ){  
      int p;  
      register float c_00_reg,   c_01_reg,   c_02_reg,   c_03_reg,  b_0p_reg;  
      float *ap0_pntr, *ap1_pntr, *ap2_pntr, *ap3_pntr;  
      ap0_pntr = &A( 0, 0 );  
      ap1_pntr = &A( 1, 0 );  
      ap2_pntr = &A( 2, 0 );  
      ap3_pntr = &A( 3, 0 );  
      c_00_reg = 0.0;   
      c_01_reg = 0.0;   
      c_02_reg = 0.0;   
      c_03_reg = 0.0;  
      for ( p=0; p<k; p++ ){  
        b_0p_reg = B( p, 0 );  
        c_00_reg += b_0p_reg * *ap0_pntr++;  
        c_01_reg += b_0p_reg * *ap1_pntr++;  
        c_02_reg += b_0p_reg * *ap2_pntr++;  
        c_03_reg += b_0p_reg * *ap3_pntr++;  
      }  
      C( 0, 0 ) += c_00_reg;   
      C( 1, 0 ) += c_01_reg;   
      C( 2, 0 ) += c_02_reg;   
      C( 3, 0 ) += c_03_reg;  
    }  
    

이 작은 변경만으로 0.98gflops를 얻었고, 하드웨어 이용률은 9%에 도달했다. 이는 확실히 큰 폭의 향상을 보인 최적화이다. 이 버전의 코드는 `https://github.com/BBuf/how-to-optimize-gemm/blob/master/armv7a/src/MMult_1x4_7.h` 에 대응한다.

# 6\. 세 번째 최적화, 더 큰 blocking 시도

위의 몇 차례 최적화에서 우리는 한 번에 C 행렬의 원소 1개 또는 4개를 계산했다. 이 절에서는 이를 한 번에 C 행렬의 16개 원소를 계산하도록 확장한다. 즉, blocking 방법이다. 또한 C의 원소를 register 변수로 누적하고, A의 원소에도 register 변수를 사용한다. 이 부분의 코드 구현도 비교적 간단하며, `https://github.com/BBuf/how-to-optimize-gemm/blob/master/armv7a/src/MMult_4x4_6.h` 에서 확인할 수 있다. blocking 후에 1.75gflops의 결과를 얻었으며, 하드웨어 이용률은 약 16% 수준이었다.

다음으로 5절의 아이디어를 참고하여 `/MMult_4x4_6` 기반으로 포인터로 B의 원소를 주소 지정한다. 그러나 blocking 자체가 이미 메모리 접근을 크게 개선했기 때문에 이 최적화는 여기서는 큰 효과를 보이지 않았으며, 뚜렷한 gflops 향상을 얻지 못했다. 이 부분의 코드 구현은 `https://github.com/BBuf/how-to-optimize-gemm/blob/master/armv7a/src/MMult_4x4_7.h` 에 대응한다.

# 7\. 네 번째 최적화, NEON 명령어 셋 최적화

C의 원소를 계산할 때 SIMD를 사용하여 최적화할 수 있다. Armv7a 아키텍처에서는 `https://github.com/BBuf/how-to-optimize-gemm/blob/master/armv7a/src/MMult_4x4_10.h` 의 핵심 구현 부분을 NEON 명령어 셋으로 최적화하는 것이며, 여기서는 먼저 NEON Intrinsics로 최적화한다.
    
    
      
    void AddDot4x4( int k, float *a, int lda,  float *b, int ldb, float *c, int ldc ){  
      float  *a_0p_pntr, *a_1p_pntr, *a_2p_pntr, *a_3p_pntr;  
      a_0p_pntr = &A(0, 0);  
      a_1p_pntr = &A(1, 0);  
      a_2p_pntr = &A(2, 0);  
      a_3p_pntr = &A(3, 0);  
      float32x4_t c_p0_sum = {0};  
      float32x4_t c_p1_sum = {0};  
      float32x4_t c_p2_sum = {0};  
      float32x4_t c_p3_sum = {0};  
      register float a_0p_reg, a_1p_reg, a_2p_reg, a_3p_reg;  
      
      for (int p = 0; p < k; ++p) {  
        float32x4_t b_reg = vld1q_f32(&B(p, 0));  
      
        a_0p_reg = *a_0p_pntr++;  
        a_1p_reg = *a_1p_pntr++;  
        a_2p_reg = *a_2p_pntr++;  
        a_3p_reg = *a_3p_pntr++;  
      
        c_p0_sum = vmlaq_n_f32(c_p0_sum, b_reg, a_0p_reg);  
        c_p1_sum = vmlaq_n_f32(c_p1_sum, b_reg, a_1p_reg);  
        c_p2_sum = vmlaq_n_f32(c_p2_sum, b_reg, a_2p_reg);  
        c_p3_sum = vmlaq_n_f32(c_p3_sum, b_reg, a_3p_reg);  
      }  
      
      float *c_pntr = 0;  
      c_pntr = &C(0, 0);  
      float32x4_t c_reg = vld1q_f32(c_pntr);  
      c_reg = vaddq_f32(c_reg, c_p0_sum);  
      vst1q_f32(c_pntr, c_reg);  
      
      c_pntr = &C(1, 0);  
      c_reg = vld1q_f32(c_pntr);  
      c_reg = vaddq_f32(c_reg, c_p1_sum);  
      vst1q_f32(c_pntr, c_reg);  
      
      c_pntr = &C(2, 0);  
      c_reg = vld1q_f32(c_pntr);  
      c_reg = vaddq_f32(c_reg, c_p2_sum);  
      vst1q_f32(c_pntr, c_reg);  
      
      c_pntr = &C(3, 0);  
      c_reg = vld1q_f32(c_pntr);  
      c_reg = vaddq_f32(c_reg, c_p3_sum);  
      vst1q_f32(c_pntr, c_reg);  
    }  
      
    

이전에 더펑(Depeng)과 함께 《AI 모바일 최적화》 칼럼에서 NEON 명령어 셋 최적화 예제를 많이 소개했기 때문에, 여기서는 위 코드의 각 줄이 무엇을 의미하는지 자세히 설명하지 않겠다. 관심 있는 독자는 `MMult_4x4_7.h` 의 코드와 비교하여 이해할 수 있다. NEON Intrinsics 최적화 후 2.6gflops의 성적을 얻었으며, 23.8%의 하드웨어 이용률에 도달했다.

# 8\. 다섯 번째 최적화, 데이터 Pack

위의 최적화에서 알 수 있듯이, matrix multiplication 계산에서는 row-major이든 column-major이든 항상 한쪽 행렬의 메모리는 contiguous하게 접근할 수 없다. 이것이 blocking 후 gflops가 크게 향상된 중요한 이유이기도 하다. 따라서 이 상황을 개선하기 위해 데이터 Pack을 수행하여, A 행렬과 B 행렬에 접근할 때의 메모리를 contiguous하게 만든다.

이론적으로 이렇게 하면 분명히 향상이 있어야 하지만, Armv7a에서 실측한 결과 gflops는 향상되지 않았다(x86에서는 약 4배 정도의 gflops 향상이 있었다). 그 이유는 추측건대 주로 데이터 Pack 자체에도 시간이 필요하기 때문이며, 또한 blocking이 이미 메모리 비연속성으로 인한 메모리 접근 시간 소비를 잘 회피하고 있어 데이터 Pack 시간을 무시할 수 없을 때는 가속 효과가 매우 적기 때문일 것이다. 또한 x86 아키텍처에서의 데이터 pack 속도가 armv7a 아키텍처보다 우수하기 때문일 것으로 추측된다(고수분들 중 더 좋은 설명이 있으시면 연락 바란다).

따라서 여기서 얻은 깨달음은 **데이터 Pack은 가능한 한 핵심 계산 과정의 외부에서 완료해야 한다** 는 것이다.

# 9\. 여섯 번째 최적화, 한 번에 여러 행 계산 + NEON Assembly

먼저, CNN에서 convolution은 Kernel 행렬과 입력 feature map 행렬이 직접 matrix multiplication을 수행하는 것으로 볼 수 있다. **convolution kernel을 matrix multiplication의 행렬 A로 본다** 면, 그 차원은 다음과 같다. 그리고 **입력 feature map을 matrix multiplication의 행렬 B로 본다** 면, 그 차원도 다음과 같다. 이렇게 하면 **행렬 C는 우리의 convolution 결과** 가 되며, 차원은 다음과 같다. convolution이고 stride가 1일 때, 출력 feature map의 가로세로는 입력 feature map과 완전히 일치하기 때문이다.

여기서:

  * inChannel은 convolution layer의 입력 채널 수를 나타낸다
  * outChannel은 convolution layer의 출력 채널 수를 나타낸다
  * inHeight는 입력 feature map의 높이를 나타낸다
  * inWidth는 입력 feature map의 너비를 나타낸다



이를 바탕으로 NCNN의 convolution 첫 버전 구현 방법을 참고하여 이번 최적화 버전을 얻었다. 전체 구현은 `https://github.com/BBuf/how-to-optimize-gemm/blob/master/armv7a/src/convolution1x1s1.h#L10` 에 있다. 주요 아이디어는 한 번에 여러 행의 출력을 계산하고, 각 행에서 NEON 명령어 셋(NEON Assembly)을 사용하여 최적화하는 것이다. 즉, 열 방향으로 한 번 더 여러 원소를 계산한다. 이 아이디어를 더 잘 이해할 수 있도록, 다음에는 이 함수에서 NEON 최적화 관련 부분을 제거하고 일반 구현 코드만 남겨 독자가 이 알고리즘을 빠르게 이해할 수 있도록 하였다.
    
    
    void conv1x1s1(float *const &src, const int &inWidth, const int &inHeight,  const int &inChannel, float *const &kernel,  
                                     float* &dest, const int &outWidth, const int &outHeight, const int &outChannel){  
            int ccOutChannel = outChannel >> 2;  
            int ccRemainOutChannel = ccOutChannel << 2;  
            const int in_size = inWidth * inHeight;  
            const int out_size = outWidth * outHeight;  
            for(int cc = 0; cc < ccOutChannel; cc++){  
                int c = cc << 2;  
                  
                float *dest0 = dest + c * out_size;  
                float *dest1 = dest + (c + 1) * out_size;  
                float *dest2 = dest + (c + 2) * out_size;  
                float *dest3 = dest + (c + 3) * out_size;  
      
                int q = 0;  
      
                for(q = 0; q + 3 < inChannel; q += 4){  
                    float *destptr0 = dest0;  
                    float *destptr1 = dest1;  
                    float *destptr2 = dest2;  
                    float *destptr3 = dest3;  
      
                    const float *src0 = src + q * in_size;  
                    const float *src1 = src + (q + 1) * in_size;  
                    const float *src2 = src + (q + 2) * in_size;  
                    const float *src3 = src + (q + 3) * in_size;  
      
                    const float *r0 = src0;  
                    const float *r1 = src1;  
                    const float *r2 = src2;  
                    const float *r3 = src3;  
      
                    const float *kernel0 = kernel + c * inChannel + q;  
                    const float *kernel1 = kernel + (c + 1) * inChannel + q;  
                    const float *kernel2 = kernel + (c + 2) * inChannel + q;  
                    const float *kernel3 = kernel + (c + 3) * inChannel + q;  
                    int remain = out_size;  
                    for(; remain > 0;  remain--){  
                        float sum0 = *r0 * kernel0[0] + *r1 * kernel0[1] + *r2 * kernel0[2] + *r3 * kernel0[3];  
                        float sum1 = *r0 * kernel1[0] + *r1 * kernel1[1] + *r2 * kernel1[2] + *r3 * kernel1[3];  
                        float sum2 = *r0 * kernel2[0] + *r1 * kernel2[1] + *r2 * kernel2[2] + *r3 * kernel2[3];  
                        float sum3 = *r0 * kernel3[0] + *r1 * kernel3[1] + *r2 * kernel3[2] + *r3 * kernel3[3];  
      
                        *destptr0 += sum0;  
                        *destptr1 += sum1;  
                        *destptr2 += sum2;  
                        *destptr3 += sum3;  
      
                        r0++;  
                        r1++;  
                        r2++;  
                        r3++;  
                        destptr0++;  
                        destptr1++;  
                        destptr2++;  
                        destptr3++;  
                    }  
                }  
      
                for(; q < inChannel; q++){  
                    float *destptr0 = dest0;  
                    float *destptr1 = dest1;  
                    float *destptr2 = dest2;  
                    float *destptr3 = dest3;  
      
                    const float *src0 = src + q * in_size;  
                    const float *kernel0 = kernel + c * inChannel + q;  
                    const float *kernel1 = kernel + (c + 1) * inChannel + q;  
                    const float *kernel2 = kernel + (c + 2) * inChannel + q;  
                    const float *kernel3 = kernel + (c + 3) * inChannel + q;  
                    const float *r0 = src0;  
                    int remain = out_size;  
                    for(; remain > 0; remain--){  
                        float sum0 = *r0 * kernel0[0];  
                        float sum1 = *r0 * kernel1[0];  
                        float sum2 = *r0 * kernel2[0];  
                        float sum3 = *r0 * kernel3[0];  
      
                        *destptr0 += sum0;  
                        *destptr1 += sum1;  
                        *destptr2 += sum2;  
                        *destptr3 += sum3;  
      
                        r0++;  
                        destptr0++;  
                        destptr1++;  
                        destptr2++;  
                        destptr3++;  
                    }  
      
                }  
            }  
            for(int cc = ccRemainOutChannel; cc < outChannel; cc++){  
                float *dest0 = dest + cc * out_size;  
                int q = 0;  
                for(; q + 3 < inChannel; q += 4){  
                    float *destptr0 = dest0;  
                    const float *src0 = src + q * in_size;  
                    const float *src1 = src + (q + 1) * in_size;  
                    const float *src2 = src + (q + 2) * in_size;  
                    const float *src3 = src + (q + 3) * in_size;  
                    const float *r0 = src0;  
                    const float *r1 = src1;  
                    const float *r2 = src2;  
                    const float *r3 = src3;  
                    const float *kernel0 = kernel + cc * inChannel + q;  
                    int remain = out_size;  
                    for(; remain > 0; remain--){  
                        float sum0 = *r0 * kernel0[0] + *r1 * kernel0[1] + *r2 * kernel0[2] + *r3 * kernel0[3];  
      
                        *destptr0 += sum0;  
      
                        r0++;  
                        r1++;  
                        r2++;  
                        r3++;  
                        destptr0++;  
                    }  
      
                }  
      
                for(; q < inChannel; q++){  
                    float *destptr0 = dest0;  
                    const float *src0 = src + q * in_size;  
                    const float *kernel0 = kernel + cc * inChannel + q;  
                    const float *r0 = src0;  
                    int remain = out_size;  
                    for(; remain > 0; remain--){  
                        float sum0 = *r0 * kernel0[0];  
      
                        *destptr0 += sum0;  
      
                        r0++;  
                        destptr0++;  
                    }  
      
                }  
            }  
    }  
    

위 코드에 NEON Assembly 최적화를 적용한 후 테스트하여, 3.4gflops의 성적을 얻었으며 하드웨어 이용률 31%에 도달했다. 현재까지의 최고 성적이다.

# 10\. 일곱 번째 최적화, 데이터 Pack의 위력

여섯 번째 최적화 구현에서 데이터 Pack을 고려하지 않았기 때문에 메모리 접근이 비교적 좋지 않다. 여기서는 Pack 전략을 사용하여 최적화할 수 있다. 이 아이디어는 이미 [Im2Col + Pack + Sgemm 전략으로 더 나은 convolution 최적화 상세 해설](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247489576&idx=1&sn=6db06170d21414d15169b140ddae93d8&scene=21#wechat_redirect>) 에서 다양한 그림으로 비교적 명확하게 설명했고, MsnhNet 작성자도 이전에 NC4HW4에 관한 그림 해설 [신비로운 NC4HW4 도해](<https://mp.weixin.qq.com/s?__biz=MzA4MjY4NTk0NQ==&mid=2247490948&idx=1&sn=b129d4e915127dc3720d733c8a04414a&scene=21#wechat_redirect>) 를 작성한 바 있다. 따라서 여기서는 데이터 Pack의 장점과 필자가 구체적으로 어떻게 데이터 Pack을 했는지 다시 반복하지 않겠으며, 관심이 있다면 직접 소스 코드를 참고하기 바란다.

convolution kernel에 데이터 Pack을 수행하고(한 번만 하면 되며 gflops에 영향을 미치지 않는다), 그 다음 입력 데이터에 Pack을 수행한 후(Version2에서는 매 계산 과정마다 한 번씩 입력 데이터의 Pack을 수행해야 하므로 데이터 입력 Pack 시간도 gflops에 영향을 미친다는 점에 유의) 계산을 진행한다. 이 부분의 코드 구현은 `https://github.com/msnh2012/Msnhnet/blob/master/src/layers/arm/MsnhConvolution1x1.cpp#L598` 에 있다.

이 버전을 테스트한 결과, **4.9gflops** 의 성적을 얻었으며 하드웨어 이용률 49.5%에 도달했다.

또한, 입력 배치를 NC4HW4 방식으로 변경하면 입력 데이터의 Pack 시간도 절약할 수 있다. 이 조작을 통해 **5.5gflops** 의 결과를 얻었고, 하드웨어 이용률 50.5%에 도달했다. 코드 구현은 `https://github.com/BBuf/how-to-optimize-gemm/blob/master/armv7a/src/convolution1x1s1.h` 에 있다.

# 11\. 정리

이 글은 주로 지난 2주간 GEMM 알고리즘 최적화에 대해 진행한 연구와 필자가 어떻게 한 걸음씩 matrix multiplication의 하드웨어 이용률을 50%까지 끌어올렸는지 기록한 것이다. 물론 이 하드웨어 이용률은 높지 않으며, 앞으로도 지속적으로 학습하고 최적화할 예정이다. 여러분의 제안과 우리 공식 계정 **GiantPandaCV** 에 대한 관심을 환영하며, 여러분의 관심이 필자에게 가장 큰 격려이다.
